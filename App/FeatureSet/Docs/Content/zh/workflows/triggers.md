# 工作流触发器

触发器是工作流的起始节点。它没有输入端口——执行从这里开始。OneUptime 支持四类触发器；每个工作流恰好用其中一种。

## 手动

在工作流页面点击 **手动运行** 来按需运行一个工作流。你可以粘贴一个可选的 JSON 负载，工作流可通过 `{{Manual.JSON}}` 读取它。

当你想要一个按钮触发一段自动化时用这种方式——一键"轮换值班 key"或"重建搜索索引"的工作流，并不需要周期性调度或事件去触发。

**参数**：无。

**返回值**：

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `JSON` | JSON | 运行时提供的 JSON 负载，或空对象。 |

## 调度

按 cron 调度运行一个工作流。用标准 cron 表达式配置节奏。

用于周期性任务：夜间清理、每小时同步、每周导出。

**参数**：

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `Schedule at` | CronTab | 标准 5 字段 cron 表达式。例如 `0 * * * *` 在每小时的整点运行，`*/5 * * * *` 每五分钟运行一次。 |

**返回值**：

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `executedAt` | Date | 计划的运行时间。 |

调度工作流在项目所在区域的工作流 Worker 上运行。如果 Worker 短暂不可用，恢复时这次运行会被派发——对于短时故障，你不需要专门防范错过的触发节拍。

## Webhook

暴露一个独有的 HTTPS URL，外部系统可以向它 `POST`。请求头、查询参数和请求体作为返回值暴露给下游组件读取。

用它从第三方系统*接收*数据进入 OneUptime：CI/CD 回调、来自另一个监控工具的告警、你的 CRM 中新增的客户。

**参数**：无。工作流保存时会自动分配 URL 并显示在触发器节点上。把它当作密钥对待——任何拿到 URL 的人都可以触发这个工作流。

**返回值**：

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| `Request Headers` | JSON | 进入的 HTTP 请求的全部头。 |
| `Request Query Params` | JSON | 解析后的查询字符串。 |
| `Request Body` | JSON | 解析后的请求体。如果请求体不是合法 JSON，会以字符串形式出现在 `raw` 键下。 |

Webhook 接受 `GET` 和 `POST`。一旦运行入队，给调用方的响应就是 `200 OK` 并附带一个 JSON 确认——工作流本身是异步运行的，所以不要指望在 HTTP 响应里读取下游组件的结果。

## 模型事件触发器

几乎每个 OneUptime 实体——监控、事件、告警、计划维护事件、状态页、值班策略、团队、遥测服务以及更多——都暴露三种触发器：

- **On Create** — 当此类型的新记录被创建时触发。
- **On Update** — 当已有记录发生更改时触发。该触发器同时暴露旧值和新值。
- **On Delete** — 当一条记录被删除时触发。

这就是你不靠轮询而构建"当 OneUptime 里发生 X 时，做 Y"自动化的方式。

模型本身作为返回值暴露，字段名与你在资源上看到的一致。例如 **Incident → On Create** 触发器返回完整的 `Incident` 对象，下游节点可以读取 `{{Incident.title}}`、`{{Incident.description}}`、`{{Incident.incidentSeverityId}}` 等。

**参数**：创建/删除通常无参数。更新触发器可能让你缩小希望响应的字段范围，这样你不会因为某些表面性的变化就触发。

**返回值**（视模型而定）：

| 名称 | 类型 | 描述 |
| --- | --- | --- |
| 模型字段 | （视情况） | 实体的每一列——名称、状态、时间戳、外键。 |
| `previous`（仅 Update） | JSON | 改变之前的记录状态。 |

### 常见模型触发器

团队最常使用的模型事件的不完全列表：

- **Incident** — `On Create`、`On Update`（用于响应 Acknowledged 或 Resolved 等状态变化）、`On Delete`。
- **Alert** — 告警模型上同样的三种事件。
- **Monitor** — 当监控被新增、修改或移除时响应；结合条件可以只对生产监控生效。
- **Scheduled Maintenance** — 当维护窗口创建或状态变化时，自动化下游通告。
- **Status Page Subscriber** — 有人订阅时触发欢迎流程。
- **On-Call Duty Policy** — 把排班变更同步到外部排班系统。

如果某个模型在 OneUptime API 中暴露，那它几乎一定能触发工作流——在触发器面板里按实体名称搜索。

## 选对触发器

| 如果你想要…… | 用 |
| --- | --- |
| 给工作流加一个有人点的按钮 | **手动** |
| 每 N 分钟/小时/天跑一个任务 | **调度** |
| 让外部系统把数据推进 OneUptime | **Webhook** |
| 响应 OneUptime *内部* 发生的事情 | **模型事件** |

工作流只能有一个触发器。如果你需要两个不同的起始信号共享大部分相同的逻辑，把共享步骤抽到一个工作流里，并用 **Execute Workflow** 组件从两个轻量"包装"工作流里调用它（见 [工作流组件](/docs/workflows/components)）。
