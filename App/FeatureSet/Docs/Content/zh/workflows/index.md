# 工作流概览

工作流是 OneUptime 的可视化自动化构建器。把一个触发器拖到画布上，将它连接到一串动作——HTTP 调用、Slack 消息、JavaScript 片段、条件分支、数据库查询——然后你就有了一个会在 OneUptime（或外部世界）发生事件时运行的自动化。

如果说 Runbook 是事件期间给人用的清单，那工作流就是给你的项目跑的后台作业——它们无人值守地运行、对事情作出反应，并把 OneUptime 和你技术栈的其它部分粘在一起。

## 一览

- OneUptime 仪表板 **工作流** 下的**顶层功能**。
- **三种触发方式**：手动、调度（cron）、Webhook——外加一种**模型事件触发器**，当任意 OneUptime 实体（事件、告警、监控、状态页等）被创建、更新或删除时触发。
- **可视化画布**：从组件面板拖入节点，把输出端口连到输入端口。
- **混合自动化**：HTTP 请求、Slack / Discord / Microsoft Teams / Telegram 消息、自定义 JavaScript、JSON 解析、条件判断、邮件、子工作流调用，以及对 OneUptime 模型的增删改查。
- **全局变量**：项目级别的密钥和配置，你可以从任意工作流引用，而不必复制粘贴。
- **运行与日志**：每次执行都会被记录，包含状态、耗时和每步的输出。

## 为什么用工作流？

大多数团队会在以下场景中用到工作流：

- **把 OneUptime 接入另一个系统**——把事件推到 PagerDuty、把告警镜像到 Jira、ping 你技术栈中的某个 webhook。
- **响应 OneUptime 事件**——当 `Sev 1` 事件打开时，呼叫值班经理、*同时*在 Linear 创建工单、*同时*锁定一个功能开关。
- **调度周期性任务**——每五分钟查询一次内部 API，并把结果写入外部系统。
- **从 OneUptime 外部接收数据**——来自 CI 系统的 webhook 触发一连串 OneUptime 更新。
- **复用小段粘合逻辑**——一个工作流调用另一个，让常见模式集中在一个地方。

## 关键概念

| 术语 | 含义 |
| --- | --- |
| **工作流** | 画布。一个命名的、可复用的触发器与组件图，带有 `isEnabled` 开关。 |
| **触发器** | 启动工作流运行的节点。手动、调度、Webhook，或者模型事件。每个工作流恰好有一个触发器。 |
| **组件** | 执行工作的节点——HTTP 调用、Slack 消息、JavaScript 片段、条件等。 |
| **端口** | 节点上的输入或输出接口。组件有 `success`、`error` 等输出端口；你把一个端口连到下一节点的输入端口。 |
| **运行 / 日志** | 工作流的一次执行。包含时间戳、状态（运行中、成功、失败、超时），以及每个运行过的节点捕获的输出。 |
| **全局变量** | 在项目级别定义一次的命名值（通常是密钥或 API key），任意工作流通过 `{{variable.NAME}}` 引用。 |
| **本地变量** | 作用域限于一次工作流运行的值——通常是早先节点的返回值，通过 `{{ComponentId.portName}}` 引用。 |

## 工作流在仪表板的位置

| 页面 | 在那里做什么 |
| --- | --- |
| **工作流** | 浏览、创建、搜索工作流模板。 |
| 工作流的 **构建器** 标签 | 拖放式画布。添加节点、连接端口、配置参数。 |
| 工作流的 **日志** 标签 | 这个工作流的每一次运行，可按状态和时间范围筛选。点击某次运行查看每个节点的输出。 |
| 工作流的 **设置** 标签 | 重命名、启用/禁用、修改描述、管理标签、删除。 |
| **工作流 → 全局变量** | 定义在任意工作流中引用的项目级值。把值标为密钥，保存后会从界面隐藏。 |
| **工作流 → 运行与日志** | 项目级所有工作流的执行历史。 |

## 工作流的生命周期

1. **撰写** — 创建一个工作流，把触发器丢到画布上，拖入需要的组件，连接它们，配置每一个。
2. **启用** — 工作流出厂是禁用的。在确认连线无误后，到设置里打开开关。
3. **触发** — 手动：用可选的 JSON 负载点击 **手动运行**。调度：cron 触发。Webhook：外部系统 `POST` 到工作流 URL。模型事件：有人（或另一个工作流）创建/更新/删除监控、事件、告警等。
4. **执行** — 工作流 Worker 按顺序遍历图。每个组件读取参数（字面值或插值变量），完成工作，写入返回值，并选择一个输出端口。下一节点继续触发。
5. **审计** — 这次运行会出现在 **日志** 里。状态、总耗时、每个组件的输出和任何错误，在项目的生命周期内都会保留。

## 完整示例

目标：当一个标题里含 `Sev 1` 的事件被创建时，发一条 Slack 消息，并在你的内部管理工具中开一张工单。

**1. 创建一个工作流**，命名为 "Sev 1 fan-out"。

**2. 丢一个触发器。** 从面板选择 **Incident → On Create** 触发器。触发器把新事件作为返回值暴露。

**3. 丢一个条件组件。** 把触发器的输出端口连到它的输入端口。设置条件：`{{Incident.title}}` *contains* `Sev 1`。

**4. 从条件的 `yes` 端口，丢一个 Slack 组件。** 频道：`#incident-room`。消息体：`Sev 1 declared: {{Incident.title}} — {{Incident.dashboardUrl}}`。

**5. 从同一个 `yes` 端口（并行）丢一个 API 组件。** `POST` 到 `https://admin.internal/incidents`。请求体：从事件构建的一个小 JSON 对象。

**6. 启用工作流。** 在另一个标签页打开一个标题为 "Sev 1 — checkout 500s" 的事件。几秒内 Slack 消息就会到达，**日志** 下会出现一条新运行，包含每个节点捕获的输出。

## 工作流如何与 OneUptime 的其他部分配合

- **监控** 检测问题；**事件/告警** 记录它们；**工作流** 响应它们——发消息、开工单、启动自动化。
- **Runbook** 是给人用的响应流程（可选包含脚本步骤）。工作流是无人值守的后台自动化。两者互补——某个 Runbook 步骤可能会 `POST` 到一个工作流的 Webhook 触发器。
- **工作区连接**（Slack、Microsoft Teams）是工作流通知的典型目的地。
- **仪表板** 是只读视图；工作流是写入侧——它们更新 OneUptime 状态、调用外部 API、在系统之间搬运数据。
