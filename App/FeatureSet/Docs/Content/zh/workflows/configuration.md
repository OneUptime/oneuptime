# 工作流配置与安全

本页收集了在你把一个工作流指向生产流量之前值得了解的设置与安全限制。

## 启用 / 禁用

每个工作流在 **设置** 里都有一个 **isEnabled** 开关。禁用的工作流永远不会触发——模型事件、Webhook 和调度运行都会被忽略。新工作流出厂是禁用的。

把它当作你的"准备好上线"开关：

1. 构建工作流。
2. 用有代表性的负载点击 **手动运行**。
3. 检查 **日志**——确认每个节点都走了你预期的端口。
4. 打开 **isEnabled**。

禁用一个工作流不影响已经在进行中的运行；它只会阻止新的运行被创建。

## 所有权与标签

- **Owners** — 列为所有者的用户和团队会获得基于权限的访问，并且（可选）在工作流失败时收到通知。在 **设置 → Owners** 中配置。
- **Labels** — 用于组织工作流的多对多标签。可在工作流列表中按标签过滤。当一个项目有几十个按团队、集成或环境组织的工作流时很有用。
- **标签规则** — 在 **工作流 → 设置 → Label Rules** 下，基于名称或描述的正则匹配自动给新工作流应用标签。
- **所有者规则** — 在 **工作流 → 设置 → Owner Rules** 下，给新工作流自动指派所有者。

## 密钥

全局变量可以被标记为 **密钥**。值在静态存储时加密、保存后界面变为只写、并在运行日志中被脱敏（替换为 `[REDACTED]`）。

把密钥变量用于：

- 出站集成的 API key。
- Bearer token。
- Webhook 签名 key。
- 任何不应该被有工作流读取权限的攻击者看到的值。

不要把密钥直接粘到组件的参数里——像 `Authorization: Bearer eyJh...` 这样的引用会以明文出现在工作流 JSON 和运行日志里。换成 `{{variable.MY_SECRET}}` 来引用。

## 运行超时

每次运行有最大时长。如果运行在超时时间内没结束，会被标记为 `Timeout` 并取消正在进行的组件。默认值比较宽松（分钟级，而不是秒级）——具体数值见你这套安装中 Worker 的环境配置。

大多数组件在运行超时内还有自己的每调用超时——例如 API 组件会在卡住的出站请求耗尽整次运行超时之前就放弃。

## 递归限制

**Execute Workflow** 组件让一个工作流可以调用另一个。为了防止 A 调 B、B 又调 A 的失控循环，Worker 跟踪调用链并停止超过固定深度（通常是 5 之类的小数字）的链。终止的运行会被标记为 `Error`，并附上关于递归限制的清晰说明。

如果你有合理需要的长调用链（例如递归遍历文件夹，每次运行处理一层），把它重构成单个工作流，通过 **Custom Code** 在内部迭代——这种模式不受链限制约束。

## Webhook 安全

Webhook 触发器暴露一个独有的 HTTPS URL。任何知道这个 URL 的人都能调它。为了防范误调用或恶意调用：

- 把 URL 当作共享密钥对待。不要把它粘到公开聊天或提交到公开仓库。
- 对高价值工作流，让调用系统在头里附一个共享密钥（例如 `X-Webhook-Token`），并在做任何破坏性动作前用 **Conditions** 节点校验。把期望的 token 定义为密钥型全局变量。
- 对极高价值的工作流，宁可使用模型事件触发器加手动导入步骤，也不要用公开 Webhook。

## 出站网络出口

API 等 HTTP 风格的组件从 OneUptime 工作流 Worker 的网络发出请求。如果你自托管 OneUptime，Worker 的出站网络是你的事——确保它能访问到你要调用的第三方 API。如果你用 OneUptime Cloud，我们的 IP 出口范围已在 [IP Addresses](/docs/configuration/ip-addresses) 公布，便于你在接收侧设置允许名单。

## 权限

工作流是受项目级基于角色访问控制的一等资源：

- `CreateWorkflow`、`ReadWorkflow`、`EditWorkflow`、`DeleteWorkflow` — 工作流模板上的四种 CRUD 权限。
- `RunWorkflow` — 点击 **手动运行** 或通过 API 派发工作流所需。
- `ReadWorkflowLog` — 查看 **运行与日志** 页面所需。
- `ReadWorkflowVariable`、`CreateWorkflowVariable`、`EditWorkflowVariable`、`DeleteWorkflowVariable` — 对全局变量列表的控制。

大多数工程师应该有创建/编辑/读取工作流的权限，但没有对变量的权限。把变量编辑权限留给管理项目密钥的人。

## 配额

OneUptime Cloud 在较小的计划上限制每个项目每月的运行数。上限在 **项目设置 → 计费** 中显示。达到上限后，新触发会被拒绝（并以 "quota exceeded" 原因记录在受影响的工作流上），直到下一个计费周期。自托管安装不受配额约束。

## 工作流*不*擅长的场景

几种你应该转向其它工具的模式：

- **长耗时计算** — 工作流面向系统间的胶水，不适合啃大数据集。在你自己的基础设施里跑重活，用工作流来启动它。
- **跨数分钟/数小时的有状态工作流** — 单次运行的目的就是快速完成。如果你需要"做 A，等两小时，做 B"，把等待建模为一个外部调度器，让它回调到一个 Webhook 触发器。
- **带人工检查点的逐步事件响应** — 那是 [Runbook](/docs/runbooks/index) 的事。回路里没有人时用工作流；有人时用 Runbook。
