# 工作流运行与日志

每当工作流的触发器触发，OneUptime 都会创建一次 **运行**——一条包含耗时、状态和每节点输出的执行记录。运行是你确认工作流是否工作、排查未生效的工作流、以及在自动化失常时写事后回顾的依据。

## 在哪里查看

| 页面 | 范围 |
| --- | --- |
| **工作流 → 运行与日志** | 项目级。所有工作流的所有运行。可按工作流、状态和时间范围过滤。 |
| 工作流的 **日志** 标签 | 仅此工作流的运行。 |
| 一次运行的详情页 | 单次执行的展开视图，包含每节点输出和任何错误消息。 |

## 运行状态

| 状态 | 含义 |
| --- | --- |
| **Scheduled** | 触发器已触发、运行已入队，但 Worker 还没拣选它。通常是不到一秒的事。 |
| **Running** | Worker 正在遍历图。长耗时的组件（慢 HTTP 调用、刻意的延迟）会让运行停在这个状态。 |
| **Success** | 所有运行过的节点都无错完成。（故意走了 `error` 分支的工作流整体仍然是 `Success`——工作流自身没失败。） |
| **Error** | 某个节点失败，并且没有 `error` 端口接住它。运行在那个节点停止。 |
| **Timeout** | 运行超出了每次运行的超时时长。见 [工作流配置与安全](/docs/workflows/configuration)。 |

## 阅读一次运行

从列表点击一次运行打开它的详情页。你会看到：

- **头部** — 触发的触发器、开始与结束时间戳、总耗时、状态。
- **节点列表** — 每个按顺序执行过的节点，及其捕获的参数、返回值和它所选的输出端口。
- **错误** — 若有节点失败，展示错误消息以及（在可用时）堆栈。

捕获的参数显示*插值之后*的值——也就是变量解析完成后节点真正看到的字符串。这是最有用的单一调试视图：如果一条 Slack 消息里出现字面文本 `{{Incident.title}}`，你就知道变量引用没解析成功。

## 常见排查模式

### "我的工作流没触发。"

1. 在 **设置** 里确认工作流是 **启用** 状态。新工作流出厂是禁用的。
2. 对模型事件触发器：确认事件真的发生了。打开实体（事件、告警、监控），查看它的历史。
3. 对 Webhook 触发器：确认外部系统在调对的 URL。很多工具会记录出站 webhook 投递——在那边查。
4. 对调度触发器：确认 cron 表达式解析到你预期的时间。不确定时用 cron 解析器跑一下。

如果触发器触发了但没有运行出现，到 **项目设置 → 计费** 里检查项目的运行额度。

### "工作流跑起来了，但某个下游节点从未执行。"

不被执行的节点通常是连线问题。打开画布并检查：

- 上游节点的输出端口是不是真的连到这个节点的输入端口？
- 上游节点是否走了别的端口（比如走了 `error` 而不是 `success`、或者 `no` 而不是 `yes`）？看运行详情就能知道它选了哪个端口。

### "某个变量得到的是空值。"

打开运行详情，看出问题那个节点捕获的参数。如果你看到字面的 `{{NodeId.field}}` 文本，说明引用没解析——很可能 `NodeId` 或 `field` 有拼写错误。如果你看到空字符串，说明上游节点跑了，但没有产出那个字段。

### "手动跑没问题，但触发器跑就不行。"

用 **手动运行** 加上一份模仿真实触发器发布内容的 JSON 负载。然后把手动运行和生产运行的捕获参数并排比较——差别通常就藏在某个字段名或类型上。

## 重新运行一个工作流

没有"重试这次运行"按钮——这是有意的设计，因为 OneUptime 永远不会重跑旧的运行，因为出站副作用（Slack 消息、API 调用）可能不是幂等的。如果你想重做这段工作，修好工作流，让下一次真实触发器来跑它。

对手动工作流，直接用同样的负载点击 **手动运行**。

## 日志留存

运行在项目上无限期保留。如果你需要清理高量噪音的工作流（比如一个每分钟都触发的调试工作流），禁用或删除它——没有按工作流的留存开关。
