# 创建仪表板

在 **仪表板 → 创建仪表板** 下新建一个仪表板，给它起个名字并打开。画布以 **Edit** 模式打开，准备放置小部件。

## 画布

仪表板是一个网格。默认画布是 **12 仪表板单位宽**、**60 单位高**——你可以通过在底部之外加行来增加高度。每个单位是一个会随视口缩放的方格：在桌面上比手机上更宽，但每个小部件保持其比例。

小部件占据一个单位矩形。你决定位置（左上角，从画布左上角起以单位计）和尺寸（以单位为单位的宽度和高度）。最小尺寸约束保证小部件再小也仍然可读。

## Edit 与 View

页面头部的切换在两种模式间切换：

- **Edit** — 小部件面板打开，小部件可拖动、可调大小，每个小部件有设置齿轮。构建时使用。
- **View** — 仪表板以只读方式渲染，与拥有只读访问的人（或公开访问者）看到的完全一致。在分享前用它检查效果。

同一个仪表板在两种模式中显示——没有单独的"发布"步骤。保存编辑后，对每个查看者立即生效。

## 添加小部件

1. 打开小部件面板（Edit 模式下的 **+** 按钮）。
2. 选小部件类型。清单见 [仪表板小部件](/docs/dashboards/widgets)。
3. 小部件以默认尺寸落在画布下一个空闲位置。
4. 点击小部件的齿轮打开它的设置面板。
5. 配置数据源（指标查询、列表过滤器、文本正文等）以及任何展示选项（阈值、单位、轴、列）。
6. 拖动小部件以定位。拖动角部以调整大小。

重复。网格会把小部件吸附到整单位边界上。

## 配置数据源

大多数小部件从以下三处之一读数据：

- **指标** — 由 ClickHouse 支撑的指标查询。小部件构造一个 `metricQueryConfig`（单个序列）或 `metricQueryConfigs`（多个序列堆叠或重叠）。可选 `transformAsRate` 把 OpenTelemetry 累积计数器转成变化率。可选 `formula` 让你组合两个查询（例如 错误数 / 总数）。
- **实时资源列表** — 事件、告警、监控、Kubernetes 资源、Docker 资源、主机。每个列表小部件接收一个过滤器（例如标签、状态、命名空间），并实时显示匹配的行。
- **静态内容** — **Text** 小部件接收 Markdown 正文。用于标题、分隔、Runbook 链接和"这个仪表板是什么？"这种注释。

对指标小部件，配置镜像了你在 OneUptime 别处看到的内联查询构建器——挑指标、挑聚合方式、加 `WHERE` 过滤、选时间分组。查询针对项目的遥测数据运行。

## 阈值与格式

显示单一数字的小部件（**Value**、**Gauge**）接收可选阈值：

- **Warning threshold** — 越过它时以黄色显示。
- **Critical threshold** — 越过它时以红色显示。

图表让你设 Y 轴单位、图例位置以及是否堆叠序列。表格让你选要显示的列和行数上限。

## 时间范围与刷新

仪表板头部带两个影响每个指标小部件的全局控件：

- **时间范围** — 选一个预设（过去 1 小时、24 小时、7 天、30 天）或自定义范围。每个指标小部件都针对这个窗口查询。
- **刷新间隔** — 关闭、5s、10s、30s、1m、5m、15m。按所选节奏重新跑每个小部件的查询。原生支持 websocket 的列表小部件无论选了什么间隔，都会按推送更新。

对忽略全局时间范围的小部件（例如文本块），该控件无效。

## 保存

画布会随编辑自动保存。头部有一个小指示器告诉你最新更改是否已持久化。没有"发布"——每次编辑保存的那一刻就生效。如果你要做风险较高的修改，先复制一份仪表板。

## 行之有效的模式

- **每个仪表板专注一个主题。** 抵制把"所有我们监控的东西"放在一页上的诱惑。三个分别标为 `oncall-checkout`、`oncall-payments`、`oncall-search` 的仪表板，比一个巨型仪表板更经得起时间考验。
- **页面顶部放最重要的小部件。** 人是从顶部扫的——确保他们看到的第一件事就是回答"这个系统健康吗？"。
- **用 Text 小部件给区块贴标签。** 每隔几行加一个简短的标题（"Latency" / "Errors" / "Capacity"）让仪表板隔着房间也能扫读。
- **用变量代替重复。** 如果你发现自己在为两个服务搭两遍同样的仪表板，那你需要一个 `service` 变量。见 [仪表板变量与过滤器](/docs/dashboards/variables)。
