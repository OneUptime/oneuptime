# 仪表板配置与权限

本页收集了当你已经有一个真的想长期保留的仪表板时，值得了解的设置和访问控制旋钮。

## 所有权

仪表板的 **owners** 是被显式授予该仪表板权限的用户和团队（独立于项目级角色）。

在 **仪表板 → Owners** 下：

- 添加 **user owner** 来给某个具体的人在该仪表板上额外的访问权。
- 添加 **team owner** 来给团队的每个成员同样的权限。

当项目级读取角色范围太宽时使用所有权——例如包含客户级敏感细节的仪表板只应被客户成功团队看到。

## 标签

标签是用于组织仪表板的多对多标签。在 **仪表板 → Overview** 下应用。

常见标签模式：

- **按团队**：`team:platform`、`team:checkout`、`team:growth`。
- **按环境**：`env:prod`、`env:staging`。
- **按用途**：`purpose:oncall`、`purpose:exec`、`purpose:investigation`。

**仪表板** 列表支持按标签过滤——在已经积累了几十个仪表板的项目里，这是找到一个仪表板最快的方式。

## 权限

仪表板是 OneUptime 基于角色的访问控制中的一等资源。相关权限：

| 权限 | 允许 |
| --- | --- |
| `CreateDashboard` | 在项目中创建新仪表板。 |
| `ReadDashboard` | 查看仪表板（私有模式）。 |
| `EditDashboard` | 修改仪表板上的小部件、变量、设置。 |
| `DeleteDashboard` | 删除仪表板。 |

支持实体（用户/团队所有者、自定义域名）有各自的 create / read / edit / delete 权限对，因此你可以授予"管理所有者"而不授予"编辑仪表板本身"。

在 **项目设置 → Teams & Roles** 下把这些权限分配到项目角色上。

## 公开模式访问控制

公开模式访问（见 [共享与公开仪表板](/docs/dashboards/sharing)）由三层按顺序治理：

1. **Public Dashboard** 开关 — 关闭时，公开 URL 返回 404。
2. **Master Password** — 设置后，访客必须先输入它才能渲染仪表板。
3. **IP Whitelist**（Scale 计划） — 设置后，来自非列表 IP 的请求收到 403。

一个仪表板可以采用任意组合。最具防御性的配置是"Public 打开、设置密码、IP 允许名单生效"——对你想要三层防护的合作伙伴门户有用。

## 留存

仪表板本身不会过期。它们展示的数据遵循项目的遥测留存——指标、日志、追踪能查询多久取决于你的计划保留多久。在留存 30 天的计划上，指向"过去 90 天"的小部件会渲染存储中还剩下的部分。

## 克隆仪表板

要复制一个已有仪表板，打开它并使用仪表板列表中的 **Duplicate** 动作。副本包含每一个小部件、变量和设置，公开模式配置除外（副本总是从关闭开始——是否再启用由你决定）。

当你想把一个模板（"我们的值班仪表板"）派生成针对某个服务的版本时，这就是合适的模式。

## 删除仪表板

在 **仪表板 → Delete** 下。这是不可逆的——画布配置和任何自定义域名绑定都会被移除。遥测数据不受影响（它在指标/日志/追踪存储里，不在仪表板上）。

如果一个仪表板带自定义域名公开发布，删除它的那一刻公开 URL 就停止解析。如果你需要重新指向，请先把域名解绑下来。

## 迁移与备份

自托管安装：仪表板的完整配置（小部件、变量、设置）保存在 Postgres 的 `Dashboard` 表里。定期数据库备份就够了——没有单独的仪表板导出格式。

OneUptime Cloud：定期备份由我们处理。如果你想要仪表板配置的本地副本，用 [OneUptime API](/docs/api-reference/api-reference) 读取 `Dashboard` 记录。
