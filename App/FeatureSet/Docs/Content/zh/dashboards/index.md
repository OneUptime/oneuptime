# 仪表板概览

仪表板把 OneUptime 已经在采集的遥测数据——指标、日志、追踪、事件、监控、Kubernetes 与 Docker 资源——变成一个能让人一眼看懂系统健康的页面。

把请求延迟的图、打开事件的列表、CPU 利用率的仪表盘和一句话英文状态摘要放在一起。保存、把链接分享出去。

## 一览

- OneUptime 仪表板 **仪表板** 下的**顶层功能**。
- **基于网格的画布** — 默认 12 单位宽、60 单位高。把小部件拖入、调整大小、对齐到网格。
- **20+ 种小部件类型** — 图表、单值、仪表盘、表格、文本块、日志流、追踪列表，以及事件、告警、监控、Kubernetes（Pod、节点、Deployment 等）、Docker 和主机的实时资源列表。
- **变量与过滤器** — 把一个仪表板变成模板视图，可针对每个集群、服务、客户或环境复用。
- **公开分享** — 打开开关，仪表板就能通过公开 URL 访问，可选密码保护与 IP 允许名单。
- **自定义域名** — 用 `status.your-domain.com` 而不是 OneUptime 的域名托管公开仪表板。

## 为什么用仪表板？

只要满足其中一种情况，仪表板就值回票价：

- **你需要一个"一切是否正常？"页面** — 给值班轮值、团队站会，或者经过墙上电视的 CEO。
- **你需要把信号关联起来** — 同一分钟的 CPU 飙升、追踪延迟上升和打开事件，放在同一个仪表板上远比开三个标签页明显。
- **你正在排查** — 调试过程中即兴搭一个仪表板，比手动跑十次查询快得多。
- **你要对外发布** — 给客户看的性能仪表板、给合作伙伴看的汇总、开源服务的公开健康看板。

## 关键概念

| 术语 | 含义 |
| --- | --- |
| **仪表板** | 画布。一个命名的可复用视图，包含小部件列表、时间范围控件和一组变量。 |
| **小部件** | 画布上的一个组件——图表、值、表格、文本块、列表。每个有类型和 JSON 风格的配置。 |
| **仪表板单位** | 网格方格。小部件以仪表板单位为单位调整尺寸（例如"4 宽 × 6 高"）。单位会按视口转换成像素。 |
| **变量** | 一个命名值，查看者通过下拉框选择（或输入），仪表板把它注入每个小部件的查询。集群、服务、客户、环境——任何你想要过滤的维度。 |
| **时间范围** | 每个小部件查询所针对的时间窗。可选预设（"过去 24 小时"）或自定义范围。 |
| **刷新间隔** | 在 **View** 模式下小部件重新查询的频率。关闭、5s、10s、30s、1m、5m、15m。 |
| **模式** | `Edit`（拖动、调整大小、配置）或 `View`（只读）。两者共享同一个画布。 |

## 小部件清单

仪表板上能放什么的不完全地图：

| 类别 | 小部件 |
| --- | --- |
| **时间序列** | Chart |
| **单值** | Value、Gauge |
| **表格** | Table |
| **注释** | Text |
| **日志与追踪** | LogStream、TraceList |
| **运营列表** | IncidentList、AlertList、MonitorList |
| **Kubernetes** | KubernetesPodList、KubernetesNodeList、KubernetesNamespaceList、KubernetesDeploymentList、KubernetesStatefulSetList、KubernetesDaemonSetList、KubernetesJobList、KubernetesCronJobList |
| **Docker** | DockerHostList、DockerContainerList、DockerImageList、DockerNetworkList、DockerVolumeList |
| **基础设施** | HostList |

各小部件的参数和适用场景见 [仪表板小部件](/docs/dashboards/widgets)。

## 仪表板在仪表板中的位置

| 页面 | 在那里做什么 |
| --- | --- |
| **仪表板** | 浏览、创建、搜索、给仪表板打标签。 |
| 仪表板 → **View** | 画布——作者的 Edit 模式，其他所有人的 View 模式。在头部切换。 |
| 仪表板 → **Overview** | 描述、所有权、标签。 |
| 仪表板 → **Settings** | 公开分享、主密码、IP 允许名单、自定义域名、品牌（页面标题、描述、Logo、favicon）。 |
| 仪表板 → **Owners** | 拥有明确所有权的用户和团队。 |
| 仪表板 → **Delete** | 删除仪表板（不可逆）。 |

## 仪表板的生命周期

1. **创建** — 在 **仪表板 → 创建仪表板** 下命名。画布以空白打开。
2. **拖入小部件** — 从面板挑类型，配置数据源（指标查询、列表过滤器、纯文本正文）。定位并调整大小。
3. **（可选）添加变量** — 定义诸如 `cluster` 或 `service` 的下拉，让同一个仪表板为每个值渲染。
4. **设置时间范围和刷新间隔** — 默认就够用；以后再调。
5. **（可选）公开分享** — 在 **设置** 中打开 **Public Dashboard**。如果你想加门槛，加个主密码；或按 IP 限制。
6. **（可选）自定义域名** — 添加 `dashboard.your-domain.com` 记录并验证 DNS，然后在你自己的 URL 上提供仪表板。

## 完整示例

目标：一个 checkout 服务的值班页面，包含延迟、错误率、打开的事件、最近日志。

1. 创建仪表板 "Checkout oncall"。
2. 添加一个 `service` 变量，类型为 **Telemetry Attribute**，绑定到属性 key `service.name`。默认值 `checkout`。
3. 加一个 **Chart** 小部件：来自 APM 指标的 P95 延迟，按 `service.name = {{service}}` 过滤。时间范围跟随仪表板。
4. 旁边加一个 **Value** 小部件：错误率百分比，警告阈值 1%、严重阈值 5%。
5. 下面加一个 **IncidentList** 小部件，按包含 `checkout` 的标签过滤。
6. 再下面，一个 **LogStream** 小部件，按 `service.name = {{service}}` 过滤。
7. 保存。把变量下拉切到 `payments`——整个仪表板会以 payments 服务的视角重新渲染。同一份模板，不同过滤。

## 仪表板如何与 OneUptime 的其他部分配合

- **监控与遥测** 给仪表板提供原始数据——你配置的每个指标、采集的每条日志、每个 trace span 都可在小部件上查询。
- **事件与告警** 出现在 **IncidentList** 与 **AlertList** 小部件中——仪表板是它们的只读视图；创建/编辑这些实体在别处完成。
- **状态页** 是面向客户的沟通工具（"系统现在是否在线？"）。仪表板是分析工具（"系统具体表现如何？"）。两者互补，不能相互替代。
- **工作流** 是 OneUptime 的写入侧——仪表板是读取侧。
