# 仪表板概览

仪表板把 OneUptime 已经在采集的数据——指标、日志、追踪、事件、监控、Kubernetes 资源、主机——变成一个让人一眼就能看清现状的页面。

把一个请求延迟的图表放在一个未解决事件列表旁边、再加一个 CPU 仪表、再附一段说明。保存。分享链接。

## 仪表板适合做什么

- **一个"一切都正常吗?"页面**——给值班、给团队站会,或者挂在墙上的电视上。
- **发现关联**——CPU 飙升的同时延迟在涨、还伴随着一个未解决的事件,这种联系在一张页面上比在三个标签页之间切换容易看多了。
- **排查问题**——调试时,一张你即兴搭的仪表板比一个个跑十次查询要快得多。
- **对外分享**——给客户的性能页面、给合作伙伴的状态页面、为开源项目搭的公共仪表板。

## 仪表板上可以放什么

- **图表** 用于时序趋势——延迟、错误、吞吐量。
- **单值瓦片和仪表盘**——当前错误率、CPU、未解决事件数。
- **表格** 用于分组对比——最吵的 10 台主机、各服务的错误数。
- **文本块** 用于标题、说明,以及指向 Runbook 的链接。
- **实时列表** 包含事件、告警、监控、日志、追踪、Kubernetes 资源、Docker 资源,以及主机。

完整清单和每个组件的展示内容见 [组件](/docs/dashboards/widgets)。

## 关键术语

| 术语 | 含义 |
| --- | --- |
| **仪表板** | 整个页面——一个名称、一组网格化的组件、时间范围控件,以及变量列表。 |
| **组件** | 页面上的一块瓦片——一个图表、一个数字、一个列表、一段文字。 |
| **变量** | 顶部的下拉框,可以一次性筛选所有组件(集群、服务、客户、环境)。 |
| **时间范围** | 所有图表和数字使用的时间窗口。在页面顶部设置一次。 |
| **刷新** | 组件重新查询数据的频率。可以关闭、几秒一次、几分钟一次。 |
| **模式** | 要么是 **编辑**(拖动组件),要么是 **查看**(只读,访客看到的样子)。 |

## 在哪里找到仪表板

在左侧导航打开 **仪表板**。

| 页面 | 你在那里可以做什么 |
| --- | --- |
| **仪表板** | 你的仪表板列表。新建一个、搜索,或按标签筛选。 |
| **仪表板 → 查看** | 画布。在头部切换 **编辑** 与 **查看**。 |
| **仪表板 → 概览** | 描述、所有者和标签。 |
| **仪表板 → 设置** | 公开分享、密码、IP 白名单、自定义域名、品牌定制。 |
| **仪表板 → 所有者** | 拥有显式访问权限的用户和团队。 |
| **仪表板 → 删除** | 删除仪表板。 |

## 构建仪表板

1. **创建**——起一个名称。画布打开时是空的。
2. **添加组件**——选择组件类型,配置数据,然后拖到你想要的位置。
3. **(可选)添加变量**——例如,加一个 `service` 下拉框,让同一张仪表板适用于每个服务。
4. **设置时间范围**——默认值够用;以后再调整。
5. **(可选)公开分享**——在设置中切换开关,如有需要再加密码或 IP 白名单。
6. **(可选)自定义域名**——把仪表板部署到 `status.your-domain.com`。

## 一个简单的例子

目标:为 checkout 服务做一个值班页面,包含延迟、错误率、未解决事件,以及实时日志尾部。

1. 创建一个名为 "Checkout 值班" 的仪表板。
2. 添加一个 `service` 变量。默认设为 `checkout`。
3. 添加一个 **图表** 组件,显示 P95 延迟,按 `service` 变量过滤。
4. 在它旁边添加一个 **单值** 组件显示错误率,警告阈值 1%、严重阈值 5%。
5. 下面添加一个 **事件列表** 组件,展示打了 `checkout` 标签的事件。
6. 再下面加一个 **日志流** 组件,显示同一服务的日志。
7. 保存。把下拉切换到 `payments`——同一张仪表板现在展示的是 payments 服务。

## 仪表板与 OneUptime 其他部分的关系

- **监控与遥测** 是数据来源。你采集的每个指标、日志和追踪都可以在组件上查询。
- **事件与告警** 会显示在 **事件列表** 和 **告警列表** 组件中。仪表板对这些是只读的——创建和更新在别处进行。
- **状态页** 是面向客户的沟通("系统在线吗?")。仪表板用于详细查看系统的运行情况。两者协同工作,不互相替代。
- **工作流** 是 OneUptime 采取行动的方式。仪表板是你查看现状的方式。
