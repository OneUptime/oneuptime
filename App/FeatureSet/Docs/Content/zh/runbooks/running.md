# 运行 Runbook

Runbook 执行有三种生成方式：

1. **通过规则自动触发** — 见 [Runbook 规则](/docs/runbooks/rules)。
2. **从 Runbook 页面手动触发** — 在 Runbook 概览页点击 **立即运行**。不绑定任何事件、告警或维护。
3. **从实体动态手动触发** — 在事件、告警或计划维护事件上点击 **运行 Runbook**。执行会绑定到该实体上。

## 执行视图

打开任意一次执行即可看到清单 UI。每个步骤显示：

- **状态徽标** — 待处理、运行中、等待你、完成、已跳过、失败。
- **标题与描述** — 在执行时从 Runbook 复制。
- **输出**（可折叠）— stdout、返回值、HTTP 响应。
- **错误信息** （步骤失败时）。
- 处于 `WaitingForUser` 的手动步骤：**标记完成** 与 **跳过** 按钮。

只要执行未达终态，页面每 3 秒轮询一次，因此自动步骤几乎实时显示完成。

## 手动与自动步骤交错

经典流程：

1. **脚本步骤**：捕获系统状态，写入 S3。
2. **手动步骤**："通过状态页横幅通知客户。"响应者勾选。
3. **HTTP 步骤**：通过 PagerDuty 呼叫 DBA。
4. **手动步骤**："确认副本 DB 已切为主库。"响应者勾选。
5. **脚本步骤**：向 Slack 发送"恢复正常"消息。

第 2、4 步会让执行暂停直到勾选。第 1、3、5 步自动运行。整次流程是一个执行、一条时间线、一个事实来源。

## 取消执行

在执行页点击 **取消执行**。当前步骤（如果有）跑完；后续步骤不再启动。状态变为 `Cancelled`。

## 输出保留

每步输出上限 **50KB**，防止失控脚本撑爆数据库。需要更大产物时，请在脚本里把数据写到 S3 或日志系统，再把 URL 放进返回值。

## 重新运行 Runbook

执行是一次性的、不可变的记录。要重跑，再点一次 **立即运行**——这会用 Runbook 当前步骤的新快照创建一次新执行。原执行保持不动，留作审计。

## 找历史执行

每个 Runbook 都有 **执行** 标签，列出它的所有运行，可以按状态、日期范围、源实体过滤。在事件、告警或维护上，**Runbooks** 标签会显示绑定到该实体的运行。
