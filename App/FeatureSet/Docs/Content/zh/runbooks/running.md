# 运行 Runbook

Runbook 执行有三种生成方式：

1. **通过规则自动触发** — 见 [Runbook 规则](/docs/runbooks/rules)。
2. **从 Runbook 页面手动触发** — 在 Runbook 概览页点 **立即运行**。不会附属任何事件、告警或计划维护。
3. **从实体动态手动触发** — 在事件、告警或计划维护事件上点 **运行 Runbook**。执行会被附属到该实体。

## 执行视图

打开任意一次执行查看其清单 UI。每个步骤显示：

- **状态徽标** — Pending、Running、等你处理、Done、Skipped、Failed。
- **标题与描述** — 在执行创建时从 Runbook 拷贝过来。
- **输出**（可折叠） — stdout、返回值、HTTP 响应。
- 步骤失败时的 **错误信息**。
- 对于 `WaitingForUser` 状态的 Manual 步骤：**标记完成** 与 **跳过** 按钮。

执行未进入终态期间，页面每 3 秒轮询一次，所以你能近实时看到自动步骤完成。

## 手动与自动步骤交织

经典流程：

1. **脚本步骤**：捕获系统状态，写到 S3。
2. **手动步骤**："通过状态页横幅通知客户。" 响应者勾选。
3. **HTTP 步骤**：通过 PagerDuty 呼叫 DBA。
4. **手动步骤**："确认副本 DB 现在是主库。" 响应者勾选。
5. **脚本步骤**：发送恢复 Slack 消息。

步骤 2 与 4 会让执行暂停直到被勾选。步骤 1、3、5 自动运行。整个运行是一次执行、一条时间线、一份事实来源。

## 取消运行

在执行页点 **取消执行**。当前步骤（如果有）会跑完；后续步骤不再启动。状态变为 `Cancelled`。

## 输出保留

每步输出上限为 **50KB**，以防失控的脚本把数据库撑爆。如果需要更大产物，从脚本写到 S3 或日志器，把 URL 放在返回值里。

## 重新运行一个 Runbook

Runbook 的执行是一次性的、不可变的记录。要重新运行，再点一次 **立即运行** — 那会创建一次新的执行，并附上当前步骤的新快照。原来的执行会原样保留，作为审计轨迹。

## 查找过往执行

每个 Runbook 都有 **执行** 标签，列出它的所有运行，可按状态、日期范围和来源实体过滤。从事件、告警或计划维护事件页面，**Runbooks** 标签会显示附属到该实体的运行。
