# 编写 Runbook

在 **Runbooks → 创建 Runbook** 中创建 Runbook，然后打开它并进入 **步骤** 标签。

## 步骤的结构

每个步骤包含：

| 字段 | 用途 |
| --- | --- |
| **标题** | 清单 UI 中的短标签。必填。 |
| **描述** | 给响应者的可选上下文。支持 Markdown。 |
| **失败时继续** | 开启时，失败的步骤不会停止执行——下一步仍然会运行。 |
| **类型相关配置** | 脚本、URL 等——见下文。 |

步骤**按顺序**执行。用步骤编辑器中的上下箭头调整顺序。

## 步骤类型

### Manual

由响应者勾选的复选框。执行到达 Manual 步骤时会暂停，并保持在 `WaitingForManualStep` 状态，直到有人将其标记为完成（或跳过）。

适合那些只有人才能验证的事情："负载均衡器仪表板显示流量已切到副本区域——确认。"

### JavaScript

在 `isolated-vm` 沙箱中运行的 JavaScript 片段（没有文件系统，没有网络，除非你自己注入 API）。

```js
const start = Date.now();
// ... 你的逻辑 ...
return { durationMs: Date.now() - start };
```

返回值会记录到步骤执行上。`console.log` 输出会作为日志行被捕获。默认超时：30 秒。

### HTTP 请求

发起一次出站 HTTP 调用。配置方法（GET/POST/PUT/PATCH/DELETE/HEAD）、URL、可选 JSON 头部、可选请求体。响应状态、头部和正文会被记录（合计上限 50KB）。

适合：开 PagerDuty 事件、发 Slack、调用自家管理 API 等。

### Bash

在 [Runbook 代理](/docs/runbooks/agents) 上运行的 bash 脚本——一个你安装在自己基础设施某台主机上的小进程。Bash 步骤绝不会在 OneUptime Worker 上执行。

在 Bash 步骤上配置两件事：

- **Agent Tag** — 标识哪个/哪些代理应执行此步骤的标签。项目中带有该标签的任一健康代理都会认领并运行该作业。
- **脚本** — 要运行的 bash。输出（stdout + stderr）最多捕获 50 KB；超时则强制终止进程。

如果 Runbook 到达此步骤时没有带所选标签的代理在线，该步骤将等待至 **认领超时**（默认 2 分钟）然后失败。在依赖 Bash 步骤之前，先在 **Runbooks → Agents** 下添加一个代理。

## 保存与编辑

点击 **保存步骤** 持久化。Runbook 旧版本的进行中执行不会受到影响——它们沿用自己的快照。

## 多步骤与失败处理

默认情况下，一个失败的步骤会终止执行并将执行标记为 `Failed`。如果在某个步骤上开启 **失败时继续**，失败会被记录但下一个步骤仍会运行。适合"试这三件事，然后通知"这种模式。

## 完整示例

针对"主数据库不可达"的简单 Runbook：

1. **JavaScript** — 从配置服务取当前主库地址并记录。
2. **Manual** — "副本的复制延迟小于 5 秒——确认。"
3. **HTTP 请求** — POST 到 failover 协调器 API。
4. **Manual** — "写入已切到新主库——确认。"
5. **HTTP 请求** — 向 Slack POST "恢复正常" 消息。

响应者看着自动步骤跑完，勾选一个手动步骤，再看下一个自动步骤，依此类推。每步输出都保留下来供事后回顾。
