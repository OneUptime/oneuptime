# 執行 Runbook

建立 runbook 執行的方式有三種：

1. **透過規則自動執行** — 請參閱 [Runbook Rules](/docs/runbooks/rules)。
2. **從 runbook 頁面手動執行** — 在 runbook 的總覽頁面上點按 **Run Now**。不會附加到任何事件、警示或排程維護事件。
3. **從實體動態消息手動執行** — 在事件、警示或排程維護事件上點按 **Run Runbook**。該執行會附加到該實體。

## 執行檢視畫面

開啟任何執行即可看到其檢查清單 UI。每個步驟會顯示：

- **狀態標籤** — Pending、Running、Waiting for you、Done、Skipped、Failed。
- **標題與描述** — 在執行時從 runbook 複製而來。
- **輸出**（可折疊） — stdout、回傳值、HTTP 回應。
- 若步驟失敗則顯示**錯誤訊息**。
- 對於處於 `WaitingForUser` 的手動步驟：**Mark Complete** 與 **Skip** 按鈕。

當執行尚未進入終止狀態時，頁面每 3 秒輪詢一次，因此您會看到自動化步驟近乎即時地完成。

## 交錯手動與自動化步驟

典型流程：

1. **Script 步驟**：擷取系統狀態，寫入 S3。
2. **手動步驟**：「透過狀態頁橫幅通知客戶。」回應人員勾選完成。
3. **HTTP 步驟**：透過 PagerDuty 呼叫 DBA。
4. **手動步驟**：「確認次要資料庫現已成為主要資料庫。」回應人員勾選完成。
5. **Script 步驟**：傳送全部解除的 Slack 訊息。

步驟 2 與 4 會暫停執行，直到被勾選為止。步驟 1、3、5 會自動執行。整個執行過程是一次執行、一條時間軸、一個真實來源。

## 取消執行

在執行頁面上點按 **Cancel Execution**。目前的步驟（若有）會完成；後續步驟則不會開始。狀態會變成 `Cancelled`。

## 輸出保留

每個步驟的輸出上限為 **50KB**，以防失控的指令碼撐大資料庫。如果您需要更大的產物，請從指令碼將它們寫入 S3 或記錄器，並將 URL 存放在回傳值中。

## 重新執行 Runbook

runbook 執行是一次性、不可變的記錄。若要重新執行，請再次點按 **Run Now** — 這會建立一個全新的執行，並對 runbook 目前的步驟建立全新的快照。原始執行會保持完整以供稽核軌跡使用。

## 尋找過往執行

每個 runbook 都有一個 **Executions** 分頁，列出其所有的執行，並提供依狀態、日期範圍與來源實體的篩選功能。從事件、警示或排程維護事件中，**Runbooks** 分頁會顯示附加到該實體的執行。
