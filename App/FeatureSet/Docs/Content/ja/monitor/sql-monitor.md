# SQL クエリモニター

SQL クエリモニターは、読み取り専用の SQL クエリをプローブからスケジュールに従って実行し、その結果（返された行数、スカラー値、クエリの実行時間、またはクエリエラー）に基づいてアラートを発行します。「クエリを実行してインシデントを起票する」というユースケースのために作られており、たとえば直近 5 分間のキャンセル済み注文の数が急増したとき、キュー用テーブルが大きくなりすぎたとき、あるいは重要な行が消失したときにアラートを発行できます。

クエリはネットワーク内のプローブから実行されるため、OneUptime がデータベースへ直接接続する必要は一切なく、完全な結果セットがプローブの外に出ることもありません。プローブからは、結果を小さく制限した射影だけが報告されます。

## サポートされているデータベース

SQL クエリモニターは、以下のデータベースエンジンをサポートしています。

- **PostgreSQL**（デフォルトポート `5432`）
- **MySQL**（デフォルトポート `3306`）
- **Microsoft SQL Server**（デフォルトポート `1433`）

同じワイヤープロトコルと SQL 方言を扱う MySQL 互換および PostgreSQL 互換のエンジンも一般的に動作しますが、公式にテストされているのは上記の 3 つのエンジンのみです。

## 仕組み

各チェックのたびに、プローブはデータベースに接続し、読み取り専用のコンテキストでクエリを実行し、制限された行数までを読み戻して、コンパクトな射影を OneUptime に報告します。モニターの条件は、その射影に対して評価されます。

プローブが報告するのは、以下のみです。

- **行数（Row Count）** — クエリが返した行数（Max Rows の上限で制限されます）。
- **スカラー値（Scalar Value）** — 最初の行の最初の列。これは `SELECT COUNT(*)` のようなクエリにとって自然な値です。
- **最初の行（First Row）** — 列と値のペアの集合としての最初の行。コンテキストのためにチェックのサマリーに表示されます。
- **実行時間（Execution Time）** — クエリの実行にかかった時間（ミリ秒）。
- **クエリエラー（Query Error）** — クエリが失敗した場合のサニタイズ済みエラーメッセージ。

完全な結果セットが OneUptime に送信されることはないため、顧客データが OneUptime のストレージに複製されることはありません。

## セキュリティモデル

本番データベースに対して顧客が指定したクエリを実行することは慎重を要するため、SQL クエリモニターは設計上読み取り専用であり、いくつかの制御を重ねています。

- **最小権限のデータベースユーザー（主要な制御）。** 常に、クエリが必要とするテーブルにのみアクセスできる、専用の読み取り専用データベースユーザーで接続してください。これが最も重要な制御です。以下の 読み取り専用ユーザーの作成 を参照してください。
- **読み取り専用の実行。** PostgreSQL と MySQL では、プローブは `READ ONLY` トランザクションを開始し、クエリのテキストにかかわらずあらゆる書き込み（書き込み可能な CTE を含む）を拒否します。読み取り専用トランザクションを持たない Microsoft SQL Server では、プローブは常にロールバックされるトランザクション内で実行します。
- **単一ステートメントの許可リスト方式クエリ。** クエリは `SELECT`、`WITH`、`VALUES`、または `TABLE` で始まる単一のステートメントである必要があります。スタックされたステートメント（`SELECT 1; DROP TABLE …`）や書き込み／DDL は、プローブが接続する前に拒否されます。このチェックはコメントと文字列リテラルを認識するため、コメントや文字列内に隠されたキーワードが通過することはありません。
- **ステートメントタイムアウト。** すべてのクエリには厳密な時間制限があります。実行に時間がかかりすぎるクエリはキャンセルされます。
- **行数の制限。** 読み戻されるのは最大で Max Rows（切り捨てを検出するためにプラス 1）行までであり、これによりプローブのメモリとペイロードのサイズが制限されます。
- **認証情報の秘匿。** データベースエラーは保存前にサニタイズされます。パスワードと接続文字列は秘匿されるため、認証情報がエラーメッセージに漏れることはありません。

## 前提条件

- データベースのホストとポートにネットワークアクセスできる **プローブ**。これは、（データベースがインターネットから到達可能であれば）OneUptime がホストするプローブでも、ネットワーク内で実行されるセルフホスト型のプローブでも構いません。カスタムプローブのインストール方法については、プローブのドキュメントを参照してください。
- **読み取り専用のデータベースユーザー** と接続情報（ホスト、ポート、データベース名、ユーザー名、パスワード）。

## 設定

新しいモニターを作成し、モニタータイプとして **SQL クエリ** を選択してから、接続情報を入力します。

- **データベースタイプ（Database Type）** — PostgreSQL、MySQL、または Microsoft SQL Server。タイプを選択すると、デフォルトのポートが設定されます。
- **ホスト（Host）** — プローブから到達可能なデータベースホスト（例：`db.internal`）。
- **ポート（Port）** — データベースのポート。
- **データベース名（Database Name）** — クエリを実行する対象のデータベース。
- **ユーザー名（Username）** — 読み取り専用の最小権限データベースユーザー。
- **パスワード（Password）** — データベースのパスワード。パスワードをプレーンテキストで入力する代わりに、`{{monitorSecrets.name}}` を使って [モニターシークレット](/docs/monitor/monitor-secrets) を参照することを強く推奨します（以下を参照）。
- **SQL クエリ（SQL Query）** — 実行する読み取り専用クエリ（クエリの記述 を参照）。
- **SSL/TLS を使用（Use SSL/TLS）** — TLS 経由で接続する場合に有効にします。有効にすると、データベースが自己署名証明書を使用している場合は **サーバー証明書を検証（Verify server certificate）** をオフにできます。

### 詳細オプション

- **接続タイムアウト（ミリ秒）（Connection Timeout (ms)）** — 接続の確立を待つ時間。デフォルト `10000`、最大 `30000`。
- **ステートメントタイムアウト（ミリ秒）（Statement Timeout (ms)）** — クエリが実行できる時間の厳密な上限。デフォルト `15000`、最大 `60000`。
- **最大行数（Max Rows）** — データベースから読み戻される行数の上限。デフォルト `100`、最大 `1000`。

## クエリの記述

クエリは **単一の読み取り専用ステートメント** である必要があります。`SELECT`、`WITH`、`VALUES`、または `TABLE` のいずれかで始まる必要があります。末尾のセミコロンは 1 つだけ許可されますが、複数のステートメントは許可されません。

クエリは軽量かつ適切にスコープを絞ってください。クエリはチェックのたびに実行されるため、インデックス付きの列や狭い時間範囲を使うことをおすすめします。

```sql
-- 直近のキャンセル数をカウント（PostgreSQL）
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL '5 minutes';
```

```sql
-- MySQL での同じ考え方
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL 5 MINUTE;
```

```sql
-- Microsoft SQL Server での同じ考え方
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > DATEADD(minute, -5, GETDATE());
```

`COUNT(*)` のようなクエリでは、そのカウント値は **行数（Row Count）**（1 行が返されるため `1` になります）としても、**スカラー値（Scalar Value）**（最初の列から得られるカウント値そのもの）としても利用できます。「いくつあるか」でアラートを発行するには、**スカラー値（Scalar Value）** と比較してください。

## パスワードへのモニターシークレットの使用

データベースのパスワードがモニター上にプレーンテキストで保存されないように、[モニターシークレット](/docs/monitor/monitor-secrets) を作成し、パスワードフィールドからそれを参照します。

1. OneUptime Dashboard → プロジェクト設定 → モニターシークレット → モニターシークレットの作成 に移動します。
2. シークレット（たとえば `dbPassword`）を作成し、このモニターにそのシークレットへのアクセスを許可します。
3. モニターのパスワードフィールドに `{{monitorSecrets.dbPassword}}` と入力します。

OneUptime は、設定がプローブに渡される前に、サーバー側でシークレットを解決します。OneUptime がこれらのシークレットを代わりに作成することはありません。シークレットを参照するかどうかはあなたの選択です。

## 条件の設定

モニターがオンライン、劣化、またはオフラインとみなされるタイミングを判断するために条件を追加します。SQL クエリモニターでは、以下のチェックを利用できます。

- **SQL がオンライン（SQL Is Online）** — データベースに到達可能で、クエリが成功したかどうか。
- **SQL クエリの行数（SQL Query Row Count）** — 返された行数。「より大きい」「より小さい」「等しい」などの演算子で比較します。
- **SQL クエリのスカラー値（SQL Query Scalar Value）** — 最初の行の最初の列。両辺が数値のように見える場合は数値として、そうでない場合は文字列として比較されます。これは `COUNT(*)` のようなクエリに使用するチェックです。
- **SQL クエリの実行時間（ミリ秒）（SQL Query Execution Time (in ms)）** — クエリの実行にかかった時間。遅いデータベースを検出するのに役立ちます。
- **SQL クエリエラー（SQL Query Error）** — クエリのエラーメッセージ。空である（または空でない）とき、あるいは特定の文字列に一致するときにアラートを発行します。
- **JavaScript 式（JavaScript Expression）** — 完全に制御するためにカスタム JavaScript 式を評価します。[JavaScript 式](/docs/monitor/javascript-expression) を参照してください。

### 例：キャンセルが急増したときにアラートを発行する

上記のクエリを使用します。

- **条件：劣化（Degraded）** — `SQL Query Scalar Value` が `10` より大きい。
- **条件：オフライン（Offline）** — `SQL Query Scalar Value` が `50` より大きい、または `SQL Is Online` が `false` である。

適切な担当者に通知が届くように、条件にオンコールポリシーを紐づけてください。

## 読み取り専用ユーザーの作成

常に専用の読み取り専用ユーザーで接続してください。例：

```sql
-- PostgreSQL
CREATE USER oneuptime_ro WITH PASSWORD 'a-strong-password';
GRANT CONNECT ON DATABASE orders TO oneuptime_ro;
GRANT USAGE ON SCHEMA public TO oneuptime_ro;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO oneuptime_ro;
-- 将来作成されるテーブルも含める:
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO oneuptime_ro;
```

```sql
-- MySQL
CREATE USER 'oneuptime_ro'@'%' IDENTIFIED BY 'a-strong-password';
GRANT SELECT ON orders.* TO 'oneuptime_ro'@'%';
FLUSH PRIVILEGES;
```

```sql
-- Microsoft SQL Server
CREATE LOGIN oneuptime_ro WITH PASSWORD = 'a-strong-password';
USE orders;
CREATE USER oneuptime_ro FOR LOGIN oneuptime_ro;
ALTER ROLE db_datareader ADD MEMBER oneuptime_ro;
```

## 考慮すべき点

- クエリはチェックのたびに実行されるため、軽量に保ってください。インデックスと狭い時間範囲を使い、最後の砦としてステートメントタイムアウトに頼ってください。
- 報告されるのは行数、最初のセル（スカラー）、および最初の行のみです。アラートを発行したい値が最初の列になるようにクエリを設計してください。
- 結果が Max Rows を超えたために切り捨てられた場合、チェックのサマリーで上限に達したことが示されます。必要な場合にのみ Max Rows を増やしてください。結果セットが大きいほど、プローブ上でより多くのメモリを消費します。
- 書き込みと DDL は常に拒否されます。書き込みの経路をテストする必要がある場合、このモニターはその用途のためのものではありません。
- 認証情報が保存時に暗号化されたまま保たれるように、プレーンテキストのパスワードよりもモニターシークレットを優先してください。
