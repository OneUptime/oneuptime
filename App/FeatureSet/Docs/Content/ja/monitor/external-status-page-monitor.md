# 外部ステータスページモニター

外部ステータスページモニタリングを使用すると、サードパーティのステータスページを監視し、依存しているサービスで障害やパフォーマンス低下が発生した際にアラートを受け取ることができます。OneUptimeは定期的に外部ステータスページ（AWS、GCP、Azure、GitHub、OpenAI、Anthropicなど）を確認し、そのステータスを評価します。

## 概要

外部ステータスページモニターは、公開されているステータスページに問い合わせることで、依存しているサービスの正常性を確認します。これにより、以下のことが可能になります。

- アプリケーションが依存しているサードパーティサービスの可用性の監視
- アップストリームプロバイダーで障害が発生した際のアラート受信
- 個々のコンポーネントのステータス追跡（例：「AWS EC2 us-east-1」）
- 監視対象を単一のコンポーネントグループに絞り込み（例：OpenAIの「APIs」のみ）、ページ上の他の無関係なインシデントによってモニターが誤作動しないようにする
- ユーザーへの影響が出る前のパフォーマンス低下の検出
- 自社のインシデントとアップストリームプロバイダーの問題の相関分析

## サポートされるプロバイダー

OneUptimeは以下の方法によるステータスページの監視をサポートしています。

| プロバイダータイプ       | 説明                                                       |
| ------------------------ | ---------------------------------------------------------- |
| **自動**（デフォルト）   | ステータスページの形式を自動検出                           |
| **Atlassian Statuspage** | Atlassian Statuspage（JSON API）を利用したステータスページ |
| **incident.io**          | incident.io を利用したステータスページ（例：`https://status.openai.com`） |
| **RSS**                  | RSSフィードを提供するステータスページ                      |
| **Atom**                 | Atomフィードを提供するステータスページ                     |

### 自動検出

**自動** に設定すると、OneUptimeはステータスページの形式を以下の順序で自動的に検出しようとします。

1. まず、Atlassian Statuspage JSON API（`/api/v2/status.json`、`/api/v2/components.json`、`/api/v2/incidents/unresolved.json`）を試みます
2. 次に、incident.io ステータスページAPI（`/proxy/<host>`）を試みます
3. それらが失敗した場合、RSSまたはAtomフィードとしてページの解析を試みます
4. 最後のフォールバックとして、基本的なHTTPの到達可能性チェックを実行します

## 外部ステータスページモニターの作成

1. OneUptime ダッシュボードで **モニター** を開きます
2. **モニターの作成** をクリックします
3. モニタータイプとして **外部ステータスページ** を選択します
4. 監視するステータスページのURLを入力します
5. 必要に応じて特定のプロバイダータイプを選択します（または **自動** のままにします）
6. 必要に応じて **コンポーネントグループ** を入力し、「APIs」などのグループに絞り込みます
7. 必要に応じて **コンポーネント名** を入力し、単一のコンポーネントに絞り込みます（グループが設定されている場合はそのグループ内で絞り込みます）
8. 必要に応じて監視条件を設定します

## 設定オプション

### ステータスページURL

監視する外部ステータスページのURLを入力します。Atlassian Statuspageおよびincident.ioを使用したサイトの場合、通常はルートURL（例：`https://status.example.com`）です。RSS/AtomフィードはフィードURLを直接入力してください。

### プロバイダータイプ

ステータスページのプロバイダータイプを選択します。OneUptimeが形式を自動検出するには **自動**（デフォルト）を使用し、形式が分かっている場合は **Atlassian Statuspage**、**incident.io**、**RSS**、**Atom** のいずれかを指定します。

### コンポーネントグループフィルター

ステータスページがコンポーネントをグループに整理している場合、モニターを単一のグループに絞り込むことができます。たとえば、`https://status.openai.com` で `APIs` を入力すると、モニターはOpenAIのAPIサービスに絞り込まれます。

コンポーネントグループが設定されている場合、**アクティブなインシデント数** と **全体ステータス** は、そのグループ内のコンポーネントのみを使用して計算されます。無関係なグループ（たとえばChatGPT）に影響するインシデントは、「APIs」グループに絞り込まれたモニターを誤作動させることはありません。

コンポーネントグループフィルタリングは **Atlassian Statuspage** および **incident.io** プロバイダーでサポートされています。（RSS/Atomフィードはコンポーネントグループを公開しません。）

### コンポーネント名フィルター

ステータスページが複数のコンポーネントを報告している場合、オプションでコンポーネント名を指定して特定のコンポーネントのみを監視できます。たとえば、us-east-1のAWS EC2のみを監視する場合は、`EC2 us-east-1`（ステータスページに表示される正確なコンポーネント名）を入力します。

コンポーネントグループも設定されている場合、コンポーネント名フィルターはそのグループ **内** で適用され、より大きなグループ内の単一のコンポーネントを対象にできます。どちらのフィルターも指定されていない場合、対象範囲内のすべてのコンポーネントが監視されます。

### 詳細オプション

#### タイムアウト

ステータスページからのレスポンスを待つ最大時間（ミリ秒）。デフォルトは10000ミリ秒（10秒）。

#### リトライ回数

リクエストが失敗した場合の再試行回数。デフォルトは3回。

## 監視条件

以下に基づいて、外部サービスがオンラインまたはオフラインと判定されるタイミングを設定できます。

- **オンラインか** — ステータスページが到達可能でステータスデータを返しているか
- **全体ステータス** — ステータスページの全体的なステータス指標（例：`operational`、`degraded_performance`、`partial_outage`、`major_outage`）
- **コンポーネントステータス** — 対象範囲内のコンポーネントのステータス（コンポーネントグループ／コンポーネント名フィルターを反映）
- **アクティブなインシデント** — ステータスページで報告されている現在アクティブなインシデント数（フィルターが設定されている場合はコンポーネントグループ／コンポーネントに絞り込まれます）
- **レスポンスタイム** — ステータスページのデータを取得するまでの時間

### デフォルトの条件

デフォルトでは、OneUptimeは単なる到達可能性ではなく、ステータスページにとって実際に重要なもの、つまりアクティブなインシデントとコンポーネントの正常性に基づいて条件を設定します。

- 対象範囲内にアクティブなインシデントがない場合、モニターは **オンライン** とマークされます。
- 対象範囲内に少なくとも1件のアクティブなインシデントがある場合、または対象範囲内のコンポーネントが `degraded_performance`、`partial_outage`、`major_outage`、`full_outage` を報告した場合、モニターは **オフライン** とマークされ（インシデントが作成されます）。

アクティブなインシデント数とコンポーネントステータスはコンポーネントグループ／コンポーネント名フィルターを反映するため、これらのデフォルトの条件は、あなたが関心を持つコンポーネントのみを自動的に対象とします。

## インシデント・アラートテンプレート

外部ステータスページモニターからインシデントやアラートを作成する際に、以下のテンプレート変数を使用できます。

| 変数                      | 説明                                                                             |
| ------------------------- | -------------------------------------------------------------------------------- |
| `{{isOnline}}`            | ステータスページがオンラインか（true/false）                                     |
| `{{responseTimeInMs}}`    | レスポンスタイム（ミリ秒）                                                       |
| `{{failureCause}}`        | 障害の原因（存在する場合）                                                       |
| `{{overallStatus}}`       | 全体ステータス指標の値                                                           |
| `{{activeIncidentCount}}` | アクティブなインシデント数（フィルターが設定されている場合はそれに絞り込まれます） |
| `{{componentStatuses}}`   | コンポーネントステータスのJSON配列（`name`、`status`、`description`、`groupName`） |
| `{{provider}}`            | 検出されたプロバイダー（Atlassian Statuspage、incident.io、RSS、Atom）           |
| `{{componentGroup}}`      | モニターが絞り込まれているコンポーネントグループ（存在する場合）                 |
| `{{componentName}}`       | モニターが絞り込まれているコンポーネント（存在する場合）                         |

## ベストプラクティス

- **自動プロバイダータイプを使用する** — 正確な形式が分かっている場合を除き、自動検出はほとんどのステータスページで適切に機能します
- **コンポーネントグループに絞り込む** — プロバイダーの一部のみに依存している場合（例：OpenAIの「APIs」のみ）は、無関係なインシデントがノイズを生成しないようにグループに絞り込みます
- **特定のコンポーネントを監視する** — 特定のサービス（特定のAWSリージョンなど）のみに依存している場合は、そのコンポーネントを監視します
- **インシデントの相関分析を設定する** — 自社モニターが問題を検出し、アップストリームのステータスページも問題を示している場合、根本原因の特定が速くなります
- **他のモニターと組み合わせる** — 包括的な可視性のために、外部ステータスページモニターと自社のAPIやウェブサイトモニターを組み合わせます
