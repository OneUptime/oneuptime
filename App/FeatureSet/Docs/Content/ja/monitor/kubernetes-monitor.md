# Kubernetesモニター

Kubernetesモニタリングを使用すると、ノード、Pod、ワークロード、コントロールプレーンコンポーネントを含むKubernetesクラスターの正常性とパフォーマンスを監視できます。OneUptimeはクラスターからメトリクスを収集し、設定した条件に基づいてそれらを評価します。

## 概要

Kubernetesモニターはクラスターのメトリクスを使用して、インフラストラクチャの詳細な可視性を提供します。これにより、以下のことが可能になります。

- クラスター、ネームスペース、ワークロード、ノード、Podの正常性の監視
- リソース全体のCPU、メモリ、ディスク、ネットワーク使用量の追跡
- Podのクラッシュ、再起動、スケジューリング失敗の検出
- Deploymentのレプリカ可用性の監視
- コントロールプレーンの問題の検出（etcd、APIサーバー、スケジューラー）
- リソースリクエストと制限の追跡

## Kubernetesモニターの作成

1. OneUptime ダッシュボードで **モニター** を開きます
2. **モニターの作成** をクリックします
3. モニタータイプとして **Kubernetes** を選択します
4. 監視するクラスターとリソーススコープを選択します
5. リソースフィルターとメトリクスクエリを設定します
6. 必要に応じて監視条件を設定します

## 設定オプション

### クラスター

監視するKubernetesクラスターを選択します。クラスターはOpenTelemetryを介してOneUptimeと連携している必要があります。

### リソーススコープ

リソースを監視するレベルを選択します。

| スコープ | 説明 |
|-------|-------------|
| クラスター | クラスター全体を監視 |
| ネームスペース | 特定のネームスペース内のリソースを監視 |
| ワークロード | 特定のDeployment、StatefulSet、DaemonSet、Job、CronJobを監視 |
| ノード | 特定のクラスターノードを監視 |
| Pod | 特定のPodを監視 |

### リソースフィルター

オプションのフィルターでスコープを絞り込みます。

| フィルター | 説明 | 適用可能なスコープ |
|--------|-------------|-------------------|
| ネームスペース | Kubernetesネームスペース | ネームスペース、ワークロード、Pod |
| ワークロードタイプ | deployment、statefulset、daemonset、job、cronjob | ワークロード |
| ワークロード名 | ワークロードの名前 | ワークロード |
| ノード名 | ノードの名前 | ノード |
| Pod名 | Podの名前 | Pod |

### メトリクスクエリ

評価する1つ以上のメトリクスクエリを設定します。各クエリは以下を指定します。

- **メトリクス名** — 照会するKubernetesメトリクス
- **集計** — メトリクス値の集計方法
- **フィルター** — 追加の属性ベースのフィルタリング

複数のメトリクスクエリを数式で組み合わせた**フォーミュラ**も作成できます。

### ローリング時間ウィンドウ

メトリクス評価の時間ウィンドウを選択します。

- 過去1分
- 過去5分
- 過去10分
- 過去15分
- 過去30分
- 過去60分

## 主要なKubernetesメトリクス

### Podメトリクス

| メトリクス | 説明 |
|--------|-------------|
| Pod CPU使用量 | PodによるCPU消費量 |
| Podメモリ使用量 | Podによるメモリ消費量 |
| Podファイルシステム使用量 | Podによるディスク使用量 |
| Podネットワーク受信/送信 | ネットワークトラフィック |
| Podフェーズ | 現在のPodフェーズ（実行中、保留中、失敗など） |

### ノードメトリクス

| メトリクス | 説明 |
|--------|-------------|
| ノードCPU使用量 | ノードごとのCPU使用率 |
| ノードメモリ使用量 | ノードごとのメモリ使用率 |
| ノードファイルシステム使用量 | ノードごとのディスク使用量 |
| ノードディスクI/O | 読み取り/書き込み操作 |
| ノード準備状態 | ノードが準備完了かどうか |

### コンテナメトリクス

| メトリクス | 説明 |
|--------|-------------|
| コンテナ再起動数 | コンテナの再起動回数 |
| コンテナCPU/メモリ制限 | リソース制限 |
| コンテナCPU/メモリリクエスト | リソースリクエスト |
| コンテナ準備状態 | コンテナが準備完了かどうか |

### ワークロードメトリクス

| メトリクス | 説明 |
|--------|-------------|
| Deployment利用可能/利用不可レプリカ数 | レプリカ数 |
| DaemonSetのミススケジュールされたノード | スケジューリングの問題 |
| StatefulSetの準備完了レプリカ数 | 準備完了レプリカ数 |
| Jobのアクティブ/失敗/成功Pod数 | Jobのステータス |

## 監視条件

### 利用可能なチェックタイプ

| チェックタイプ | 説明 |
|------------|-------------|
| メトリクス値 | 設定したメトリクスクエリまたはフォーミュラの値 |

### 集計タイプ

| 集計 | 説明 |
|-------------|-------------|
| 平均 | 時間ウィンドウ内の平均値 |
| 合計 | すべての値の合計 |
| 最大値 | 時間ウィンドウ内の最大値 |
| 最小値 | 時間ウィンドウ内の最小値 |
| すべての値 | すべての値が条件を満たす必要がある |
| いずれかの値 | 少なくとも1つの値が条件を満たす |

### フィルタータイプ

- **より大きい**、**より小さい**、**以上**、**以下**、**等しい**、**等しくない**

## 事前定義されたアラートテンプレート

OneUptimeは一般的なKubernetesモニタリングシナリオのテンプレートを提供しています。

| テンプレート | 説明 | しきい値 |
|----------|-------------|-----------|
| CrashLoopBackOff検出 | コンテナの再起動数 | > 5回 |
| 保留中のPod | PendingフェーズのPod | > 0個 |
| ノードが準備完了でない | ノードの準備完了状態 | = 0（準備完了でない） |
| ノードのCPU高使用率 | ノードのCPU使用率 | > 90% |
| ノードのメモリ高使用率 | ノードのメモリ使用率 | > 85% |
| Deploymentのレプリカ不一致 | 利用不可レプリカ | > 0個 |
| Jobの失敗 | Job内の失敗したPod | > 0個 |
| etcdリーダーなし | etcdクラスターのリーダー欠如 | = 0（リーダーなし） |
| APIサーバーのスロットリング | ドロップされたAPIリクエスト | > 0リクエスト |
| スケジューラーのバックログ | スケジューラー内の保留中Pod | > 0個 |
| ノードのディスク高使用率 | ノードのファイルシステム使用量 | > 90% |
| DaemonSet利用不可 | ミススケジュールされたノード | > 0個 |

## セットアップ要件

Kubernetesモニタリングを使用するには、クラスターにOneUptime Kubernetesエージェントをインストールする必要があります。エージェントはOTLPを介してクラスターのメトリクス、イベント、PodログをOneUptimeに送信します。

[Kubernetesエージェントのインストール](/docs/monitor/kubernetes-agent)ガイドを参照してください。1コマンドのHelmインストールと、クラスター（Standard、GKE Autopilot、EKS Fargate）に適した設定を選択するための `preset` オプションについて説明しています。
