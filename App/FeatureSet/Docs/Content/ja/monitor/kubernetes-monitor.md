# Kubernetes モニター

Kubernetes モニタリングを使用すると、ノード、Pod、ワークロード、コントロールプレーンコンポーネントを含む Kubernetes クラスターの健全性とパフォーマンスを監視できます。OneUptime はクラスターからメトリクスを収集し、設定された基準に対して評価します。

## 概要

Kubernetes モニターはクラスターのメトリクスを使用して、インフラストラクチャを深く可視化します。これにより、以下のことが可能になります:

- クラスター、namespace、ワークロード、ノード、Pod の健全性を監視する
- リソース全体の CPU、メモリ、ディスク、ネットワーク使用率を追跡する
- Pod のクラッシュ、再起動、スケジューリング失敗を検出する
- Deployment のレプリカ可用性を監視する
- コントロールプレーンの問題(etcd、API サーバー、scheduler)をアラートする
- リソースのリクエストとリミットを追跡する

## Kubernetes モニターの作成

1. OneUptime ダッシュボードで **Monitors** に移動します
2. **Create Monitor** をクリックします
3. モニタータイプとして **Kubernetes** を選択します
4. 監視するクラスターとリソーススコープを選択します
5. リソースフィルターとメトリッククエリを設定します
6. 必要に応じて監視基準を設定します

## 設定オプション

### クラスター

監視する Kubernetes クラスターを選択します。クラスターは OpenTelemetry 経由で OneUptime と統合されている必要があります。

### リソーススコープ

リソースを監視するレベルを選択します:

| スコープ | 説明 |
|-------|-------------|
| クラスター | クラスター全体を監視します |
| Namespace | 特定の namespace 内のリソースを監視します |
| ワークロード | 特定の deployment、statefulset、daemonset、job、または cronjob を監視します |
| ノード | 特定のクラスターノードを監視します |
| Pod | 特定の Pod を監視します |

### リソースフィルター

オプションのフィルターでスコープを絞り込みます:

| フィルター | 説明 | 適用可能なスコープ |
|--------|-------------|-------------------|
| Namespace | Kubernetes namespace | Namespace、ワークロード、Pod |
| ワークロードタイプ | deployment、statefulset、daemonset、job、cronjob | ワークロード |
| ワークロード名 | ワークロードの名前 | ワークロード |
| ノード名 | ノードの名前 | ノード |
| Pod 名 | Pod の名前 | Pod |

### メトリッククエリ

評価対象とするメトリッククエリを 1 つ以上設定します。各クエリは以下を指定します:

- **メトリック名** — クエリ対象の Kubernetes メトリック
- **集計** — メトリック値の集計方法
- **フィルター** — 属性に基づく追加のフィルタリング

数式を使用して複数のメトリッククエリを組み合わせる**フォーミュラ**を作成することもできます。

### ローリングタイムウィンドウ

メトリック評価のためのタイムウィンドウを選択します:

- 過去 1 分
- 過去 5 分
- 過去 10 分
- 過去 15 分
- 過去 30 分
- 過去 60 分

## 一般的な Kubernetes メトリクス

### Pod メトリクス

| メトリック | 説明 |
|--------|-------------|
| Pod CPU 使用率 | Pod による CPU 消費 |
| Pod メモリ使用率 | Pod によるメモリ消費 |
| Pod ファイルシステム使用率 | Pod によるディスク使用量 |
| Pod ネットワーク受信/送信 | ネットワークトラフィック |
| Pod フェーズ | 現在の Pod フェーズ(Running、Pending、Failed など) |

### ノードメトリクス

| メトリック | 説明 |
|--------|-------------|
| ノード CPU 使用率 | ノードごとの CPU 使用率 |
| ノードメモリ使用率 | ノードごとのメモリ使用率 |
| ノードファイルシステム使用率 | ノードごとのディスク使用量 |
| ノードディスク I/O | 読み取り/書き込み操作 |
| ノード Ready 状態 | ノードが Ready かどうか |

### コンテナメトリクス

| メトリック | 説明 |
|--------|-------------|
| コンテナ再起動回数 | コンテナの再起動回数 |
| コンテナ CPU/メモリリミット | リソースリミット |
| コンテナ CPU/メモリリクエスト | リソースリクエスト |
| コンテナ Ready ステータス | コンテナが Ready かどうか |

### ワークロードメトリクス

| メトリック | 説明 |
|--------|-------------|
| Deployment 利用可能/利用不可レプリカ | レプリカ数 |
| DaemonSet 誤スケジュールノード | スケジューリングの問題 |
| StatefulSet Ready レプリカ | Ready レプリカ数 |
| Job アクティブ/失敗/成功 Pod | Job ステータス |

## 監視基準

### 利用可能なチェックタイプ

| チェックタイプ | 説明 |
|------------|-------------|
| メトリック値 | 設定されたメトリッククエリまたはフォーミュラの値 |

### 集計タイプ

| 集計 | 説明 |
|-------------|-------------|
| 平均 | タイムウィンドウ全体の平均値 |
| 合計 | すべての値の合計 |
| 最大値 | タイムウィンドウ内の最高値 |
| 最小値 | タイムウィンドウ内の最低値 |
| すべての値 | すべての値が基準に一致する必要があります |
| いずれかの値 | 少なくとも 1 つの値が一致する必要があります |

### フィルタータイプ

- **より大きい**、**より小さい**、**以上**、**以下**、**等しい**、**等しくない**

## 構築済みのアラートテンプレート

OneUptime は、一般的な Kubernetes 監視シナリオ向けのテンプレートを提供しています:

| テンプレート | 説明 | しきい値 |
|----------|-------------|-----------|
| CrashLoopBackOff 検出 | コンテナ再起動回数 | 5 回を超える再起動 |
| Pod が Pending で停止 | Pending フェーズの Pod | 0 個を超える Pod |
| ノード Not Ready | ノードの Ready 状態 | = 0(Ready ではない) |
| ノード CPU 高負荷 | ノード CPU 使用率 | 90% 超 |
| ノードメモリ高負荷 | ノードメモリ使用率 | 85% 超 |
| Deployment レプリカ不一致 | 利用不可レプリカ | 0 個を超えるレプリカ |
| Job の失敗 | Job 内の失敗した Pod | 0 件を超える失敗 |
| etcd リーダーなし | etcd クラスターのリーダー欠落 | = 0(リーダーなし) |
| API サーバースロットリング | ドロップされた API リクエスト | 0 件を超えるリクエスト |
| Scheduler バックログ | Scheduler 内の Pending Pod | 0 個を超える Pod |
| ノードディスク高使用率 | ノードファイルシステム使用率 | 90% 超 |
| DaemonSet 利用不可 | 誤スケジュールされたノード | 0 個を超えるノード |

## セットアップ要件

Kubernetes モニタリングを使用するには、クラスターに OneUptime Kubernetes エージェントをインストールする必要があります。エージェントは、クラスターメトリクス、イベント、Pod ログ、そしてデフォルトで **eBPF 経由でキャプチャされたアプリケーショントレースおよび HTTP RED メトリクス**を、OTLP で OneUptime に送信します。サービスレベルのトラフィックを確認するために、コード変更やアプリごとの SDK は不要です。

[Kubernetes エージェントのインストール](/docs/monitor/kubernetes-agent) ガイドを参照してください。ワンコマンドの Helm インストール、クラスターに適した構成を選択するための `preset` オプション(standard、GKE Autopilot、EKS Fargate)、および個別のシグナルファミリー(HTTP RED メトリクス、サービスグラフ、ネットワークフロー、TCP 統計)用の `ebpf.features.*` トグルについて説明されています。
