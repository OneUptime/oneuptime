# SCIM（クロスドメイン ID 管理システム）

OneUptime は、自動的なユーザーのプロビジョニングとデプロビジョニングのために SCIM v2.0 プロトコルをサポートしています。SCIM により、Azure AD、Okta などのエンタープライズ ID システムを含むアイデンティティプロバイダー（IdP）が、OneUptime プロジェクトとステータスページへのユーザーアクセスを自動的に管理できます。

## 概要

SCIM 統合は以下のメリットを提供します。

- **自動ユーザープロビジョニング**: IdP でユーザーが割り当てられると、OneUptime に自動的にユーザーを作成します
- **自動ユーザーデプロビジョニング**: IdP でユーザーの割り当てが解除されると、OneUptime からユーザーを自動的に削除します
- **ユーザー属性の同期**: IdP と OneUptime 間でユーザー情報を同期します
- **集中アクセス管理**: 既存の ID 管理システムから OneUptime アクセスを管理します

## プロジェクトの SCIM

プロジェクト SCIM を使用すると、アイデンティティプロバイダーが OneUptime プロジェクト内のチームメンバーを管理できます。

### プロジェクト SCIM のセットアップ

1. **プロジェクト設定に移動する**
   - OneUptime プロジェクトに移動します
   - **プロジェクト設定** > **チーム** > **SCIM** に移動します

2. **SCIM 設定を構成する**
   - **ユーザーの自動プロビジョニング** を有効にして、IdP でユーザーが割り当てられたときに自動的に追加します
   - **ユーザーの自動デプロビジョニング** を有効にして、IdP でユーザーの割り当てが解除されたときに自動的に削除します
   - 新しいユーザーが追加される **デフォルトチーム** を選択します
   - IdP の設定に使用する **SCIM ベース URL** と **Bearer トークン** をコピーします

3. **アイデンティティプロバイダーを設定する**
   - SCIM ベース URL を使用: `https://oneuptime.com/scim/v2/{scimId}`
   - 提供されたトークンで Bearer トークン認証を設定します
   - ユーザー属性をマッピングします（メールアドレスは必須）

### プロジェクト SCIM エンドポイント

- **サービスプロバイダー設定**: `GET /scim/v2/{scimId}/ServiceProviderConfig`
- **スキーマ**: `GET /scim/v2/{scimId}/Schemas`
- **リソースタイプ**: `GET /scim/v2/{scimId}/ResourceTypes`
- **ユーザー一覧**: `GET /scim/v2/{scimId}/Users`
- **ユーザー取得**: `GET /scim/v2/{scimId}/Users/{userId}`
- **ユーザー作成**: `POST /scim/v2/{scimId}/Users`
- **ユーザー更新**: `PUT /scim/v2/{scimId}/Users/{userId}` または `PATCH /scim/v2/{scimId}/Users/{userId}`
- **ユーザー削除**: `DELETE /scim/v2/{scimId}/Users/{userId}`
- **グループ一覧**: `GET /scim/v2/{scimId}/Groups`
- **グループ取得**: `GET /scim/v2/{scimId}/Groups/{groupId}`
- **グループ作成**: `POST /scim/v2/{scimId}/Groups`
- **グループ更新**: `PUT /scim/v2/{scimId}/Groups/{groupId}` または `PATCH /scim/v2/{scimId}/Groups/{groupId}`
- **グループ削除**: `DELETE /scim/v2/{scimId}/Groups/{groupId}`

### プロジェクト SCIM ユーザーライフサイクル

1. **IdP でのユーザー割り当て**: IdP でユーザーが OneUptime に割り当てられます
2. **SCIM プロビジョニング**: IdP が OneUptime SCIM API を呼び出してユーザーを作成します
3. **チームメンバーシップ**: ユーザーが設定されたデフォルトチームに自動的に追加されます
4. **アクセス付与**: ユーザーが OneUptime プロジェクトにアクセスできるようになります
5. **ユーザー割り当て解除**: IdP でユーザーの割り当てが解除されます
6. **SCIM デプロビジョニング**: IdP が OneUptime SCIM API を呼び出してユーザーを削除します
7. **アクセス取り消し**: ユーザーはプロジェクトへのアクセスを失います

## ステータスページの SCIM

ステータスページ SCIM を使用すると、アイデンティティプロバイダーがプライベートステータスページのサブスクライバーを管理できます。

### ステータスページ SCIM のセットアップ

1. **ステータスページ設定に移動する**
   - OneUptime ステータスページに移動します
   - **ステータスページ設定** > **プライベートユーザー** > **SCIM** に移動します

2. **SCIM 設定を構成する**
   - **ユーザーの自動プロビジョニング** を有効にして、IdP でユーザーが割り当てられたときにサブスクライバーを自動的に追加します
   - **ユーザーの自動デプロビジョニング** を有効にして、IdP でユーザーの割り当てが解除されたときにサブスクライバーを自動的に削除します
   - IdP の設定に使用する **SCIM ベース URL** と **Bearer トークン** をコピーします

3. **アイデンティティプロバイダーを設定する**
   - SCIM ベース URL を使用: `https://oneuptime.com/status-page-scim/v2/{scimId}`
   - 提供されたトークンで Bearer トークン認証を設定します
   - ユーザー属性をマッピングします（メールアドレスは必須）

### ステータスページ SCIM エンドポイント

- **サービスプロバイダー設定**: `GET /status-page-scim/v2/{scimId}/ServiceProviderConfig`
- **スキーマ**: `GET /status-page-scim/v2/{scimId}/Schemas`
- **リソースタイプ**: `GET /status-page-scim/v2/{scimId}/ResourceTypes`
- **ユーザー一覧**: `GET /status-page-scim/v2/{scimId}/Users`
- **ユーザー取得**: `GET /status-page-scim/v2/{scimId}/Users/{userId}`
- **ユーザー作成**: `POST /status-page-scim/v2/{scimId}/Users`
- **ユーザー更新**: `PUT /status-page-scim/v2/{scimId}/Users/{userId}` または `PATCH /status-page-scim/v2/{scimId}/Users/{userId}`
- **ユーザー削除**: `DELETE /status-page-scim/v2/{scimId}/Users/{userId}`

### ステータスページ SCIM ユーザーライフサイクル

1. **IdP でのユーザー割り当て**: IdP でユーザーが OneUptime ステータスページに割り当てられます
2. **SCIM プロビジョニング**: IdP が OneUptime SCIM API を呼び出してサブスクライバーを作成します
3. **アクセス付与**: ユーザーがプライベートステータスページにアクセスできるようになります
4. **ユーザー割り当て解除**: IdP でユーザーの割り当てが解除されます
5. **SCIM デプロビジョニング**: IdP が OneUptime SCIM API を呼び出してサブスクライバーを削除します
6. **アクセス取り消し**: ユーザーはステータスページへのアクセスを失います

## アイデンティティプロバイダーの設定

### Microsoft Entra ID（旧 Azure AD）

Microsoft Entra ID は、堅牢な SCIM プロビジョニング機能を備えたエンタープライズグレードの ID 管理を提供します。OneUptime との SCIM プロビジョニングを設定する詳細な手順を以下に示します。

#### 前提条件

- Premium P1 または P2 ライセンスを持つ Microsoft Entra ID テナント（自動プロビジョニングに必要）
- Scale プラン以上の OneUptime アカウント
- Microsoft Entra ID と OneUptime の両方への管理者アクセス

#### ステップ 1: OneUptime から SCIM 設定を取得する

1. OneUptime ダッシュボードにログインします
2. **プロジェクト設定** > **チーム** > **SCIM** に移動します
3. **SCIM 設定を作成** をクリックします
4. わかりやすい名前を入力します（例: "Microsoft Entra ID Provisioning"）
5. 以下のオプションを設定します。
   - **ユーザーの自動プロビジョニング**: 有効にしてユーザーを自動作成します
   - **ユーザーの自動デプロビジョニング**: 有効にしてユーザーを自動削除します
   - **デフォルトチーム**: 新しいユーザーが追加されるチームを選択します
   - **グループのプッシュを有効にする**: Entra ID グループでチームメンバーシップを管理する場合は有効にします
6. 設定を保存します
7. **SCIM ベース URL** と **Bearer トークン** をコピーします - Entra ID の設定に必要です

#### ステップ 2: Microsoft Entra ID でエンタープライズアプリケーションを作成する

1. [Microsoft Entra 管理センター](https://entra.microsoft.com) にサインインします
2. **ID** > **アプリケーション** > **エンタープライズアプリケーション** に移動します
3. **+ 新しいアプリケーション** をクリックします
4. **+ 独自のアプリケーションを作成する** をクリックします
5. 名前を入力します（例: "OneUptime"）
6. **ギャラリーに見つからない他のアプリケーションを統合する（非ギャラリー）** を選択します
7. **作成** をクリックします

#### ステップ 3: SCIM プロビジョニングを設定する

1. OneUptime エンタープライズアプリケーションで **プロビジョニング** に移動します
2. **作業の開始** をクリックします
3. **プロビジョニングモード** を **自動** に設定します
4. **管理者の資格情報** で:
   - **テナント URL**: OneUptime の SCIM ベース URL を入力します（例: `https://oneuptime.com/api/identity/scim/v2/{your-scim-id}`）
   - **シークレットトークン**: OneUptime の Bearer トークンを入力します
5. **接続テスト** をクリックして設定を確認します
6. **保存** をクリックします

#### ステップ 4: 属性マッピングを設定する

1. プロビジョニングセクションで **マッピング** をクリックします
2. **Azure Active Directory ユーザーをプロビジョニングする** をクリックします
3. 以下の属性マッピングを設定します。

| Azure AD 属性 | OneUptime SCIM 属性 | 必須 |
|-------------------|-------------------------|----------|
| `userPrincipalName` | `userName` | はい |
| `mail` | `emails[type eq "work"].value` | 推奨 |
| `displayName` | `displayName` | 推奨 |
| `givenName` | `name.givenName` | 任意 |
| `surname` | `name.familyName` | 任意 |
| `Switch([IsSoftDeleted], , "False", "True", "True", "False")` | `active` | 推奨 |

4. 不要なマッピングを削除してプロビジョニングを簡略化します
5. **保存** をクリックします

#### ステップ 5: グループプロビジョニングを設定する（任意）

OneUptime で **グループのプッシュ** を有効にした場合:

1. **マッピング** に戻ります
2. **Azure Active Directory グループをプロビジョニングする** をクリックします
3. **有効** を **はい** に設定してグループプロビジョニングを有効にします
4. 以下の属性マッピングを設定します。

| Azure AD 属性 | OneUptime SCIM 属性 |
|-------------------|-------------------------|
| `displayName` | `displayName` |
| `members` | `members` |

5. **保存** をクリックします

#### ステップ 6: ユーザーとグループを割り当てる

1. OneUptime エンタープライズアプリケーションで **ユーザーとグループ** に移動します
2. **+ ユーザー/グループの追加** をクリックします
3. OneUptime にプロビジョニングするユーザーやグループを選択します
4. **割り当て** をクリックします

#### ステップ 7: プロビジョニングを開始する

1. **プロビジョニング** > **概要** に移動します
2. **プロビジョニングの開始** をクリックします
3. 最初のプロビジョニングサイクルが始まります（初回同期は最大 40 分かかる場合があります）
4. エラーがないか **プロビジョニングログ** を監視します

#### Microsoft Entra ID のトラブルシューティング

- **接続テストが失敗する**: SCIM ベース URL に `/api/identity` プレフィックスが含まれており、Bearer トークンが正しいことを確認します
- **ユーザーがプロビジョニングされない**: ユーザーがアプリケーションに割り当てられており、属性マッピングが正しいことを確認します
- **プロビジョニングエラー**: Entra ID のプロビジョニングログで具体的なエラーメッセージを確認します
- **同期の遅延**: 初回プロビジョニングは最大 40 分かかる場合があります。その後の同期は 40 分ごとに実行されます

### Okta

Okta は優れた SCIM サポートを備えた柔軟な ID 管理を提供します。OneUptime との SCIM プロビジョニングを設定する詳細な手順を以下に示します。

#### 前提条件

- プロビジョニング機能を持つ Okta テナント（Lifecycle Management 機能）
- Scale プラン以上の OneUptime アカウント
- Okta と OneUptime の両方への管理者アクセス

#### ステップ 1: OneUptime から SCIM 設定を取得する

1. OneUptime ダッシュボードにログインします
2. **プロジェクト設定** > **チーム** > **SCIM** に移動します
3. **SCIM 設定を作成** をクリックします
4. わかりやすい名前を入力します（例: "Okta Provisioning"）
5. 以下のオプションを設定します。
   - **ユーザーの自動プロビジョニング**: 有効にしてユーザーを自動作成します
   - **ユーザーの自動デプロビジョニング**: 有効にしてユーザーを自動削除します
   - **デフォルトチーム**: 新しいユーザーが追加されるチームを選択します
   - **グループのプッシュを有効にする**: Okta グループでチームメンバーシップを管理する場合は有効にします
6. 設定を保存します
7. **SCIM ベース URL** と **Bearer トークン** をコピーします - Okta の設定に必要です

#### ステップ 2: Okta アプリケーションを作成または設定する

**既存の SSO アプリケーションがある場合:**
1. Okta 管理コンソールにサインインします
2. **アプリケーション** > **アプリケーション** に移動します
3. 既存の OneUptime アプリケーションを見つけて選択します

**新しいアプリケーションを作成する場合:**
1. Okta 管理コンソールにサインインします
2. **アプリケーション** > **アプリケーション** に移動します
3. **アプリ統合を作成** をクリックします
4. **SAML 2.0** を選択して **次へ** をクリックします
5. アプリ名に "OneUptime" を入力します
6. SAML 設定を完了します（SSO ドキュメントを参照）
7. **完了** をクリックします

#### ステップ 3: SCIM プロビジョニングを有効にする

1. OneUptime アプリケーションで **全般** タブに移動します
2. **アプリ設定** セクションで **編集** をクリックします
3. **プロビジョニング** で **SCIM** を選択します
4. **保存** をクリックします
5. 新しい **プロビジョニング** タブが表示されます

#### ステップ 4: SCIM 接続を設定する

1. **プロビジョニング** タブに移動します
2. 左サイドバーの **統合** をクリックします
3. **API 統合を設定** をクリックします
4. **API 統合を有効にする** をチェックします
5. 以下を設定します。
   - **SCIM コネクタのベース URL**: OneUptime の SCIM ベース URL を入力します（例: `https://oneuptime.com/api/identity/scim/v2/{your-scim-id}`）
   - **ユーザーの一意識別子フィールド**: `userName` を入力します
   - **サポートされるプロビジョニングアクション**: 有効にするアクションを選択します:
     - 新しいユーザーとプロファイル更新をインポート
     - 新しいユーザーをプッシュ
     - プロファイルの更新をプッシュ
     - グループをプッシュ（グループベースのプロビジョニングを使用する場合）
   - **認証モード**: **HTTP ヘッダー** を選択します
   - **承認**: `Bearer {your-bearer-token}` を入力します（実際のトークンに置き換えてください）
6. **API 認証情報をテスト** をクリックして接続を確認します
7. **保存** をクリックします

#### ステップ 5: アプリへのプロビジョニングを設定する

1. **プロビジョニング** タブで左サイドバーの **アプリへ** をクリックします
2. **編集** をクリックします
3. 以下のオプションを有効にします。
   - **ユーザーを作成**: 有効にして新しいユーザーをプロビジョニングします
   - **ユーザー属性を更新**: 有効にして属性の変更を同期します
   - **ユーザーを無効化**: 有効にして割り当て解除時にユーザーをデプロビジョニングします
4. **保存** をクリックします

#### ステップ 6: 属性マッピングを設定する

1. **属性マッピング** までスクロールします
2. 以下のマッピングを確認または設定します。

| Okta 属性 | OneUptime SCIM 属性 | 方向 |
|---------------|-------------------------|-----------|
| `userName` | `userName` | Okta からアプリへ |
| `user.email` | `emails[primary eq true].value` | Okta からアプリへ |
| `user.firstName` | `name.givenName` | Okta からアプリへ |
| `user.lastName` | `name.familyName` | Okta からアプリへ |
| `user.displayName` | `displayName` | Okta からアプリへ |

3. 不要なマッピングを削除します
4. 変更した場合は **保存** をクリックします

#### ステップ 7: グループのプッシュを設定する（任意）

OneUptime で **グループのプッシュ** を有効にした場合:

1. **グループのプッシュ** タブに移動します
2. **+ グループをプッシュ** をクリックします
3. **名前でグループを検索** または **ルールでグループを検索** を選択します
4. プッシュするグループを検索して選択します
5. **保存** をクリックします

#### ステップ 8: ユーザーを割り当てる

1. **割り当て** タブに移動します
2. **割り当て** > **ユーザーに割り当て** または **グループに割り当て** をクリックします
3. プロビジョニングするユーザーまたはグループを選択します
4. 各選択で **割り当て** をクリックします
5. **完了** をクリックします

#### ステップ 9: プロビジョニングを確認する

1. Okta 管理コンソールの **レポート** > **システムログ** に移動します
2. OneUptime アプリケーションに関連するイベントをフィルタリングします
3. プロビジョニングイベントが成功していることを確認します
4. OneUptime でユーザーが作成されたことを確認します

#### Okta のトラブルシューティング

- **API 認証情報テストが失敗する**: SCIM ベース URL と Bearer トークンが正しいことを確認します
- **ユーザーがプロビジョニングされない**: ユーザーがアプリケーションに割り当てられており、プロビジョニングが有効になっていることを確認します
- **重複ユーザー**: `userName` 属性が一意であり、メールアドレスに正しくマッピングされていることを確認します
- **グループプッシュの失敗**: グループが存在し、正しいメンバーシップを持っていることを確認します
- **エラー: 401 Unauthorized**: OneUptime で Bearer トークンを再生成し、Okta を更新します

### その他のアイデンティティプロバイダー

OneUptime の SCIM 実装は SCIM v2.0 仕様に準拠しており、任意の準拠したアイデンティティプロバイダーで動作します。一般的な設定手順は以下の通りです。

1. **SCIM ベース URL**: `https://oneuptime.com/api/identity/scim/v2/{scim-id}`（プロジェクト用）または `https://oneuptime.com/api/identity/status-page-scim/v2/{scim-id}`（ステータスページ用）
2. **認証**: HTTP Bearer トークン
3. **必須ユーザー属性**: `userName`（有効なメールアドレスである必要があります）
4. **サポートされる操作**: ユーザーとグループの GET、POST、PUT、PATCH、DELETE

#### サポートされている SCIM エンドポイント

| エンドポイント | メソッド | 説明 |
|----------|---------|-------------|
| `/ServiceProviderConfig` | GET | SCIM サーバーの機能 |
| `/Schemas` | GET | 利用可能なリソーススキーマ |
| `/ResourceTypes` | GET | 利用可能なリソースタイプ |
| `/Users` | GET、POST | ユーザーの一覧表示と作成 |
| `/Users/{id}` | GET、PUT、PATCH、DELETE | 個々のユーザーの管理 |
| `/Groups` | GET、POST | グループ/チームの一覧表示と作成（プロジェクト SCIM のみ） |
| `/Groups/{id}` | GET、PUT、PATCH、DELETE | 個々のグループの管理（プロジェクト SCIM のみ） |

#### SCIM ユーザースキーマ

```json
{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "userName": "user@example.com",
  "name": {
    "givenName": "John",
    "familyName": "Doe",
    "formatted": "John Doe"
  },
  "displayName": "John Doe",
  "emails": [
    {
      "value": "user@example.com",
      "type": "work",
      "primary": true
    }
  ],
  "active": true
}
```

#### SCIM グループスキーマ

```json
{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
  "displayName": "Engineering Team",
  "members": [
    {
      "value": "user-id-here",
      "display": "user@example.com"
    }
  ]
}
```

## よくある質問

### ユーザーがデプロビジョニングされるとどうなりますか？

ユーザーがデプロビジョニングされると（DELETE リクエストまたは `active: false` の設定により）、SCIM 設定で構成されたチームから削除されます。ユーザーアカウント自体は OneUptime に残りますが、プロジェクトへのアクセスを失います。

### SSO なしで SCIM を使用できますか？

はい、SCIM と SSO は独立した機能です。ユーザーが OneUptime パスワードやその他の認証方法でログインできるようにしながら、SCIM をユーザープロビジョニングに使用できます。

### OneUptime に既存するユーザーをどのように扱いますか？

SCIM が既存のユーザー（メールアドレスで照合）を作成しようとした場合、OneUptime は重複ユーザーを作成するのではなく、設定されたデフォルトチームにそのユーザーを追加するだけです。

### デフォルトチームとグループのプッシュの違いは何ですか？

- **デフォルトチーム**: SCIM 経由でプロビジョニングされたすべてのユーザーが同じ事前定義されたチームに追加されます
- **グループのプッシュ**: チームメンバーシップはアイデンティティプロバイダーによって管理され、IdP グループのメンバーシップに基づいてユーザーが異なるチームに所属できます

### プロビジョニングの同期はどのくらいの頻度で実行されますか？

これはアイデンティティプロバイダーによって異なります。
- **Microsoft Entra ID**: 初回同期は最大 40 分かかる場合があります。その後の同期は 40 分ごとに実行されます
- **Okta**: ほとんどの操作でほぼリアルタイム、定期的な完全同期も実行されます
