# ワークフロー 実行とログ

ワークフローのトリガーが発火するたびに、OneUptime は**実行**を作成します — タイミング、ステータス、ノードごとの出力を持つ 1 回の実行の記録です。実行はワークフローが動いたことを確認する方法であり、動かなかったときにデバッグする方法であり、自動化が誤動作したときのポストモーテムを書く方法です。

## 見つける場所

| ページ | スコープ |
| --- | --- |
| **Workflows → Runs & Logs** | プロジェクト全体。すべてのワークフローのすべての実行。ワークフロー、ステータス、時間範囲でフィルタ可能。 |
| ワークフローの **Logs** タブ | このワークフローの実行だけ。 |
| 実行の詳細ページ | 1 回の実行。ノードごとの出力やエラーメッセージまで展開して表示。 |

## 実行ステータス

| ステータス | 意味 |
| --- | --- |
| **Scheduled** | トリガーが発火し実行がキューに入ったが、ワーカーがまだ拾っていない。通常は 1 秒未満。 |
| **Running** | ワーカーが現在グラフを辿っている。長時間実行のコンポーネント (遅い HTTP コール、意図的な遅延) はこの状態を保ちます。 |
| **Success** | 動いたすべてのノードがエラーなしで終了。(意図的に `error` ブランチを取ったワークフローでも全体としては `Success` です — ワークフロー自体が失敗したわけではないため。) |
| **Error** | ノードが失敗し、それを処理する `error` ポートが配線されていなかった。実行はそのノードで停止しました。 |
| **Timeout** | 実行が実行ごとのタイムアウトを超えた。[設定と安全性](/docs/workflows/configuration) を参照。 |

## 実行を読む

リストから実行をクリックすると詳細ページが開きます。表示される内容:

- **ヘッダー** — 発火したトリガー、開始と終了のタイムスタンプ、合計所要時間、ステータス。
- **ノードリスト** — 順に実行されたすべてのノード、それぞれのキャプチャされた引数、戻り値、選ばれた出力ポート。
- **エラー** — ノードが失敗した場合、エラーメッセージと (利用可能なら) スタックトレース。

キャプチャされた引数は*補間後*の値を表示します — つまり変数解決後にノードが実際に見た文字列そのものです。これは最も役立つデバッグビューです: Slack メッセージにリテラルのテキスト `{{Incident.title}}` が入っていれば、変数参照が解決できなかったと分かります。

## よくあるデバッグパターン

### 「ワークフローが発火しなかった。」

1. ワークフローが **Settings** で**有効化**されていることを確認。新しいワークフローは無効状態で出荷されます。
2. モデルイベントトリガーの場合: イベントが実際に起きたことを確認。エンティティ (インシデント、アラート、モニター) を開いてその履歴を見ます。
3. Webhook トリガーの場合: 外部システムが正しい URL を叩いていることを確認。多くのツールはアウトバウンド Webhook の配信をログに残します — そこをチェックします。
4. Schedule トリガーの場合: cron 式が期待する時刻に評価されることを確認。疑わしければ cron パーサーを使ってください。

トリガーは発火したのに実行が出てこない場合、**Project Settings → Billing** でプロジェクトの実行クォータをチェックしてください。

### 「動いているのに下流のノードが実行されない。」

実行されないノードは通常、配線の問題です。キャンバスを開いて確認:

- 上流ノードの出力ポートは本当にこのノードの入力ポートにつながっていますか?
- 上流ノードが別のポートを取った可能性は (例: `success` ではなく `error`、`yes` ではなく `no`)? 実行詳細でどのポートが選ばれたか見てください。

### 「変数が空になって流れてくる。」

実行詳細を開き、失敗しているノードのキャプチャされた引数を見ます。リテラルの `{{NodeId.field}}` テキストが見えれば、参照が解決できていません — おそらく `NodeId` か `field` のタイポです。空文字列が見えれば、上流ノードは動いたがそのフィールドを生成しなかったということです。

### 「手動なら動くがトリガーからは動かない。」

実際のトリガーが公開する内容をミラーする JSON ペイロードで **Run Manually** を使います。それから手動実行と本番実行のキャプチャされた引数を並べて比較してください — 違いは通常、1 つのフィールド名や型にあります。

## ワークフローを再実行する

「この実行をリトライ」ボタンはありません — 設計上、OneUptime は古い実行を再実行しません。アウトバウンドの副作用 (Slack メッセージ、API コール) が冪等でないかもしれないからです。やり直したければ、ワークフローを修正して次の本物のトリガーで発火させてください。

手動ワークフローの場合は、同じペイロードで **Run Manually** をもう一度クリックするだけです。

## ログ保持

実行はプロジェクトに無期限に保持されます。大量のノイジーなワークフロー (例えば毎分発火するデバッグワークフロー) をクリーンアップする必要があれば、無効化または削除してください — ワークフローごとの保持トグルはありません。
