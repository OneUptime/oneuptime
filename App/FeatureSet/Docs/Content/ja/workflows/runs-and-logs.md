# 実行とログ

ワークフローが実行されるたびに、OneUptime は実行内容の記録 — 実行時刻、成功したかどうか、各ブロックが何をしたか — を保存します。この記録を **実行 (run)** と呼びます。実行は、ワークフローが動いたことを確認し、動かなかったときにデバッグし、過去のアクティビティを振り返るための手段です。

## 見つける場所

| ページ                      | 表示内容                                                                                             |
| --------------------------- | ---------------------------------------------------------------------------------------------------- |
| **Workflows → Runs & Logs** | プロジェクト内のすべてのワークフローのすべての実行。ワークフロー、ステータス、時間でフィルター可能。 |
| **Workflow → Logs タブ**    | このワークフローの実行のみ。                                                                         |
| **個別の実行**              | 1 回の実行と、すべてのブロックの出力。                                                               |

## 実行ステータス

| ステータス    | 意味                                                                                                                                                           |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scheduled** | トリガーが発火し、実行が開始されようとしています。通常はほんの一瞬です。                                                                                       |
| **Running**   | ワークフローが進行中です。長く動くブロックがあると、この状態のままになります。                                                                                 |
| **Success**   | 実行されたすべてのブロックがエラーなく完了しました。(意図的に **error** ブランチを通った場合も成功とみなされます — ワークフロー自体は失敗していないからです。) |
| **Error**     | ブロックが失敗し、それを処理する **error** パスがつながっていませんでした。実行はそこで停止しました。                                                          |
| **Timeout**   | 実行が許可された時間を超えました。[構成と安全性](/docs/workflows/configuration) を参照してください。                                                           |

## 実行内容の読み方

任意の実行をクリックすると詳細が開きます。以下が表示されます:

- **ヘッダー** — トリガー、開始・終了時刻、合計時間、ステータス。
- **ブロック一覧** — 実行されたすべてのブロックを順番に表示。それぞれに、与えられた値、出力、通ったパスが示されます。
- **エラー** — ブロックが失敗した場合のエラーメッセージと、利用可能な場合は詳細情報。

表示される値は、ブロックが実際に見たもの — すべての変数が解決された後の値 — です。これは最も役立つデバッグビューです: もし Slack メッセージに実際のタイトルではなくリテラルの `{{Incident.title}}` と表示されていれば、変数が解決されなかったことがわかります。

## よくあるデバッグ

### 「ワークフローが実行されなかった」

1. Settings でワークフローが **Enabled** になっているか確認します。新しいワークフローは無効状態で始まります。
2. OneUptime のイベントトリガーの場合: そのイベントが本当に発生したか確認します。レコードを開いて履歴をチェックします。
3. Webhook トリガーの場合: 他のシステムが正しい URL に送信しているか確認します。ほとんどのツールは Webhook 送信時にログを残すので、そこを確認します。
4. スケジュールトリガーの場合: cron 式が想定した時刻と一致しているか確認します。

トリガーは発火したのに実行が表示されない場合は、**Project Settings → Billing** で実行クォータを確認してください。

### 「後続のブロックが実行されなかった」

実行されないブロックは、たいてい配線の問題です。キャンバスを開いて確認します:

- 前のブロックの出力が、このブロックの入力に接続されているか?
- 前のブロックが期待と違う出力 (たとえば **success** ではなく **error**、**Yes** ではなく **No**) を取ったのではないか? 実行詳細にどのパスが通ったかが表示されます。

### 「変数が空で来た」

実行を開いて、失敗したブロックの値を見てください。

- リテラルの `{{BlockName.field}}` テキストが見える場合、参照が解決されていません — おそらくブロック名やフィールド名のタイプミスです。
- 空文字列が見える場合、前のブロックは実行されたものの、そのフィールドを生成しなかったということです。

### 「手動で実行するとうまく動くが、トリガーからは動かない」

実際のトリガーが送るのと同じような JSON ペイロードで **Run Manually** を使います。そして、手動実行と実際の実行の値を並べて比較します。違いはたいてい 1 つのフィールド名や型です。

## ワークフローの再実行

「この実行を再試行する」ボタンはありません。古い実行を自動的に再実行することはしません — 副作用 (Slack メッセージ、API 呼び出し、チケット) を繰り返すのは安全ではないことがあるためです。やり直したい場合は、ワークフローを修正して、次の本物のトリガーで発火させます。

手動ワークフローの場合は、同じペイロードで **Run Manually** をクリックするだけです。

## 実行はどのくらい保持される?

実行はプロジェクトに対して無期限に保持されます。ワークフローが非常に頻繁に実行されて履歴が散らかる (毎分発火するデバッグワークフローなど) 場合は、それを無効化または削除してノイズへの追加を止めてください。
