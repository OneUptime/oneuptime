# ワークフロー 概要

ワークフローを使えば、コードを書かずに OneUptime のタスクを自動化できます。いくつかのブロックをキャンバスにドラッグ&ドロップしてつなぎ合わせるだけで、何かが起きたとき — インシデントが発生したとき、スケジュールが発火したとき、または別のツールが OneUptime にデータを送ってきたとき — に動く自動化のできあがりです。

ワークフローはプロジェクトのバックグラウンドヘルパーのようなものだと考えてください。イベントに反応し、他のツールとやり取りし、あなたが本来の業務に集中している間、静かにいろいろなものを同期し続けてくれます。

## ワークフローでできること

- **OneUptime を他のツールと連携する** — インシデントを Slack に送信、Jira チケットを作成、自社スタックの Webhook に投稿。
- **OneUptime で起きたことに反応する** — クリティカルなインシデントが作成されたら、オンコールチームに通知してチケットを自動で作成。
- **ジョブをスケジュール実行する** — 5 分ごと、毎晩、毎週月曜日の朝など。
- **外部からデータを受け取る** — 他のシステムが固有の URL を通じて OneUptime にデータをプッシュできるようにする。
- **共通の自動化を再利用する** — 一度作って、別のワークフローから呼び出す。

## ワークフローの仕組み

すべてのワークフローには 3 つの要素があります。

1. **トリガー** — ワークフローを開始するもの。手動ボタン、スケジュール、受信 Webhook、OneUptime のイベント (新しいインシデントなど) のいずれかです。
2. **1 つ以上のコンポーネント** — ワークフローが行うこと。メッセージを送る、HTTP 呼び出しを行う、簡単なチェックを実行する、条件で分岐する、など。
3. **それらの間の接続** — あるブロックから次のブロックへ線を引いて、実行順序を決めます。

これらをすべてキャンバス上で視覚的に組み立てます。ほとんどのワークフローはコーディング不要ですが、必要に応じて JavaScript スニペットを差し込むこともできます。

## キー概念

| 用語               | 意味                                                                                                                  |
| ------------------ | --------------------------------------------------------------------------------------------------------------------- |
| **ワークフロー**   | 自動化全体 — 名前、キャンバス、オン / オフのスイッチを持つもの。                                                      |
| **トリガー**       | 最初のブロック。ワークフローがいつ実行されるかを決定します。すべてのワークフローはちょうど 1 つのトリガーを持ちます。 |
| **コンポーネント** | アクションブロック — メッセージを送ったり、リクエストを行ったり、条件をチェックしたりします。                         |
| **実行**           | ワークフローの 1 回の実行。タイムスタンプとすべてのブロックの出力とともに保存されます。                               |
| **グローバル変数** | 一度保存して、どのワークフローからでも再利用できる値 (API キーなど)。                                                 |

## OneUptime でワークフローを見つける場所

左ナビゲーションの **Workflows** を開きます。そこから:

- **Workflows** — ワークフローのリスト。新規作成、または既存のワークフローを開きます。
- **Builder タブ** — ワークフローを設計するキャンバス。
- **Logs タブ** — このワークフローのすべての実行と詳細。
- **Settings タブ** — 名前、説明、オーナー、ラベル、有効化 / 無効化。
- **Global Variables** — すべてのワークフローで共有される値。
- **Runs & Logs** — プロジェクト内のすべてのワークフローの実行履歴。

## 最初のワークフローを作る

1. **作成** — ワークフローに名前と簡単な説明をつけます。
2. **トリガーを選ぶ** — 手動、スケジュール、Webhook、または OneUptime のイベント。
3. **コンポーネントを追加** — アクションをキャンバスにドラッグして接続します。
4. **テスト** — **Run Manually** をクリックし、ログで何が起きるかを確認します。
5. **オンにする** — 準備ができたら、Settings の **Enabled** スイッチを切り替えます。

## 簡単な例

クリティカルなインシデントが作成されるたびに Slack に投稿したい場合:

1. 「Critical incidents to Slack」という名前のワークフローを作成します。
2. **Incident → On Create** トリガーを選びます。
3. **Conditions** ブロックを追加します。インシデントのタイトルに「Sev 1」が含まれているかをチェックするように設定します。
4. **Yes** ブランチから **Slack** ブロックを追加します。チャンネルを選んでメッセージを書きます。
5. ワークフローをオンにします。

次に誰かがタイトルに「Sev 1」を含むインシデントを開くと、Slack が反応します。

## ワークフローと OneUptime の他の機能との関係

- **モニター** が問題を発見します。**インシデント** がそれを記録します。**ワークフロー** がそれに反応します。
- **Runbook** は人のためのステップバイステップのガイドです。ワークフローは無人で動く自動化です。人が判断する必要がある場合は Runbook を、ステップが自動の場合はワークフローを使います。
- **ワークスペース接続** (Slack、Teams) は、ワークフローがメッセージを送る先です。
