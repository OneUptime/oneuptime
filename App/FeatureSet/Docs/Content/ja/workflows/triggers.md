# ワークフロー トリガー

トリガーはワークフローの開始ノードです。入力ポートはありません — そこから実行が始まります。OneUptime は 4 つのトリガーファミリーをサポートし、各ワークフローは必ず 1 つを使います。

## Manual

ワークフローページで **Run Manually** をクリックして、オンデマンドでワークフローを実行します。任意の JSON ペイロードを貼り付けて、ワークフローから `{{Manual.JSON}}` として読むことができます。

これを使うのは、自動化の片鱗を起動するボタンが欲しいときです — 定期スケジュールやイベントを必要としない、「オンコールキーをローテートする」「検索インデックスを再構築する」といったワンクリックワークフローに向いています。

**引数**: なし。

**戻り値**:

| 名前 | 型 | 説明 |
| --- | --- | --- |
| `JSON` | JSON | 実行時に渡された JSON ペイロード、または空オブジェクト。 |

## Schedule

cron スケジュールでワークフローを実行します。標準的な cron 式で頻度を設定します。

定期ジョブに使います: 夜間クリーンアップ、毎時同期、週次エクスポート。

**引数**:

| 名前 | 型 | 説明 |
| --- | --- | --- |
| `Schedule at` | CronTab | 標準的な 5 フィールド cron 式。たとえば `0 * * * *` は毎時 0 分に、`*/5 * * * *` は 5 分ごとに実行します。 |

**戻り値**:

| 名前 | 型 | 説明 |
| --- | --- | --- |
| `executedAt` | Date | スケジュールされた実行時刻。 |

スケジュールされたワークフローはプロジェクトのリージョンの Workflow Worker で実行されます。ワーカーが一時的に利用できない場合、復旧時に実行がディスパッチされます — 短時間の停止であれば、tick の取りこぼしを心配する必要はありません。

## Webhook

外部システムが `POST` する固有の HTTPS URL を公開します。リクエストのヘッダー、クエリパラメータ、ボディは下流のコンポーネントが読めるように戻り値として公開されます。

これを使うと、サードパーティシステムから OneUptime にデータを*取り込めます*: CI/CD のコールバック、別の監視ツールからのアラート、CRM の顧客サインアップなど。

**引数**: なし。URL はワークフローが保存されたときに自動的に割り当てられ、トリガーノードに表示されます。シークレットのように扱ってください — URL を知っている人なら誰でもワークフローをトリガーできます。

**戻り値**:

| 名前 | 型 | 説明 |
| --- | --- | --- |
| `Request Headers` | JSON | 受信した HTTP リクエストのすべてのヘッダー。 |
| `Request Query Params` | JSON | パースされたクエリ文字列。 |
| `Request Body` | JSON | パースされたリクエストボディ。ボディが有効な JSON でない場合、`raw` キーの下に文字列として届きます。 |

Webhook は `GET` と `POST` を受け入れます。呼び出し元へのレスポンスは、実行がエンキューされ次第すぐに `200 OK` と JSON の確認応答で返ります — ワークフロー自体は非同期で動くので、HTTP レスポンスで下流コンポーネントの結果が読めるとは期待しないでください。

## モデルイベントトリガー

OneUptime のほぼすべてのエンティティ — モニター、インシデント、アラート、予定メンテナンスイベント、ステータスページ、オンコールポリシー、チーム、テレメトリーサービス、その他多数 — は 3 つのトリガーを公開します:

- **On Create** — このタイプの新しいレコードが作成されたときに発火。
- **On Update** — 既存のレコードが変更されたときに発火。トリガーは古い値と新しい値の両方を公開します。
- **On Delete** — レコードが削除されたときに発火。

これがポーリングなしで「OneUptime で X が起きたら、Y をする」という自動化を構築する方法です。

モデル自体は、リソース上で見ているのと同じフィールド名で戻り値として公開されます。たとえば **Incident → On Create** トリガーは完全な `Incident` オブジェクトを返すので、下流のノードは `{{Incident.title}}`、`{{Incident.description}}`、`{{Incident.incidentSeverityId}}` などを読めます。

**引数**: 通常は create/delete には何もありません。Update トリガーは反応したいフィールドを絞り込めるので、見た目だけの変更で発火させずに済みます。

**戻り値** (モデルによる):

| 名前 | 型 | 説明 |
| --- | --- | --- |
| モデルフィールド | (種類による) | エンティティのすべてのカラム — 名前、ステータス、タイムスタンプ、外部キー。 |
| `previous` (Update のみ) | JSON | 変更前のレコード。 |

### よく使われるモデルトリガー

チームが最もよく手を伸ばすモデルイベントの非網羅的なリスト:

- **Incident** — `On Create`、`On Update` (Acknowledged や Resolved のような状態変化に反応する)、`On Delete`。
- **Alert** — アラートモデル上の同じ 3 イベント。
- **Monitor** — モニターが追加・編集・削除されたときに反応。条件と組み合わせて本番モニターだけに作用させる。
- **Scheduled Maintenance** — メンテナンスウィンドウが作成されたり状態が変わったりしたときに下流アナウンスを自動化。
- **Status Page Subscriber** — 誰かが購読したらウェルカムフローを発火。
- **On-Call Duty Policy** — スケジュール変更を外部の名簿に同期。

OneUptime API でモデルが公開されていれば、ほぼ確実にワークフローをトリガーできます — エンティティ名でトリガーパレットを検索してください。

## 適切なトリガーを選ぶ

| やりたいこと | 使うトリガー |
| --- | --- |
| ワークフロー上に誰かがクリックするボタンを作る | **Manual** |
| N 分/時/日ごとにジョブを実行する | **Schedule** |
| 外部システムから OneUptime にデータをプッシュさせる | **Webhook** |
| OneUptime *内部*で起きたことに反応する | **モデルイベント** |

ワークフローは 1 つのトリガーしか持てません。2 つの異なる開始シグナルでほとんど同じロジックを共有させたい場合は、共有ステップを 1 つのワークフローに切り出し、**Execute Workflow** コンポーネントを使って 2 つの薄い「ラッパー」ワークフローから呼び出します ([コンポーネント](/docs/workflows/components) を参照)。
