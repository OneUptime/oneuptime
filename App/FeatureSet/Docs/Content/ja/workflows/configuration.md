# ワークフロー 設定と安全性

このページは、ワークフローを本番トラフィックに向ける前に知っておくべき設定と安全性の制限をまとめています。

## 有効化 / 無効化

すべてのワークフローは **Settings** に **isEnabled** フラグを持ちます。無効なワークフローは発火しません — モデルイベント、Webhook、スケジュールされた実行はすべて無視されます。新しいワークフローは無効状態で出荷されます。

これを「本番投入準備完了」スイッチとして扱ってください:

1. ワークフローを構築。
2. 代表的なペイロードで **Run Manually** をクリック。
3. **Logs** をチェック — すべてのノードが期待したポートを取ったか確認。
4. **isEnabled** をオン。

ワークフローを無効化しても進行中の実行には影響しません。新しい実行が作成されなくなるだけです。

## 所有者とラベル

- **所有者** — 所有者として一覧化されたユーザーとチームは権限ベースのアクセスを受け、(任意で) ワークフローが失敗したときに通知を受けます。**Settings → Owners** で設定します。
- **ラベル** — ワークフローを整理するための多対多のタグ。ワークフローリストをラベルでフィルタできます。プロジェクトがチームや連携や環境ごとに何十ものワークフローを抱えるときに便利です。
- **ラベルルール** — **Workflows → Settings → Label Rules** で、名前または説明への正規表現マッチに基づいて新しいワークフローに自動的にラベルを付けます。
- **所有者ルール** — **Workflows → Settings → Owner Rules** で、新しいワークフローに所有者を自動割り当てします。

## シークレット

グローバル変数は**シークレット**としてマークできます。値は保存時に暗号化され、保存後の UI では書き込み専用となり、実行ログから秘匿されます (`[REDACTED]` に置換)。

シークレット変数を使うのは:

- アウトバウンド連携の API キー。
- ベアラートークン。
- Webhook 署名キー。
- ワークフローへの読み取りアクセスを持つ攻撃者が見るべきでない任意の値。

シークレットをコンポーネントの引数に直接貼り付けないでください — `Authorization: Bearer eyJh...` のような参照はワークフロー JSON と実行ログに平文で現れます。代わりに `{{variable.MY_SECRET}}` を参照してください。

## 実行タイムアウト

各実行には最大所要時間があります。実行がタイムアウト内に終わらなければ `Timeout` とマークされ、進行中のコンポーネントはキャンセルされます。デフォルトは寛大 (秒ではなく分単位) です — インストール環境の正確な値はワーカーの環境設定を参照してください。

ほとんどのコンポーネントは実行タイムアウトの内側に独自の呼び出しごとのタイムアウトを持ちます — 例えば API コンポーネントは、実行全体が終わるよりずっと前にハングしたアウトバウンドリクエストを諦めます。

## 再帰の上限

**Execute Workflow** コンポーネントは 1 つのワークフローから別のワークフローを呼べます。A が B を呼び B が A を呼ぶといった無限ループを防ぐため、ワーカーは呼び出しチェーンを追跡し、固定の深さ (通常は 5 のような小さな数) を超えるチェーンを止めます。終了する実行は、再帰の上限についての明確なメッセージとともに `Error` とマークされます。

長いチェーンに正当な必要性があれば (例えば、レベルごとに 1 実行で処理する再帰的なフォルダウォーク)、**Custom Code** で内部的に反復する単一のワークフローにリファクタしてください — そのパターンはチェーンの上限の対象外です。

## Webhook のセキュリティ

Webhook トリガーは固有の HTTPS URL を公開します。URL を知った人は誰でも叩けます。意図しない呼び出しや悪意ある呼び出しを防ぐには:

- URL を共有シークレットとして扱う。公開チャットに貼ったり、公開リポジトリにコミットしないこと。
- 高価値のワークフローでは、呼び出し側システムに共有シークレットをヘッダーとして含めるよう要求し (例: `X-Webhook-Token`)、破壊的な処理を行う前に **Conditions** ノードで検証します。期待するトークンはシークレットグローバル変数として定義してください。
- 非常に高価値のワークフローでは、公開 Webhook ではなくモデルイベントトリガーと手動インポートステップを使ってください。

## アウトバウンドネットワーク

API やその他の HTTP 系コンポーネントは OneUptime Workflow Worker のネットワークからリクエストを送信します。OneUptime をセルフホストしている場合、ワーカーのアウトバウンドネットワークはお客様の管轄です — 呼び出すサードパーティ API に到達できることを確認してください。OneUptime Cloud を使っている場合、IP の送信元レンジは [IP アドレス](/docs/configuration/ip-addresses) で公開されているので、受信側でアローリストに登録できます。

## 権限

ワークフローはプロジェクトレベルのロールベースアクセス制御に従う第一級リソースです:

- `CreateWorkflow`、`ReadWorkflow`、`EditWorkflow`、`DeleteWorkflow` — ワークフローテンプレート上の 4 つの CRUD 権限。
- `RunWorkflow` — **Run Manually** をクリックしたり API でワークフローをディスパッチするのに必要。
- `ReadWorkflowLog` — **Runs & Logs** ページを表示するのに必要。
- `ReadWorkflowVariable`、`CreateWorkflowVariable`、`EditWorkflowVariable`、`DeleteWorkflowVariable` — グローバル変数リストの制御。

ほとんどのエンジニアはワークフローの create/edit/read を持つべきですが、変数についてはそうではありません。変数の編集アクセスはプロジェクトのシークレットを管理する人だけに留保してください。

## クォータ

OneUptime Cloud では、小さなプランではプロジェクトあたり月間の実行数に上限があります。上限は **Project Settings → Billing** に表示されます。上限に達すると、新しいトリガーは拒否され (影響を受けるワークフローには「クォータ超過」の理由で記録されます)、次の課金サイクルまで続きます。セルフホストのインストールはクォータの対象外です。

## ワークフローが*得意でない*こと

別のツールに手を伸ばすべきパターンをいくつか:

- **長時間の計算** — ワークフローはシステム間の糊付けに向いており、大きなデータセットの処理には向きません。重い処理は自社インフラで実行し、ワークフローはそれを起動するのに使ってください。
- **数分・数時間にまたがる状態を持つワークフロー** — 単一の実行は素早く終わるよう意図されています。「A をして、2 時間待ち、B をする」が必要なら、待ちを Webhook トリガーにポストバックする外部スケジューラとしてモデル化してください。
- **人のチェックポイントを含むステップバイステップのインシデント対応** — それは [Runbook](/docs/runbooks/index) の役目です。人が介在しないならワークフロー、介在するなら Runbook を使ってください。
