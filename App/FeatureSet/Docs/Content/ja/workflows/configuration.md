# 構成と安全性

このページでは、ワークフローを本番のトラフィックに向ける前に知っておくべき設定と安全性の制限についてまとめます。

## ワークフローのオン / オフ

すべてのワークフローには **Settings** に **Enabled** スイッチがあります。オフのときはワークフローは実行されず、Webhook 呼び出し、スケジュール時刻、OneUptime のイベントはすべて無視されます。新しいワークフローは無効状態で始まります。

このスイッチを「準備完了」のゲートとして使ってください:

1. ワークフローを構築する。
2. 現実的なペイロードで **Run Manually** をクリックする。
3. **Logs** を確認し、すべてのブロックが期待通りに進んだことを確認する。
4. **Enabled** をオンにする。

ワークフローをオフにしても、すでに進行中の実行は停止しません — 新しい実行が始まらなくなるだけです。

## オーナーとラベル

- **オーナー** — オーナーとして登録されたユーザーとチームは、ワークフローへのアクセス権を持ち、ワークフローが失敗したときの通知をオプトインできます。**Settings → Owners** で設定します。
- **ラベル** — ワークフローをグループ化するためのタグです。ワークフロー一覧をラベルでフィルターできるので、忙しいプロジェクトでもナビゲートが楽になります。チーム、統合、環境ごとに整理する場合に便利です。
- **ラベルルール** — **Workflows → Settings → Label Rules** で、名前や説明のパターンに基づいて新しいワークフローに自動的にラベルを適用します。
- **オーナールール** — **Workflows → Settings → Owner Rules** で、新しいワークフローに自動的にオーナーを割り当てます。

## シークレット

機密情報を含むグローバル変数は **シークレット** としてマークします。値は暗号化され、保存後は UI で非表示になり、実行ログでも非表示 (`[REDACTED]` と表示) になります。

シークレット変数の用途:

- 外部サービスの API キー。
- 認証トークン。
- Webhook の署名キー。
- 読み取り専用アクセス権の人に見られたくないもの全般。

シークレットをブロックに直接貼り付けないでください — `Authorization: Bearer eyJh...` のような値はワークフローとログに表示されてしまいます。代わりに `{{variable.MY_SECRET}}` を使ってください。

## 実行時間の上限

各実行には最大時間があります。実行が時間内に完了しない場合、**Timeout** とマークされ、進行中のブロックはキャンセルされます。デフォルトは余裕があり、通常の HTTP 呼び出しやブロックの連鎖には十分な長さです。

個別のブロックにもそれぞれの制限時間があります — 例えば API ブロックは、ハングした送信リクエストを実行全体よりずっと前に諦めます。

## 他のワークフローを呼び出す制限

**Execute Workflow** コンポーネントを使うと、あるワークフローから別のワークフローを呼び出せます。ワークフロー A が B を呼び、B がさらに A を呼ぶような偶発的なループを防ぐため、連鎖の深さには上限があります。上限を超えた実行は明確なエラーで終了します。

長い連鎖が本当に必要な場合 (1 回の実行で 1 アイテムを処理するジョブなど) は、たいてい **Custom Code** を使って 1 つのワークフロー内でループする方がシンプルです。

## Webhook のセキュリティ

Webhook トリガーには固有の URL が与えられます。その URL を知っている人なら誰でもアクセスできます。意図しない呼び出しから守るために:

- URL はパスワードのように扱ってください。公開で共有したり、公開リポジトリにコミットしたりしないでください。
- 機密性の高いワークフローでは、呼び出し側のシステムにヘッダー (`X-Webhook-Token` など) で共有トークンを送ってもらい、重要なことを行う前に **Conditions** ブロックでチェックしましょう。期待するトークンはシークレット変数として保存します。
- 極めて機密性の高いワークフローでは、公開 Webhook ではなく、OneUptime のイベントトリガーと手動インポートのステップを選んでください。

## 送信ネットワークアクセス

API や他の HTTP ブロックは、OneUptime からリクエストを送ります。セルフホストしている場合は、呼び出し先のサービスにインストール環境から到達できることを確認してください。OneUptime Cloud を利用している場合は、送信元 IP 範囲が [IP アドレス](/docs/configuration/ip-addresses) に記載されているので、相手側で許可リストに追加できます。

## アクセス権

ワークフローはプロジェクトのロールベースアクセス制御に従います。関連する権限:

- **Create / Read / Edit / Delete Workflow** — ワークフロー自体に対する基本権限。
- **Run Workflow** — **Run Manually** をクリックしたり、API でワークフローをトリガーしたりするために必要。
- **Read Workflow Log** — 実行を閲覧するために必要。
- **Read / Create / Edit / Delete Workflow Variable** — グローバル変数一覧を管理する権限。

ほとんどのエンジニアにはワークフローの作成・編集・閲覧権限は与えるべきですが、変数の権限は与えるべきではありません。変数の編集権限は、プロジェクトのシークレットを管理する人だけに与えてください。

## プランの制限

OneUptime Cloud では、小規模プランで月間の実行回数に上限があります。現在の上限は **Project Settings → Billing** に表示されています。上限に達すると、次の請求サイクルまで新しいトリガーは拒否されます。セルフホストのインストールにはこの制限はありません。

## ワークフローが適切でない場合

別のものを選ぶべきケースをいくつか挙げます:

- **重い計算や大規模なデータセット** — ワークフローは軽い糊付け作業のために設計されており、数値計算のためではありません。重い処理は自分のインフラで実行し、ワークフローはそれを開始するのに使ってください。
- **数時間にわたる長時間プロセス** — 1 回の実行は素早く終わるべきです。「A を実行し、2 時間待ち、B を実行する」が必要な場合は、外部のスケジューラから、時間が来たら OneUptime に Webhook を送り返してもらってください。
- **人間が関与するステップバイステップのインシデント対応** — それは [Runbook](/docs/runbooks/index) の役目です。ワークフローは無人で動く自動化のためのものです。
