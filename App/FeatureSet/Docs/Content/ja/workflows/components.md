# ワークフロー コンポーネント

コンポーネントはトリガーの後に配置するアクションノードです。それぞれが 1 つの仕事をします — HTTP リクエストを送る、Slack メッセージを送る、条件で分岐する、JavaScript スニペットを実行する — そして次のノードがつなげる 1 つ以上の出力ポートを公開します。

このページはカタログです。配線ルールとキャンバス自体については[ワークフローを作成する](/docs/workflows/authoring) を参照してください。

## API

任意の URL に対してアウトバウンドの HTTP リクエストを送信します。

**引数**:

- **Method** — `GET`、`POST`、`PUT`、`PATCH`、`DELETE`。
- **URL** — リクエストの URL。補間されます。
- **Request Headers** — ヘッダーの JSON オブジェクト。
- **Request Body** — `POST` / `PUT` / `PATCH` 用の JSON またはテキストボディ。

**出力ポート**:

- `success` — レスポンスステータスが 2xx のときに発火。戻り値: `response-status`、`response-headers`、`response-body`。
- `error` — ネットワーク障害または非 2xx レスポンスで発火。戻り値: `error` メッセージ。

これを使うのは: あらゆるサードパーティ REST API、自社の管理エンドポイント、専用コンポーネントのない軽量な連携。

## Webhook (アウトバウンド)

よくある「投げて忘れる」ケース向けの API コンポーネントの薄いラッパーです。URL に JSON ボディを送信し、単一の `success` / `error` ペアを公開します。

下流でレスポンスボディを読む必要があれば **API** を選び、別のシステムに通知するだけなら **Webhook** を選んでください。

## Slack

プロジェクトの Slack ワークスペース接続を使って Slack チャンネルにメッセージを投稿します。

**引数**:

- **Channel name** — 投稿先のチャンネル。ボットが事前にそのチャンネルのメンバーになっている必要があります。
- **Message text** — 本文。補間されます。Slack mrkdwn をサポートします。

最初に **Project Settings → Workspace Connections → Slack** でワークスペース接続をセットアップします。[Slack ワークスペース接続](/docs/workspace-connections/slack) を参照してください。

## Microsoft Teams

プロジェクトの Teams 接続を使って Microsoft Teams のチャンネルにメッセージを投稿します。

**引数**:

- **Team & channel** — 送信先。
- **Message text** — 本文。

接続のセットアップについては [Microsoft Teams ワークスペース接続](/docs/workspace-connections/microsoft-teams) を参照してください。

## Discord

コンポーネントに設定された incoming webhook URL 経由で Discord チャンネルにメッセージを投稿します。

## Telegram

コンポーネントに設定されたボットトークンとチャット ID 経由で Telegram のチャットにメッセージを送信します。

## Email

OneUptime の SMTP 設定を使ってメールを送信します。

**引数**:

- **To** — 受信者のメールアドレス。
- **Subject** — 補間されます。
- **Body** — Markdown または HTML。

メールはプロジェクトに設定された送信者アドレスから送信されます ([SMTP](/docs/emails/smtp) を参照)。

## Custom Code

ワークフローの変数と上流ノードの戻り値にアクセスできる JavaScript スニペットを実行します。

**引数**:

- **Code** — JavaScript の本体。最後の式の値 (または `(async () => { ... })()` から返されたもの) がコンポーネントの戻り値になります。
- **Arguments** — `args` として渡される任意の名前付き値。

**出力ポート**: `success` (戻り値)、`error` (キャッチされた例外)。

これを使うのは: 2 つのシステム間でペイロードを変換する、独自のコンポーネントに値しない小さな計算をする、JS でしかできないロジックを呼ぶ。自社インフラ内で実行する必要のあるより重いスクリプト処理は [Runbook](/docs/runbooks/index) の Bash または JavaScript ステップが適しています。

## JSON

テキストと JSON を相互変換します。

- **JSON → Text** — JSON オブジェクトを文字列にシリアライズ (テキストを期待するアウトバウンドコンポーネントの `body` 引数にパイプするのに便利)。
- **Text → JSON** — 文字列を JSON オブジェクトにパース。上流の API がテキストとしてボディを返したが、フィールドを読む必要があるときに便利。

## Conditions

比較で分岐します。設定:

- **Left value** — 通常は `{{Incident.title}}` のような補間参照。
- **Operator** — `==`、`!=`、`>`、`>=`、`<`、`<=`、`contains`、`starts with`、`ends with`。
- **Right value** — 比較対象の値。

**出力ポート**: `yes` と `no`。意図に合うブランチでワークフローの残りを配線します。

## Schedule (delay)

設定された時間だけワークフローを一時停止してから続行します。状態を確認する前に外部システムが落ち着く時間を与えたいときに便利です。

## Log

ワークフロー実行ログに行を書き込みます。純粋なデバッグ補助です。行は実行に記録され、**Logs** で見られます。外部への副作用はありません。

## Execute Workflow

別のワークフローをサブステップとして呼び出します。呼び出されたワークフローは独立して実行され (投げて忘れる)、呼び出しがディスパッチされるとすぐに制御が呼び出し元に戻ります。

これを使って共通ロジックを複数のワークフローから切り出します: 「インシデントチャンネルに投稿する」ワークフローを 1 つ作り、チャンネルに通知する必要のある他のすべてのワークフローから呼び出します。

再帰の上限により、ワークフローが互いを無限ループで呼び合うのを防ぎます。[設定と安全性](/docs/workflows/configuration) を参照してください。

## モデルコンポーネント (OneUptime エンティティ上の CRUD)

ワークフローをサポートするすべての OneUptime エンティティ (モニター、インシデント、アラート、ステータスページ、オンコールポリシーなど) について、パレットは自動的に次のコンポーネントを公開します — エンティティ名で検索できます:

- **Find One {Entity}** — クエリで単一のレコードを取得。
- **Find {Entity}** — クエリでレコードのリストを取得 (ページング)。
- **Create {Entity}** — 新しいレコードを挿入。
- **Update {Entity}** — ID で 1 つのレコードを更新。
- **Delete {Entity}** — ID で 1 つのレコードを削除。
- **Count {Entity}** — クエリに一致するレコードを数える。

これがワークフローがプラットフォームを離れずに OneUptime の状態を読み書きする方法です。たとえば: CI ツールからの Webhook がビルドの失敗メッセージで **Create Incident** を呼ぶ、スケジュールされたワークフローが 5 分ごとに **Find Incident** を実行してサマリーをメールで送る。

## 適切なコンポーネントを選ぶ

経験則をいくつか:

- やりたいことに専用コンポーネントがあれば (Slack、Email、OneUptime エンティティの CRUD)、それを使う — 自前で組むよりエラーハンドリングがきれいで、ログも明確です。
- 専用コンポーネントのない外部 HTTP API を呼ぶ必要があれば **API** を使う。
- 2 つのコンポーネント間でデータを*整形*する必要があれば **Custom Code** または **JSON** を使う。
- 値に応じて異なるアクションを取る必要があれば **Conditions** を使う。
