# コンポーネント

コンポーネントはトリガーの後に追加するビルディングブロックです。それぞれが 1 つの仕事 — メッセージを送る、API を呼び出す、条件をチェックする — を行い、次に続くものに接続されます。

このページはカタログです。キャンバス上でのドラッグ、ドロップ、接続については [ワークフローの作成](/docs/workflows/authoring) を参照してください。

## API

任意の URL に HTTP リクエストを行います。

**設定**:

- **Method** — `GET`、`POST`、`PUT`、`PATCH`、`DELETE`。
- **URL** — 呼び出すアドレス。
- **Headers** — 送信するヘッダー。
- **Body** — `POST` / `PUT` / `PATCH` のリクエストボディ。

**出力**:

- **Success** — 呼び出しが成功した (2xx 応答) ときに発火。ステータス、ヘッダー、ボディを渡します。
- **Error** — ネットワーク障害や 2xx 以外の応答で発火。エラーメッセージを渡します。

向いている用途: 外部の API、自社の管理エンドポイント、専用コンポーネントがないあらゆる統合。

## Webhook (送信)

「投げて忘れる」ケース向けの API コンポーネントの簡略版。URL に JSON ボディを POST します。

応答を読む必要がある場合は **API** を使ってください。単に通知を送って次に進みたい場合は **Webhook** を使います。

## Slack

Slack チャンネルにメッセージを投稿します。

**設定**:

- **Channel** — チャンネル名。ボットが事前にそのチャンネルに参加している必要があります。
- **Message** — 送信するテキスト。Slack のフォーマットに対応しています。

事前に **Project Settings → Workspace Connections → Slack** で Slack をプロジェクトに接続してください。[Slack ワークスペース接続](/docs/workspace-connections/slack) を参照してください。

## Microsoft Teams

Microsoft Teams のチャンネルにメッセージを投稿します。

**設定**:

- **Team and channel** — 投稿先。
- **Message** — 送信するテキスト。

セットアップについては [Microsoft Teams ワークスペース接続](/docs/workspace-connections/microsoft-teams) を参照してください。

## Discord

受信 Webhook URL を通じて Discord チャンネルにメッセージを投稿します。

## Telegram

ボットトークンとチャット ID を使って Telegram のチャットにメッセージを送信します。

## Email

OneUptime 経由でメールを送信します。

**設定**:

- **To** — 受信者のメールアドレス。
- **Subject** — 件名。
- **Body** — Markdown または HTML のメッセージ本文。

メールはプロジェクトに設定された送信元から送られます — [SMTP](/docs/emails/smtp) を参照してください。

## Custom Code

他のブロックでは実現できないことを行いたい場合に、小さな JavaScript を実行します。

**設定**:

- **Code** — あなたの JavaScript。最後の値 (または非同期関数から return した値) がブロックの出力になります。
- **Arguments** — 渡せる名前付きの値。

**出力**: success (return した値) と error (発生した例外)。

向いている用途: 2 つのシステム間のデータの整形、ちょっとした計算、専用ブロックを作るほどでもないこと。本格的なスクリプトには [Runbook](/docs/runbooks/index) を使ってください。

## JSON

テキストと JSON を相互変換します。

- **JSON → Text** — JSON オブジェクトを文字列に変換します。次のブロックがテキストを期待する場合に便利です。
- **Text → JSON** — 文字列を JSON オブジェクトにパースします。テキストとして到着したデータからフィールドを読みたい場合に便利です。

## Conditions

比較に基づいて分岐します。

**設定**:

- **Left value** — 通常は前のブロックからの値。
- **Operator** — `==`、`!=`、`>`、`>=`、`<`、`<=`、`contains`、`starts with`、`ends with`。
- **Right value** — 比較対象。

**出力**: **Yes** と **No**。次のブロックを希望するブランチに接続します。

## Delay

続行する前にワークフローを一定時間停止します。別のシステムが追いつくまで少し待たせたい場合に便利です。

## Log

実行ログに 1 行書き込みます。外部への影響はなく、ワークフローのログに表示されるだけです。デバッグに便利です。

## Execute Workflow

このワークフローから別のワークフローを呼び出します。呼び出されたワークフローは独立して実行されます — 呼び出し元のワークフローは終了を待たずに続行します。

これを使って共通ロジックを共有します。「インシデントチャンネルへの投稿」ワークフローを一度作っておけば、そのチャンネルへの通知が必要な他のワークフローから呼び出せます。

ワークフロー同士がループで呼び合うことを防ぐための安全制限があります。[構成と安全性](/docs/workflows/configuration) を参照してください。

## OneUptime データコンポーネント

OneUptime の各種レコード (モニター、インシデント、アラート、ステータスページ、オンコールポリシーなど多数) について、パレットには以下のコンポーネントがあります — 種類の名前で検索してください:

- **Find One** — ID やフィルターで 1 件のレコードを取得。
- **Find** — レコードのリストを取得。
- **Create** — 新しいレコードを追加。
- **Update** — レコードを 1 件変更。
- **Delete** — レコードを 1 件削除。
- **Count** — フィルターに一致するレコードをカウント。

これがワークフローで OneUptime のデータを読み書きする方法です。たとえば: CI ツールからの Webhook が **Create Incident** を使って、失敗の詳細を含むインシデントを開けます。

## どのコンポーネントを使うべきか?

簡単な指針:

- やりたいことに専用ブロック (Slack、Email、OneUptime のレコード) があれば、それを使ってください — エラーハンドリングがきれいで、ログもわかりやすくなります。
- それ以外の外部 API には **API** を使います。
- ブロック間でデータを整形するには **Custom Code** か **JSON** を使います。
- 値に応じて別のアクションを取るには **Conditions** を使います。
