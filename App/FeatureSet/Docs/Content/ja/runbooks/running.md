# Runbook を実行する

Runbook 実行は次の 3 つの方法で作成されます:

1. **ルールによる自動実行** — [Runbook ルール](/docs/runbooks/rules) を参照。
2. **Runbook ページから手動で** — Runbook の概要で **今すぐ実行** をクリック。インシデント、アラート、メンテナンスのいずれにも紐付きません。
3. **エンティティのフィードから手動で** — インシデント、アラート、予定メンテナンスで **Runbook を実行** をクリック。実行はそのエンティティに紐付きます。

## 実行ビュー

任意の実行を開くと、チェックリスト UI が表示されます。各ステップには次のものがあります:

- **ステータスピル** — 待機中、実行中、あなた待ち、完了、スキップ、失敗。
- **タイトルと説明** — 実行時に Runbook から複製。
- **出力** (折りたためる) — stdout、返り値、HTTP レスポンス。
- **エラーメッセージ** (ステップが失敗した場合)。
- `WaitingForUser` の Manual ステップでは **完了にする** と **スキップ** のボタン。

実行が終端状態でない間、ページは 3 秒ごとにポーリングするため、自動ステップの完了がほぼリアルタイムで見えます。

## 手動と自動ステップの組み合わせ

典型的なフロー:

1. **スクリプトステップ**: システム状態を取得して S3 に書き出す。
2. **手動ステップ**: 「ステータスページのバナーで顧客に通知。」対応者がチェック。
3. **HTTP ステップ**: PagerDuty で DBA を呼ぶ。
4. **手動ステップ**: 「セカンダリ DB が primary になったことを確認。」対応者がチェック。
5. **スクリプトステップ**: Slack へ「異常なし」を送信。

ステップ 2 と 4 はチェックされるまで実行を停止します。ステップ 1、3、5 は自動で走ります。全体で 1 つの実行、1 つのタイムライン、1 つの真実の情報源になります。

## 実行のキャンセル

実行ページで **実行をキャンセル** をクリック。現在のステップ (あれば) は完了しますが、以降のステップは開始されません。ステータスは `Cancelled` になります。

## 出力の保持

ステップごとの出力は **50KB** までに制限されます。暴走スクリプトが DB を膨らませるのを防ぐためです。より大きな成果物が必要な場合は、スクリプトから S3 やロガーに書き出し、返り値に URL を保存してください。

## Runbook の再実行

実行は 1 回限り、変更不可の記録です。再実行するには再度 **今すぐ実行** をクリックします — 現在の Runbook ステップの新しいスナップショットを伴う新規実行が作成されます。元の実行は監査用にそのまま残ります。

## 過去の実行を探す

各 Runbook には **実行** タブがあり、すべての走行を一覧でき、ステータス、日付範囲、ソースエンティティでフィルタできます。インシデント、アラート、メンテナンスでは **Runbooks** タブにそのエンティティに紐付く実行が表示されます。
