# Runbook を実行する

Runbook の実行が作成されるパスは 3 つあります:

1. **ルールによる自動実行** — [Runbook ルール](/docs/runbooks/rules) を参照。
2. **Runbook ページから手動** — Runbook の概要ページで **今すぐ実行** をクリック。インシデント、アラート、予定メンテナンスのどれにも紐付かない。
3. **エンティティのフィードから手動** — インシデント、アラート、予定メンテナンスイベント上で **Runbook を実行** をクリック。実行はそのエンティティに紐付く。

## 実行ビュー

任意の実行を開くとチェックリスト UI が見られます。各ステップは次を表示します:

- **ステータスバッジ** — Pending、Running、あなた待ち、Done、Skipped、Failed。
- **タイトルと説明** — 実行時に Runbook からコピーされたもの。
- **出力** (折り畳み可) — stdout、返り値、HTTP レスポンス。
- ステップが失敗した場合の **エラーメッセージ**。
- `WaitingForUser` の Manual ステップでは **完了マーク** と **スキップ** のボタン。

実行が終了状態でない間、ページは 3 秒ごとにポーリングするので、自動ステップが完了するのをほぼリアルタイムで見られます。

## Manual ステップと自動ステップを混ぜる

典型的なフロー:

1. **スクリプトステップ**: システム状態を取得して S3 に書き込む。
2. **Manual ステップ**: 「ステータスページのバナーで顧客に通知。」対応者がチェックを入れる。
3. **HTTP ステップ**: PagerDuty で DBA を呼び出し。
4. **Manual ステップ**: 「セカンダリ DB が新プライマリになったことを確認。」対応者がチェックを入れる。
5. **スクリプトステップ**: 復旧の Slack メッセージを送信。

ステップ 2 と 4 はチェックされるまで実行を一時停止します。ステップ 1、3、5 は自動で動きます。全体は 1 つの実行、1 つのタイムライン、1 つの真実のソースです。

## 実行をキャンセルする

実行ページで **実行をキャンセル** をクリック。現在のステップ (もしあれば) は完了し、それ以降のステップは開始しません。ステータスは `Cancelled` になります。

## 出力の保持

ステップごとの出力は暴走スクリプトでデータベースが肥大化しないように **50KB** で打ち切られます。もっと大きな成果物が必要なら、スクリプト内で S3 やロガーに書き出して、URL を返り値に格納してください。

## Runbook を再実行する

Runbook の実行は 1 回限り・不変の記録です。再実行するには再び **今すぐ実行** をクリック — それは Runbook の現在のステップの新しいスナップショットを伴う新規実行を作成します。元の実行は監査証跡のためそのまま残ります。

## 過去の実行を探す

すべての Runbook には **実行** タブがあり、すべての実行が、ステータス、日付範囲、ソースエンティティでフィルタ可能な形で並びます。インシデント、アラート、予定メンテナンスイベントからは、**Runbooks** タブにそのエンティティに紐付く実行が表示されます。
