# Runbook エージェント

**Runbook エージェント**は、Runbook の Bash *および* JavaScript ステップを**お客様自身のインフラ内で**実行する、セルフホスト型の小さなプロセスです。OneUptime のワーカーがあなたのスクリプトを実行することは決してありません — ワーカーはキューに積むだけで、あなたが環境にインストールした Runbook エージェントがそれを取りに来て、実行し、結果を返します。

JavaScript は引き続き `isolated-vm` サンドボックスで動きますが、そのサンドボックスが我々のホストではなく、あなたのエージェントホストに置かれるのが違いです。

このページでは、エージェントのインストール方法、Bash と JavaScript ステップのルーティング方法、日常運用について説明します。

## なぜエージェントが必要なのか

以前の OneUptime は Bash と JavaScript ステップをワーカー上で実行していました。JavaScript はサンドボックス (`isolated-vm`) 内でしたが、Bash は違いました。シングルテナントのセルフホスト構成を超えると、いずれも問題でした:

- **信頼境界**: Runbook を作成できる人なら誰でもワーカー上でコードを実行でき、ワーカーが持つ全ての環境変数とファイルシステムにアクセスできてしまいました。JavaScript のサンドボックスは明白なことをブロックしますが、本気のユーザーが我々のネットワークから何に到達できるかを調査するのを止めることはできませんでした。
- **到達範囲**: 役に立つステップの多くは**お客様側の**インフラを操作したい (「このサービスを再起動」「私たちのクラスタで kubectl」「社内 DB のレコードを引く」) のであって、OneUptime 側ではありません。

Runbook エージェントはこれを反転させます。Bash と JavaScript ステップは我々の側では動かず、あなたが管理するホスト上で動き、そのホストに何を許すかはあなたが決めます。

## 仕組み

1. OneUptime で Runbook エージェントを作成すると、OneUptime が ID と秘密鍵を生成します。
2. その ID/鍵と OneUptime URL を指定して、エージェントのコンテナをあなたのインフラ内のホストで起動します。
3. エージェントは数秒ごとに OneUptime に「私にやる仕事はありますか?」と問い合わせます。
4. Bash または JavaScript ステップが実行されるとき、ワーカーはそのステップの **Agent Tag** とステップ種別 (Bash または JavaScript) を付けたジョブ行を挿入し、ステータスを `Pending` にします。
5. 同じプロジェクト内でそのタグを持つ任意の健全なエージェントがアトミックにそのジョブを取得し (2 つのエージェントが同じジョブを実行することはありません)、ローカルで実行します — Bash は `bash -c <スクリプト>` で、JavaScript は `isolated-vm` サンドボックスで — 結果を収集して返送します。
6. ワーカーはその結果で Runbook を続行します。

エージェントが必要とするのは、OneUptime インスタンスへの**送信 HTTPS** のみです。受信接続は一切受け付けません。

## エージェントのインストール

### 1. エージェントレコードを作成

**Runbooks → Agents → 新規作成**へ進み、入力します:

| フィールド | メモ |
| --- | --- |
| **名前** | 分かりやすい名前 — 通常は `どこで動いて何ができるか` 形式、例えば `prod-eu-west-1`。 |
| **説明** | 任意。このホストが到達できるものについての一文。未来のあなたが感謝します。 |
| **タグ** | カンマ区切り。Bash と JavaScript ステップはタグを指定し、プロジェクト内でそのタグを持つどのエージェントも実行できます。よくある形: `prod`, `staging`, `eu-west-1`, `db-host`。 |

### 2. インストールコマンドをコピー

エージェント作成後、行の **セットアップ手順を表示** をクリックします。このエージェントの ID と鍵があらかじめ埋め込まれた `docker run` コマンドが表示されます。**今すぐ鍵を保存してください** — 後でリセットはできますが、モーダルを閉じた後に同じ鍵の値を再表示することはできません。

### 3. インフラ内のホストで実行

次のことができる任意のホストで Docker コマンドを実行します:

- HTTPS で OneUptime インスタンスに到達できる、
- Bash ステップにやらせたいこと (他ホストへの SSH、`kubectl`、DB との通信など) ができる。

```bash
docker run --name oneuptime-runbook-agent --restart unless-stopped \
  -e RUNBOOK_AGENT_ID=<agent-id> \
  -e RUNBOOK_AGENT_KEY=<agent-key> \
  -e ONEUPTIME_URL=https://oneuptime.あなたのドメイン.com \
  -d oneuptime/runbook-agent:release
```

### 4. 接続状態を確認

**Runbooks → Agents** に戻ります。約 60 秒以内に、エージェントの行が新しい **Last seen** タイムスタンプ付きで `Connected` に切り替わるはずです。`Disconnected` のままなら:

- コンテナログ (`docker logs oneuptime-runbook-agent`) で認証エラーやネットワーク障害を確認。
- ホストから `curl` で OneUptime URL に到達できることを確認。
- ID と鍵が空白なしでコピーされていることを確認。

## タグとルーティング

タグは Bash または JavaScript ステップがエージェントを見つける手段です。いくつかのパターン:

- **環境ごとに 1 タグ**: prod のエージェントに `prod`、staging に `staging`。`prod` を指定した Bash ステップは prod 上だけで動きます。
- **リージョンごとに 1 タグ**: `eu-west-1`, `us-east-1`。操作対象リソースの近くで動かしたいときに便利。
- **同じタグの複数エージェント**: 同じく `prod` をタグ付けしたエージェントを 2 つ動かします。どちらでもジョブを取得できる — 可用性が上がり、Runbook を壊さずにローリング再起動ができます。
- **1 エージェントに複数タグ**: prod の EU クラスタにあるエージェントが `prod`, `eu-west-1`, `kubernetes` を持つ、など。Bash ステップはそのいずれを指定してもよい。

Bash と JavaScript ステップはそれぞれちょうど 1 つのエージェントタグを**指定しなければなりません**。マルチタグルーティング (`prod` AND `db` の両方を持つ任意のエージェントで実行) はロードマップにありますが、このリリースには含まれません。

## ステップをエージェントに向ける

Runbook に Bash または JavaScript ステップを追加すると、フォームが **Agent Tag** を尋ねます:

- 実行させたいエージェントに一致するタグを入力。
- その下のエディタでスクリプトを書く。

Runbook が走ってステップに到達すると、ワーカーがそのタグとステップ種別でジョブをキューに入れます。そのタグを持つ健全なエージェントが少なくとも 1 つオンラインなら、数秒以内にジョブが取得され実行されます。Bash は `bash -c` で実行され、JavaScript はエージェント上の `isolated-vm` サンドボックスで動きます (ファイルシステムなし、ネットワークなし、`Function`/`eval` なし)。

## 運用上の注意

### タイムアウト

各 Bash または JavaScript ステップには 2 つのタイムアウトが適用されます:

| タイムアウト | 既定 | 制御内容 |
| --- | --- | --- |
| **Claim タイムアウト** | 2 分 | *いずれかの*エージェントがジョブを取得するまでワーカーが待つ時間。誰も時間内に取得しない場合、ステップは `TimedOut` で失敗し、Runbook は (**失敗時継続**設定に応じて) 続行または停止します。 |
| **実行タイムアウト** | 30 秒 | エージェントがスクリプトを終了させるまでに走らせる時間。ステップごとに設定可能。(Bash は `SIGKILL`、JavaScript はアイソレートが破棄されます。) |

ワーカーの全体待機ウィンドウは `claim タイムアウト + 実行タイムアウト + 数秒の余裕` です。ステップに合った値を選んでください。

### リースとハートビート

エージェントがジョブを取得すると、短いリース (既定 30 秒) が付与されます。スクリプト実行中、エージェントは 10 秒ごとにリースを更新します。スクリプト途中でエージェントが死んだりネットワークを失ったりすると、リースが期限切れになり、ワーカーは無限に待つのではなくジョブを `TimedOut` とマークします。

Bash の子プロセスはリース期限切れ時に**自動的にはキャンセルされません** (JavaScript のアイソレートも、終わるなら終わるまで走り続けます) — ただしワーカーはそれを待つのをやめ、他のクレームが引き継いだ後ではエージェントは結果を送信できなくなります。Exactly-once が重要ならスクリプトは再実行可能なよう設計してください。

### エージェントが 1 つもオンラインでない場合

ステップ実行時にそのステップのタグを持つ健全なエージェントが 1 つもオンラインでなければ、ジョブは claim タイムアウトが切れるまで `Pending` のままで、その後「no agent claimed the job」という明確なメッセージで失敗します。Runbook を本番で動かす前にカバレッジを確認する場所が Agents ページです。

### 出力上限

stdout + stderr の合計はステップごとに **50 KB** で打ち切られます。それより大きい出力はマーカー付きで切り詰められます。完全なログが必要なら、スクリプト内から S3 やログストアに書き出し、URL を `echo` してください。

### キャンセル

Runbook 実行をキャンセル (実行ビューや API から) すると、その `Pending`/`Claimed`/`Running` 状態の Bash ジョブすべてが直ちに `Cancelled` とマークされます。すでにスクリプト実行中のエージェントは作業を完了しますが、その結果はサーバーに受け入れられません。

### 並行度

各エージェントは既定で同時に 1 ジョブを実行します。それ以上許可するには、エージェントコンテナに `RUNBOOK_AGENT_CONCURRENCY` を設定してください — ただしエージェントはホスト上の他のものとリソースを共有することを覚えておいてください。

## 環境変数

エージェントは起動時に次を読みます:

| 変数 | 必須 | 既定 | メモ |
| --- | --- | --- | --- |
| `ONEUPTIME_URL` | はい | — | OneUptime インスタンスのベース URL、例 `https://oneuptime.あなたのドメイン.com`。 |
| `RUNBOOK_AGENT_ID` | はい | — | エージェントのセットアップモーダルに表示される UUID。 |
| `RUNBOOK_AGENT_KEY` | はい | — | エージェントのセットアップモーダルに表示される秘密値。 |
| `RUNBOOK_AGENT_POLL_INTERVAL_MS` | いいえ | `5000` | エージェントが新しいジョブを問い合わせる頻度。 |
| `RUNBOOK_AGENT_HEARTBEAT_INTERVAL_MS` | いいえ | `60000` | エージェントが生存報告する頻度。 |
| `RUNBOOK_AGENT_JOB_HEARTBEAT_INTERVAL_MS` | いいえ | `10000` | エージェントが実行中ジョブのリースを更新する頻度。 |
| `RUNBOOK_AGENT_CONCURRENCY` | いいえ | `1` | このエージェント上での最大同時ジョブ数。 |

## エージェント鍵のローテーション

鍵が漏洩した場合、OneUptime でそのエージェントを開いて鍵をリセットしてください。古い鍵は即座に無効になります。新しい鍵でエージェントコンテナを更新して再起動します。

## 権限

エージェントの管理は既存の Runbooks 権限グループに属します:

- `CreateRunbookAgent`, `EditRunbookAgent`, `DeleteRunbookAgent`, `ReadRunbookAgent` — エージェントレコードを管理。
- `RunbookManager` (ロール) — 上記すべてを束ねる。

Runbook を*実行する* (したがって Bash ステップをディスパッチする) 権限は依然として `CreateRunbookExecution` / `EditRunbookExecution` です。

## エージェント向け API

参考までに — エージェントは `/runbook-agent-ingest` 配下にマウントされた次のエンドポイントを使用します。認証は JSON ボディ内のエージェント ID + 鍵 (または `x-agent-id` / `x-agent-key` ヘッダー) で行われます。

| エンドポイント | 目的 |
| --- | --- |
| `POST /heartbeat` | 生存報告; `lastAlive`、`connectionStatus`、`hostInfo`、`agentVersion` を更新。 |
| `POST /claim-next-job` | このエージェントのタグのいずれかに一致する最も古い `Pending` ジョブをアトミックに取得。やることがなければ `{ job: null }` を返す。 |
| `POST /job/:jobId/heartbeat` | ジョブのリースを更新。リース失効後やジョブ終了後は 404 を返す。 |
| `POST /job/:jobId/result` | 最終結果を送信。リースがすでに他に移っていれば無視される。 |

これらを手動で呼ぶ必要はありません — 同梱のエージェントが代わりに呼びます。私たちのエージェントでは制約に合わない場合に独自エージェントを作れるよう、参考として文書化しています。
