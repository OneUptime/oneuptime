# Runbook 概要

Runbook は再利用できる対応手順 — 手動または自動のステップを順序付けたリスト — で、インシデント、アラート、または予定メンテナンスに紐付けます。「これからどうする?」という Slack の即興のやり取りを、夜中の 3 時にチームメイトが何も知らずに引き継げる形にしてくれます。

## ひと目で

- OneUptime ダッシュボードの **分析 & 自動化 → Runbooks** にある**トップレベル機能**。
- **4 種類のステップ**: 手動チェックリスト、JavaScript (サンドボックス) と Bash (どちらもお客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) で実行)、HTTP リクエスト。
- **3 つのトリガー経路**: インシデント/アラート/予定メンテナンスに一致するルール、または任意のイベント上の手動「Runbook を実行」ボタン。
- **スナップショット方式**: Runbook が開始すると、そのステップが実行にコピーされます。後でテンプレートを編集しても進行中の実行は変わりません。
- **完全な監査証跡**: 各ステップのステータス、出力、エラーメッセージ、所要時間が実行に永久に記録されます。

## なぜ Runbook を使うのか?

インシデント対応はしばしば、1 分の小さな途切れと数時間の障害の分かれ目になります。Runbook は次のことを助けます:

- **暗黙知をコード化する** — 「キューが詰まったらどうする?」の答えがチームの見える場所に置かれる。
- **平均復旧時間 (MTTR) を短縮する** — 自動ステップは秒で動き、手動ステップは決断の麻痺を取り除く。
- **対応アクションを監査する** — 実行したステップ、出力、対応者のクリックがすべて実行に記録される。
- **ジュニアの即戦力化** — シニアを夜中の 3 時に呼ばずに自信を持って Runbook を実行できる。
- **記憶ではなくデータからポストモーテムを書く** — 取り込まれた実行は何が起きたかをそのまま凍結した記録。

## キー概念

残りのドキュメントを通じて繰り返し登場する用語。まずここを揃えておきましょう:

| 用語 | 意味 |
| --- | --- |
| **Runbook** | テンプレート。名前付きの再利用可能な手順で、順序付きステップ一覧と `isEnabled` フラグを持つ。 |
| **ステップ** | Runbook 内の 1 項目。タイプ (Manual / JavaScript / HTTP / Bash)、タイトル、説明、タイプ固有の設定を持つ。 |
| **Runbook ルール** | タイトルや説明が正規表現に一致したときにインシデント、アラート、予定メンテナンスに 1 つ以上の Runbook を自動で添付するパターン。 |
| **実行** | Runbook の 1 回の走行。ルールが発火、誰かがイベントで「Runbook を実行」をクリック、または Runbook 自身で「今すぐ実行」をクリックしたときに作成される。ステップのスナップショットとステップごとのステータス/出力を保持する。 |
| **スナップショット** | 各実行に張り付く、Runbook のステップの凍結コピー。後でテンプレートを編集しても履歴を書き換えずに済む。 |

## Runbook のライフサイクル

1. **作成** — Runbook を作り、Manual、JavaScript、HTTP、Bash のステップを混ぜて並べ、保存する。
2. **(任意) ルールを追加** — インシデント、アラート、予定メンテナンスの設定で、イベントのタイトルまたは説明が正規表現に一致したらこの Runbook を起動するよう OneUptime に指示する。
3. **トリガー** — ルールが一致するイベントの作成時に自動発火するか、対応者がイベント上で **Runbook を実行** を手動でクリックする。
4. **実行** — ステップのスナップショットを伴って新しい実行が作成される。自動ステップは Runbook ワーカーで実行され、手動ステップごとに誰かがチェックを入れるまで一時停止する。
5. **監査** — 実行はイベントの **Runbooks** タブと Runbook の実行リストに永続的に残る。ステップごとの出力、エラー、タイミングはポストモーテム用に保持される。

## どのステップタイプをいつ使うか

簡単な判断ガイド。詳しくは [Runbook を作成する](/docs/runbooks/authoring) を参照。

| ステップタイプ | こんなときに使う… | 例 |
| --- | --- | --- |
| **Manual** | 人が確認・判断・OneUptime には観察できない操作を行う必要があるとき。 | 「ロードバランサーのダッシュボードでセカンダリリージョンへのトラフィックを確認。」 |
| **JavaScript** | 小さな閉じた計算が必要 — 設定サービスへの問い合わせ、ペイロードの変換、次のステップ前のロジック実行。お客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) 上でサンドボックスで実行されます。 | 現在のレプリカ遅延を計算し、続行するか判断。 |
| **HTTP リクエスト** | 既存の API を呼ぶ場合 — 自社の管理エンドポイント、クラウドプロバイダ、PagerDuty、Slack。 | フェイルオーバーオーケストレーターへの `POST`。 |
| **Bash** | お客様自身のインフラでシェルコマンドを実行する必要があるとき — サービス再起動、`kubectl` 実行、デプロイスクリプトの呼び出し。お客様の環境に [Runbook エージェント](/docs/runbooks/agents) のインストールが必要。 | サービス再起動、`kubectl rollout restart` 実行、リカバリスクリプト実行。 |

4 種類すべてを 1 つの Runbook に混ぜられます — Runbook の強みは人による検証と自動化を織り交ぜることです。

## ダッシュボードのどこに Runbook があるか

| ページ | そこで行うこと |
| --- | --- |
| **分析 & 自動化 → Runbooks** | Runbook テンプレートを閲覧・作成・編集する。 |
| Runbook の **ステップ** タブ | ステップ一覧の作成と並び替え。 |
| Runbook の **実行** タブ | この Runbook の実行をすべて表示、ステータスでフィルタ可能。 |
| Runbook の **今すぐ実行** ボタン | どのイベントにも紐付かないアドホックな実行を開始する。 |
| **インシデント / アラート / 予定メンテナンス → 設定 → Runbook ルール** | エンティティ種別ごとに自動トリガールールを作成する。 |
| インシデント / アラート / メンテナンスイベント → **Runbooks** タブ | このイベントに紐付く実行を表示、手動で **Runbook を実行** をクリックして実行する。 |

## よくあるユースケース

チームが Runbook を使う典型的なパターン:

- **データベースフェイルオーバー** — JavaScript で現状を取得、Manual で当番 DBA にレプリカ正常性を確認させる、HTTP でオーケストレーター API を呼び出す、Manual で「DNS 更新済み」をチェック、HTTP で Slack に「復旧」を投稿。
- **キャッシュフラッシュ** — 1 つの HTTP ステップに加えて「ダッシュボードでキャッシュヒット率が回復していることを確認」する Manual ステップ。
- **顧客影響インシデント** — Manual: 「ステータスページ更新を投稿」。HTTP: 「`#customer-incidents` で CS チームに通知」。JavaScript: 「内部 API から影響アカウントの一覧を取得」。
- **予定メンテナンスのプリフライト** — JavaScript: 現在のメトリクスをスナップショット。Manual: 「ステークホルダーと変更ウィンドウを確認」。HTTP: ロードバランサーでメンテナンスモードを有効化。
- **常時実行の衛生チェック** — タイトルパターンを空にしたルールがどんなインシデントでもシステム状態を取り込みます — ポストモーテムに便利。

## 具体例

「db-primary」をタイトルに含むすべてのインシデントで、5 ステップの DB フェイルオーバー Runbook を自動で動かしたいとします。

**1. Runbook を作成。** **Runbooks → Runbook を作成** から「DB primary failover」と名付け、次のステップを追加:

| # | タイプ | タイトル |
| --- | --- | --- |
| 1 | JavaScript | フェイルオーバー前のレプリカ遅延を取得 |
| 2 | Manual | DBA ダッシュボードでレプリカ正常性を確認 |
| 3 | HTTP | フェイルオーバーオーケストレーターに `POST` |
| 4 | Manual | 書き込みが新プライマリに向いていることを検証 |
| 5 | HTTP | `#db-incidents` Slack に復旧を投稿 |

**2. ルールを追加。** **インシデント → 設定 → Runbook ルール** で次を作成:

```
タイトルパターン:  ^db-primary
Runbooks:        [DB primary failover]
```

**3. トリガー。** モニターのアラートがインシデント `INC-4821 · db-primary connection timeout` を開く。ルールが一致し、実行が作成され、以下が起きる:

- ステップ 1 (JavaScript) は即座にワーカー上で実行 — `return { lagMs: 412 }` が記録される。
- ステップ 2 (Manual) は実行を一時停止。当番者がインシデントページで「あなたを待っています」バッジを見て、ダッシュボードをクリックし、ステップをチェックオフする。
- ステップ 3 (HTTP) はステップ 2 のチェックと同時に実行 — `POST` のレスポンスボディが記録される。
- ステップ 4 (Manual) で再び一時停止。
- ステップ 5 (HTTP) が実行され、実行が完了する。

**4. 監査。** 実行はインシデントの **Runbooks** タブに残ります。各ステップの出力はクリック 1 回で見られます。翌週ポストモーテムを書くときに「あのスクリプトは何を返したっけ?」と聞き直す必要はありません — そこにあります。

## OneUptime の他機能との関わり

- **モニター** はインシデントとアラートを開き、**Runbook ルール** はそれを Runbook の実行に変換します。2 つで「検出 → トリガー → 対応 → 記録」の閉ループになります。
- **ワークスペース連携** (Slack、Microsoft Teams) は Runbook の HTTP ステップの自然な宛先 — ステータス更新の投稿、チャンネル通知に。
- **ステータスページ** は顧客影響 Runbook の Manual ステップでよく更新されます。
- **オンコールスケジュール** は誰が呼ばれるかを決め、Runbook はその人が起きてから何をするかを決めます。
