# Runbook 概要

Runbook は再利用できる対応手順 — 手動または自動のステップを順序付けたリスト — で、インシデント、アラート、または予定メンテナンスに紐付けます。「これからどうする?」という Slack の即興のやり取りを、夜中の 3 時にチームメイトが何も知らずに引き継げる形にしてくれます。

## ひと目で

- OneUptime ダッシュボードの **分析 & 自動化 → Runbooks** にある**トップレベル機能**。
- **4 種類のステップ**: 手動チェックリスト、JavaScript (サンドボックス)、HTTP リクエスト、Bash (お客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) で実行)。
- **3 つのトリガー経路**: インシデント/アラート/予定メンテナンスに一致するルール、または任意のイベント上の手動「Runbook を実行」ボタン。
- **スナップショット方式**: Runbook が開始すると、そのステップが実行にコピーされます。後でテンプレートを編集しても進行中の実行は変わりません。
- **完全な監査証跡**: 各ステップのステータス、出力、エラーメッセージ、所要時間が実行に永久に記録されます。

## なぜ Runbook を使うのか?

インシデント対応はしばしば、1 分の小さな途切れと数時間の障害の分かれ目になります。Runbook は次のことを助けます:

- **暗黙知をコード化する** — 「キューが詰まったらどうする?」の答えがチームの見える場所に置かれる。
- **平均復旧時間 (MTTR) を短縮する** — 自動ステップは秒で動き、手動ステップは決断の麻痺を取り除く。
- **対応アクションを監査する** — 実行したステップ、出力、対応者のクリックがすべて実行に記録される。
- **ジュニアの即戦力化** — シニアを夜中の 3 時に呼ばずに自信を持って Runbook を実行できる。
- **記憶ではなくデータからポストモーテムを書く** — 取り込まれた実行は何が起きたかをそのまま凍結した記録。

## キー概念

残りのドキュメントを通じて繰り返し登場する用語。まずここを揃えておきましょう:

| 用語 | 意味 |
| --- | --- |
| **Runbook** | テンプレート。名前付きの再利用可能な手順で、順序付きステップ一覧と `isEnabled` フラグを持つ。 |
| **ステップ** | Runbook 内の 1 項目。タイプ (Manual / JavaScript / HTTP / Bash)、タイトル、説明、タイプ固有の設定を持つ。 |
| **Runbook ルール** | タイトルや説明が正規表現に一致したときにインシデント、アラート、予定メンテナンスに 1 つ以上の Runbook を自動で添付するパターン。 |
| **実行** | Runbook の 1 回の走行。ルールが発火、誰かがイベントで「Runbook を実行」をクリック、または Runbook 自身で「今すぐ実行」をクリックしたときに作成される。ステップのスナップショットとステップごとのステータス/出力を保持する。 |
| **スナップショット** | 各実行に張り付く、Runbook のステップの凍結コピー。後でテンプレートを編集しても履歴を書き換えずに済む。 |

## Runbook のライフサイクル

1. **作成** — Runbook を作り、Manual、JavaScript、HTTP、Bash のステップを混ぜて並べ、保存する。
2. **(任意) ルールを追加** — インシデント、アラート、予定メンテナンスの設定で、イベントのタイトルまたは説明が正規表現に一致したらこの Runbook を起動するよう OneUptime に指示する。
3. **トリガー** — ルールが一致するイベントの作成時に自動発火するか、対応者がイベント上で **Runbook を実行** を手動でクリックする。
4. **実行** — ステップのスナップショットを伴って新しい実行が作成される。自動ステップは Runbook ワーカーで実行され、手動ステップごとに誰かがチェックを入れるまで一時停止する。
5. **監査** — 実行はイベントの **Runbooks** タブと Runbook の実行リストに永続的に残る。ステップごとの出力、エラー、タイミングはポストモーテム用に保持される。

## どのステップタイプをいつ使うか

簡単な判断ガイド。詳しくは [Runbook を作成する](/docs/runbooks/authoring) を参照。

| ステップタイプ | こんなときに使う… | 例 |
| --- | --- | --- |
| **Manual** | 人が確認・判断・OneUptime には観察できない操作を行う必要があるとき。 | 「ロードバランサーのダッシュボードでセカンダリリージョンへのトラフィックを確認。」 |
| **JavaScript** | 小さな閉じた計算が必要 — 設定サービスへの問い合わせ、ペイロードの変換、次のステップ前のロジック実行。 | 現在のレプリカ遅延を計算し、続行するか判断。 |
| **HTTP リクエスト** | 既存の API を呼び出している — 自社の管理エンドポイント、クラウドプロバイダ、PagerDuty、Slack。 | フェイルオーバーオーケストレータへ `POST`。 |
| **Bash** | 自分のインフラ上でシェルコマンドを実行する必要がある — サービスを再起動、`kubectl` を実行、デプロイスクリプトを呼び出す。お客様の環境にインストールした [Runbook エージェント](/docs/runbooks/agents) が必要。 | サービス再起動、`kubectl rollout restart`、復旧スクリプトの実行。 |

ひとつの Runbook で 4 種類すべてを混ぜられます — Runbook の強みは、人の確認と自動化を織り合わせることにあります。

## ダッシュボードでの居場所

| ページ | そこで行うこと |
| --- | --- |
| **分析 & 自動化 → Runbooks** | Runbook テンプレートの閲覧、作成、編集。 |
| **Runbook の「ステップ」タブ** | ステップ一覧の作成と並び替え。 |
| **Runbook の「実行」タブ** | ステータスフィルタ付きでこの Runbook のすべての走行を確認。 |
| **Runbook の「今すぐ実行」ボタン** | どのイベントにも紐付かないアドホック実行を起動。 |
| **インシデント / アラート / 予定メンテナンス → 設定 → Runbook ルール** | エンティティタイプごとの自動トリガールールを作成。 |
| **インシデント / アラート / メンテナンスイベント → Runbooks タブ** | そのイベントに紐付く実行を確認し、**Runbook を実行** で手動実行を起動。 |

## よくあるユースケース

チームが Runbook を活用するパターンの例:

- **データベースフェイルオーバー** — JavaScript で現在の状態を取得、オンコール DBA にレプリカの健全性を確認させ (Manual)、オーケストレータ API を呼び (HTTP)、「DNS 更新済み」をチェック (Manual)、Slack に「異常なし」を投稿 (HTTP)。
- **キャッシュフラッシュ** — HTTP ステップ 1 つと、ダッシュボードでキャッシュヒット率の回復を確認する Manual。
- **顧客影響のあるインシデント** — Manual: 「ステータスページのバナーを更新。」HTTP: 「#customer-incidents で CS チームに通知。」JavaScript: 「内部 API から影響顧客リストを取得。」
- **予定メンテナンスのプレフライト** — JavaScript: 現在のメトリクスのスナップショット。Manual: 「ステークホルダーに変更ウィンドウを確認。」HTTP: ロードバランサーをメンテナンスモードに。
- **常時実行する衛生ルール** — タイトルパターンが空のルールで、どんなインシデントでもシステムの状態を取得 — ポストモーテムに最適。

## 詳細な例

「db-primary」をタイトルに含むすべてのインシデントで、5 ステップの DB フェイルオーバー Runbook を自動で起動したいとします。

**1. Runbook を作成。** **Runbooks → Runbook を作成** で「DB primary failover」と名付け、以下のステップを追加します:

| # | タイプ | タイトル |
| --- | --- | --- |
| 1 | JavaScript | フェイルオーバー前のレプリカ遅延を取得 |
| 2 | Manual | DBA ダッシュボードでレプリカの健全性を確認 |
| 3 | HTTP | フェイルオーバーオーケストレータへ `POST` |
| 4 | Manual | 書き込みが新しい primary に向いていることを確認 |
| 5 | HTTP | Slack の `#db-incidents` に「異常なし」を投稿 |

**2. ルールを追加。** **インシデント → 設定 → Runbook ルール** で次を作成します:

```
タイトルパターン:  ^db-primary
Runbooks:           [DB primary failover]
```

**3. トリガー。** モニターアラートで `INC-4821 · db-primary connection timeout` が起票。ルールが一致し、実行が作成され:

- ステップ 1 (JavaScript) はワーカーで即実行され、`return { lagMs: 412 }` の値が取り込まれる。
- ステップ 2 (Manual) で実行は一時停止。オンコールはインシデントページの「あなた待ち」ピルを見てダッシュボードを確認し、ステップにチェックを入れる。
- ステップ 3 (HTTP) はステップ 2 のチェック直後に走り、`POST` レスポンス本文が取り込まれる。
- ステップ 4 (Manual) は再び一時停止。
- ステップ 5 (HTTP) が走り、実行は終了。

**4. 監査。** 実行はインシデントの **Runbooks** タブに残ります。各ステップの出力はワンクリックで開ける場所にあります。来週ポストモーテムを書くとき、「あのスクリプトは何を返した?」と尋ねる必要はありません — そこに書いてあります。

## OneUptime の他の部分との組み合わせ

- **モニター**はインシデントとアラートを起票し、**Runbook ルール**はそれらのイベントを Runbook 実行へ変換します。一緒で、検知 → トリガー → 対応 → 記録の閉じたループになります。
- **ワークスペース連携** (Slack、Microsoft Teams) は HTTP ステップの自然な投稿先で、ステータス更新やチャンネル通知に使えます。
- **ステータスページ**は顧客影響のある Runbook で Manual ステップとして更新されることが多いです。
- **オンコールスケジュール**は誰を呼ぶかを決め、Runbook は呼ばれた人が何をするかを決めます。
