# Runbook ルール

Runbook ルールは、**インシデント**、**アラート**、または**予定メンテナンスイベント**が作成されたときに Runbook を自動で添付します。各エンティティの設定メニューから管理します:

- インシデント → 設定 → **Runbook ルール**
- アラート → 設定 → **Runbook ルール**
- 予定メンテナンス → 設定 → **Runbook ルール**

3 つのページはすべて、同じルールモデルを編集します — エンティティ種別ごとにルールを絞り込んで表示しているだけです。

## ルールの構造

| フィールド | 用途 |
| --- | --- |
| **名前** | 短く人間向けのラベル。監査ログに表示される。 |
| **説明** | チームメイト向けの任意のコンテキスト。 |
| **有効化** | 削除せずルールを停止するためのトグル。 |
| **タイトルパターン** | エンティティのタイトルに対する大文字小文字を無視する正規表現。空 = どのタイトルにもマッチ。 |
| **説明パターン** | エンティティの説明に対する大文字小文字を無視する正規表現。空 = どの説明にもマッチ。 |
| **起動する Runbook** | ルールが発火したときに起動する 1 つ以上の Runbook。 |

## マッチの意味論

ルールは**指定された条件がすべて通る**ときにマッチします。空の条件はスキップされるので:

- パターンを設定していないルールは、その種別のすべてのイベントで動きます (グローバルな「常に動く」ルール)。
- タイトルパターンだけ設定したルールは、タイトルが正規表現にマッチするイベントで発火します。
- 同じイベントに複数のルールがマッチすることもあります — マッチしたルールはすべて発火し、それらの Runbook の和集合が動きます (各 Runbook はそれぞれ独自の実行になります)。

## 例: データベースインシデント向け DB フェイルオーバー

```
名前:             Start DB failover for DB incidents
トリガー:          インシデント
タイトルパターン:    (?:^|\b)(db|database|postgres|mysql|mongo)
Runbooks:         [DB failover playbook, Notify DBA team]
```

これは「db」「database」「postgres」などをタイトルに含むインシデントが作成されるたびに、2 つの Runbook 実行を作成します。

## 例: 常に動かす衛生チェックルール

```
名前:               Always-run pre-flight check
トリガー:            インシデント
タイトルパターン:      (空)
説明パターン:         (空)
Runbooks:           [Capture pre-incident state]
```

どのインシデントでも発火します — システム状態のスナップショット取得、ページメトリクスなどに便利。

## ルールが発火したときに起こること

1. Runbook が読み込まれます。
2. そのステップが新しい Runbook 実行に**スナップショット**されます。
3. 実行が Runbook キューワーカーへ投入されます。
4. 実行はソースエンティティにリンクされます — インシデント、アラート、または予定メンテナンスイベントのページと Runbook の実行リストに表示されます。

ルールによる実行はすべて **Runbooks → 実行** で、ステータス、Runbook、日付でフィルタして見ることができます。

## 無効化された Runbook

ルールが `isEnabled = false` の Runbook を参照していると、ルールはマッチしてもその Runbook の実行はスキップされます。Runbook を再度有効化すれば動くようになります。

## ルールのテスト

本番でルールに頼る前に、パターンにマッチするタイトルでテストインシデント (またはアラート) を作成して、期待した Runbook が発火するか確認してください。ルールは作成時に評価されます — 後でインシデントのタイトルを編集してもルールは再トリガーされません。
