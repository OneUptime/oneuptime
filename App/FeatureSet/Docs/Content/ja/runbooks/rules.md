# Runbook ルール

Runbook ルールは、**インシデント**、**アラート**、**予定メンテナンスイベント**が作成されたときに自動で Runbook を添付します。管理は各エンティティの設定メニューから行います:

- インシデント → 設定 → **Runbook ルール**
- アラート → 設定 → **Runbook ルール**
- 予定メンテナンス → 設定 → **Runbook ルール**

3 つのページは同じルールモデルを編集します — そのエンティティタイプに該当するルールだけが表示されるようフィルタされているだけです。

## ルールの構成

| フィールド | 目的 |
| --- | --- |
| **名前** | 短く読みやすいラベル。監査ログに表示。 |
| **説明** | チームメンバー向けの任意のコンテキスト。 |
| **有効** | ルールを削除せず一時停止するためのスイッチ。 |
| **タイトルパターン** | エンティティのタイトルに対する大小無視の正規表現。空 = すべてのタイトルに一致。 |
| **説明パターン** | エンティティの説明に対する大小無視の正規表現。空 = すべての説明に一致。 |
| **起動する Runbook** | ルール発火時に起動する 1 つ以上の Runbook。 |

## マッチング仕様

ルールは **指定したすべての条件が満たされたとき**に一致します。空の条件はスキップされます:

- パターンを設定していないルールは、そのタイプのすべてのイベントで実行されます (グローバル「常時実行」ルール)。
- タイトルパターンだけのルールは、タイトルが一致するイベントで発火します。
- 同じイベントに複数のルールが一致する場合、それぞれが発火し、それぞれの Runbook の和集合が走ります (各 Runbook はそれぞれの実行を持ちます)。

## 例: データベースインシデントの DB フェイルオーバー

```
名前:               DB インシデントで DB フェイルオーバーを開始
トリガー:           インシデント
タイトルパターン:    (?:^|\b)(db|database|postgres|mysql|mongo)
Runbook:            [DB フェイルオーバー Playbook, DBA チームに通知]
```

「db」「database」「postgres」などをタイトルに含むインシデントが作成されるたびに、2 つの Runbook 実行が作成されます。

## 例: 常時実行する衛生ルール

```
名前:                    すべてのインシデントでプリフライト
トリガー:                インシデント
タイトルパターン:        (空)
説明パターン:            (空)
Runbook:                 [インシデント前の状態を取得]
```

すべてのインシデントで発火 — システム状態、ページのメトリクス等の取得に便利です。

## ルールが発火するとどうなるか

1. Runbook が読み込まれる。
2. そのステップが新しい Runbook 実行へ **スナップショット** として複製される。
3. 実行が Runbook ワーカーキューに積まれる。
4. 実行はソースのエンティティに紐付けられ、インシデント、アラート、メンテナンスのページや Runbook の実行一覧に表示される。

ルールでトリガーされた実行はすべて **Runbooks → 実行** で、ステータス、Runbook、日付でフィルタできます。

## 無効化された Runbook

ルールが `isEnabled = false` の Runbook を参照している場合、ルールは依然として一致しますが実行はスキップされます。再度有効化すれば再開します。

## ルールのテスト

本番でルールに頼る前に、パターンに一致するタイトルでテストインシデント (またはアラート) を作成し、想定どおりの Runbook が発火することを確認します。ルールは作成時点で評価されます — 後でインシデントのタイトルを編集してもルールは再発火しません。
