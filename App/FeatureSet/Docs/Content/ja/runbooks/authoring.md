# Runbook を作成する

**Runbooks → Runbook を作成** で Runbook を作成し、開いて **ステップ** タブへ進みます。

## ステップの構成

各ステップには次のものがあります:

| フィールド | 目的 |
| --- | --- |
| **タイトル** | チェックリスト UI に表示される短いラベル。必須。 |
| **説明** | 対応者向けの任意のコンテキスト。Markdown 対応テキスト。 |
| **失敗時も継続** | オンの場合、失敗したステップでも実行を止めず、次のステップが走る。 |
| **タイプ固有の設定** | スクリプト、URL など — 下記参照。 |

ステップは**順番に**実行されます。ステップエディタの上下矢印で並び替えてください。

## ステップタイプ

### Manual

対応者がチェックを入れるチェックボックス。Manual ステップに達すると実行は一時停止し、誰かが完了とマークする (またはスキップする) まで `WaitingForManualStep` のままです。

人にしか確認できないことに使います: 「ロードバランサーのダッシュボードでセカンダリリージョンへトラフィックが移動したことを確認 — 完了。」

### JavaScript

`isolated-vm` サンドボックス内で動く JavaScript スニペット (ファイルシステムなし、API を持ち込まない限りネットワークなし) — ただしサンドボックスは OneUptime Worker ではなく、お客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) 上で動きます。ステップに **Agent Tag** を設定し、実行させたいエージェントを指定してください。そのタグを持つエージェントが 1 つもオンラインでない場合、ステップは **claim タイムアウト** (既定 2 分) まで待ってから失敗します。

```js
const start = Date.now();
// ... ロジック ...
return { durationMs: Date.now() - start };
```

返却値はステップ実行に取り込まれます。`console.log` 出力はログ行として記録されます。デフォルトタイムアウト: 30 秒。

### HTTP リクエスト

外向きの HTTP 呼び出し。メソッド (GET/POST/PUT/PATCH/DELETE/HEAD)、URL、任意の JSON ヘッダ、任意のボディを設定します。レスポンスのステータス、ヘッダ、ボディが取り込まれます (合計 50KB まで)。

PagerDuty のインシデント起票、Slack 投稿、自前の管理 API 呼び出しなどに便利です。

### Bash

[Runbook エージェント](/docs/runbooks/agents) 上で実行される bash スクリプト — お客様自身のインフラ内のホストにインストールする小さなプロセスです。Bash ステップが OneUptime Worker 上で実行されることはありません。

Bash ステップでは次の 2 つを設定します:

- **Agent Tag** — このステップを実行すべきエージェントを識別するタグ。プロジェクト内で当該タグを保有する任意の健全なエージェントがジョブを取得して実行します。
- **スクリプト** — 実行する bash。出力 (stdout + stderr) は 50 KB まで取り込まれ、タイムアウト時にプロセスは強制終了されます。

Runbook がこのステップに到達した時点で選択タグを持つエージェントが 1 つもオンラインでない場合、ステップは **claim タイムアウト** (既定 2 分) まで待機し、その後失敗します。Bash ステップに頼る前に **Runbooks → Agents** でエージェントを追加してください。

## 保存と編集

**ステップを保存** で永続化します。古いバージョンの Runbook を実行中のものは影響を受けません — 自身のスナップショットで走り続けます。

## 複数ステップと失敗ハンドリング

既定では、失敗したステップは実行を止め、実行を `Failed` にします。ステップに **失敗時も継続** を設定すると、失敗は記録しつつ次のステップが実行されます。「これら 3 つを試し、その後通知」のようなパターンに有用です。

## 詳細な例

「primary DB に到達不可」のシンプルな Runbook:

1. **JavaScript** — 設定サービスから現在の primary ホストを取得してログに残す。
2. **Manual** — 「セカンダリのレプリケーション遅延 5 秒未満 — 確認済み。」
3. **HTTP リクエスト** — フェイルオーバーオーケストレータの API に POST。
4. **Manual** — 「書き込みが新しい primary に向いている — 確認済み。」
5. **HTTP リクエスト** — Slack へ「異常なし」を POST。

対応者は自動ステップが走るのを見守り、Manual を 1 つチェックし、次の自動ステップを見守る、を繰り返します。各ステップの出力はポストモーテム用に保存されます。
