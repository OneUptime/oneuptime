# Runbook を作成する

**Runbooks → Runbook を作成** で Runbook を作り、開いて **ステップ** タブへ進みます。

## ステップの構造

すべてのステップには次のものがあります:

| フィールド | 用途 |
| --- | --- |
| **タイトル** | チェックリスト UI に表示される短いラベル。必須。 |
| **説明** | 対応者向けの任意のコンテキスト。Markdown 対応のテキスト。 |
| **失敗時に続行** | オンなら、失敗したステップで実行が止まらず、次のステップが実行される。 |
| **承認を必須にする** | オンなら、このステップ後に Runbook が一時停止し、ユーザーが承認するまで次のステップに進まない。 |
| **タイプ固有の設定** | スクリプト、URL、エージェントなど — 下記参照。 |

ステップは**順番に**実行されます。ステップエディタの上下矢印で並び替えできます。

## ステップタイプ

### Manual

対応者がチェックを入れるチェックボックス。Runbook の実行が Manual ステップに達すると一時停止し、誰かが完了 (またはスキップ) のマークを付けるまで `WaitingForManualStep` のままです。

人だけが確認できることに使います:「ロードバランサーのダッシュボードでトラフィックがセカンダリリージョンに移ったことを確認した。」

### JavaScript

サンドボックス化された `isolated-vm` 内で実行される JavaScript スニペット。サンドボックスはお客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) 上にあります — OneUptime ワーカー上ではありません。

JavaScript ステップでは 2 つを設定します:

- **Runbook エージェント** — このステップを実行すべきエージェントをドロップダウンから選択。選択されたエージェントだけがジョブを取得できる。
- **スクリプト** — 実行する JavaScript。

```js
const start = Date.now();
// ... your logic ...
return { durationMs: Date.now() - start };
```

返り値はステップ実行に記録されます。`console.log` 出力はログ行として記録されます。デフォルトの実行タイムアウト: 30 秒。デフォルトの取得タイムアウト (エージェントがジョブを取得するまでワーカーが待つ時間): 2 分。

### HTTP リクエスト

アウトバウンド HTTP 呼び出しを行います。メソッド (GET/POST/PUT/PATCH/DELETE/HEAD)、URL、任意の JSON ヘッダー、任意のボディを設定します。レスポンスのステータス、ヘッダー、ボディが記録されます (合計 50KB で打ち切り)。

便利な使い道: PagerDuty のインシデント起動、Slack への投稿、自社の管理 API 呼び出しなど。HTTP ステップは OneUptime ワーカー上で直接動きます。エージェントは不要です。

### Bash

bash スクリプト (`bash -c <script>`) で、お客様自身のインフラ内の [Runbook エージェント](/docs/runbooks/agents) 上で実行されます。Bash が OneUptime ワーカー上で実行されることはありません。

Bash ステップでは 2 つを設定します:

- **Runbook エージェント** — このステップを実行すべきエージェントをドロップダウンから選択。選択されたエージェントだけがジョブを取得できる。
- **スクリプト** — 実行する bash。出力 (stdout + stderr) は 50 KB まで記録され、タイムアウトでプロセスは kill される。

Runbook がこのステップに達したときに選択されたエージェントがオフラインだと、ステップは**取得タイムアウト** (デフォルト 2 分) まで待ってから `TimedOut` で失敗します。Bash ステップに頼る前に **Runbooks → 設定 → エージェント** でエージェントを追加してください。

## 保存と編集

**ステップを保存** を押して永続化します。古いバージョンの Runbook の進行中の実行には影響しません — スナップショットを使い続けます。

## 複数ステップと失敗のハンドリング

デフォルトでは、失敗したステップが実行を止め、実行は `Failed` とマークされます。**失敗時に続行** をステップに設定すると失敗は記録されつつ次のステップが動きます。「3 つ試してから通知」のようなパターンに便利です。

## 具体例

「DB プライマリ到達不能」のシンプルな Runbook:

1. **JavaScript** — 設定サービスから現在のプライマリホストを取得してログ出力。
2. **Manual** — 「セカンダリのレプリケーション遅延が 5 秒未満であることを確認。」
3. **HTTP リクエスト** — フェイルオーバーオーケストレーターの API に POST。
4. **Manual** — 「書き込みが新プライマリに向いていることを検証。」
5. **HTTP リクエスト** — Slack に「復旧」メッセージを POST。

対応者は自動ステップが動くのを見て、手動ステップにチェックを入れて、次の自動ステップを見て、と繰り返します。各ステップの出力はポストモーテム用に記録されます。
