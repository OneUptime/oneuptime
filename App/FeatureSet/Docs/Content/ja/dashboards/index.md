# ダッシュボード 概要

ダッシュボードは、OneUptime がすでに収集しているテレメトリー — メトリクス、ログ、トレース、インシデント、モニター、Kubernetes と Docker のリソース — を、誰かが一目で見てシステムの健全性を理解できる 1 枚のページに変える方法です。

リクエストレイテンシーのチャートを、開いているインシデントのリストの隣に、CPU 利用率のゲージの隣に、平易な英語のステータス文の隣に置きます。保存します。リンクを共有します。

## ひと目で

- OneUptime ダッシュボードの **Dashboards** にある**トップレベル機能**。
- **グリッドベースのキャンバス** — デフォルトは横 12 ユニット × 縦 60 ユニット。ウィジェットをドラッグで追加、リサイズ、グリッドにスナップ。
- **20 以上のウィジェットタイプ** — チャート、単一値、ゲージ、テーブル、テキストブロック、ログストリーム、トレースリスト、そしてインシデント、アラート、モニター、Kubernetes (Pod、ノード、デプロイメント、…)、Docker、ホスト用のライブリソースリスト。
- **変数とフィルター** — 1 つのダッシュボードをすべてのクラスター、サービス、顧客、環境に再利用できるテンプレートビューに変えます。
- **公開共有** — スイッチを入れるとダッシュボードが公開 URL で到達可能になります。任意でパスワード保護や IP アローリストもかけられます。
- **カスタムドメイン** — OneUptime のドメインではなく `status.your-domain.com` で公開ダッシュボードをホストします。

## なぜダッシュボードを使うのか?

次のいずれかが当てはまるとき、ダッシュボードはその価値を発揮します:

- **「全部 OK?」と尋ねるページ**が必要なとき — オンコールのローテーション、チームのスタンドアップ、壁掛けテレビの前を通る CEO のために。
- **シグナルを相関させたい**とき — トレースのレイテンシー増加と開いているインシデントと同じ分の CPU スパイクは、3 つのタブを行き来するより 1 つのダッシュボードでずっと明白です。
- **調査している**とき — デバッグセッション中に組み立てる自由形式のダッシュボードは、10 個のクエリを手で実行するより速いです。
- **外部に公開している**とき — 顧客向けのパフォーマンスダッシュボード、パートナー向けのロールアップ、オープンソースサービスの公開ヘルスボード。

## キー概念

| 用語 | 意味 |
| --- | --- |
| **ダッシュボード** | キャンバス。ウィジェットのリスト、時間範囲コントロール、変数セットを持つ名前付きの再利用可能なビュー。 |
| **ウィジェット** | キャンバス上の 1 つのコンポーネント — チャート、値、テーブル、テキストブロック、リスト。それぞれにタイプと JSON 形式の設定があります。 |
| **ダッシュボードユニット** | グリッドの 1 マス。ウィジェットはダッシュボードユニット単位でサイズ指定されます (例: 「横 4 × 縦 6」)。ユニットはビューポートに基づいてピクセルに変換されます。 |
| **変数** | 閲覧者がドロップダウンから選ぶ (またはタイプする) 名前付きの値で、ダッシュボードはそれをすべてのウィジェットのクエリに注入します。クラスター、サービス、顧客、環境 — フィルタしたい何でも。 |
| **時間範囲** | すべてのウィジェットがクエリする時間ウィンドウ。プリセット (「過去 24 時間」) かカスタム範囲を選びます。 |
| **更新間隔** | **View** モードでウィジェットが再クエリする頻度。Off、5s、10s、30s、1m、5m、15m。 |
| **モード** | `Edit` (ドラッグ、リサイズ、設定) または `View` (読み取り専用)。両者は同じキャンバスを共有します。 |

## ウィジェットカタログ

ダッシュボードに置けるものの非網羅的なマップ:

| カテゴリ | ウィジェット |
| --- | --- |
| **時系列** | Chart |
| **単一の数値** | Value、Gauge |
| **表形式** | Table |
| **注釈** | Text |
| **ログとトレース** | LogStream、TraceList |
| **運用リスト** | IncidentList、AlertList、MonitorList |
| **Kubernetes** | KubernetesPodList、KubernetesNodeList、KubernetesNamespaceList、KubernetesDeploymentList、KubernetesStatefulSetList、KubernetesDaemonSetList、KubernetesJobList、KubernetesCronJobList |
| **Docker** | DockerHostList、DockerContainerList、DockerImageList、DockerNetworkList、DockerVolumeList |
| **インフラ** | HostList |

各ウィジェットの引数といつ使うべきかについては[ウィジェット](/docs/dashboards/widgets) を参照してください。

## ダッシュボードのどこにあるか

| ページ | そこで行うこと |
| --- | --- |
| **Dashboards** | ダッシュボードを閲覧・作成・検索・ラベル付けする。 |
| ダッシュボードの **View** | キャンバス — 作成者には Edit モード、他の全員には View モード。ヘッダーで切り替え可能。 |
| ダッシュボードの **Overview** | 説明、所有者、ラベル。 |
| ダッシュボードの **Settings** | 公開共有、マスターパスワード、IP アローリスト、カスタムドメイン、ブランディング (ページタイトル、説明、ロゴ、ファビコン)。 |
| ダッシュボードの **Owners** | 明示的な所有権を持つユーザーとチーム。 |
| ダッシュボードの **Delete** | ダッシュボードを削除 (取り消し不可)。 |

## ダッシュボードのライフサイクル

1. **作成** — **Dashboards → Create Dashboard** で名前を付けます。キャンバスは空の状態で開きます。
2. **ウィジェットをドロップ** — ウィジェットパレットからタイプを選び、ソース (メトリクスクエリ、リストフィルター、自由なテキスト本文) を設定。位置とサイズを決める。
3. **(任意) 変数を追加** — 同じダッシュボードを各値に対してレンダーするために `cluster` や `service` のようなドロップダウンを定義。
4. **時間範囲と更新間隔を設定** — デフォルトで十分動きます。後で調整してください。
5. **(任意) 公開共有** — **Settings** で **Public Dashboard** をオン。ゲートが欲しければマスターパスワードを追加、または IP で制限。
6. **(任意) カスタムドメイン** — `dashboard.your-domain.com` レコードを追加して DNS を検証し、自社の URL でダッシュボードを提供。

## 具体例

ゴール: チェックアウトサービスのオンコールページ。レイテンシー、エラー率、開いているインシデント、最近のログテイル。

1. 「Checkout oncall」というダッシュボードを作成。
2. 属性キー `service.name` にバインドされた **Telemetry Attribute** タイプの `service` 変数を追加。デフォルト値 `checkout`。
3. **Chart** ウィジェットを追加: APM メトリクスから `service.name = {{service}}` でフィルタした P95 レイテンシー。時間範囲はダッシュボードに従う。
4. その隣に **Value** ウィジェットを追加: 警告閾値 1%、危険閾値 5% のエラー率パーセンテージ。
5. 下に、`checkout` を含むラベルでフィルタした **IncidentList** ウィジェットを追加。
6. その下に、`service.name = {{service}}` でフィルタした **LogStream** ウィジェットを追加。
7. 保存。変数ドロップダウンを `payments` に変更 — ダッシュボード全体が payments サービス用に再レンダーされます。同じテンプレート、違うフィルター。

## OneUptime の他機能との関わり

- **モニターとテレメトリー**はダッシュボードに生データを供給します — 設定したすべてのメトリクス、取り込まれたすべてのログ行、すべてのトレーススパンがウィジェットでクエリ可能です。
- **インシデントとアラート**は **IncidentList** と **AlertList** ウィジェットに現れます — ダッシュボードはそれらに対する読み取り専用のビューです。エンティティの作成/編集は別の場所で行います。
- **ステータスページ**は顧客向けのコミュニケーションツール (「システムは今動いていますか?」) です。ダッシュボードは分析ツール (「システムは詳細にはどう振る舞っていますか?」) です。両者は補完的で代替ではありません。
- **ワークフロー**は OneUptime の書き込み側で、ダッシュボードは読み取り側です。
