# ダッシュボード 設定とアクセス権

このページは、保持しておきたいダッシュボードを手にしたら知っておくべき設定とアクセス制御のつまみをまとめています。

## 所有権

ダッシュボードの**所有者**は、明示的に権限を付与されたユーザーとチームです (プロジェクト全体のロールとは別)。

**Dashboard → Owners** で:

- 特定の人にこのダッシュボードへの追加アクセスを付与するために**ユーザー所有者**を追加。
- 同じことをチームのすべてのメンバーに付与するために**チーム所有者**を追加。

所有権を使うのは、プロジェクト全体の読み取りロールが広すぎるとき — 例えば、カスタマーサクセスチームだけが見るべき機密の顧客レベルの詳細を持つダッシュボード。

## ラベル

ラベルはダッシュボードを整理するための多対多のタグです。**Dashboard → Overview** で適用します。

よくあるラベルパターン:

- **チーム別**: `team:platform`、`team:checkout`、`team:growth`。
- **環境別**: `env:prod`、`env:staging`。
- **目的別**: `purpose:oncall`、`purpose:exec`、`purpose:investigation`。

**Dashboards** リストはラベルでフィルタできます。これは何十ものダッシュボードを蓄えたプロジェクトでダッシュボードを見つける最速の方法です。

## 権限

ダッシュボードは OneUptime のロールベースアクセス制御における第一級リソースです。関連する権限:

| 権限 | 許可される操作 |
| --- | --- |
| `CreateDashboard` | プロジェクトで新しいダッシュボードを作成。 |
| `ReadDashboard` | ダッシュボードを表示 (プライベートモード)。 |
| `EditDashboard` | ダッシュボードのウィジェット、変数、設定を変更。 |
| `DeleteDashboard` | ダッシュボードを削除。 |

サポートエンティティに対応する権限もあります: ダッシュボード所有者 (ユーザー / チーム) とカスタムドメインには独自の create / read / edit / delete ペアがあり、「ダッシュボード自体を編集」を付与せずに「所有者を管理」を付与できます。

これらを **Project Settings → Teams & Roles** でプロジェクトロールに割り当てます。

## 公開モードのアクセス制御

公開モードのアクセス ([共有と公開ダッシュボード](/docs/dashboards/sharing) を参照) は次の 3 つの層で順に統制されます:

1. **Public Dashboard** トグル — オフなら公開 URL は 404 を返します。
2. **Master Password** — 設定されていれば、訪問者はダッシュボードがレンダーされる前に入力する必要があります。
3. **IP Whitelist** (Scale プラン) — 設定されていれば、リストにない IP からのリクエストは 403 を受けます。

ダッシュボードは任意の組み合わせを持てます。最も防御的な構成は「Public オン、パスワード設定、IP アローリスト有効」 — 3 つすべてが欲しいパートナーポータルに便利です。

## 保持

ダッシュボード自体は期限切れになりません。表示されるデータはプロジェクトのテレメトリー保持に従います — メトリクス、ログ、トレースは、プランが保持する期間だけクエリ可能です。30 日保持のプランで「過去 90 日」を指すウィジェットは、ストアに残っているものを何でもレンダーします。

## ダッシュボードを複製する

既存のダッシュボードを複製するには、それを開き、ダッシュボードリストから **Duplicate** アクションを使います。コピーは公開モードの設定を除くすべてのウィジェット、変数、設定を含みます (公開モードは常にオフから始まる — コピーで再有効化するかを決めます)。

これは、テンプレート (「自チームのオンコールダッシュボード」) をサービス固有のバージョンにフォークするときの正しいパターンです。

## ダッシュボードを削除する

**Dashboard → Delete** で行います。これは取り消し不可です — キャンバス構成とカスタムドメインバインディングは削除されます。テレメトリーデータは影響を受けません (ダッシュボードではなくメトリクス / ログ / トレースストアにあります)。

ダッシュボードがカスタムドメインで公開されている場合、削除した瞬間に公開 URL の解決が止まります。再ポイントが必要なら、先にドメインを外してください。

## 移行とバックアップ

セルフホストのインストールの場合: ダッシュボードの完全な構成 (ウィジェット、変数、設定) は Postgres の `Dashboard` テーブルに存在します。通常のデータベースバックアップで十分です — 別のダッシュボードエクスポート形式はありません。

OneUptime Cloud の場合: 通常のバックアップは代行されます。ダッシュボード構成のローカルコピーが欲しい場合は、[OneUptime API](/docs/api-reference/api-reference) を使って `Dashboard` レコードを読んでください。
