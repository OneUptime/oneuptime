# ダッシュボード 変数とフィルター

変数は単一のダッシュボードをテンプレートに変えます。`service` 変数を定義すれば、同じチャートが `checkout`、`payments`、`search` 用に再レンダーされます — ほぼ同一の 3 つのダッシュボードを構築する代わりに、上部のドロップダウンから選びます。

このページは 4 つの変数タイプ、その値がウィジェットクエリにどう注入されるか、そしてその隣に座るグローバルな時間範囲と更新コントロールを扱います。

## 変数タイプ

**Dashboard → Settings → Variables** で変数を追加します。各変数には名前 (ウィジェットクエリで `{{name}}` として参照)、任意のラベル、タイプがあります。

### Custom List

静的なドロップダウン。カンマ区切りの値リストを供給し、閲覧者が 1 つを選びます。

これを使うのは: 選択肢の集合が小さく、固定で、自チームにのみ意味があるとき。値 `prod, staging, dev` を持つ `environment`。値 `us-east-1, eu-west-1, ap-south-1` を持つ `region`。

### Query

ドロップダウンの選択肢はレンダー時に ClickHouse クエリで計算されます。

これを使うのは: 選択肢が動的でテレメトリー内に存在するとき。「過去 24 時間にログインしたすべての顧客 ID」を `SELECT DISTINCT customer_id FROM ...` で。クエリはプロジェクトのデータに対して実行されます。自分のデータでも、結果は信頼できない入力として扱ってください。

### Text Input

自由テキストフィールド。閲覧者がタイプしたものが何でも注入されます。

これを使うのは: ダッシュボードを検索ツールのように振る舞わせたいとき。「IP でフィルター」や「リクエスト ID でフィルター」ダッシュボード。

### Telemetry Attribute

選択肢は、ダッシュボードの時間範囲にわたるプロジェクトのテレメトリーにおける OpenTelemetry 属性キーの個別の値です。

**属性キー** (例: `k8s.cluster.name`、`service.name`、`host.name`) を設定します。ウィジェットはログ / メトリクス / トレースから個別の値を取得し、ドロップダウンとして提供します。

これを使うのは: 選択肢がまさにテレメトリーにすでにタグ付けしているエンティティであるとき。クラスター名、サービス名、リージョン、顧客 ID、デプロイメント環境 — OpenTelemetry のリソースまたはスパン属性として送っている何か。

これは自動更新するため、サービス指向のダッシュボードで最もよく使われる変数タイプです: `service.name = inventory` でタグ付けされた新しいサービスを出荷すると、誰もダッシュボードを編集することなくその値がドロップダウンに現れます。

## 複数選択

各変数を**複数選択**として設定できます。オンにすると、閲覧者は 1 つ以上の値を選び、ダッシュボードは `value = ...` の代わりに `value IN (...)` でフィルターします。

複数選択を使うのは: ダッシュボードを離れずに「checkout と payments を一緒に」見たいとき。選択された値間でチャートの数学が成り立たないとき (例: 平均の平均) は避けてください。

## デフォルト値

各変数は任意のデフォルトを取ります。閲覧者がドロップダウンを変更するまで、ダッシュボードはデフォルトでレンダーされます。公開ダッシュボードでは、デフォルトが訪問者が最初に着地する状態です。

## 補間の仕組み

ウィジェットクエリが文字列フィルターを取る場所ならどこでも — メトリクスクエリの `WHERE` 句、リストウィジェットのフィルター、ログストリームの属性マッチ — `{{variable_name}}` を参照できます。

たとえば、Chart のメトリクスクエリは次のようになります:

```
SELECT avg(latency_ms) FROM spans WHERE service.name = '{{service}}'
```

`service` が `checkout` に設定されると、クエリは `service.name = 'checkout'` で実行されます。閲覧者が `payments` に切り替えると、クエリは `service.name = 'payments'` で再実行されます。

**Telemetry Attribute** 変数については特に、OneUptime は属性キーを知っているので、同じ属性に言及するすべてのウィジェットにフィルターを注入します — 変数が変わるたびに各ウィジェットのクエリを手で編集する必要はありません。これがサービステンプレート化されたダッシュボードが箱から出してすぐ動く魔法です。

## 時間範囲

ダッシュボードヘッダーにはグローバルな**時間範囲**ピッカーがあります。すべてのメトリクスウィジェットはこのウィンドウに対してクエリします。選択肢:

- **プリセット** — 過去 1 時間、24 時間、7 日、30 日、90 日 (保持期間による)。
- **カスタム範囲** — 開始と終了のタイムスタンプを選ぶ。

時間範囲はダッシュボードの URL の一部です — URL を共有するとウィンドウも共有されます。これはインシデント中に便利です: 時間範囲を「本日 10:00–10:30 UTC」に固定して、インシデントチャンネルでリンクを共有します。

## 更新間隔

時間範囲の隣で、ウィジェットが再クエリする頻度を選びます:

- **Off** — ウィジェットは読み込み時に一度クエリします。
- **5s / 10s / 30s / 1m / 5m / 15m** — 自動更新。

自動更新は壁掛けスクリーンや進行中のインシデントビューに便利です。アドホックな調査には、ビューがスクロール中に安定するようオフにしてください。

## まとめると

サービステンプレート化されたダッシュボードは通常:

1. `service.name` にバインドされた **Telemetry Attribute** タイプの `service` 変数。デフォルト: 最も見ているサービス。複数選択: オフ (チャートは常に 1 サービスを表示)。
2. **Custom List** タイプの `environment` 変数。デフォルト: `prod`。
3. `k8s.cluster.name` にバインドされた **Telemetry Attribute** タイプの `cluster` 変数。複数選択: オン (クラスター横断でロールアップできる)。
4. ダッシュボードのウィジェットはフィルターでこれらの変数を参照します。

結果: 1 つのダッシュボード、フリート全体のカバレッジ、上部にいくつかのドロップダウン。
