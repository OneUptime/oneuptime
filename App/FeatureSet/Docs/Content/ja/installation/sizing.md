# サイジングとキャパシティプランニング

このガイドは、Kubernetes (Helm) 上でセルフホストの OneUptime デプロイメントをサイジングするのに役立ちます。OneUptime が依存する 3 つのデータストア — **PostgreSQL**、**Redis**、**ClickHouse** — に加えてアプリケーションのコンピュートを取り上げ、実際の数値が得られた後に調整できる出発点となる階層を示します。

> **まずこれを読んでください:** Helm チャートには **CPU/メモリのリクエストやリミットが設定されておらず**、PostgreSQL と ClickHouse 用に小さな **25 Gi** のデフォルトボリュームが付属しています。これらのデフォルトは、チャートがあらゆるクラスターでインストールおよび実行できるように存在しています — これらは本番環境向けのサイジングでは **ありません**。簡単な試用を超えるものについては、以下の数値を使用してリソースとストレージを明示的に設定してください。

代わりにシングルサーバーの Docker Compose インストールを実行している場合、サイジングはよりシンプルです — [Docker Compose](/docs/installation/docker-compose) を参照してください (推奨: 16 GB RAM、8 コア、400 GB ディスク)。

## 各データストアを左右する要因

OneUptime は本番環境で 3 つのデータストアを必要とします。これらはまったく異なる入力に応じてスケールするため、独立してサイジングしてください。

| データストア   | 保存する内容                                                                                                                  | サイズを左右する要因                                                                                                      |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **ClickHouse** | すべてのテレメトリー — ログ、メトリクス、トレース、例外、プロファイル                                                         | テレメトリーの **取り込みレート × 保持期間**。これがストレージの約 95% を占め、最大のコストとなります。                   |
| **PostgreSQL** | 構成と状態 — モニター、インシデント、アラート、ユーザー、チーム、プロジェクト、ワークフロー、ステータスページ、ダッシュボード | テレメトリー量ではなく **エンティティ数と履歴**。緩やかに増加します。                                                     |
| **Redis**      | キャッシュ、作業キュー、セッション                                                                                            | **キューの深さとアクティブなセッション**。メモリに依存し、控えめです。信頼できる情報源 (source of truth) ではありません。 |

オブジェクトストレージ (S3/MinIO) は OneUptime の実行に **必須ではありません**。これはデータベースの **バックアップ** のためにオプションで使用されるだけです (PostgreSQL 用の CloudNativePG Barman プラグイン経由、または ClickHouse 用の `clickhouse-backup` 経由)。OneUptime はテレメトリーをオブジェクトストレージに階層化しません — 以下の「保持期間とそれがストレージに与える影響」セクションを参照してください。

## ClickHouse — 最大の要因

ストレージのほぼすべてと RAM の大部分は ClickHouse に割り当てられます。なぜなら、すべてのログ行、メトリクスポイント、トレーススパン、例外がそこに存在するからです。

### ストレージの計算式

```
ClickHouse disk ≈ (daily raw telemetry GB ÷ compression) × retention days × replicas × 1.3 (headroom)
```

圧縮率はシグナルによって異なります:

- **ログ** はよく圧縮されます — おおよそ **5:1**。
- **メトリクス** は圧縮率が低く — おおよそ **2:1** — 高いラベル **カーディナリティ** は生のボリュームよりも速くディスクと RAM の両方を膨張させます。ラベルは低カーディナリティに保ってください。
- **トレース** はスパン属性に応じて、その中間に位置します。

### 計算例

**10 クラスター** のフリート (それぞれ約 10 ノード / 約 100 ポッド、INFO レベルの冗長性) は、**30 日間でクラスターあたりおよそ 50–150 GB の生ログ** (≈ クラスターあたり 1.7–5 GB/day) を生成します。フリート全体では、メトリクスとトレースを加えて圧縮した後、**5–15 GB/day の圧縮済みテレメトリー** をおおよその予算とします。

| 保持期間 | シングルレプリカ | 2 レプリカ + 30% ヘッドルーム |
| -------- | ---------------- | ----------------------------- |
| 30日     | ~150–450 GB      | **~0.4–1.2 TB**               |
| 90日     | ~0.45–1.35 TB    | **~1.2–3.5 TB**               |

ストレージは **保持期間に比例して線形に** スケールします — 90 日間のウィンドウは 30 日間のウィンドウの約 3 倍のコストがかかります。

### RAM とディスクタイプ

- **NVMe/SSD を使用してください。** テレメトリーは書き込み負荷が高く、バースト的な集計読み取りを伴います。回転ディスク上の ClickHouse は苦戦します。
- **ClickHouse に潤沢な RAM を割り当ててください。** 集計クエリはメモリを大量に消費します。経験則として、RAM は _ホット_ な (最近クエリされた) 圧縮データセットの意味のある割合 (25–50%) にサイジングし、実際の本番フリートには実用上の下限として 16 GB を設定してください。
- **メトリクスのカーディナリティを管理してください。** これは ClickHouse の RAM とディスクの両方に対する最大のレバーです。収集レイヤーで低カーディナリティのラベル規約を徹底し、アクティブなシリーズ数を監視してください。

## PostgreSQL — 構成と状態

PostgreSQL はテレメトリーではなく構成と運用状態を保存するため、緩やかに増加し、ClickHouse に対して小さいままです。大規模なデプロイメントでも通常は数十 GB の範囲です。デフォルトの **25 Gi** ボリュームは小規模なインストールには十分です。より大規模なものには、インシデント/アラート履歴のヘッドルームを含めて 50–100 GB を計画してください。

多数のアプリケーション、ワーカー、プローブのレプリカを実行する場合、データベース接続数がストレージよりも先にボトルネックになることがあります。OneUptime の Helm チャートには、まさにこのためのオプションの **PgBouncer** 接続プーラー (`pgbouncer.enabled`) が含まれています — 高レプリカのデプロイメントでは有効にしてください。

## Redis — キャッシュ、キュー、セッション

Redis はキャッシュ、作業キュー、セッションストアとして使用されます。これは **メモリに依存** し、永続化は **デフォルトで無効** になっています (ここでの Redis は信頼できる情報源ではありません — 再構築可能です)。予想されるキューの深さと同時セッションでサイジングしてください。2–8 GB のメモリでほとんどのデプロイメントをカバーできます。デフォルトの退避ポリシーは `noeviction` であることに注意してください。そのため、持続的な過負荷下でキューが滞留する場合は、Redis のメモリを監視してください。

## アプリケーションのコンピュート

データストアに加えて、ステートレスなワークロード (イングレス、Web/API、ワーカー、プローブ) をサイジングしてください。すべてリソースリミットなしの **1 レプリカ** がデフォルトです — 明示的に設定してください。チャートには **KEDA** がバンドルされており、ワーカーとプローブがキューの深さに応じてオートスケールできます。可変負荷の場合は有効にしてください。ワーカーはテレメトリー/取り込み処理のボリュームに応じてスケールし、プローブはアクティブなモニターの数に応じてスケールします。

## 出発点となる階層

自分の環境に最も近い階層を出発点として選び、その後実際の使用状況 (`kubectl top pods`、ClickHouse/Postgres のディスク増加) を監視して調整してください。

- **Small / PoC** — 1–3 クラスター、≤30 ノード、≤5 GB/day の生テレメトリー、30 日間の保持。
- **Medium / Production fleet** — 約 10 クラスター、約 100 ノード、10–30 GB/day の生テレメトリー、30–90 日間の保持。
- **Large / Multi-fleet** — 50 以上のクラスター、500 以上のノード、100 GB/day 以上の生テレメトリー、90 日間の保持。

|                      | Small / PoC                  | Medium / Production fleet    | Large / Multi-fleet                              |
| -------------------- | ---------------------------- | ---------------------------- | ------------------------------------------------ |
| **ClickHouse**       | 4 vCPU / 16 GB / 200 GB NVMe | 8 vCPU / 32 GB / 1–3 TB NVMe | 16+ vCPU / 64–128 GB / 5–15 TB NVMe, **sharded** |
| **PostgreSQL**       | 2 vCPU / 4 GB / 50 GB SSD    | 4 vCPU / 8 GB / 100 GB SSD   | 8 vCPU / 16–32 GB / 250 GB SSD (+ PgBouncer)     |
| **Redis**            | 1 vCPU / 2 GB                | 2 vCPU / 4 GB                | 4 vCPU / 8–16 GB                                 |
| **想定する保持期間** | 30日                         | 30–90日                      | 90日                                             |

これらは OneUptime の **バックエンド** をサイジングします。監視対象の各クラスターで実行される OneUptime コレクターは別途サイジングされます — [Kubernetes Agent](/docs/telemetry/kubernetes-agent) のサイジング階層を参照してください。

## 高可用性

チャートの組み込みデータストアは、デフォルトで **シングルインスタンス** として実行されます。本番環境の HA のために:

- **PostgreSQL** — 自動フェイルオーバーのために、バンドルされた [CloudNativePG](https://cloudnative-pg.io) オペレーター (`postgresOperator.cnpg.enabled`) を **3 インスタンス** (1 プライマリ + 2 ホットスタンバイ) で有効にしてください。
- **ClickHouse** — バンドルされた [Altinity](https://github.com/Altinity/clickhouse-operator) オペレーター (`clickhouseOperator.altinity.enabled`) を **シャードあたり ≥2 レプリカ** と、クォーラムのための **3 つの ClickHouse Keeper** ノードで有効にしてください。単一ノードのディスクまたは RAM が制限になったら、シャードを追加してください。
- **Redis** — チャートにはチャート内レプリケーションがありません。HA のためには、OneUptime を **外部のマネージド Redis** (または Sentinel/クラスターデプロイメント) に向けてください。

## 保持期間とそれがストレージに与える影響

テレメトリーの保持期間は、**プロジェクトごと** に設定され、**シグナルごと** (ログ、メトリクス、トレース、プロファイル) およびバケットごと (例えばログの重大度別) に細かく調整できる **日数で設定された ClickHouse TTL** として強制されます。ハードコードされたデフォルトは 15 日です。

保持期間は ClickHouse のストレージを直接的に倍増させるため、ディスクをサイジングする前に決定してください。OneUptime は古いテレメトリーをオブジェクトストレージに自動的にアーカイブまたは階層化 **しません** — 複数年にわたるコンプライアンス保持のためには、保持ウィンドウを延長し、それに合わせて ClickHouse のストレージをサイジングしてください (または、選択した外部アーカイブにエクスポートしてください)。

## コミットする前に計測する

テレメトリーのボリュームは、アプリケーションのログの冗長性、名前空間数、スクレイプ間隔、そしてどこかで DEBUG ログが有効になっているかどうかによって大きく異なります。上記の階層は出発点として扱ってください: **少なくとも 4 週間環境を計測** し、シグナルごとの実際の GB/day を測定してから、実データから保持期間とストレージをサイジングしてください。
