# 설정 및 보안

이 페이지는 워크플로우를 프로덕션 트래픽에 향하게 하기 전에 알아둘 만한 설정과 안전 한도를 모은 것입니다.

## 활성화 / 비활성화

모든 워크플로우는 **Settings**에 **isEnabled** 플래그를 가집니다. 비활성 워크플로우는 절대 발동되지 않습니다 — 모델 이벤트, 웹훅, 스케줄된 실행이 무시됩니다. 새 워크플로우는 비활성 상태로 출시됩니다.

이를 "프로덕션 준비 완료" 스위치로 취급하세요:

1. 워크플로우를 만듭니다.
2. 대표적인 페이로드로 **Run Manually**를 클릭합니다.
3. **Logs**를 확인 — 모든 노드가 기대한 포트를 탔는지 확인하세요.
4. **isEnabled**를 켭니다.

워크플로우를 비활성화해도 이미 진행 중인 실행에는 영향이 없습니다; 새 실행이 생성되지 않을 뿐입니다.

## 소유권과 라벨

- **Owners** — 소유자로 나열된 사용자와 팀은 권한 기반 접근과 (선택적으로) 워크플로우 실패 시 알림을 받습니다. **Settings → Owners**에서 설정합니다.
- **Labels** — 워크플로우 정리를 위한 다대다 태그. 라벨로 워크플로우 목록을 필터링합니다. 프로젝트가 팀, 통합, 환경별로 정리된 수십 개의 워크플로우를 가질 때 유용합니다.
- **Label rules** — **워크플로우 → Settings → Label Rules**에서 이름이나 설명에 대한 정규식 일치에 기반해 새 워크플로우에 라벨을 자동 적용합니다.
- **Owner rules** — **워크플로우 → Settings → Owner Rules**에서 새 워크플로우에 소유자를 자동 할당합니다.

## 비밀

글로벌 변수는 **secret**으로 표시할 수 있습니다. 값은 저장 시 암호화되고, 저장 후 UI에서 쓰기 전용이며, 실행 로그에서 가려집니다(`[REDACTED]`로 대체).

비밀 변수를 사용하는 경우:

- 아웃바운드 통합용 API 키.
- 베어러 토큰.
- 웹훅 서명 키.
- 워크플로우에 대한 읽기 권한을 가진 공격자가 보아서는 안 되는 모든 값.

비밀을 구성 요소의 인자에 직접 붙여 넣지 마세요 — `Authorization: Bearer eyJh...` 같은 참조는 워크플로우 JSON과 실행 로그에 평문으로 나타납니다. 대신 `{{variable.MY_SECRET}}`을 참조하세요.

## 실행 타임아웃

각 실행은 최대 소요 시간을 가집니다. 실행이 타임아웃 내에 끝나지 않으면 `Timeout`으로 표시되고 진행 중인 구성 요소는 취소됩니다. 기본값은 넉넉합니다(초가 아닌 분) — 정확한 값은 워커의 환경 설정에서 확인하세요.

대부분의 구성 요소는 실행 타임아웃 안에 자체 호출별 타임아웃을 가집니다 — 예를 들어, API 구성 요소는 전체 실행이 끝나기 훨씬 전에 멈춰 있는 아웃바운드 요청을 포기합니다.

## 재귀 한도

**Execute Workflow** 구성 요소는 한 워크플로우가 다른 워크플로우를 호출하게 합니다. A가 B를 호출하고 B가 다시 A를 무한히 호출하는 폭주를 방지하기 위해, 워커는 호출 체인을 추적하고 고정된 깊이(보통 5 같은 작은 수)를 초과하는 체인을 중지시킵니다. 종료된 실행은 재귀 한도에 대한 명확한 메시지와 함께 `Error`로 표시됩니다.

긴 체인에 대한 정당한 필요가 있다면(예: 실행당 한 레벨씩 처리하는 재귀 폴더 워크), **Custom Code**로 내부적으로 반복하는 단일 워크플로우로 리팩터링하세요 — 그 패턴은 체인 한도의 적용을 받지 않습니다.

## 웹훅 보안

웹훅 트리거는 고유 HTTPS URL을 노출합니다. URL을 알게 된 누구든 그것을 칠 수 있습니다. 우발적이거나 적대적인 호출자로부터 방어하려면:

- URL을 공유 비밀로 취급하세요. 공개 채팅에 붙여 넣거나 공개 저장소에 커밋하지 마세요.
- 가치가 높은 워크플로우의 경우, 호출 시스템이 헤더에 공유 비밀(예: `X-Webhook-Token`)을 포함하도록 요청하고, 파괴적 작업을 하기 전에 **Conditions** 노드에서 검증하세요. 기대 토큰을 비밀 글로벌 변수로 정의하세요.
- 매우 가치가 높은 워크플로우의 경우, 공개 웹훅 대신 모델 이벤트 트리거와 수동 임포트 단계를 선호하세요.

## 아웃바운드 네트워크 이그레스

API와 다른 HTTP 스타일 구성 요소는 OneUptime 워크플로우 워커의 네트워크에서 요청을 보냅니다. OneUptime을 셀프 호스팅한다면, 워커의 아웃바운드 네트워크는 당신의 책임입니다 — 호출하는 서드파티 API에 도달할 수 있는지 확인하세요. OneUptime Cloud를 사용한다면, 우리의 IP 이그레스 범위는 [IP Addresses](/docs/configuration/ip-addresses)에 게시되어 있으므로 수신 측에서 허용 목록에 추가할 수 있습니다.

## 권한

워크플로우는 프로젝트 단위 역할 기반 접근 제어의 대상이 되는 일급 리소스입니다:

- `CreateWorkflow`, `ReadWorkflow`, `EditWorkflow`, `DeleteWorkflow` — 워크플로우 템플릿에 대한 네 가지 CRUD 권한.
- `RunWorkflow` — **Run Manually**를 클릭하거나 API를 통해 워크플로우를 디스패치하는 데 필요.
- `ReadWorkflowLog` — **실행 및 로그** 페이지를 보는 데 필요.
- `ReadWorkflowVariable`, `CreateWorkflowVariable`, `EditWorkflowVariable`, `DeleteWorkflowVariable` — 글로벌 변수 목록에 대한 제어.

대부분의 엔지니어는 워크플로우에 대한 create/edit/read는 가져야 하지만 변수는 갖지 않아야 합니다. 변수 편집 접근은 프로젝트의 비밀을 관리하는 사람에게 예약하세요.

## 쿼터

OneUptime Cloud는 작은 플랜에서 프로젝트당 월 실행 수에 상한을 둡니다. 상한은 **Project Settings → Billing**에 표시됩니다. 도달하면, 새 트리거는 거부되고(영향받은 워크플로우에 "quota exceeded" 사유로 기록됨) 다음 청구 주기까지 거부됩니다. 셀프 호스팅 설치는 쿼터의 적용을 받지 않습니다.

## 워크플로우가 *잘하지 못하는* 것

다른 도구가 필요한 몇 가지 패턴:

- **오래 걸리는 연산** — 워크플로우는 시스템 사이의 연결을 중심으로 합니다. 큰 데이터셋을 가공하기 위한 것이 아닙니다. 무거운 작업은 자체 인프라에서 실행하고 워크플로우를 사용해 그것을 시작하세요.
- **분/시간에 걸친 상태 유지 워크플로우** — 한 번의 실행은 빠르게 끝나야 합니다. "작업 A를 하고 두 시간 기다린 다음 작업 B를 하라"가 필요하다면, 대기를 웹훅 트리거에 다시 게시하는 외부 스케줄러로 모델링하세요.
- **사람의 체크포인트가 있는 단계별 인시던트 대응** — 그것이 [Runbook](/docs/runbooks/index)의 용도입니다. 루프에 사람이 없으면 워크플로우를 사용하고, 있으면 Runbook을 사용하세요.
