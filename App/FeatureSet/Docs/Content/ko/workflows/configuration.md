# 구성 및 안전

이 페이지는 실제 트래픽으로 워크플로를 운영하기 전에 알아두면 유용한 설정과 안전 한도를 다룹니다.

## 워크플로 켜기 및 끄기

모든 워크플로에는 **Settings**에 **Enabled** 스위치가 있습니다. 꺼져 있으면 워크플로가 실행되지 않습니다. webhook 호출, 예정된 시간, OneUptime 이벤트가 모두 무시됩니다. 새 워크플로는 비활성화 상태로 시작합니다.

이 스위치를 "출시 준비" 게이트로 활용하시기 바랍니다.

1. 워크플로를 빌드합니다.
2. 실제와 유사한 페이로드로 **Run Manually**를 클릭합니다.
3. **Logs**를 확인합니다. 모든 블록이 예상한 경로로 갔는지 확인합니다.
4. **Enabled**를 켭니다.

워크플로를 끈다고 해서 이미 진행 중인 실행이 멈추지는 않으며, 새 실행이 시작되지 않을 뿐입니다.

## 소유자 및 라벨

- **소유자(Owners)** — 소유자로 지정된 사용자와 팀은 워크플로에 접근할 수 있으며, 실패 시 알림 수신을 선택할 수 있습니다. **Settings → Owners**에서 설정합니다.
- **라벨(Labels)** — 워크플로를 그룹화하기 위한 태그입니다. 워크플로 목록에서 라벨로 필터링할 수 있어 바쁜 프로젝트를 훨씬 쉽게 탐색할 수 있습니다. 팀, 통합, 환경별로 워크플로를 정리할 때 유용합니다.
- **라벨 규칙(Label rules)** — **Workflows → Settings → Label Rules**에서 이름이나 설명 패턴에 따라 새 워크플로에 라벨을 자동으로 적용합니다.
- **소유자 규칙(Owner rules)** — **Workflows → Settings → Owner Rules**에서 새 워크플로에 자동으로 소유자를 할당합니다.

## 시크릿

전역 변수에 민감한 정보가 포함되어 있다면 **시크릿(secret)**으로 표시하시기 바랍니다. 값은 암호화되고, 저장 후 UI에서 숨겨지며, 실행 로그에서도 숨겨집니다(`[REDACTED]`로 표시).

다음과 같은 경우 시크릿 변수를 사용하시기 바랍니다.

- 외부 서비스의 API 키.
- 인증 토큰.
- Webhook 서명 키.
- 읽기 전용 권한을 가진 사람이 보지 않았으면 하는 모든 정보.

시크릿을 블록에 직접 붙여넣지 마시기 바랍니다. `Authorization: Bearer eyJh...`와 같은 값은 워크플로와 로그에 그대로 노출됩니다. 대신 `{{variable.MY_SECRET}}`을 사용하시기 바랍니다.

## 실행에 걸리는 최대 시간

각 실행에는 최대 시간이 있습니다. 실행이 시간 내에 완료되지 않으면 **Timeout**으로 표시되고 진행 중인 블록이 취소됩니다. 기본값은 충분히 여유로워, 정상적인 HTTP 호출과 블록 체인에 충분한 시간입니다.

개별 블록은 그 안에서 자체 시간 제한을 가집니다. 예를 들어, API 블록은 전체 실행이 끝나기 훨씬 전에 응답이 없는 아웃바운드 요청을 포기합니다.

## 다른 워크플로 호출 한도

**Execute Workflow** 컴포넌트를 사용하면 한 워크플로가 다른 워크플로를 호출할 수 있습니다. 워크플로 A가 B를 호출하고 B가 다시 A를 호출하는 식의 의도치 않은 무한 루프를 방지하기 위해, 체인이 얼마나 깊어질 수 있는지에 제한이 있습니다. 한도를 넘는 실행은 명확한 오류로 종료됩니다.

긴 체인이 실제로 필요한 경우(예: 실행당 항목 하나를 처리하는 작업), **Custom Code**를 사용해 단일 워크플로 내부에서 반복하는 것이 더 간단할 수 있습니다.

## Webhook 보안

Webhook 트리거는 고유한 URL을 제공합니다. URL을 아는 사람은 누구나 호출할 수 있습니다. 의도하지 않거나 원치 않는 호출자로부터 보호하려면 다음을 고려하시기 바랍니다.

- URL을 비밀번호처럼 취급하세요. 공개적으로 공유하거나 공개 리포지토리에 커밋하지 마시기 바랍니다.
- 민감한 워크플로의 경우, 호출 시스템에 헤더(예: `X-Webhook-Token`)로 공유 토큰을 보내도록 요청하고, 중요한 작업을 수행하기 전에 **Conditions** 블록으로 확인하세요. 예상되는 토큰을 시크릿 변수로 저장하시기 바랍니다.
- 매우 민감한 워크플로의 경우 공개 webhook 대신 OneUptime 이벤트 트리거와 수동 가져오기 단계를 사용하는 것을 권장합니다.

## 아웃바운드 네트워크 접근

API 및 기타 HTTP 블록은 OneUptime에서 요청을 보냅니다. 자체 호스팅을 사용한다면 설치 환경이 호출 대상 서비스에 도달할 수 있는지 확인하시기 바랍니다. OneUptime Cloud를 사용하신다면 [IP 주소](/docs/configuration/ip-addresses)에 아웃바운드 IP 범위가 나와 있으므로 다른 쪽에서 허용할 수 있습니다.

## 권한

워크플로는 프로젝트의 역할 기반 액세스 제어를 따릅니다. 관련 권한은 다음과 같습니다.

- **Create / Read / Edit / Delete Workflow** — 워크플로 자체에 대한 기본 권한.
- **Run Workflow** — **Run Manually**를 클릭하거나 API를 통해 워크플로를 트리거하는 데 필요합니다.
- **Read Workflow Log** — 실행을 보는 데 필요합니다.
- **Read / Create / Edit / Delete Workflow Variable** — 전역 변수 목록에 대한 제어.

대부분의 엔지니어는 워크플로에 대한 생성/편집/읽기 권한은 가지되 변수에는 권한이 없어야 합니다. 변수 편집 권한은 프로젝트의 시크릿을 관리하는 사람만 가지도록 하시기 바랍니다.

## 요금제 한도

OneUptime Cloud는 더 작은 요금제에서 월별 실행 횟수에 한도를 둡니다. 현재 한도는 **Project Settings → Billing**에 표시됩니다. 한도에 도달하면 다음 청구 주기까지 새 트리거가 거부됩니다. 자체 호스팅 설치는 이런 한도가 없습니다.

## 워크플로가 적합하지 않은 경우

워크플로 대신 다른 도구를 사용해야 하는 경우는 다음과 같습니다.

- **무거운 연산이나 큰 데이터셋** — 워크플로는 가벼운 글루 작업을 위해 설계되었으며 숫자 처리용은 아닙니다. 무거운 작업은 자체 인프라에서 실행하고 워크플로가 시작만 하도록 하시기 바랍니다.
- **몇 시간에 걸친 장기 실행 프로세스** — 단일 실행은 빠르게 완료되도록 설계되었습니다. "A를 수행하고 두 시간 기다린 후 B를 수행"하는 경우라면, 외부 스케줄러가 적절한 시점에 OneUptime에 webhook을 다시 보내는 방식을 사용하시기 바랍니다.
- **사람의 개입이 포함된 단계별 인시던트 대응** — [런북](/docs/runbooks/index)의 용도입니다. 워크플로는 무인 자동화를 위한 것입니다.
