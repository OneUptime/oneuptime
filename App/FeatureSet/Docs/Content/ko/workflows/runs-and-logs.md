# 실행 및 로그

워크플로가 실행될 때마다 OneUptime은 무슨 일이 일어났는지 기록을 저장합니다. 언제 실행되었는지, 정상 동작했는지, 각 블록이 무엇을 했는지 등이 저장됩니다. 이 기록을 **실행(run)**이라고 합니다. 실행 기록을 통해 워크플로가 정상 동작했는지 확인하고, 실패한 워크플로를 디버깅하며, 과거 활동을 되돌아볼 수 있습니다.

## 어디에서 찾나요

| 페이지                      | 보이는 내용                                                                                   |
| --------------------------- | --------------------------------------------------------------------------------------------- |
| **Workflows → Runs & Logs** | 프로젝트의 모든 워크플로에서 발생한 모든 실행. 워크플로, 상태, 시간으로 필터링할 수 있습니다. |
| **Workflow → Logs 탭**      | 이 워크플로 하나의 실행만.                                                                    |
| **단일 실행**               | 한 번의 실행과 모든 블록의 출력.                                                              |

## 실행 상태

| 상태          | 의미                                                                                                                                                            |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scheduled** | 트리거가 발생했고 실행이 곧 시작됩니다. 보통 1초도 걸리지 않습니다.                                                                                             |
| **Running**   | 워크플로가 진행 중입니다. 오래 실행되는 블록은 실행을 이 상태로 유지합니다.                                                                                     |
| **Success**   | 실행된 모든 블록이 오류 없이 완료되었습니다. (의도적으로 **error** 분기를 사용한 경우에도 success로 간주됩니다. 워크플로 자체가 실패한 것은 아니기 때문입니다.) |
| **Error**     | 블록이 실패했고, 이를 처리할 **error** 경로가 연결되어 있지 않았습니다. 실행이 거기서 중단되었습니다.                                                           |
| **Timeout**   | 실행이 허용된 시간보다 오래 진행되었습니다. [구성 및 안전](/docs/workflows/configuration)을 참조하시기 바랍니다.                                                |

## 실행 읽기

실행을 클릭하면 세부 정보가 열립니다. 다음을 확인할 수 있습니다.

- **헤더** — 트리거, 시작 및 종료 시간, 총 소요 시간, 상태.
- **블록 목록** — 실행된 모든 블록을 순서대로 표시합니다. 각각에 대해 받은 값, 출력, 선택한 경로가 표시됩니다.
- **오류** — 블록이 실패한 경우 오류 메시지와(가능한 경우) 추가 세부 정보가 표시됩니다.

표시되는 값은 변수가 모두 채워진 후 블록이 실제로 본 값입니다. 이것은 가장 유용한 디버깅 뷰입니다. Slack 메시지에 실제 제목 대신 `{{Incident.title}}`이라는 문자가 그대로 나타난다면 변수가 해석되지 않았음을 알 수 있습니다.

## 일반적인 디버깅

### "워크플로가 실행되지 않았어요."

1. Settings에서 워크플로가 **활성화(enabled)** 되어 있는지 확인하세요. 새 워크플로는 비활성화 상태로 시작합니다.
2. OneUptime 이벤트 트리거의 경우 이벤트가 실제로 발생했는지 확인합니다. 레코드를 열고 이력을 확인하세요.
3. webhook 트리거의 경우 다른 시스템이 올바른 URL로 보내고 있는지 확인합니다. 대부분의 도구는 webhook을 보낼 때 기록을 남기므로 거기서 확인할 수 있습니다.
4. 스케줄 트리거의 경우 cron 표현식이 예상한 시간과 일치하는지 확인합니다.

트리거는 발생했지만 실행이 나타나지 않는 경우, **Project Settings → Billing**에서 실행 할당량을 확인하시기 바랍니다.

### "이후 블록이 실행되지 않았어요."

블록이 실행되지 않는 것은 대개 연결 문제입니다. 캔버스를 열고 확인하시기 바랍니다.

- 이전 블록의 출력이 이 블록의 입력에 연결되어 있나요?
- 이전 블록이 예상과 다른 출력을 사용했나요? (예: **success** 대신 **error** 또는 **Yes** 대신 **No**) 실행 세부 정보에 선택한 경로가 표시됩니다.

### "변수가 비어 있게 들어왔어요."

실행을 열고 실패한 블록의 값을 확인하시기 바랍니다.

- 그대로 `{{BlockName.field}}` 텍스트가 보인다면 참조가 해석되지 않은 것입니다. 블록 이름이나 필드 이름의 오타일 가능성이 큽니다.
- 빈 문자열이 보인다면 이전 블록은 실행되었지만 해당 필드를 생성하지 않은 것입니다.

### "수동으로 실행하면 동작하는데, 트리거에서는 안 돼요."

실제 트리거가 보내는 것과 유사한 JSON 페이로드로 **Run Manually**를 사용하시기 바랍니다. 그런 다음 수동 실행의 값과 실제 실행의 값을 나란히 비교합니다. 차이는 보통 필드 이름 하나 또는 타입 하나입니다.

## 워크플로 다시 실행하기

"이 실행 다시 시도하기" 버튼은 없습니다. 부작용(Slack 메시지, API 호출, 티켓 등)이 반복해도 안전하지 않을 수 있기 때문에 과거 실행을 자동으로 다시 실행하지 않습니다. 작업을 다시 수행하려면 워크플로를 수정하고 다음 실제 트리거가 발생하도록 하시기 바랍니다.

수동 워크플로의 경우 같은 페이로드로 **Run Manually**를 다시 클릭하시면 됩니다.

## 실행은 얼마나 오래 보관되나요?

실행은 프로젝트에 무기한으로 보관됩니다. 워크플로가 매우 자주 실행되어 이력을 어지럽힌다면(예: 1분마다 실행되는 디버그 워크플로) 비활성화하거나 삭제하여 노이즈를 줄이시기 바랍니다.
