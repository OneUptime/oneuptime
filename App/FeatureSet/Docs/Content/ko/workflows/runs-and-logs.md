# 실행 및 로그

워크플로우의 트리거가 발동될 때마다, OneUptime은 **실행**을 만듭니다 — 타이밍, 상태, 노드별 출력을 가진 한 번의 실행 기록입니다. 실행은 워크플로우가 작동했음을 확인하는 방법이자, 작동하지 않았을 때 디버깅하는 방법이며, 자동화가 잘못 동작했을 때 포스트모템을 쓰는 방법입니다.

## 어디서 찾는가

| 페이지 | 범위 |
| --- | --- |
| **워크플로우 → 실행 및 로그** | 프로젝트 전체. 모든 워크플로우의 모든 실행. 워크플로우, 상태, 시간 범위로 필터. |
| 워크플로우의 **Logs 탭** | 이 워크플로우의 실행만. |
| 실행의 상세 페이지 | 노드별 출력과 오류 메시지로 확장된 한 번의 실행. |

## 실행 상태

| 상태 | 의미 |
| --- | --- |
| **Scheduled** | 트리거가 발동되어 실행이 큐에 들어갔지만, 워커가 아직 가져가지 않음. 보통 1초의 일부. |
| **Running** | 워커가 현재 그래프를 따라가는 중. 오래 걸리는 구성 요소(느린 HTTP 호출, 의도적 지연)는 실행을 이 상태로 유지합니다. |
| **Success** | 실행된 모든 노드가 오류 없이 끝남. (의도적으로 `error` 분기를 탄 워크플로우도 전체적으로는 여전히 `Success`입니다 — 워크플로우 자체는 실패하지 않았습니다.) |
| **Error** | 노드가 실패하고 처리할 `error` 포트가 배선되지 않았습니다. 실행은 그 노드에서 멈췄습니다. |
| **Timeout** | 실행이 실행당 타임아웃을 초과했습니다. [설정 및 보안](/docs/workflows/configuration)을 참고하세요. |

## 실행 읽기

목록에서 실행을 클릭하여 상세 페이지를 엽니다. 다음이 보입니다:

- **헤더** — 발동된 트리거, 시작과 끝 타임스탬프, 총 소요 시간, 상태.
- **노드 목록** — 순서대로 실행된 모든 노드, 각각의 캡처된 인자, 반환값, 선택된 출력 포트.
- **오류** — 노드가 실패했다면, 오류 메시지와 (가능한 경우) 스택 트레이스.

캡처된 인자는 *보간 후* 값을 보여줍니다 — 즉, 변수가 해결된 후 노드가 본 정확한 문자열입니다. 이것이 단연 가장 유용한 디버깅 뷰입니다: Slack 메시지에 리터럴 텍스트 `{{Incident.title}}`이 있다면, 변수 참조가 해결되지 않았음을 알 수 있습니다.

## 흔한 디버깅 패턴

### "내 워크플로우가 발동되지 않았어요."

1. **Settings**에서 워크플로우가 **활성화**되어 있는지 확인. 새 워크플로우는 비활성 상태로 출시됩니다.
2. 모델 이벤트 트리거의 경우: 이벤트가 실제로 일어났는지 확인. 엔티티(인시던트, 알람, 모니터)를 열어 그 이력을 보세요.
3. 웹훅 트리거의 경우: 외부 시스템이 올바른 URL에 접근하는지 확인. 많은 도구가 아웃바운드 웹훅 전송을 로깅합니다 — 거기를 확인하세요.
4. 스케줄 트리거의 경우: cron 표현식이 기대하는 시간에 평가되는지 확인. 의심스러우면 cron 파서를 사용하세요.

트리거가 발동되었지만 실행이 나타나지 않으면, **Project Settings → Billing**에서 프로젝트의 실행 쿼터를 확인하세요.

### "실행되지만 하위 노드가 절대 실행되지 않아요."

실행되지 않는 노드는 보통 배선 문제입니다. 캔버스를 열고 확인하세요:

- 상위 노드의 출력 포트가 실제로 이 노드의 입력 포트에 연결되어 있나요?
- 상위 노드가 다른 포트를 탔나요(예: `success` 대신 `error`, 또는 `yes` 대신 `no`)? 실행 상세를 보아 어떤 포트가 선택되었는지 확인하세요.

### "변수가 빈 값으로 전달돼요."

실행 상세를 열고 실패한 노드의 캡처된 인자를 보세요. 리터럴 `{{NodeId.field}}` 텍스트가 보이면, 참조가 해결되지 않았습니다 — 아마 `NodeId` 또는 `field`의 오타일 것입니다. 빈 문자열이 보이면, 상위 노드는 실행되었지만 그 필드를 생산하지 않았습니다.

### "수동으로는 작동하지만 트리거에서는 안 돼요."

실제 트리거가 게시하는 것을 미러링하는 JSON 페이로드로 **Run Manually**를 사용하세요. 그런 다음 수동 실행과 프로덕션 실행의 캡처된 인자를 나란히 비교하세요 — 차이는 보통 단일 필드 이름이나 타입에 있습니다.

## 워크플로우 재실행

"이 실행을 다시 시도" 버튼은 없습니다 — 설계상, OneUptime은 오래된 실행을 다시 실행하지 않습니다. 아웃바운드 부작용(Slack 메시지, API 호출)이 idempotent하지 않을 수 있기 때문입니다. 작업을 다시 하고 싶다면, 워크플로우를 고치고 다음 실제 트리거가 발동하게 하세요.

수동 워크플로우의 경우, 같은 페이로드로 **Run Manually**를 클릭하세요.

## 로그 보존

실행은 프로젝트에서 무기한 보관됩니다. 대량의 노이즈 워크플로우(예: 매분마다 발동하는 디버그 워크플로우)를 정리해야 한다면, 비활성화하거나 삭제하세요 — 워크플로우별 보존 토글은 없습니다.
