# 트리거

트리거는 워크플로의 첫 번째 블록으로, 워크플로가 언제 실행될지 결정합니다. 모든 워크플로에는 정확히 하나의 트리거가 있습니다. 네 가지 종류 중에서 선택할 수 있습니다.

## Manual

워크플로 페이지에서 **Run Manually**를 클릭하여 필요할 때 워크플로를 실행합니다. 나머지 워크플로가 읽을 수 있는 JSON 페이로드를 붙여넣을 수도 있습니다.

용도: "키 회전" 또는 "테스트 알림 전송"처럼 버튼 하나로 실행하고 싶은 원클릭 자동화에 적합합니다.

**출력**: 붙여넣은 JSON, 또는 입력하지 않은 경우 빈 객체.

## Schedule

cron 표현식을 사용해 반복되는 스케줄에 따라 워크플로를 실행합니다.

용도: 야간 정리, 시간별 동기화, 주간 리포트 등.

**설정**: cron 표현식. 자주 사용되는 몇 가지 예시는 다음과 같습니다.

- `0 * * * *` — 매시 정각.
- `*/5 * * * *` — 5분마다.
- `0 9 * * 1` — 매주 월요일 오전 9시.

시스템이 잠시 사용 불가 상태가 되더라도 복구되는 즉시 실행이 이어집니다. 짧은 장애로 인한 누락은 걱정하지 않으셔도 됩니다.

## Webhook

OneUptime이 고유한 URL을 생성합니다. 해당 URL에 요청이 들어오면 워크플로가 시작됩니다. 요청의 헤더, 쿼리 파라미터, 본문이 모두 전달됩니다.

용도: 다른 도구에서 OneUptime으로 데이터를 받을 때 사용합니다. CI/CD 콜백, 다른 모니터링 도구의 알림, CRM의 가입 이벤트 등에 적합합니다.

**출력**:

- **Request Headers** — 들어오는 요청의 모든 헤더.
- **Request Query Params** — 파싱된 쿼리 문자열.
- **Request Body** — 파싱된 본문 (JSON이 아닌 경우 원시 텍스트).

URL은 `GET`과 `POST`를 모두 허용합니다. 호출자는 빠른 확인 응답을 받으며, 워크플로 자체는 백그라운드에서 실행됩니다.

URL을 비밀번호처럼 취급하세요. URL을 가진 사람은 누구나 워크플로를 시작할 수 있습니다.

## OneUptime 이벤트 트리거

OneUptime의 거의 모든 요소(모니터, 인시던트, 알림, 예정된 점검, 상태 페이지, 온콜 정책, 팀 등)는 워크플로를 트리거할 수 있습니다. 각 요소는 세 가지 이벤트를 제공합니다.

- **On Create** — 새로 추가될 때 발생합니다.
- **On Update** — 변경될 때 발생합니다.
- **On Delete** — 삭제될 때 발생합니다.

이렇게 하면 반복문으로 확인할 필요 없이 "OneUptime에서 X가 일어나면 Y를 수행"하는 패턴을 구성할 수 있습니다.

전체 레코드가 다음 블록으로 전달됩니다. 예를 들어, **Incident → On Create** 트리거는 새로 생성된 인시던트를 전달하므로 다음 블록에서 제목, 설명, 심각도 등 모든 필드를 읽을 수 있습니다.

### 팀에서 가장 많이 사용하는 이벤트

- **Incident** — 인시던트가 열리거나, 업데이트(확인, 해결)되거나, 삭제될 때 반응합니다.
- **Alert** — 알림에 대해 동일한 세 가지 이벤트가 있습니다.
- **Monitor** — 모니터가 추가, 편집, 제거될 때 반응합니다.
- **Scheduled Maintenance** — 점검 일정이 잡힐 때 자동으로 공지합니다.
- **Status Page Subscriber** — 상태 페이지를 구독한 사람을 환영합니다.
- **On-Call Duty Policy** — 일정 변경을 다른 명부 시스템과 동기화합니다.

원하는 트리거를 이름으로 검색하여 팔레트에서 찾을 수 있습니다.

## 어떤 트리거를 사용해야 할까요?

| 원하는 작업 | 선택 |
| --- | --- |
| 버튼을 클릭해 워크플로 실행 | **Manual** |
| 반복되는 스케줄로 실행 | **Schedule** |
| 다른 시스템이 데이터를 푸시 | **Webhook** |
| OneUptime 내부 이벤트에 반응 | **OneUptime 이벤트** |

워크플로는 트리거를 하나만 가질 수 있습니다. 같은 자동화를 두 가지 방식으로 시작하고 싶다면, 공통 로직을 하나의 워크플로에 만들고 **Execute Workflow** 컴포넌트를 사용하는 두 개의 얇은 "래퍼" 워크플로에서 호출하면 됩니다.
