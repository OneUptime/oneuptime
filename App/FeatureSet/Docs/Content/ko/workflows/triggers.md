# 트리거

트리거는 워크플로우의 시작 노드입니다. 입력 포트가 없습니다 — 실행이 여기서 시작됩니다. OneUptime은 네 가지 트리거 계열을 지원하며, 모든 워크플로우는 정확히 하나를 사용합니다.

## 수동(Manual)

워크플로우 페이지에서 **Run Manually**를 클릭해 온디맨드로 워크플로우를 실행합니다. 워크플로우가 `{{Manual.JSON}}`으로 읽을 수 있는 선택적 JSON 페이로드를 붙여 넣을 수 있습니다.

자동화 한 조각을 발동하는 버튼을 원할 때 사용합니다 — 반복 스케줄이나 발동 이벤트가 필요 없는 원클릭 "온콜 키 회전" 또는 "검색 인덱스 재구성" 워크플로우.

**인자**: 없음.

**반환값**:

| 이름 | 타입 | 설명 |
| --- | --- | --- |
| `JSON` | JSON | 실행 시 제공된 JSON 페이로드, 또는 빈 객체. |

## 스케줄(Schedule)

cron 스케줄로 워크플로우를 실행합니다. 표준 cron 표현식으로 주기를 설정합니다.

반복 작업에 사용합니다: 야간 정리, 시간별 동기화, 주간 익스포트.

**인자**:

| 이름 | 타입 | 설명 |
| --- | --- | --- |
| `Schedule at` | CronTab | 표준 5-필드 cron 표현식. 예를 들어 `0 * * * *`은 매시 정각, `*/5 * * * *`은 5분마다 실행됩니다. |

**반환값**:

| 이름 | 타입 | 설명 |
| --- | --- | --- |
| `executedAt` | Date | 예정된 실행 시각. |

스케줄된 워크플로우는 프로젝트 리전의 워크플로우 워커에서 실행됩니다. 워커가 잠시 사용 불가능해도, 복구되면 실행이 디스패치됩니다 — 짧은 중단에 대해 누락된 틱을 방어할 필요는 없습니다.

## 웹훅(Webhook)

외부 시스템이 `POST`할 수 있는 고유 HTTPS URL을 노출합니다. 요청 헤더, 쿼리 파라미터, 본문이 하위 구성 요소가 읽을 수 있는 반환값으로 노출됩니다.

OneUptime *안으로* 서드파티 시스템에서 데이터를 받는 데 사용합니다: CI/CD 콜백, 다른 모니터링 도구의 알림, CRM의 고객 가입.

**인자**: 없음. URL은 워크플로우 저장 시 자동으로 할당되어 트리거 노드에 표시됩니다. 비밀처럼 다루세요 — URL을 가진 사람은 누구나 워크플로우를 발동할 수 있습니다.

**반환값**:

| 이름 | 타입 | 설명 |
| --- | --- | --- |
| `Request Headers` | JSON | 들어오는 HTTP 요청의 모든 헤더. |
| `Request Query Params` | JSON | 파싱된 쿼리 문자열. |
| `Request Body` | JSON | 파싱된 요청 본문. 본문이 유효한 JSON이 아니면 `raw` 키 아래 문자열로 도착합니다. |

웹훅은 `GET`과 `POST`를 받습니다. 호출자에 대한 응답은 실행이 큐에 들어가는 즉시 JSON 확인과 함께 `200 OK`입니다 — 워크플로우 자체는 비동기로 실행되므로, HTTP 응답에서 하위 구성 요소의 결과를 읽으리라 기대하지 마세요.

## 모델 이벤트 트리거

거의 모든 OneUptime 엔티티 — 모니터, 인시던트, 알람, 예정 유지보수 이벤트, 상태 페이지, 온콜 정책, 팀, 텔레메트리 서비스 등 — 가 세 가지 트리거를 노출합니다:

- **On Create** — 이 타입의 새 레코드가 생성될 때 발동.
- **On Update** — 기존 레코드가 변경될 때 발동. 트리거는 이전 값과 새 값 모두 노출합니다.
- **On Delete** — 레코드가 삭제될 때 발동.

이것이 폴링 없이 "OneUptime에서 X가 일어나면 Y를 한다" 자동화를 만드는 방법입니다.

모델 자체는 리소스에서 보는 같은 필드 이름으로 반환값으로 노출됩니다. 예를 들어, **Incident → On Create** 트리거는 전체 `Incident` 객체를 반환하여 하위 노드가 `{{Incident.title}}`, `{{Incident.description}}`, `{{Incident.incidentSeverityId}}` 등을 읽을 수 있게 합니다.

**인자**: create/delete의 경우 일반적으로 없음. Update 트리거는 외관상 변경에 발동하지 않도록 반응할 필드를 좁힐 수 있게 해줄 수 있습니다.

**반환값**(모델에 따라 다름):

| 이름 | 타입 | 설명 |
| --- | --- | --- |
| 모델 필드 | (다양) | 엔티티의 모든 컬럼 — 이름, 상태, 타임스탬프, 외래 키. |
| `previous` (Update 전용) | JSON | 변경 전 상태의 레코드. |

### 자주 쓰는 모델 트리거

팀이 가장 자주 손에 닿는 모델 이벤트의 비포괄적 목록:

- **Incident** — `On Create`, `On Update`(Acknowledged 또는 Resolved 같은 상태 변경에 반응), `On Delete`.
- **Alert** — 알람 모델에 대한 같은 세 가지 이벤트.
- **Monitor** — 모니터가 추가/편집/제거될 때 반응; 조건과 결합해 프로덕션 모니터에만 동작.
- **Scheduled Maintenance** — 유지보수 창이 생성되거나 상태가 바뀔 때 하위 안내를 자동화.
- **Status Page Subscriber** — 누군가 구독할 때 환영 플로우를 발동.
- **On-Call Duty Policy** — 일정 변경을 외부 명부로 동기화.

OneUptime API에 노출된 모델이라면 거의 확실히 워크플로우를 트리거할 수 있습니다 — 엔티티 이름으로 트리거 팔레트를 검색하세요.

## 알맞은 트리거 고르기

| 원하는 것 | 사용 |
| --- | --- |
| 누군가 클릭하는 버튼을 워크플로우에 만들기 | **수동** |
| N분/시간/일마다 작업 실행 | **스케줄** |
| 외부 시스템이 OneUptime으로 데이터 푸시 | **웹훅** |
| OneUptime *내부에서* 일어나는 일에 반응 | **모델 이벤트** |

워크플로우는 트리거를 하나만 가질 수 있습니다. 같은 로직 대부분을 공유하는 두 개의 다른 시작 신호가 필요하다면, 공유 단계를 하나의 워크플로우로 인수하고 **Execute Workflow** 구성 요소를 사용하여 두 개의 얇은 "wrapper" 워크플로우에서 호출하세요([구성 요소](/docs/workflows/components) 참고).
