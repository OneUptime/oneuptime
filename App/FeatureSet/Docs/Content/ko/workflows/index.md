# 워크플로우 개요

워크플로우는 OneUptime의 시각적 자동화 빌더입니다. 트리거를 캔버스에 끌어다 놓고, 일련의 액션 — HTTP 호출, Slack 메시지, JavaScript 스니펫, 조건 분기, 데이터베이스 조회 — 에 연결하면, OneUptime(또는 외부 세계)에서 이벤트가 발생할 때마다 실행되는 자동화가 완성됩니다.

Runbook이 인시던트 동안 사람을 위한 체크리스트라면, 워크플로우는 프로젝트를 위한 백그라운드 작업입니다 — 무인으로 실행되고, 무언가에 반응하며, OneUptime을 나머지 스택과 이어줍니다.

## 한눈에 보기

- OneUptime 대시보드의 **워크플로우** 아래에 있는 **최상위 기능**.
- **세 가지 트리거 스타일**: 수동, 스케줄(cron), 웹훅 — 그리고 OneUptime 엔티티(인시던트, 알람, 모니터, 상태 페이지 등) 누구든 생성, 업데이트, 삭제될 때 발동되는 **모델 이벤트 트리거**.
- **시각적 캔버스**: 구성 요소 팔레트에서 노드를 끌어다 놓고, 출력 포트를 입력 포트에 연결합니다.
- **혼합 자동화**: HTTP 요청, Slack / Discord / Microsoft Teams / Telegram 메시지, 커스텀 JavaScript, JSON 파싱, 조건 분기, 이메일, 서브 워크플로우 호출, OneUptime 모델에 대한 CRUD 연산.
- **글로벌 변수**: 어떤 워크플로우에서도 복사-붙여넣기 없이 참조할 수 있는 프로젝트 단위의 비밀과 설정.
- **실행 및 로그**: 모든 실행이 상태, 타이밍, 단계별 출력과 함께 기록됩니다.

## 왜 워크플로우를 쓰는가?

대부분의 팀이 워크플로우를 찾는 경우는 다음과 같습니다:

- **OneUptime을 다른 시스템에 연결** — PagerDuty에 인시던트 게시, Jira에 알람 미러링, 자체 스택의 웹훅 호출.
- **OneUptime 이벤트에 반응** — `Sev 1` 인시던트가 열리면, 온콜 매니저를 호출하고 *동시에* Linear 티켓을 만들고 *동시에* 기능 플래그를 잠금.
- **반복 작업 스케줄링** — 5분마다 내부 API를 조회하고 그 결과를 외부 시스템에 기록.
- **OneUptime 외부의 데이터 수신** — CI 시스템의 웹훅이 OneUptime 업데이트 체인을 발동.
- **작은 연결 로직 재사용** — 한 워크플로우가 다른 워크플로우를 호출해서 공통 패턴이 한곳에 모입니다.

## 핵심 개념

| 용어 | 의미 |
| --- | --- |
| **워크플로우** | 캔버스. 이름이 붙고 재사용 가능한 트리거와 구성 요소의 그래프이며 `isEnabled` 플래그를 가진다. |
| **트리거** | 워크플로우 실행을 시작하는 노드. 수동, 스케줄, 웹훅, 또는 모델 이벤트. 모든 워크플로우는 정확히 하나의 트리거를 갖는다. |
| **구성 요소** | 작업을 수행하는 노드 — HTTP 호출, Slack 메시지, JavaScript 스니펫, 조건 분기 등. |
| **포트** | 노드의 입력 또는 출력 소켓. 구성 요소는 `success`, `error` 같은 출력 포트를 가지며, 포트를 다음 노드의 입력 포트에 연결한다. |
| **실행 / 로그** | 워크플로우의 한 번의 실행. 타임스탬프, 상태(Running, Success, Failed, Timeout), 실행된 모든 노드의 캡처된 출력을 보관. |
| **글로벌 변수** | 프로젝트 단위에서 한 번 정의되어 어떤 워크플로우에서도 `{{variable.NAME}}`으로 참조되는 이름 붙은 값(보통 비밀이나 API 키). |
| **로컬 변수** | 단일 워크플로우 실행에 한정된 값 — 보통 앞선 노드의 반환값으로, `{{ComponentId.portName}}`으로 참조. |

## 워크플로우가 대시보드의 어디에 있는가

| 페이지 | 거기서 하는 일 |
| --- | --- |
| **워크플로우** | 워크플로우 템플릿을 둘러보고, 만들고, 검색한다. |
| 워크플로우의 **Builder 탭** | 드래그 앤 드롭 캔버스. 노드를 추가하고, 포트를 연결하고, 인자를 설정한다. |
| 워크플로우의 **Logs 탭** | 이 워크플로우의 모든 실행을 상태와 시간 범위 필터와 함께 본다. 실행을 클릭하면 노드별 출력이 보인다. |
| 워크플로우의 **Settings 탭** | 이름 변경, 활성화/비활성화, 설명 변경, 라벨 관리, 삭제. |
| **워크플로우 → 글로벌 변수** | 어떤 워크플로우에서도 참조되는 프로젝트 단위 값을 정의한다. 비밀로 표시한 값은 저장 후 UI에서 가려진다. |
| **워크플로우 → 실행 및 로그** | 모든 워크플로우에 걸친 프로젝트 단위 실행 이력. |

## 워크플로우의 라이프사이클

1. **작성** — 워크플로우를 만들고, 캔버스에 트리거를 놓고, 필요한 구성 요소를 끌어다 연결하고, 각각을 설정합니다.
2. **활성화** — 워크플로우는 비활성 상태로 출시됩니다. 배선이 올바른지 확신이 들면 Settings에서 스위치를 켭니다.
3. **트리거** — 수동: 선택적 JSON 페이로드와 함께 **Run Manually**를 클릭. 스케줄: cron이 발동. 웹훅: 외부 시스템이 워크플로우 URL로 `POST`. 모델 이벤트: 누군가(또는 다른 워크플로우)가 모니터, 인시던트, 알람 등을 생성/업데이트/삭제.
4. **실행** — 워크플로우 워커가 그래프를 순서대로 따라갑니다. 각 구성 요소는 자신의 인자(리터럴 값 또는 보간된 변수)를 읽고, 작업을 수행하고, 반환값을 기록하고, 출력 포트를 선택합니다. 다음 노드가 발동합니다.
5. **감사** — 실행은 **Logs**에 나타납니다. 상태, 총 소요 시간, 구성 요소별 출력과 모든 오류가 프로젝트 수명 동안 보존됩니다.

## 작동하는 예제

목표: 제목에 `Sev 1`이 들어간 인시던트가 생성되면 Slack 채널에 게시하고 내부 어드민 도구에 티켓을 엽니다.

**1. "Sev 1 fan-out"이라는 이름의 워크플로우를 만듭니다.**

**2. 트리거를 놓습니다.** 팔레트에서 **Incident → On Create** 트리거를 고릅니다. 이 트리거는 새 인시던트를 반환값으로 노출합니다.

**3. 조건 구성 요소를 놓습니다.** 트리거의 출력 포트를 그 입력에 연결합니다. 조건 설정: `{{Incident.title}}` *contains* `Sev 1`.

**4. 조건의 `yes` 포트에서 Slack 구성 요소를 놓습니다.** 채널: `#incident-room`. 메시지 본문: `Sev 1 declared: {{Incident.title}} — {{Incident.dashboardUrl}}`.

**5. 같은 `yes` 포트에서 (병렬로) API 구성 요소를 놓습니다.** `POST`로 `https://admin.internal/incidents`. 본문: 인시던트로부터 만든 작은 JSON 객체.

**6. 워크플로우를 활성화합니다.** 다른 탭에서 "Sev 1 — checkout 500s" 제목의 인시던트를 엽니다. 몇 초 안에 Slack 메시지가 도착하고, **Logs**에 새 실행이 나타나며 각 노드의 출력이 캡처됩니다.

## 워크플로우가 OneUptime의 나머지와 어떻게 맞물리는가

- **모니터**가 문제를 감지하고, **인시던트/알람**이 그것을 기록하며, **워크플로우**가 그에 반응합니다 — 메시지 게시, 티켓 생성, 자동화 시작.
- **Runbook**은 사람을 위한 대응 절차(선택적 스크립트 단계 포함)입니다. 워크플로우는 무인 백그라운드 자동화입니다. 둘은 보완적입니다 — Runbook 단계가 워크플로우의 웹훅 트리거로 `POST`할 수 있습니다.
- **워크스페이스 연결**(Slack, Microsoft Teams)은 워크플로우 알림의 일반적인 목적지입니다.
- **대시보드**는 읽기 전용 뷰입니다; 워크플로우는 쓰기 측면입니다 — OneUptime 상태를 업데이트하고, 외부 API를 호출하고, 데이터를 이동시킵니다.
