# Runbook 작성

**Runbooks → Runbook 생성**에서 Runbook을 만들고, 열어서 **단계** 탭으로 이동합니다.

## 단계의 구성

모든 단계에는 다음이 있습니다:

| 필드 | 용도 |
| --- | --- |
| **제목** | 체크리스트 UI에 표시되는 짧은 라벨. 필수. |
| **설명** | 대응자를 위한 선택적 컨텍스트. 마크다운 텍스트. |
| **실패 시 계속** | 켜면 실패한 단계가 실행을 멈추지 않고 다음 단계가 진행됨. |
| **승인 필요** | 켜면 이 단계 후 Runbook이 일시 정지하고, 사용자가 승인할 때까지 다음 단계로 가지 않음. |
| **타입별 설정** | 스크립트, URL, 에이전트 등 — 아래 참조. |

단계는 **순서대로** 실행됩니다. 단계 편집기의 위/아래 화살표로 순서를 바꿀 수 있습니다.

## 단계 타입

### Manual

대응자가 체크하는 체크박스. Runbook 실행이 Manual 단계에 닿으면 일시 정지하고, 누군가 완료(또는 건너뛰기) 표시를 할 때까지 `WaitingForManualStep` 상태로 머뭅니다.

사람만이 확인할 수 있는 일에 씁니다: "로드 밸런서 대시보드에서 트래픽이 보조 리전으로 옮겨갔는지 확인."

### JavaScript

샌드박스화된 `isolated-vm` 안에서 실행되는 JavaScript 스니펫. 샌드박스는 OneUptime 워커가 아니라 자체 인프라 내 [Runbook 에이전트](/docs/runbooks/agents) 위에 있습니다.

JavaScript 단계에서 두 가지를 설정합니다:

- **Runbook 에이전트** — 이 단계를 실행해야 할 에이전트를 드롭다운에서 선택. 선택한 에이전트만 작업을 가져갈 수 있음.
- **스크립트** — 실행할 JavaScript.

```js
const start = Date.now();
// ... your logic ...
return { durationMs: Date.now() - start };
```

반환값은 단계 실행에 캡처됩니다. `console.log` 출력은 로그 라인으로 캡처됩니다. 기본 실행 타임아웃: 30초. 기본 클레임 타임아웃(에이전트가 작업을 가져갈 때까지 워커가 기다리는 시간): 2분.

### HTTP 요청

아웃바운드 HTTP 호출을 합니다. 메서드(GET/POST/PUT/PATCH/DELETE/HEAD), URL, 선택적 JSON 헤더, 선택적 본문을 설정합니다. 응답 상태, 헤더, 본문이 캡처됩니다(총 50KB 상한).

쓸 만한 곳: PagerDuty 인시던트 트리거, Slack 게시, 자체 관리 API 호출 등. HTTP 단계는 OneUptime 워커에서 직접 실행됩니다. 에이전트는 필요 없습니다.

### Bash

bash 스크립트(`bash -c <script>`)가 자체 인프라의 [Runbook 에이전트](/docs/runbooks/agents) 위에서 실행됩니다. Bash가 OneUptime 워커에서 실행되는 일은 절대 없습니다.

Bash 단계에서 두 가지를 설정합니다:

- **Runbook 에이전트** — 이 단계를 실행해야 할 에이전트를 드롭다운에서 선택. 선택한 에이전트만 작업을 가져갈 수 있음.
- **스크립트** — 실행할 bash. 출력(stdout + stderr)은 50 KB까지 캡처되고, 타임아웃이면 프로세스가 kill 됨.

Runbook이 이 단계에 닿을 때 선택된 에이전트가 오프라인이면, 단계는 **클레임 타임아웃**(기본 2분)까지 기다린 후 `TimedOut`으로 실패합니다. Bash 단계에 의지하기 전에 **Runbooks → 설정 → 에이전트**에서 에이전트를 추가하세요.

## 저장과 편집

**단계 저장**을 눌러 영속화합니다. 옛 버전 Runbook의 진행 중 실행은 영향받지 않습니다 — 자기 스냅샷을 계속 사용합니다.

## 여러 단계와 실패 처리

기본적으로 실패한 단계는 실행을 멈추게 하고, 실행은 `Failed`로 표시됩니다. **실패 시 계속**을 켜면 실패가 기록되고 다음 단계가 진행됩니다. "셋을 시도하고 알림"과 같은 패턴에 유용합니다.

## 작동하는 예제

"DB 프라이머리 도달 불가"에 대한 간단한 Runbook:

1. **JavaScript** — 설정 서비스에서 현재 프라이머리 호스트를 가져와 로그에 남김.
2. **Manual** — "보조의 복제 지연이 5초 미만인지 확인."
3. **HTTP 요청** — 페일오버 오케스트레이터 API로 POST.
4. **Manual** — "쓰기가 새 프라이머리로 가고 있는지 검증."
5. **HTTP 요청** — Slack에 "복구" 메시지 POST.

대응자는 자동 단계가 도는 것을 보고, 수동 단계에 체크하고, 다음 자동 단계를 보고, 이런 식으로 진행합니다. 각 단계의 출력은 포스트모템을 위해 캡처됩니다.
