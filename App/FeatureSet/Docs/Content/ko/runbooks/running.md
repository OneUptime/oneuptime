# Runbook 실행

Runbook 실행은 세 가지 방법으로 만들어집니다:

1. **규칙을 통한 자동** — [Runbook 규칙](/docs/runbooks/rules) 참고.
2. **Runbook 페이지에서 수동** — Runbook 개요에서 **지금 실행**을 클릭. 어떤 인시던트/알람/유지보수에도 묶이지 않습니다.
3. **엔티티 피드에서 수동** — 인시던트, 알람, 예정 유지보수 이벤트에서 **Runbook 실행**을 클릭. 실행은 해당 엔티티에 연결됩니다.

## 실행 화면

아무 실행이나 열어 체크리스트 UI를 보세요. 각 단계는 다음을 보여줍니다:

- **상태 배지** — 대기, 실행 중, 대기 중(당신), 완료, 건너뜀, 실패.
- **제목과 설명** — 실행 시점에 Runbook에서 복사.
- **출력** (접기 가능) — stdout, 반환 값, HTTP 응답.
- **오류 메시지** (단계가 실패했을 때).
- `WaitingForUser` 상태의 Manual 단계: **완료 처리** 및 **건너뛰기** 버튼.

실행이 종료 상태가 아닌 동안 페이지는 3초마다 폴링하므로, 자동 단계의 완료를 거의 실시간으로 볼 수 있습니다.

## 수동과 자동 단계의 혼합

전형적인 흐름:

1. **스크립트 단계**: 시스템 상태 캡처, S3에 기록.
2. **수동 단계**: "상태 페이지 배너로 고객에게 알림." 대응자 체크.
3. **HTTP 단계**: PagerDuty로 DBA 호출.
4. **수동 단계**: "보조 DB가 primary가 되었는지 확인." 대응자 체크.
5. **스크립트 단계**: Slack에 "정상화" 메시지 전송.

단계 2와 4는 체크될 때까지 실행을 일시 정지시킵니다. 단계 1, 3, 5는 자동으로 실행됩니다. 전체 실행은 하나의 실행, 하나의 타임라인, 하나의 진실 출처입니다.

## 실행 취소

실행 페이지에서 **실행 취소**를 클릭. 현재 단계(있다면)는 완료되고, 이후 단계는 시작되지 않습니다. 상태는 `Cancelled`이 됩니다.

## 출력 보존

단계별 출력은 데이터베이스가 폭주하는 스크립트로 부풀지 않도록 **50KB**로 제한됩니다. 더 큰 산출물이 필요하면 스크립트에서 S3나 로거에 쓰고 반환 값에 URL을 저장하세요.

## Runbook 다시 실행

실행은 일회성이며 변경 불가한 기록입니다. 다시 실행하려면 **지금 실행**을 다시 클릭하세요 — 현재 Runbook의 단계로 새 스냅샷을 가진 새 실행이 생성됩니다. 원래 실행은 감사 흔적을 위해 그대로 남습니다.

## 과거 실행 찾기

모든 Runbook에는 **실행** 탭이 있어 모든 실행을 나열하며, 상태, 날짜 범위, 소스 엔티티로 필터링할 수 있습니다. 인시던트, 알람, 유지보수에서는 **Runbooks** 탭이 해당 엔티티에 연결된 실행을 보여줍니다.
