# Runbook 실행

Runbook 실행은 세 가지 방법으로 만들어집니다:

1. **규칙을 통한 자동** — [Runbook 규칙](/docs/runbooks/rules) 참고.
2. **Runbook 페이지에서 수동** — Runbook의 개요 페이지에서 **지금 실행** 클릭. 어떤 인시던트, 알람, 예정 유지보수에도 연결되지 않음.
3. **엔티티 피드에서 수동** — 인시던트, 알람 또는 예정 유지보수 이벤트에서 **Runbook 실행** 클릭. 실행은 그 엔티티에 연결됨.

## 실행 뷰

임의의 실행을 열면 체크리스트 UI가 보입니다. 각 단계는 다음을 보여줍니다:

- **상태 배지** — Pending, Running, 당신을 기다림, Done, Skipped, Failed.
- **제목과 설명** — 실행 시 Runbook에서 복사됨.
- **출력**(접을 수 있음) — stdout, 반환값, HTTP 응답.
- 단계가 실패한 경우의 **오류 메시지**.
- `WaitingForUser` 상태의 Manual 단계에서는 **완료 표시**와 **건너뛰기** 버튼.

실행이 종료 상태가 아닌 동안 페이지는 3초마다 폴링하므로, 자동 단계가 완료되는 것을 거의 실시간으로 볼 수 있습니다.

## 수동 단계와 자동 단계를 섞기

전형적인 흐름:

1. **스크립트 단계**: 시스템 상태 캡처, S3에 기록.
2. **수동 단계**: "상태 페이지 배너로 고객에게 알림." 대응자가 체크.
3. **HTTP 단계**: PagerDuty로 DBA 호출.
4. **수동 단계**: "보조 DB가 이제 프라이머리인지 확인." 대응자가 체크.
5. **스크립트 단계**: 복구 Slack 메시지 발송.

단계 2와 4는 체크될 때까지 실행을 일시 정지합니다. 단계 1, 3, 5는 자동으로 돕니다. 전체가 하나의 실행, 하나의 타임라인, 하나의 진실의 출처입니다.

## 실행 취소

실행 페이지에서 **실행 취소**를 클릭. 현재 단계(있다면)는 끝마치고, 이후 단계는 시작하지 않습니다. 상태는 `Cancelled`가 됩니다.

## 출력 보관

단계별 출력은 폭주 스크립트가 데이터베이스를 부풀리지 않도록 **50KB**로 제한됩니다. 더 큰 산출물이 필요하면 스크립트에서 S3나 로거에 쓰고 반환값에 URL을 담으세요.

## Runbook 다시 실행

Runbook 실행은 단발적이고 불변인 기록입니다. 다시 돌리려면 **지금 실행**을 다시 클릭하세요 — 그러면 Runbook 현재 단계의 새 스냅샷으로 새 실행을 만듭니다. 원래 실행은 감사 흔적을 위해 그대로 남습니다.

## 과거 실행 찾기

모든 Runbook에는 **실행** 탭이 있어 상태, 날짜 범위, 소스 엔티티로 필터링 가능한 모든 실행을 나열합니다. 인시던트, 알람 또는 예정 유지보수 이벤트에서는 **Runbooks** 탭에 그 엔티티에 연결된 실행이 표시됩니다.
