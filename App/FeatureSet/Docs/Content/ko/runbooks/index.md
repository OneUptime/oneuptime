# Runbook 개요

Runbook은 재사용 가능한 대응 절차로, 인시던트·알람·예정 유지보수에 연결하는 수동 또는 자동 단계의 순서 있는 목록입니다. "이제 뭘 해야 하지?" 같은 즉흥적인 Slack 스레드를, 새벽 3시에 동료가 사전 지식 없이도 이어받을 수 있는 형태로 바꿔 줍니다.

## 한눈에 보기

- OneUptime 대시보드의 **분석 & 자동화 → Runbooks**에 있는 **최상위 기능**.
- **4가지 단계 타입**: 수동 체크리스트, JavaScript(샌드박스)와 Bash(둘 다 자신의 인프라 내 [Runbook 에이전트](/docs/runbooks/agents)에서 실행), HTTP 요청.
- **3가지 트리거 경로**: 인시던트/알람/예정 유지보수에 매칭되는 규칙, 또는 어떤 이벤트에서도 사용할 수 있는 수동 "Runbook 실행" 버튼.
- **스냅샷 방식**: Runbook이 시작되면 단계가 실행에 복사됩니다. 이후 템플릿을 수정해도 진행 중인 실행은 절대 바뀌지 않습니다.
- **완전한 감사 흔적**: 각 단계의 상태, 출력, 오류 메시지, 소요 시간이 실행에 영원히 남습니다.

## 왜 Runbook을 쓰는가?

인시던트 대응은 보통 1분짜리 장애와 몇 시간짜리 다운타임을 가르는 차이입니다. Runbook은 다음을 돕습니다:

- **암묵지를 문서화** — "큐가 밀릴 때 뭘 하지?"에 대한 답이 팀이 찾을 수 있는 곳에 자리한다.
- **평균 복구 시간(MTTR) 단축** — 자동 단계는 몇 초 만에 동작하고, 수동 단계는 결정 마비를 없앤다.
- **대응 행동의 감사** — 모든 단계 실행, 모든 출력, 모든 대응자의 클릭이 실행에 기록된다.
- **주니어가 자신감 있게 움직임** — 시니어를 새벽 3시에 호출하는 대신 Runbook을 자신 있게 돌릴 수 있다.
- **기억이 아닌 데이터로 포스트모템 작성** — 캡처된 실행은 정확히 무슨 일이 일어났는지의 동결된 기록이다.

## 핵심 개념

이후 문서 전반에서 반복되는 용어입니다. 먼저 정리합시다:

| 용어             | 의미                                                                                                                                                                            |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Runbook**      | 템플릿. 이름이 붙은 재사용 가능한 절차이며, 순서 있는 단계 목록과 `isEnabled` 플래그를 가진다.                                                                                  |
| **단계**         | Runbook 내 하나의 항목. 타입(Manual / JavaScript / HTTP / Bash), 제목, 설명, 타입별 설정을 가진다.                                                                              |
| **Runbook 규칙** | 인시던트·알람·예정 유지보수의 제목 또는 설명이 정규식과 일치할 때 하나 이상의 Runbook을 자동으로 연결하는 패턴.                                                                 |
| **실행**         | Runbook의 한 번의 실행. 규칙이 발동하거나, 누군가 이벤트에서 "Runbook 실행"을 클릭하거나, Runbook 자체에서 "지금 실행"을 클릭할 때 생성. 단계 스냅샷과 단계별 상태/출력을 보관. |
| **스냅샷**       | 각 실행에 남는 Runbook 단계의 동결 복사본. 이후 템플릿을 수정해도 과거를 다시 쓰지 않게 해준다.                                                                                 |

## Runbook의 라이프사이클

1. **작성** — Runbook을 만들고 Manual, JavaScript, HTTP, Bash 단계를 섞어 넣은 뒤 저장합니다.
2. **(선택) 규칙 추가** — 인시던트/알람/예정 유지보수 설정에서, 이벤트의 제목이나 설명이 정규식과 일치할 때 이 Runbook을 시작하도록 OneUptime에 알려줍니다.
3. **트리거** — 매칭되는 이벤트가 생기면 규칙이 자동 발동하거나, 대응자가 이벤트에서 **Runbook 실행** 을 수동으로 누릅니다.
4. **실행** — 단계의 스냅샷과 함께 새 실행이 생성됩니다. 자동 단계는 Runbook 워커에서 돌고, 수동 단계마다 누군가 체크할 때까지 실행이 일시 정지합니다.
5. **감사** — 실행은 이벤트의 **Runbooks** 탭과 Runbook의 실행 목록에 영원히 남습니다. 단계별 출력, 오류, 시각이 포스트모템을 위해 보존됩니다.

## 어떤 단계 타입을 언제 쓸까

빠른 판단 가이드. 자세한 설명은 [Runbook 작성](/docs/runbooks/authoring)을 참고하세요.

| 단계 타입      | 이런 때 사용                                                                                                                                                                     | 예시                                                          |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| **Manual**     | 사람이 확인하거나 판단하거나, OneUptime이 관측할 수 없는 행동을 해야 할 때.                                                                                                      | "로드 밸런서 대시보드에서 보조 리전 트래픽 확인 — 완료."      |
| **JavaScript** | 작고 자급자족적인 계산이 필요 — 설정 서비스 조회, 페이로드 변환, 다음 단계 전의 로직 실행. 자신의 인프라 내 [Runbook 에이전트](/docs/runbooks/agents)에서 샌드박스로 실행됩니다. | 현재 복제 지연을 계산하고 계속 진행할지 결정.                 |
| **HTTP 요청**  | 이미 있는 API를 호출하는 경우 — 자체 관리 엔드포인트, 클라우드 공급자, PagerDuty, Slack.                                                                                         | 페일오버 오케스트레이터로 `POST`.                             |
| **Bash**       | 자신의 인프라에서 셸 명령을 실행해야 할 때 — 서비스 재시작, `kubectl` 실행, 배포 스크립트 호출. 환경에 [Runbook 에이전트](/docs/runbooks/agents)가 설치되어 있어야 함.           | 서비스 재시작, `kubectl rollout restart`, 복구 스크립트 실행. |

네 가지 모두 하나의 Runbook에 섞을 수 있습니다 — Runbook의 강점은 사람의 검증과 자동화를 교차시키는 데 있습니다.

## Runbook이 대시보드의 어디에 있는가

| 페이지                                                    | 거기서 하는 일                                                     |
| --------------------------------------------------------- | ------------------------------------------------------------------ |
| **분석 & 자동화 → Runbooks**                              | Runbook 템플릿을 둘러보고, 만들고, 편집한다.                       |
| Runbook의 **단계** 탭                                     | 단계 목록을 작성하고 순서를 바꾼다.                                |
| Runbook의 **실행** 탭                                     | 이 Runbook의 모든 실행을 상태 필터와 함께 본다.                    |
| Runbook의 **지금 실행** 버튼                              | 어떤 이벤트에도 연결되지 않은 즉흥 실행을 시작한다.                |
| **인시던트 / 알람 / 예정 유지보수 → 설정 → Runbook 규칙** | 엔티티 유형별 자동 트리거 규칙을 만든다.                           |
| 인시던트 / 알람 / 유지보수 이벤트 → **Runbooks** 탭       | 이 이벤트에 연결된 실행을 보고 **Runbook 실행**을 수동으로 누른다. |

## 흔한 활용 사례

팀이 Runbook으로 자주 잡는 패턴:

- **데이터베이스 페일오버** — JavaScript로 현재 상태 캡처, Manual로 당직 DBA에게 복제본 정상성 확인, HTTP로 오케스트레이터 API 호출, Manual로 "DNS 업데이트" 체크, HTTP로 Slack에 복구 알림.
- **캐시 비우기** — 하나의 HTTP 단계와 "대시보드에서 캐시 히트율이 회복되고 있는지 확인"이라는 Manual 단계.
- **고객 영향 인시던트** — Manual: "상태 페이지 업데이트 게시." HTTP: "`#customer-incidents`에서 CS 팀에 알림." JavaScript: "내부 API에서 영향 받는 계정 목록 가져오기."
- **예정 유지보수 사전 점검** — JavaScript: 현재 메트릭 스냅샷. Manual: "이해관계자와 변경 창 확인." HTTP: 로드 밸런서에서 유지보수 모드 켜기.
- **항상 도는 위생 점검** — 제목 패턴이 비어 있는 규칙이 모든 인시던트에서 시스템 상태를 캡처합니다 — 포스트모템에 유용.

## 작동하는 예제

제목에 "db-primary"가 들어간 모든 인시던트가 다섯 단계짜리 DB 페일오버 Runbook을 자동으로 트리거하길 원한다고 합시다.

**1. Runbook 만들기.** **Runbooks → Runbook 생성**에서 "DB primary failover"라고 이름을 짓고 다음 단계를 추가:

| #   | 타입       | 제목                                    |
| --- | ---------- | --------------------------------------- |
| 1   | JavaScript | 페일오버 전 복제 지연 캡처              |
| 2   | Manual     | DBA 대시보드에서 복제본 정상성 확인     |
| 3   | HTTP       | 페일오버 오케스트레이터로 `POST`        |
| 4   | Manual     | 쓰기가 새 프라이머리로 가고 있는지 검증 |
| 5   | HTTP       | `#db-incidents` Slack에 복구 알림       |

**2. 규칙 추가.** **인시던트 → 설정 → Runbook 규칙**에서 다음을 만듭니다:

```
제목 패턴:    ^db-primary
Runbooks:    [DB primary failover]
```

**3. 트리거.** 모니터 알람이 인시던트 `INC-4821 · db-primary connection timeout`을 엽니다. 규칙이 일치하고, 실행이 생성되며:

- 단계 1 (JavaScript)이 워커에서 즉시 실행 — `return { lagMs: 412 }` 값이 캡처됩니다.
- 단계 2 (Manual)은 실행을 일시정지. 당직자가 인시던트 페이지에서 "당신을 기다림" 배지를 보고, 대시보드를 클릭하고, 단계에 체크합니다.
- 단계 3 (HTTP)이 단계 2가 체크되자마자 실행 — `POST` 응답 본문이 캡처됩니다.
- 단계 4 (Manual)에서 다시 일시정지.
- 단계 5 (HTTP)가 실행되고 실행이 끝납니다.

**4. 감사.** 실행은 인시던트의 **Runbooks** 탭에 그대로 남습니다. 모든 단계의 출력은 한 번의 클릭으로 볼 수 있습니다. 다음 주에 포스트모템을 쓸 때 "그 스크립트가 뭘 반환했지?"라고 물어볼 필요가 없습니다 — 거기에 있습니다.

## OneUptime의 다른 기능과 어떻게 맞물리는가

- **모니터**가 인시던트와 알람을 열고, **Runbook 규칙**이 그 이벤트를 Runbook 실행으로 바꿉니다. 둘이 함께 "감지 → 트리거 → 대응 → 기록"의 닫힌 루프를 이룹니다.
- **워크스페이스 연결**(Slack, Microsoft Teams)은 Runbook의 HTTP 단계가 자연스럽게 향하는 대상입니다 — 상태 업데이트 게시, 채널 알림에.
- **상태 페이지**는 고객 영향 Runbook의 Manual 단계로 흔히 업데이트됩니다.
- **온콜 일정**은 누가 호출될지를 정하고, Runbook은 그 사람이 깨어난 후 무엇을 할지를 정합니다.
