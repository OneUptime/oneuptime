# Runbook 규칙

Runbook 규칙은 **인시던트**, **알람**, **예정 유지보수 이벤트**가 생성될 때 Runbook을 자동으로 연결합니다. 각 엔티티의 설정 메뉴에서 관리됩니다:

- 인시던트 → 설정 → **Runbook 규칙**
- 알람 → 설정 → **Runbook 규칙**
- 예정 유지보수 → 설정 → **Runbook 규칙**

세 페이지 모두 동일한 규칙 모델을 편집합니다 — 해당 엔티티 타입의 규칙만 보이도록 필터링되었을 뿐입니다.

## 규칙의 구성

| 필드 | 용도 |
| --- | --- |
| **이름** | 짧고 읽기 쉬운 라벨. 감사 로그에 표시. |
| **설명** | 팀원을 위한 선택적 컨텍스트. |
| **활성화** | 삭제 없이 규칙을 일시 중지하는 토글. |
| **제목 패턴** | 엔티티 제목과 매칭되는 대소문자 무시 정규식. 비어 있으면 모든 제목 매칭. |
| **설명 패턴** | 엔티티 설명과 매칭되는 대소문자 무시 정규식. 비어 있으면 모든 설명 매칭. |
| **시작할 Runbook** | 규칙이 발동될 때 시작할 하나 이상의 Runbook. |

## 매칭 의미

규칙은 **지정한 모든 조건이 통과할 때** 매칭됩니다. 빈 조건은 건너뜁니다:

- 패턴이 없는 규칙은 해당 타입의 모든 이벤트에서 실행됩니다 (전역 "항상 실행" 규칙).
- 제목 패턴만 있는 규칙은 제목이 정규식과 일치하는 이벤트에서 발동합니다.
- 한 이벤트에 여러 규칙이 매칭될 수 있습니다 — 각 매칭이 발동되며 모든 규칙의 Runbook 합집합이 실행됩니다(각 Runbook은 자체 실행을 갖습니다).

## 예: 데이터베이스 인시던트의 DB 페일오버

```
이름:           DB 인시던트에 DB 페일오버 시작
트리거:         인시던트
제목 패턴:      (?:^|\b)(db|database|postgres|mysql|mongo)
Runbooks:       [DB 페일오버 Playbook, DBA 팀 알림]
```

이는 제목에 "db", "database", "postgres" 등이 포함된 인시던트가 생길 때마다 두 개의 Runbook 실행을 생성합니다.

## 예: 항상 실행되는 위생 규칙

```
이름:                       모든 인시던트의 사전 점검
트리거:                     인시던트
제목 패턴:                  (비어 있음)
설명 패턴:                  (비어 있음)
Runbooks:                   [인시던트 전 상태 캡처]
```

모든 인시던트에서 발동 — 시스템 상태, 페이지 메트릭 등을 캡처하기에 좋습니다.

## 규칙이 발동되면 무슨 일이 일어나는가

1. Runbook이 로드됩니다.
2. 단계가 **스냅샷**으로 새 Runbook 실행에 복사됩니다.
3. 실행이 Runbook 워커 큐에 추가됩니다.
4. 실행이 소스 엔티티와 연결됩니다 — 인시던트, 알람, 유지보수 이벤트 페이지와 Runbook의 실행 목록에 표시됩니다.

규칙으로 발동된 모든 실행은 **Runbooks → 실행**에서 상태, Runbook, 날짜로 필터링해 볼 수 있습니다.

## 비활성화된 Runbook

규칙이 `isEnabled = false`인 Runbook을 참조하면 규칙은 여전히 매칭되지만 Runbook 실행은 건너뜁니다. Runbook을 다시 활성화하면 재개됩니다.

## 규칙 테스트

프로덕션에서 규칙에 의존하기 전에, 패턴과 일치하는 제목으로 테스트 인시던트(또는 알람)를 만들고 기대한 Runbook이 발동되는지 확인하세요. 규칙은 생성 순간에 평가됩니다 — 이후 인시던트 제목을 수정해도 규칙이 다시 발동되지 않습니다.
