# Runbook 규칙

Runbook 규칙은 **인시던트**, **알람**, 또는 **예정 유지보수 이벤트**가 생성될 때 자동으로 Runbook을 연결합니다. 각 엔티티의 설정 메뉴에서 관리됩니다:

- 인시던트 → 설정 → **Runbook 규칙**
- 알람 → 설정 → **Runbook 규칙**
- 예정 유지보수 → 설정 → **Runbook 규칙**

세 페이지 모두 동일한 기반 규칙 모델을 편집합니다 — 단지 그 엔티티 유형의 규칙만 보이도록 필터링되어 있을 뿐입니다.

## 규칙의 구성

| 필드 | 용도 |
| --- | --- |
| **이름** | 짧은 사람용 라벨. 감사 로그에 표시. |
| **설명** | 동료를 위한 선택적 컨텍스트. |
| **활성화** | 삭제하지 않고 규칙을 잠시 멈추는 토글. |
| **제목 패턴** | 엔티티 제목에 대해 대소문자 무시 정규식. 빈 값 = 어떤 제목이든 일치. |
| **설명 패턴** | 엔티티 설명에 대해 대소문자 무시 정규식. 빈 값 = 어떤 설명이든 일치. |
| **시작할 Runbook** | 규칙이 발동할 때 시작할 하나 이상의 Runbook. |

## 매칭 의미론

규칙은 **지정된 모든 기준이 통과할 때** 일치합니다. 비어 있는 기준은 건너뛰므로:

- 패턴이 설정되지 않은 규칙은 그 유형의 모든 이벤트에서 실행됩니다(전역 "항상 실행" 규칙).
- 제목 패턴만 있는 규칙은 제목이 그 정규식과 일치하는 이벤트에서 발동합니다.
- 같은 이벤트에 여러 규칙이 일치할 수 있습니다 — 일치한 모든 규칙이 발동하고, 그 Runbook들의 합집합이 실행됩니다(각 Runbook은 자기 실행을 가집니다).

## 예: 데이터베이스 인시던트용 DB 페일오버

```
이름:          Start DB failover for DB incidents
트리거:        인시던트
제목 패턴:     (?:^|\b)(db|database|postgres|mysql|mongo)
Runbooks:     [DB failover playbook, Notify DBA team]
```

이 규칙은 제목에 "db", "database", "postgres" 등을 가진 인시던트가 생성될 때마다 두 개의 Runbook 실행을 생성합니다.

## 예: 항상 실행하는 위생 규칙

```
이름:             Always-run pre-flight check
트리거:           인시던트
제목 패턴:        (빈 값)
설명 패턴:        (빈 값)
Runbooks:        [Capture pre-incident state]
```

모든 인시던트에서 발동합니다 — 시스템 상태 스냅샷, 페이지 메트릭 등을 캡처하는 데 유용.

## 규칙이 발동하면 일어나는 일

1. Runbook이 로드됩니다.
2. 그 단계가 새 Runbook 실행에 **스냅샷**됩니다.
3. 실행이 Runbook 큐 워커로 enqueue 됩니다.
4. 실행은 소스 엔티티에 연결됩니다 — 인시던트, 알람, 또는 예정 유지보수 이벤트 페이지와 Runbook의 실행 목록에 나타납니다.

규칙으로 트리거된 실행은 모두 **Runbooks → 실행**에서 상태, Runbook, 날짜로 필터링해 볼 수 있습니다.

## 비활성화된 Runbook

규칙이 `isEnabled = false`인 Runbook을 참조하면, 규칙은 일치해도 그 Runbook 실행은 건너뜁니다. Runbook을 다시 활성화하면 재개됩니다.

## 규칙 테스트

프로덕션에서 규칙에 의지하기 전에, 패턴과 일치하는 제목으로 테스트 인시던트(또는 알람)를 만들어 기대한 Runbook이 발동하는지 확인하세요. 규칙은 생성 시점에 평가됩니다 — 나중에 인시던트 제목을 편집해도 규칙이 다시 트리거되지 않습니다.
