# SQL 쿼리 모니터

SQL 쿼리 모니터는 프로브에서 일정에 따라 읽기 전용 SQL 쿼리를 실행하고 그 결과 — 반환된 행 수, 스칼라 값, 쿼리 소요 시간, 또는 쿼리 오류 — 에 대해 알림을 보냅니다. "쿼리를 실행하고 인시던트를 여는" 사용 사례를 위해 만들어졌습니다. 예를 들어 최근 5분간 취소된 주문 수가 급증할 때, 큐 테이블이 너무 커질 때, 또는 중요한 행이 사라질 때 알림을 보낼 수 있습니다.

쿼리는 네트워크 내부의 프로브에서 실행되므로 OneUptime은 데이터베이스에 직접 연결할 필요가 전혀 없으며, 전체 결과 집합은 프로브를 벗어나지 않습니다 — 결과의 작고 제한된 일부만 다시 보고됩니다.

## 지원 데이터베이스

SQL 쿼리 모니터는 다음 데이터베이스 엔진을 지원합니다:

- **PostgreSQL** (기본 포트 `5432`)
- **MySQL** (기본 포트 `3306`)
- **Microsoft SQL Server** (기본 포트 `1433`)

동일한 와이어 프로토콜과 SQL 방언을 사용하는 MySQL 호환 및 PostgreSQL 호환 엔진도 일반적으로 작동하지만, 위의 세 가지 엔진만 공식적으로 테스트되었습니다.

## 작동 방식

각 확인 시 프로브는 데이터베이스에 연결하여 읽기 전용 컨텍스트에서 쿼리를 실행하고, 제한된 수의 행까지만 읽어들인 뒤 OneUptime에 간결한 프로젝션을 보고합니다. 그런 다음 모니터의 기준이 해당 프로젝션에 대해 평가됩니다.

프로브는 다음만 보고합니다:

- **행 수** — 쿼리가 반환한 행의 수 (최대 행 수 제한으로 제한됨).
- **스칼라 값** — 첫 번째 행의 첫 번째 열. `SELECT COUNT(*)` 스타일 쿼리에 자연스러운 값입니다.
- **첫 번째 행** — 열/값 쌍의 집합으로 표현된 첫 번째 행으로, 맥락을 위해 확인 요약에 표시됩니다.
- **실행 시간** — 쿼리에 걸린 시간(밀리초).
- **쿼리 오류** — 쿼리가 실패한 경우 정제된 오류 메시지.

전체 결과 집합은 절대 OneUptime으로 전송되지 않으므로 고객 데이터가 OneUptime 스토리지에 복제되지 않습니다.

## 보안 모델

프로덕션 데이터베이스에 대해 고객이 제공한 쿼리를 실행하는 것은 민감한 작업이므로, SQL 쿼리 모니터는 설계상 읽기 전용이며 여러 계층의 제어를 갖추고 있습니다:

- **최소 권한 데이터베이스 사용자 (기본 제어).** 항상 쿼리에 필요한 테이블에만 접근할 수 있는 전용 읽기 전용 데이터베이스 사용자로 연결해야 합니다. 이것이 가장 중요한 제어입니다 — 아래 읽기 전용 사용자 만들기를 참조하십시오.
- **읽기 전용 실행.** PostgreSQL과 MySQL에서 프로브는 `READ ONLY` 트랜잭션을 열며, 이는 쿼리 텍스트와 관계없이 모든 쓰기(쓰기 가능한 CTE 포함)를 거부합니다. 읽기 전용 트랜잭션이 없는 Microsoft SQL Server에서는 프로브가 항상 롤백되는 트랜잭션 내에서 실행됩니다.
- **단일 명령문, 허용 목록 기반 쿼리.** 쿼리는 `SELECT`, `WITH`, `VALUES`, 또는 `TABLE`로 시작하는 단일 명령문이어야 합니다. 여러 개를 이어붙인 명령문(`SELECT 1; DROP TABLE …`)과 쓰기/DDL은 프로브가 연결되기 전에 거부됩니다. 이 검사는 주석과 문자열 리터럴을 인식하므로, 주석이나 문자열에 숨겨진 키워드는 통과하지 못합니다.
- **명령문 타임아웃.** 모든 쿼리에는 엄격한 시간 제한이 있습니다. 너무 오래 실행되는 쿼리는 취소됩니다.
- **제한된 행 수.** 최대 행 수(잘림을 감지하기 위해 하나를 더한 값)까지만 읽어들이므로 프로브 메모리와 페이로드 크기가 제한됩니다.
- **자격 증명 삭제.** 데이터베이스 오류는 저장되기 전에 정제됩니다 — 비밀번호와 모든 연결 문자열이 삭제되므로 자격 증명이 오류 메시지로 유출되지 않습니다.

## 사전 요구 사항

- 데이터베이스 호스트와 포트에 네트워크로 접근할 수 있는 **프로브**. OneUptime에서 호스팅하는 프로브(데이터베이스가 인터넷에서 접근 가능한 경우) 또는 네트워크 내부에서 실행되는 자체 호스팅 프로브일 수 있습니다. 커스텀 프로브를 설치하는 방법은 프로브 문서를 참조하십시오.
- **읽기 전용 데이터베이스 사용자**와 연결 세부 정보(호스트, 포트, 데이터베이스 이름, 사용자 이름, 비밀번호).

## 구성

새 모니터를 생성하고 모니터 유형으로 **SQL 쿼리**를 선택한 다음 연결 세부 정보를 입력합니다:

- **데이터베이스 유형** — PostgreSQL, MySQL, 또는 Microsoft SQL Server. 유형을 선택하면 기본 포트가 설정됩니다.
- **호스트** — 프로브에서 접근할 수 있는 데이터베이스 호스트 (예: `db.internal`).
- **포트** — 데이터베이스 포트.
- **데이터베이스 이름** — 쿼리를 실행할 데이터베이스.
- **사용자 이름** — 읽기 전용, 최소 권한 데이터베이스 사용자.
- **비밀번호** — 데이터베이스 비밀번호. 비밀번호를 평문으로 입력하는 대신 `{{monitorSecrets.name}}`으로 [모니터 시크릿](/docs/monitor/monitor-secrets)을 참조하는 것을 강력히 권장합니다(아래 참조).
- **SQL 쿼리** — 실행할 읽기 전용 쿼리 (쿼리 작성 참조).
- **SSL/TLS 사용** — TLS로 연결하려면 활성화합니다. 활성화된 경우 데이터베이스가 자체 서명 인증서를 사용한다면 **서버 인증서 확인**을 끌 수 있습니다.

### 고급 옵션

- **연결 타임아웃 (ms)** — 연결을 설정하기 위해 대기하는 시간. 기본값 `10000`, 최대 `30000`.
- **명령문 타임아웃 (ms)** — 쿼리가 실행될 수 있는 시간의 엄격한 상한. 기본값 `15000`, 최대 `60000`.
- **최대 행 수** — 데이터베이스에서 읽어들이는 행 수의 상한. 기본값 `100`, 최대 `1000`.

## 쿼리 작성

쿼리는 **단일 읽기 전용 명령문**이어야 합니다. `SELECT`, `WITH`, `VALUES`, 또는 `TABLE` 중 하나로 시작해야 합니다. 후행 세미콜론 하나는 허용되지만 여러 명령문은 허용되지 않습니다.

쿼리는 저렴하고 범위가 잘 지정되도록 유지하십시오 — 매 확인마다 실행되므로 인덱싱된 열과 좁은 시간 창을 선호하십시오.

```sql
-- 최근 취소 건수 집계 (PostgreSQL)
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL '5 minutes';
```

```sql
-- MySQL에서 동일한 방식
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL 5 MINUTE;
```

```sql
-- Microsoft SQL Server에서 동일한 방식
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > DATEADD(minute, -5, GETDATE());
```

`COUNT(*)` 스타일 쿼리의 경우, 개수는 **행 수**(하나의 행이 반환되므로 `1`)와 **스칼라 값**(첫 번째 열에서 가져온 개수 자체) 두 가지로 모두 제공됩니다. "몇 개인지"에 대해 알림을 보내려면 **스칼라 값**과 비교하십시오.

## 비밀번호에 모니터 시크릿 사용

데이터베이스 비밀번호가 모니터에 평문으로 저장되지 않도록, [모니터 시크릿](/docs/monitor/monitor-secrets)을 생성하고 비밀번호 필드에서 참조하십시오:

1. OneUptime 대시보드 → 프로젝트 설정 → 모니터 시크릿 → 모니터 시크릿 생성으로 이동합니다.
2. 시크릿(예: `dbPassword`)을 생성하고 이 모니터에 접근 권한을 부여합니다.
3. 모니터의 비밀번호 필드에 `{{monitorSecrets.dbPassword}}`를 입력합니다.

OneUptime은 구성이 프로브에 전달되기 전에 서버 측에서 시크릿을 확인합니다. OneUptime은 이러한 시크릿을 대신 생성하지 않습니다 — 시크릿을 참조하는 것은 사용자의 선택입니다.

## 기준 설정

모니터가 온라인, 저하, 또는 오프라인으로 간주되는 시점을 결정하는 기준을 추가합니다. SQL 쿼리 모니터에는 다음 확인 항목을 사용할 수 있습니다:

- **SQL 온라인 여부** — 데이터베이스에 접근할 수 있었고 쿼리가 성공했는지 여부.
- **SQL 쿼리 행 수** — 반환된 행의 수. 초과, 미만, 또는 동일과 같은 연산자로 비교합니다.
- **SQL 쿼리 스칼라 값** — 첫 번째 행의 첫 번째 열. 양쪽이 모두 숫자로 보이면 숫자로, 그렇지 않으면 문자열로 비교됩니다. `COUNT(*)` 스타일 쿼리에 사용할 확인 항목입니다.
- **SQL 쿼리 실행 시간 (ms)** — 쿼리에 걸린 시간. 느린 데이터베이스를 포착하는 데 유용합니다.
- **SQL 쿼리 오류** — 쿼리 오류 메시지. 비어 있을 때(또는 비어 있지 않을 때), 또는 특정 문자열과 일치할 때 알림을 보냅니다.
- **JavaScript 표현식** — 완전한 제어를 위해 커스텀 JavaScript 표현식을 평가합니다. [JavaScript 표현식](/docs/monitor/javascript-expression)을 참조하십시오.

### 예시: 취소가 급증할 때 알림

위 쿼리를 사용하여:

- **기준: 저하** — `SQL 쿼리 스칼라 값`이 `10`보다 큼.
- **기준: 오프라인** — `SQL 쿼리 스칼라 값`이 `50`보다 크거나, `SQL 온라인 여부`가 `false`.

적절한 담당자가 호출되도록 기준에 온콜 정책을 연결하십시오.

## 읽기 전용 사용자 만들기

항상 전용 읽기 전용 사용자로 연결하십시오. 예시:

```sql
-- PostgreSQL
CREATE USER oneuptime_ro WITH PASSWORD 'a-strong-password';
GRANT CONNECT ON DATABASE orders TO oneuptime_ro;
GRANT USAGE ON SCHEMA public TO oneuptime_ro;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO oneuptime_ro;
-- 향후 생성되는 테이블도 포함:
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO oneuptime_ro;
```

```sql
-- MySQL
CREATE USER 'oneuptime_ro'@'%' IDENTIFIED BY 'a-strong-password';
GRANT SELECT ON orders.* TO 'oneuptime_ro'@'%';
FLUSH PRIVILEGES;
```

```sql
-- Microsoft SQL Server
CREATE LOGIN oneuptime_ro WITH PASSWORD = 'a-strong-password';
USE orders;
CREATE USER oneuptime_ro FOR LOGIN oneuptime_ro;
ALTER ROLE db_datareader ADD MEMBER oneuptime_ro;
```

## 고려 사항

- 쿼리는 매 확인마다 실행되므로 저렴하게 유지하십시오. 인덱스와 좁은 시간 창을 사용하고, 최후의 방어선으로 명령문 타임아웃에 의존하십시오.
- 행 수, 첫 번째 셀(스칼라), 첫 번째 행만 보고되므로 — 알림을 보내려는 값이 첫 번째 열이 되도록 쿼리를 설계하십시오.
- 결과가 최대 행 수를 초과하여 잘린 경우, 확인 요약에 상한에 도달했음이 표시됩니다. 필요한 경우에만 최대 행 수를 늘리십시오. 더 큰 결과 집합은 프로브에서 더 많은 메모리를 소모합니다.
- 쓰기와 DDL은 항상 거부됩니다. 쓰기 경로를 테스트해야 한다면, 이 모니터는 그 용도가 아닙니다.
- 자격 증명이 저장 시 암호화된 상태로 유지되도록 평문 비밀번호보다 모니터 시크릿을 선호하십시오.
