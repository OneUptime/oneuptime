# Написание runbook-а

Создайте runbook в **Runbooks → Создать runbook**, затем откройте его и перейдите на вкладку **Шаги**.

## Анатомия шага

У каждого шага есть:

| Поле                     | Назначение                                                                                                                       |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| **Заголовок**            | Короткая метка в чек-листе. Обязательное.                                                                                        |
| **Описание**             | Необязательный контекст для ответственного. Markdown-безопасный текст.                                                           |
| **Continue on failure**  | Если включено, упавший шаг не останавливает прогон — следующий шаг всё равно выполнится.                                         |
| **Require approval**     | Если включено, runbook ставится на паузу после этого шага и ждёт подтверждения пользователя, прежде чем запустить следующий шаг. |
| **Конфигурация под тип** | Скрипт, URL, агент и т.д. — см. ниже.                                                                                            |

Шаги выполняются **по порядку**. Меняйте порядок стрелками вверх/вниз в редакторе шагов.

## Типы шагов

### Ручной

Чек-бокс, который отмечает ответственный. Исполнение runbook-а приостанавливается на ручном шаге и остаётся в `WaitingForManualStep`, пока кто-нибудь не отметит выполнение (или не пропустит шаг).

Применяйте для того, что может подтвердить только человек: «Согласно дашборду балансировщика трафик переведён во вторичную зону — подтверждено.»

### JavaScript

Фрагмент JavaScript, выполняемый в песочнице `isolated-vm`. Песочница живёт на [Runbook-агенте](/docs/runbooks/agents) внутри вашей инфраструктуры — не на OneUptime Worker.

Настройте две вещи на JavaScript-шаге:

- **Runbook Agent** — выберите из выпадающего списка агента, который должен выполнить этот шаг. Только выбранный агент может забрать job.
- **Скрипт** — JavaScript для запуска.

```js
const start = Date.now();
// ... ваша логика ...
return { durationMs: Date.now() - start };
```

Возвращаемое значение фиксируется на исполнении шага. Вывод `console.log` сохраняется как лог-строки. Execution timeout по умолчанию: 30 секунд. Claim timeout по умолчанию (сколько Worker ждёт, пока агент возьмёт job): 2 минуты.

### HTTP-запрос

Исходящий HTTP-вызов. Настройте метод (GET/POST/PUT/PATCH/DELETE/HEAD), URL, опциональные JSON-заголовки и опциональное тело. Статус, заголовки и тело ответа сохраняются (всего до 50 КБ).

Полезно для: открытия инцидента PagerDuty, публикации в Slack, вызова собственного admin-API и т.д. HTTP-шаги выполняются напрямую на OneUptime Worker; агент не требуется.

### Bash

Bash-скрипт (`bash -c <скрипт>`), который выполняется на [Runbook-агенте](/docs/runbooks/agents) внутри вашей инфраструктуры. Bash никогда не выполняется на OneUptime Worker.

Настройте две вещи на Bash-шаге:

- **Runbook Agent** — выберите из выпадающего списка агента, который должен выполнить этот шаг. Только выбранный агент может забрать job.
- **Скрипт** — bash для запуска. Вывод (stdout + stderr) захватывается до 50 КБ; по таймауту процесс убивается.

Если выбранный агент не онлайн в момент, когда runbook доходит до этого шага, шаг ждёт до **claim timeout** (по умолчанию 2 минуты) и затем падает с `TimedOut`. Добавьте агента в **Runbooks → Настройки → Агенты** прежде чем полагаться на Bash-шаг.

## Сохранение и редактирование

Нажмите **Сохранить шаги** для сохранения. Текущие исполнения старых версий runbook-а не затрагиваются — они продолжают со своим снимком.

## Несколько шагов и обработка ошибок

По умолчанию упавший шаг останавливает прогон и переводит исполнение в `Failed`. Если включить **Continue on failure**, сбой фиксируется, но следующий шаг всё равно запускается. Полезно для паттернов «попробуйте эти три вещи, затем сообщите».

## Развёрнутый пример

Простой runbook для «Primary DB недоступна»:

1. **JavaScript** — получить текущий primary-хост из сервиса конфигурации и залогировать.
2. **Ручной** — «Задержка репликации на secondary меньше 5 секунд — подтверждено.»
3. **HTTP-запрос** — POST к API оркестратора failover-а.
4. **Ручной** — «Записи идут на нового primary — подтверждено.»
5. **HTTP-запрос** — POST в Slack с сообщением «всё в порядке».

Ответственный смотрит, как идёт автоматический шаг, отмечает ручной, смотрит следующий автоматический и так далее. Вывод каждого шага сохраняется для постмортема.
