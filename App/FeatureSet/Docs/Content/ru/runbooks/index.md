# Обзор Runbook-ов

Runbook-и — это многоразовые процедуры реагирования, упорядоченные списки ручных или автоматизированных шагов, которые вы привязываете к инцидентам, оповещениям или плановым работам. Они превращают спонтанные обсуждения в Slack типа «и что нам теперь делать?» в нечто, что коллега может подхватить «с нуля» в три часа ночи.

## Кратко

- **Функция верхнего уровня** в панели OneUptime: **Аналитика и автоматизация → Runbooks**.
- **Четыре типа шагов**: ручной чек-лист, JavaScript (в песочнице) и Bash (оба выполняются на [Runbook-агенте](/docs/runbooks/agents) внутри вашей инфраструктуры), HTTP-запрос.
- **Три способа запуска**: правила, совпадающие с инцидентами/оповещениями/плановыми работами, или ручная кнопка «Запустить runbook» на любом событии.
- **Семантика снимка**: при старте runbook-а его шаги копируются в исполнение. Последующее редактирование шаблона не меняет уже выполняющийся прогон.
- **Полный аудит**: статус, вывод, сообщение об ошибке и продолжительность каждого шага остаются в исполнении навсегда.

## Зачем нужны runbook-и?

Реагирование на инциденты часто отличает короткий сбой от многочасового простоя. Runbook-и помогают:

- **Зафиксировать неявные знания** — ответ на «что делать, если очередь забилась» лежит там, где команда его найдёт.
- **Снизить среднее время восстановления (MTTR)** — автоматические шаги выполняются за секунды, ручные шаги убирают паралич решения.
- **Аудировать действия** — каждый выполненный шаг, каждый вывод, каждый клик ответственного фиксируется в исполнении.
- **Дать самостоятельность джунам** — они могут запускать runbook с уверенностью, а не звонить сеньору в три ночи.
- **Писать постмортем по данным, а не по памяти** — захваченное исполнение — это замороженная запись того, что реально произошло.

## Ключевые понятия

Несколько терминов встречаются по всей документации. Зафиксируем их сразу:

| Термин                | Смысл                                                                                                                                                                                                              |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Runbook**           | Шаблон. Названная многоразовая процедура с упорядоченным списком шагов и флагом `isEnabled`.                                                                                                                       |
| **Шаг**               | Один элемент runbook-а. Имеет тип (Ручной / JavaScript / HTTP / Bash), заголовок, описание и типовую конфигурацию.                                                                                                 |
| **Правило runbook-а** | Шаблон, который автоматически прикрепляет один или несколько runbook-ов к инцидентам, оповещениям или плановым работам, когда их заголовок или описание совпадает с регуляркой.                                    |
| **Исполнение**        | Один прогон runbook-а. Создаётся, когда срабатывает правило, когда кто-то нажимает «Запустить runbook» на событии или «Запустить сейчас» на самом runbook-е. Содержит снимок шагов и статус/вывод по каждому шагу. |
| **Снимок**            | Замороженная копия шагов runbook-а в каждом исполнении. Позволяет редактировать шаблон позже, не переписывая историю.                                                                                              |

## Жизненный цикл runbook-а

1. **Написать** — создайте runbook, добавьте смесь ручных, JavaScript-, HTTP- и Bash-шагов. Сохраните.
2. **(Опционально) добавить правило** — в настройках Инцидентов, Оповещений или Плановых работ скажите OneUptime запускать этот runbook, когда заголовок или описание события совпадает с регуляркой.
3. **Запустить** — либо правило срабатывает автоматически при появлении подходящего события, либо ответственный вручную нажимает **Запустить runbook**.
4. **Выполнить** — создаётся новое исполнение со снимком шагов. Автоматические шаги выполняются на воркере Runbook; исполнение останавливается на каждом ручном шаге, пока кто-нибудь его не отметит.
5. **Аудит** — исполнение навсегда остаётся на вкладке **Runbooks** события и в списке **Исполнения** runbook-а. Вывод, ошибки и тайминги по шагам сохраняются для постмортема.

## Когда какой тип шага

Краткое руководство по выбору. Подробности — в [Написании runbook-а](/docs/runbooks/authoring).

| Тип шага        | Когда использовать…                                                                                                                                                                                                                      | Пример                                                                      |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **Ручной**      | Человек должен что-то проверить, оценить или выполнить действие, которое OneUptime не может наблюдать.                                                                                                                                   | «Подтвердите трафик во вторичной зоне на дашборде балансировщика.»          |
| **JavaScript**  | Нужно небольшое замкнутое вычисление — запросить конфигурационный сервис, преобразовать payload, выполнить логику перед следующим шагом. Выполняется в песочнице на [Runbook-агенте](/docs/runbooks/agents) внутри вашей инфраструктуры. | Вычислить текущую задержку реплики и решить, продолжать ли.                 |
| **HTTP-запрос** | Вы вызываете существующее API — собственный admin-endpoint, облачного провайдера, PagerDuty, Slack.                                                                                                                                      | `POST` к вашему оркестратору failover-а.                                    |
| **Bash**        | Нужно запустить shell-команды на собственной инфраструктуре — перезапустить сервис, выполнить `kubectl`, дернуть deploy-скрипт. Требует [Runbook-агент](/docs/runbooks/agents), установленный в вашей среде.                             | Перезапустить сервис, `kubectl rollout restart`, запустить recovery-скрипт. |

В одном runbook-е можно смешивать все четыре — сила runbook-ов в том, чтобы переплетать человеческое подтверждение с автоматизацией.

## Где runbook-и живут в дашборде

| Страница                                                                      | Что вы там делаете                                                                              |
| ----------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Аналитика и автоматизация → Runbooks**                                      | Просматривать, создавать и редактировать шаблоны.                                               |
| **Вкладка «Шаги» runbook-а**                                                  | Писать и переупорядочивать список шагов.                                                        |
| **Вкладка «Исполнения» runbook-а**                                            | Видеть все прогоны runbook-а с фильтрами по статусу.                                            |
| **Кнопка «Запустить сейчас» runbook-а**                                       | Запустить разовое исполнение, не привязанное к событию.                                         |
| **Инциденты / Оповещения / Плановые работы → Настройки → Правила runbook-ов** | Создавать правила автозапуска по типу сущности.                                                 |
| **Инцидент / Оповещение / Плановая работа → вкладка Runbooks**                | Видеть исполнения, привязанные к событию, и нажимать **Запустить runbook** для ручного запуска. |

## Типовые сценарии

Несколько паттернов, для которых команды берут runbook-и:

- **Failover базы данных** — захватить текущий статус JavaScript-ом, попросить дежурного DBA подтвердить состояние реплики (Ручной), вызвать API оркестратора (HTTP), отметить «DNS обновлён» (Ручной), отправить «всё в порядке» в Slack (HTTP).
- **Сброс кэша** — один HTTP-шаг плюс Ручной «убедитесь, что hit rate кэша восстанавливается на дашборде».
- **Инцидент, влияющий на клиентов** — Ручной: «Опубликуйте обновление на странице статуса.» HTTP: «Уведомить CS-команду в #customer-incidents.» JavaScript: «Достать список затронутых аккаунтов из внутреннего API.»
- **Pre-flight плановой работы** — JavaScript: снимок текущих метрик. Ручной: «Подтвердите окно изменений со стейкхолдерами.» HTTP: включить режим обслуживания на балансировщике.
- **Гигиена «всегда-вкл»** — правило с пустым шаблоном заголовка, фиксирующее состояние системы на каждом инциденте — золото для постмортемов.

## Развёрнутый пример

Предположим, вы хотите, чтобы каждый инцидент с «db-primary» в заголовке автоматически запускал пятишаговый runbook DB-failover-а.

**1. Создайте runbook.** В **Runbooks → Создать runbook** назовите его «Failover DB primary» и добавьте шаги:

| #   | Тип        | Заголовок                                            |
| --- | ---------- | ---------------------------------------------------- |
| 1   | JavaScript | Захват задержки реплики до failover                  |
| 2   | Ручной     | Подтвердить состояние реплики на дашборде DBA        |
| 3   | HTTP       | `POST` к оркестратору failover-а                     |
| 4   | Ручной     | Проверить, что записи идут на нового primary         |
| 5   | HTTP       | Опубликовать «всё в порядке» в Slack `#db-incidents` |

**2. Добавьте правило.** В **Инциденты → Настройки → Правила runbook-ов** создайте:

```
Шаблон заголовка: ^db-primary
Runbooks:         [Failover DB primary]
```

**3. Запуск.** Срабатывает алерт монитора и открывается инцидент `INC-4821 · db-primary connection timeout`. Правило совпадает, создаётся исполнение, и:

- Шаг 1 (JavaScript) запускается сразу на воркере — его `return { lagMs: 412 }` сохраняется.
- Шаг 2 (Ручной) приостанавливает прогон. Дежурный видит метку «Ждёт вас» на странице инцидента, смотрит дашборд и отмечает шаг.
- Шаг 3 (HTTP) идёт, как только шаг 2 отмечен — тело ответа `POST` сохраняется.
- Шаг 4 (Ручной) снова пауза.
- Шаг 5 (HTTP) выполняется, исполнение заканчивается.

**4. Аудит.** Исполнение остаётся на вкладке **Runbooks** инцидента. Вывод каждого шага — в одном клике. Когда вы пишете постмортем на следующей неделе, не приходится спрашивать «что вернул тот скрипт?» — это там.

## Как runbook-и стыкуются с остальной OneUptime

- **Мониторы** открывают инциденты и оповещения; **правила runbook-ов** превращают эти события в исполнения. Вместе они образуют замкнутый цикл: обнаружить → запустить → отреагировать → зафиксировать.
- **Workspace-интеграции** (Slack, Microsoft Teams) — естественная цель для HTTP-шагов: публиковать обновления, уведомлять каналы.
- **Страницы статуса** часто обновляются как ручной шаг в runbook-е, затрагивающем клиентов.
- **Расписания дежурств** решают, кого позвать; runbook-и решают, что эта персона делает после звонка.
