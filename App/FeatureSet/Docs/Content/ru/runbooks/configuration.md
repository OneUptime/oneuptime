# Конфигурация и безопасность runbook-ов

## Как на самом деле выполняются Bash и JavaScript

Bash- и JavaScript-шаги **никогда не выполняются на OneUptime Worker**. Они отправляются как job'ы на конкретный [Runbook-агент](/docs/runbooks/agents) — небольшой процесс, который вы устанавливаете на хосте в своей инфраструктуре.

Модель диспетчеризации:

1. Автор шага runbook-а выбирает Runbook-агента из выпадающего списка при написании шага.
2. Когда шаг выполняется, Worker вставляет строку в `RunbookAgentJob` с `targetAgentId`, равным ID этого агента, и статусом `Pending`.
3. Этот конкретный агент (и только он) атомарно забирает job, локально запускает скрипт — Bash через `bash -c <скрипт>`, JavaScript в песочнице `isolated-vm` — и присылает результат обратно.
4. Worker продолжает runbook с этим результатом.

Никакого флага окружения `RUNBOOK_BASH_ENABLED` больше нет. Будут ли работать Bash- или JavaScript-шаги в конкретном развёртывании, полностью зависит от того, есть ли в проекте хотя бы один подключённый Runbook-агент.

## Ограничения вывода и тайм-ауты

- Вывод на шаг: **50&nbsp;КБ**. Больший вывод усечён с маркером.
- Execution timeout на шаг по умолчанию: **30 секунд** для JavaScript, Bash и HTTP. Настраивается отдельно для каждого шага.
- **Claim timeout** на шаг для Bash- и JavaScript-шагов: **2 минуты** — сколько Worker ждёт, пока выбранный агент возьмёт job, прежде чем уронить его.

## Права

Права runbook-ов лежат в группе прав `Runbook`:

- `CreateRunbook`, `EditRunbook`, `DeleteRunbook`, `ReadRunbook` — управлять шаблонами runbook-ов.
- `CreateRunbookExecution`, `EditRunbookExecution`, `ReadRunbookExecution` — запускать, отмечать и читать исполнения.
- `CreateRunbookRule`, `EditRunbookRule`, `DeleteRunbookRule`, `ReadRunbookRule` — управлять правилами автозапуска.
- `CreateRunbookAgent`, `EditRunbookAgent`, `DeleteRunbookAgent`, `ReadRunbookAgent` — управлять Runbook-агентами, которые выполняют Bash- и JavaScript-шаги в вашей инфраструктуре.
- `RunbookAdmin`, `RunbookMember`, `RunbookViewer` (роли) — назначайте команде для полного контроля, повседневного использования или доступа только на чтение соответственно. `RunbookAdmin` объединяет все детализированные права выше.

## Очередь и воркер

Исполнения runbook-ов идут по очереди BullMQ `Runbook`. Параллелизм воркера — 25; настройте в своём развёртывании, если у вас много одновременных запусков.

Когда ручной шаг отмечается через API, исполнение снова попадает в очередь, чтобы перейти к следующему шагу. Это сохраняет воркер «горячим» для остатка runbook-а.

## Заметки о харденинге

- **JavaScript и Bash** выполняются на хосте Runbook-агента, который контролируете вы, а не на OneUptime Worker. JavaScript завёрнут в песочницу `isolated-vm` со стандартным прологом (рвёт цепи прототипов, убирает `Function`/`eval`, морозит встроенные прототипы). Bash выполняется через `bash -c` с принудительным таймаутом на агенте.
- **HTTP-шаги** используют разрешительный валидатор статуса, поэтому ответ 4xx/5xx фиксируется как упавший шаг, а не выбрасывается. Захваченный вывод отражает, что реально вернула удалённая сторона.
- **Аутентификация агента** — по ID + секретному ключу, задаваемым в контейнере агента как переменные окружения. На стороне сервера авторитетная идентичность агента берётся из строки БД, ключом для которой служат предъявленные ID/ключ — клиенты не могут выдать себя за другого агента даже со скомпрометированным ключом.

## Таблицы БД

- `Runbook` — шаблон (имя, slug, описание, isEnabled, JSON шагов).
- `RunbookExecution` — одна строка на прогон, с nullable-ссылками `incidentId`, `alertId` и `scheduledMaintenanceId` и JSON-массивом `stepExecutions`, который сохраняет снимок шагов и состояния по каждому шагу.
- `RunbookRule` — правила автозапуска с дискриминатором `triggerEntityType` (Incident, Alert, ScheduledMaintenance) и связью many-to-many с runbook-ами для запуска.
- `RunbookAgent` — одна строка на установленного агента: имя, секретный ключ, `lastAlive`, `connectionStatus`, информация о хосте.
- `RunbookAgentJob` — одна строка на отправленный Bash- или JavaScript-шаг: `targetAgentId` (агент, которого выбрал автор шага), тип шага, скрипт, статус (`Pending` → `Claimed` → `Running` → `Succeeded`/`Failed`/`TimedOut`/`Cancelled`), claim deadline, lease, вывод, exit code.

## Эксплуатационные советы

- **Убедитесь, что выбранный для шага агент здоров.** Если нужна избыточность, поднимите второго агента и разнесите шаги между ними, или держите запасной runbook, нацеленный на другого агента.
- **Сохраняйте URL, а не блобы.** Если шаг генерирует больше нескольких КБ, пишите в S3 или ваш лог-стек и возвращайте URL.
- **Важна идемпотентность.** Автоматические шаги (HTTP, JavaScript, Bash) могут запускаться повторно, если воркер перезапустился посреди шага или если lease агента истёк, пока скрипт ещё работал; проектируйте их так, чтобы повтор был безопасным.
