# Правила runbook-ов

Правила runbook-ов автоматически прикрепляют runbook-и при создании **инцидента**, **оповещения** или **плановой работы**. Управление — из меню «Настройки» каждой сущности:

- Инциденты → Настройки → **Правила runbook-ов**
- Оповещения → Настройки → **Правила runbook-ов**
- Плановые работы → Настройки → **Правила runbook-ов**

Все три страницы редактируют одну и ту же модель правил — они просто отфильтрованы по типу сущности.

## Анатомия правила

| Поле                      | Назначение                                                                            |
| ------------------------- | ------------------------------------------------------------------------------------- |
| **Имя**                   | Короткая читаемая метка. Видна в логах аудита.                                        |
| **Описание**              | Необязательный контекст для коллег.                                                   |
| **Включено**              | Переключатель, позволяющий приостановить правило без удаления.                        |
| **Шаблон заголовка**      | Регулярка без учёта регистра по заголовку сущности. Пусто = подходит любой заголовок. |
| **Шаблон описания**       | Регулярка без учёта регистра по описанию сущности. Пусто = подходит любое описание.   |
| **Runbook-и для запуска** | Один или несколько runbook-ов, которые стартуют при срабатывании правила.             |

## Семантика совпадения

Правило срабатывает, когда **все указанные критерии выполняются**. Пустые критерии пропускаются:

- Правило без шаблонов запускается на каждом событии своего типа (глобальное «всегда»).
- Правило только с шаблоном заголовка срабатывает на событиях, чей заголовок совпадает с регуляркой.
- Несколько правил могут совпасть на одном событии — каждое срабатывает, и объединение их runbook-ов запускается (каждый runbook получает своё исполнение).

## Пример: DB-failover для инцидентов БД

```
Имя:               Запустить DB-failover для DB-инцидентов
Триггер:           Инцидент
Шаблон заголовка:  (?:^|\b)(db|database|postgres|mysql|mongo)
Runbook-и:         [Playbook DB-failover, Уведомить DBA-команду]
```

Это создаст два исполнения runbook-ов при каждом инциденте с «db», «database», «postgres» и т.д. в заголовке.

## Пример: всегда-вкл гигиеническое правило

```
Имя:                  Pre-flight на каждом инциденте
Триггер:              Инцидент
Шаблон заголовка:     (пусто)
Шаблон описания:      (пусто)
Runbook-и:            [Снимок состояния до инцидента]
```

Срабатывает на каждом инциденте — полезно для снимков состояния системы, страничных метрик и т.п.

## Что происходит при срабатывании правила

1. Runbook загружается.
2. Его шаги копируются как **снимок** в новое исполнение.
3. Исполнение ставится в очередь воркера Runbook.
4. Исполнение связывается с источником — оно появляется на странице инцидента, оповещения или плановой работы и в списке исполнений runbook-а.

Все запуски, инициированные правилами, видны в **Runbooks → Исполнения** с фильтрами по статусу, runbook-у или дате.

## Отключённые runbook-и

Если правило ссылается на runbook с `isEnabled = false`, правило по-прежнему совпадает, но исполнение пропускается. Включите runbook, чтобы возобновить.

## Проверка правила

Прежде чем полагаться на правило в продакшене, создайте тестовый инцидент (или оповещение) с заголовком, подходящим под шаблон, и убедитесь, что нужные runbook-и запускаются. Правила оцениваются в момент создания — последующее редактирование заголовка инцидента правил не перезапускает.
