# Runbook-агенты

**Runbook-агент** — это небольшой самостоятельно размещаемый процесс, который выполняет Bash- *и* JavaScript-шаги ваших runbook'ов **внутри вашей собственной инфраструктуры**. OneUptime Worker никогда не запускает ваши скрипты у себя — он ставит их в очередь, а Runbook-агент, который вы установили в своей среде, забирает их, выполняет и присылает результат обратно.

JavaScript по-прежнему работает в песочнице `isolated-vm`; разница в том, что эта песочница живёт на вашем хосте-агенте, а не на нашем.

Эта страница описывает, как установить агента, маршрутизировать к нему Bash- и JavaScript-шаги и эксплуатировать его в повседневной работе.

## Зачем нужны агенты

Прежние версии OneUptime выполняли Bash- и JavaScript-шаги на Worker'е. JavaScript был в песочнице (`isolated-vm`), Bash — нет. Оба варианта создавали проблемы для всего, что выходит за рамки single-tenant самохостной установки:

- **Граница доверия.** Любой, кто мог написать runbook, мог выполнить код на Worker'е с доступом ко всем переменным окружения и файловой системе Worker'а. Песочница JavaScript блокировала очевидное, но не могла помешать решительному пользователю прозондировать, что доступно из нашей сети.
- **Досягаемость.** Большинство полезных шагов хотят работать с инфраструктурой *клиента* («перезапусти этот сервис», «kubectl на нашем кластере», «найди запись в нашей внутренней БД»), а не OneUptime.

Runbook-агенты переворачивают это. Bash- и JavaScript-шаги не выполняются у нас. Они выполняются на хосте, которым управляете вы, и вы решаете, что этот хост может делать.

## Как это работает

1. Вы создаёте Runbook-агента в OneUptime. OneUptime генерирует ID и секретный ключ.
2. Вы запускаете контейнер агента на хосте внутри вашей инфраструктуры с этим ID/ключом и вашим OneUptime URL.
3. Агент каждые несколько секунд спрашивает OneUptime: «есть для меня работа?»
4. Когда выполняется Bash- или JavaScript-шаг, Worker вставляет строку job'а, помеченную **Agent Tag** этого шага и типом шага (Bash или JavaScript), и устанавливает статус `Pending`.
5. Любой здоровый агент в том же проекте, несущий этот тег, атомарно забирает job (никогда два агента не выполняют один и тот же job), запускает его локально — `bash -c <скрипт>` для Bash, песочница `isolated-vm` для JavaScript — захватывает результат и присылает обратно.
6. Worker продолжает runbook с этим результатом.

Агенту нужен только **исходящий HTTPS** к вашему экземпляру OneUptime. Никаких входящих подключений он не принимает.

## Установка агента

### 1. Создайте запись агента

Перейдите в **Runbooks → Agents → Создать**. Заполните:

| Поле | Заметки |
| --- | --- |
| **Имя** | Говорящее имя — обычно `где-он-работает-и-что-умеет`, например `prod-eu-west-1`. |
| **Описание** | Опционально. Одно предложение о том, до чего этот хост может достучаться. Будущий вы скажет спасибо. |
| **Tags** | Через запятую. Bash- и JavaScript-шаги нацеливаются на тег; любой агент в проекте с этим тегом может их выполнять. Частые шаблоны: `prod`, `staging`, `eu-west-1`, `db-host`. |

### 2. Скопируйте команду установки

После создания нажмите **Показать инструкции по установке** в строке агента. Вы увидите команду `docker run` с уже подставленными ID и ключом этого агента. **Сохраните ключ сейчас** — позже его можно сбросить, но то же самое значение после закрытия модального окна вы уже не увидите.

### 3. Запустите её на хосте внутри вашей инфраструктуры

Запустите Docker-команду на любом хосте в вашей среде, который может:

- достучаться до вашего экземпляра OneUptime по HTTPS, и
- делать то, что должны делать ваши Bash-шаги (например, SSH к другим хостам, `kubectl`, обращение к базе).

```bash
docker run --name oneuptime-runbook-agent --restart unless-stopped \
  -e RUNBOOK_AGENT_ID=<agent-id> \
  -e RUNBOOK_AGENT_KEY=<agent-key> \
  -e ONEUPTIME_URL=https://oneuptime.ваш-домен.com \
  -d oneuptime/runbook-agent:release
```

### 4. Убедитесь, что агент подключился

Вернитесь в **Runbooks → Agents**. Примерно за 60 секунд строка агента должна переключиться на `Connected` со свежей меткой **Last seen**. Если остаётся `Disconnected`:

- Посмотрите логи контейнера (`docker logs oneuptime-runbook-agent`) на предмет ошибок аутентификации или сети.
- Убедитесь, что хост достигает OneUptime URL через `curl`.
- Убедитесь, что ID и ключ скопированы без пробелов.

## Теги и маршрутизация

Теги — это то, как Bash- или JavaScript-шаг находит агента. Несколько шаблонов:

- **Один тег на окружение.** Пометьте prod-агента `prod`, staging-агента `staging`. Bash-шаги, нацеленные на `prod`, идут только в prod.
- **Один тег на регион.** `eu-west-1`, `us-east-1`. Полезно, когда шагу нужно выполняться близко к ресурсу, который он трогает.
- **Несколько агентов, один тег.** Поднимите двух агентов, оба с тегом `prod`. Любой может забрать job — получаете высокую доступность и можете делать rolling restart, не ломая runbook'и.
- **Несколько тегов на агента.** Агент в prod-EU кластере может нести `prod`, `eu-west-1` и `kubernetes`. Bash-шаги могут нацеливаться на любой из них.

Bash- и JavaScript-шаги **обязаны** каждый указать ровно один тег агента. Маршрутизация по нескольким тегам (выполнить на любом агенте, у которого есть `prod` AND `db`) — в дорожной карте, но не в этом релизе.

## Направить шаг на агента

Добавьте в runbook Bash- или JavaScript-шаг. Форма попросит **Agent Tag**:

- Введите тег, соответствующий агенту/агентам, на которых хотите запускать.
- Напишите скрипт в редакторе ниже.

Когда runbook доходит до шага, Worker ставит в очередь job с этим тегом и типом шага. Если хотя бы один здоровый агент с этим тегом онлайн, job забирается в течение нескольких секунд и выполняется. Bash выполняется через `bash -c`; JavaScript работает внутри песочницы `isolated-vm` на агенте (без файловой системы, без сети, без `Function`/`eval`).

## Эксплуатационные заметки

### Тайм-ауты

К каждому Bash- или JavaScript-шагу применяются два тайм-аута:

| Тайм-аут | По умолчанию | Что регулирует |
| --- | --- | --- |
| **Claim timeout** | 2 минуты | Сколько Worker ждёт, пока *какой-нибудь* агент возьмёт job. Если никто не взял вовремя, шаг падает с `TimedOut`, и runbook продолжается (или останавливается, в зависимости от **Продолжать при ошибке**). |
| **Execution timeout** | 30 секунд | Сколько агент даёт скрипту отработать, прежде чем завершить его. Настраивается на каждом шаге. (Bash получает `SIGKILL`; изолят JavaScript разбирается.) |

Общее окно ожидания Worker'а равно `claim timeout + execution timeout + несколько секунд запаса`. Выбирайте значения под шаг.

### Lease и heartbeat

Когда агент забирает job, он получает короткий lease (по умолчанию 30 секунд). Пока скрипт работает, агент продлевает lease каждые 10 секунд. Если агент умирает или теряет сеть посреди скрипта, lease истекает, и Worker помечает job как `TimedOut`, а не ждёт вечно.

Дочерние процессы Bash **не** отменяются автоматически при истечении lease (изолят JavaScript тоже досчитывает до конца, если когда-нибудь это сделает) — но Worker перестаёт их ждать, и агент не сможет отправить результат, как только другой claim взял дело на себя. Проектируйте скрипты так, чтобы их было безопасно перезапускать, если для вас важно exactly-once.

### Ни одного агента онлайн

Если на момент выполнения шага ни один здоровый агент с его тегом не онлайн, job остаётся `Pending` до истечения claim timeout и затем падает с понятным сообщением («no agent claimed the job»). Страница Agents — место, где вы убеждаетесь в покрытии перед серьёзным запуском runbook'а.

### Лимит вывода

Совмещённые stdout + stderr ограничены **50 КБ** на шаг. Больший вывод обрезается с маркером. Если нужен полный лог — пишите его в S3 или ваш log-store прямо из скрипта и `echo`-айте URL.

### Отмена

Отмена выполнения runbook'а (из представления выполнения или через API) сразу помечает все его `Pending`/`Claimed`/`Running` Bash-job'ы как `Cancelled`. Агент, уже выполняющий скрипт, доработает свою часть, но сервер его результат не примет.

### Параллелизм

По умолчанию каждый агент выполняет одну job за раз. Чтобы разрешить больше, задайте `RUNBOOK_AGENT_CONCURRENCY` в контейнере агента — но помните, что агент делит хост со всем остальным, что там живёт.

## Переменные окружения

Агент читает их при старте:

| Переменная | Обязательная | По умолчанию | Заметки |
| --- | --- | --- | --- |
| `ONEUPTIME_URL` | да | — | Базовый URL вашего OneUptime, например `https://oneuptime.ваш-домен.com`. |
| `RUNBOOK_AGENT_ID` | да | — | UUID из модального окна настройки агента. |
| `RUNBOOK_AGENT_KEY` | да | — | Секрет из модального окна настройки агента. |
| `RUNBOOK_AGENT_POLL_INTERVAL_MS` | нет | `5000` | Как часто агент опрашивает наличие новых job'ов. |
| `RUNBOOK_AGENT_HEARTBEAT_INTERVAL_MS` | нет | `60000` | Как часто агент сообщает, что жив. |
| `RUNBOOK_AGENT_JOB_HEARTBEAT_INTERVAL_MS` | нет | `10000` | Как часто агент продлевает lease работающего job'а. |
| `RUNBOOK_AGENT_CONCURRENCY` | нет | `1` | Максимум одновременных job'ов на этом агенте. |

## Ротация ключа агента

Если ключ утёк, откройте агента в OneUptime и сбросьте его ключ. Старый перестаёт работать немедленно. Обновите контейнер агента новым ключом и перезапустите его.

## Права доступа

Управление агентами живёт в существующей группе прав Runbooks:

- `CreateRunbookAgent`, `EditRunbookAgent`, `DeleteRunbookAgent`, `ReadRunbookAgent` — управление записями агентов.
- `RunbookManager` (роль) — объединяет всё вышеперечисленное.

Права на *запуск* runbook'а (и, как следствие, диспетчеризацию Bash-шагов) по-прежнему `CreateRunbookExecution` / `EditRunbookExecution`.

## API для агентов

Для любопытных — агент использует эти эндпойнты, смонтированные под `/runbook-agent-ingest`. Они аутентифицируются по ID + ключу агента в JSON-теле (или заголовкам `x-agent-id` / `x-agent-key`).

| Эндпойнт | Назначение |
| --- | --- |
| `POST /heartbeat` | Живость; обновляет `lastAlive`, `connectionStatus`, `hostInfo`, `agentVersion`. |
| `POST /claim-next-job` | Атомарно забирает самый старый `Pending` job, тег которого совпадает с одним из тегов этого агента. Возвращает `{ job: null }`, если делать нечего. |
| `POST /job/:jobId/heartbeat` | Продлевает lease job'а. Возвращает 404, как только lease истёк или job достиг терминального статуса. |
| `POST /job/:jobId/result` | Отправляет финальный результат. Игнорируется, если lease уже ушёл к другому. |

Вызывать их вручную вам не понадобится — поставляемый агент это делает сам. Они задокументированы здесь на случай, если вы захотите построить свой агент, потому что наш не подходит под ваши ограничения.
