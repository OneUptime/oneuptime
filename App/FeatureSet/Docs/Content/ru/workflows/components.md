# Компоненты

Компоненты — это строительные блоки, которые вы добавляете после триггера. Каждый делает что-то одно — отправляет сообщение, вызывает API, проверяет условие — и соединяется с тем, что идёт дальше.

Эта страница — каталог. О том, как перетаскивать, ронять и соединять блоки на холсте, читайте в разделе [Создание рабочего процесса](/docs/workflows/authoring).

## API

Выполняет HTTP-запрос по произвольному URL.

**Настройки**:

- **Method** — `GET`, `POST`, `PUT`, `PATCH` или `DELETE`.
- **URL** — адрес для вызова.
- **Headers** — заголовки, которые нужно отправить.
- **Body** — тело запроса для `POST` / `PUT` / `PATCH`.

**Выходы**:

- **Success** — срабатывает, когда вызов прошёл успешно (ответ 2xx). Передаёт дальше статус, заголовки и тело.
- **Error** — срабатывает при сетевой ошибке или ответе не 2xx. Передаёт дальше сообщение об ошибке.

Используйте для: любого внешнего API, ваших собственных административных эндпоинтов или любой интеграции, у которой нет отдельного компонента.

## Webhook (исходящий)

Упрощённая версия компонента API для случаев «отправил и забыл». Отправляет JSON-тело по URL.

Используйте **API**, если нужно прочитать ответ. Используйте **Webhook**, если просто нужно отправить уведомление и двигаться дальше.

## Slack

Публикует сообщение в канал Slack.

**Настройки**:

- **Channel** — имя канала. Бот уже должен быть в этом канале.
- **Message** — текст для отправки. Поддерживает форматирование Slack.

Сначала подключите Slack к вашему проекту в разделе **Project Settings → Workspace Connections → Slack**. См. [Подключение рабочего пространства Slack](/docs/workspace-connections/slack).

## Microsoft Teams

Публикует сообщение в канал Microsoft Teams.

**Настройки**:

- **Team and channel** — куда отправлять.
- **Message** — текст для отправки.

Настройка описана в разделе [Подключение рабочего пространства Microsoft Teams](/docs/workspace-connections/microsoft-teams).

## Discord

Публикует сообщение в канал Discord через URL входящего webhook-а.

## Telegram

Отправляет сообщение в чат Telegram, используя токен бота и ID чата.

## Email

Отправляет письмо через OneUptime.

**Настройки**:

- **To** — адрес электронной почты получателя.
- **Subject** — строка темы.
- **Body** — текст письма в Markdown или HTML.

Письмо отправляется с настроенного отправителя вашего проекта — см. [SMTP](/docs/emails/smtp).

## Custom Code

Запускает небольшой фрагмент JavaScript, когда нужно сделать то, что не умеют другие блоки.

**Настройки**:

- **Code** — ваш JavaScript. Последнее значение (или то, что вы возвращаете из асинхронной функции) становится выходом блока.
- **Arguments** — именованные значения, которые можно передать внутрь.

**Выходы**: success (ваше возвращаемое значение) и error (любое исключение).

Используйте для: преобразования данных между двумя системами, небольших вычислений, всего того, что не заслуживает отдельного блока. Для более тяжёлых скриптов используйте [Runbook](/docs/runbooks/index).

## JSON

Преобразует данные между текстом и JSON.

- **JSON → Text** — превратить JSON-объект в строку. Полезно, когда следующий блок ожидает текст.
- **Text → JSON** — разобрать строку в JSON-объект. Полезно, когда что-то пришло как текст, а вам нужно прочитать поле.

## Conditions

Ветвление по сравнению.

**Настройки**:

- **Left value** — обычно значение из предыдущего блока.
- **Operator** — `==`, `!=`, `>`, `>=`, `<`, `<=`, `contains`, `starts with`, `ends with`.
- **Right value** — то, с чем сравнивать.

**Выходы**: **Yes** и **No**. Соедините следующие блоки с той ветвью, которая вам нужна.

## Delay

Приостанавливает рабочий процесс на заданное время перед продолжением. Полезно, когда нужно дать другой системе момент, чтобы догнать.

## Log

Пишет строку в журнал запуска. Никаких внешних эффектов — это значение просто появляется в журналах рабочего процесса, чтобы вы могли его прочитать. Удобно для отладки.

## Execute Workflow

Вызывает другой рабочий процесс из этого. Вызванный рабочий процесс выполняется самостоятельно — ваш рабочий процесс продолжает работу, не дожидаясь его завершения.

Используйте это, чтобы переиспользовать общую логику. Соберите один раз рабочий процесс «опубликовать в канале инцидента» — и вызывайте его из любого другого, которому нужно уведомить этот канал.

Есть предохранитель, чтобы рабочие процессы не зацикливались, вызывая друг друга по кругу. См. [Конфигурация и безопасность](/docs/workflows/configuration).

## Компоненты для данных OneUptime

Для каждой сущности в OneUptime (мониторов, инцидентов, оповещений, страниц статуса, политик дежурств и многих других) в палитре есть набор компонентов — ищите по имени типа:

- **Find One** — получить одну запись по ID или фильтру.
- **Find** — получить список записей.
- **Create** — добавить новую запись.
- **Update** — изменить одну запись.
- **Delete** — удалить одну запись.
- **Count** — подсчитать записи, подходящие под фильтр.

Так рабочий процесс может читать и изменять данные OneUptime. Например: webhook от вашего CI-инструмента может через **Create Incident** открыть инцидент с подробностями сбоя.

## Какой компонент выбрать?

Несколько быстрых правил:

- Если для нужного действия есть отдельный блок (Slack, Email, запись OneUptime), используйте его — получите более удобную обработку ошибок и более понятные журналы.
- Для любого другого внешнего API используйте **API**.
- Чтобы преобразовать данные между блоками, используйте **Custom Code** или **JSON**.
- Чтобы выполнять разные действия в зависимости от значения, используйте **Conditions**.
