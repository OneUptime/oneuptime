# Компоненты рабочего процесса

Компоненты — это узлы-действия, которые вы размещаете после триггера. Каждый выполняет одну задачу — делает HTTP-запрос, отправляет сообщение в Slack, ветвится по условию, выполняет JavaScript-сниппет — и предоставляет один или несколько выходных портов, к которым подключается следующий узел.

Эта страница — каталог. Правила соединения и сам холст см. в разделе [Создание рабочего процесса](/docs/workflows/authoring).

## API

Сделать исходящий HTTP-запрос на любой URL.

**Аргументы**:

- **Method** — `GET`, `POST`, `PUT`, `PATCH`, `DELETE`.
- **URL** — URL запроса. Поддерживает интерполяцию.
- **Request Headers** — JSON-объект заголовков.
- **Request Body** — JSON или текстовое тело для `POST` / `PUT` / `PATCH`.

**Выходные порты**:

- `success` — срабатывает, когда статус ответа 2xx. Возвращаемые значения: `response-status`, `response-headers`, `response-body`.
- `error` — срабатывает при сетевом сбое или ответе не-2xx. Возвращаемое значение: сообщение `error`.

Используйте для: любого стороннего REST API, ваших собственных admin-endpoint-ов, лёгких интеграций, для которых нет выделенного компонента.

## Webhook (исходящий)

Тонкая обёртка над компонентом API для типичного случая «выстрелил и забыл». Отправляет JSON-тело на URL и предоставляет одну пару `success` / `error`.

Предпочитайте **API**, если нужно прочитать тело ответа ниже по потоку; предпочитайте **Webhook**, если просто хотите уведомить другую систему.

## Slack

Опубликовать сообщение в канале Slack через workspace-подключение Slack вашего проекта.

**Аргументы**:

- **Channel name** — канал, в который публиковать. Бот должен уже быть участником этого канала.
- **Message text** — тело сообщения. Поддерживает интерполяцию; поддерживает Slack mrkdwn.

Сначала настройте workspace-подключение в **Project Settings → Workspace Connections → Slack**. См. [Slack Workspace Connection](/docs/workspace-connections/slack).

## Microsoft Teams

Опубликовать сообщение в канале Microsoft Teams через подключение Teams вашего проекта.

**Аргументы**:

- **Team & channel** — адресат.
- **Message text** — тело сообщения.

Настройка подключения описана в [Microsoft Teams Workspace Connection](/docs/workspace-connections/microsoft-teams).

## Discord

Опубликовать сообщение в канале Discord через входящий webhook-URL, настроенный на компоненте.

## Telegram

Отправить сообщение в чат Telegram через токен бота и chat ID, настроенные на компоненте.

## Email

Отправить email через конфигурацию SMTP в OneUptime.

**Аргументы**:

- **To** — email-адрес получателя.
- **Subject** — тема, поддерживает интерполяцию.
- **Body** — Markdown или HTML.

Email отправляется с настроенного для проекта адреса отправителя (см. [SMTP](/docs/emails/smtp)).

## Custom Code

Запустить сниппет JavaScript с доступом к переменным рабочего процесса и возвращаемым значениям узла выше по потоку.

**Аргументы**:

- **Code** — тело JavaScript. Значение последнего выражения (или то, что возвращено из `(async () => { ... })()`) становится возвращаемым значением компонента.
- **Arguments** — опциональные именованные значения, передаваемые как `args`.

**Выходные порты**: `success` (возвращаемое значение), `error` (пойманное исключение).

Используйте для: преобразования payload-а между двумя системами, небольшого вычисления, которое не заслуживает собственного компонента, вызова JS-only-логики. Более тяжёлые скрипты, которые должны выполняться внутри вашей собственной инфраструктуры, относятся к шагу Bash или JavaScript в [Runbook](/docs/runbooks/index).

## JSON

Конвертация между текстом и JSON.

- **JSON → Text** — сериализация JSON-объекта в строку (удобно для подачи в аргумент `body` исходящего компонента, ожидающего текст).
- **Text → JSON** — разбор строки в JSON-объект. Полезно, когда вышестоящий API вернул тело как текст, но нужно прочитать поле.

## Conditions

Ветвление по сравнению. Настройте:

- **Left value** — обычно интерполированная ссылка вроде `{{Incident.title}}`.
- **Operator** — `==`, `!=`, `>`, `>=`, `<`, `<=`, `contains`, `starts with`, `ends with`.
- **Right value** — значение для сравнения.

**Выходные порты**: `yes` и `no`. Подключайте остаток рабочего процесса к той ветке, которая соответствует вашему намерению.

## Schedule (задержка)

Приостановить рабочий процесс на заданное время перед продолжением. Полезно, когда нужно дать внешней системе момент устаканиться перед проверкой её состояния.

## Log

Записать строку в журнал запуска рабочего процесса. Чистое средство отладки; строка захватывается на запуске и видна в **Logs**. Никаких внешних побочных эффектов.

## Execute Workflow

Вызвать другой рабочий процесс как под-шаг. Вызванный рабочий процесс выполняется независимо (fire-and-forget) — управление возвращается вызывающему сразу после отправки вызова.

Используйте для вынесения общей логики из нескольких рабочих процессов: построите «отправить-в-канал-инцидента» один раз и вызываете его из каждого другого рабочего процесса, которому нужно уведомить канал.

Лимит рекурсии предотвращает бесконечный цикл вызовов рабочих процессов. См. [Конфигурация и безопасность](/docs/workflows/configuration).

## Компоненты моделей (CRUD над сущностями OneUptime)

Для каждой сущности OneUptime, поддерживающей рабочие процессы (мониторы, инциденты, оповещения, страницы статуса, политики дежурств и т. д.), палитра автоматически предоставляет следующие компоненты — поиск по имени сущности:

- **Find One {Entity}** — получить одну запись по запросу.
- **Find {Entity}** — получить список записей по запросу (с пагинацией).
- **Create {Entity}** — вставить новую запись.
- **Update {Entity}** — обновить одну запись по ID.
- **Delete {Entity}** — удалить одну запись по ID.
- **Count {Entity}** — подсчитать записи, соответствующие запросу.

Так рабочий процесс может читать и записывать состояние OneUptime, не покидая платформу. Например: webhook от вашего CI-инструмента вызывает **Create Incident** с сообщением о сбое сборки; или запланированный рабочий процесс каждые пять минут запускает **Find Incident** и отправляет сводку по email.

## Выбор правильного компонента

Несколько быстрых эвристик:

- Если для того, что вы хотите сделать, существует выделенный компонент (Slack, Email, CRUD над сущностью OneUptime), используйте его — он даёт лучшую обработку ошибок и более ясные журналы, чем самописное решение.
- Если нужно вызвать внешний HTTP API, для которого нет выделенного компонента, используйте **API**.
- Если нужно *преобразовать* данные между двумя компонентами, используйте **Custom Code** или **JSON**.
- Если нужно совершить разные действия в зависимости от значения, используйте **Conditions**.
