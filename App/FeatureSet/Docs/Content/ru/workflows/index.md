# Обзор рабочих процессов

Рабочие процессы позволяют автоматизировать задачи в OneUptime без написания кода. Перетащите несколько блоков на холст, соедините их между собой — и у вас готова автоматизация, которая запускается каждый раз, когда что-то происходит: открывается инцидент, срабатывает расписание или другой инструмент отправляет данные в OneUptime.

Думайте о рабочих процессах как о фоновых помощниках для вашего проекта: они реагируют на события, общаются с другими инструментами и тихо поддерживают всё в синхронизированном состоянии, пока вы сосредоточены на своей работе.

## Что можно делать с помощью рабочих процессов

- **Подключить OneUptime к другим инструментам** — отправлять инциденты в Slack, создавать тикеты в Jira, отправлять данные на webhook в вашем стеке.
- **Реагировать на события в OneUptime** — когда создаётся критический инцидент, автоматически уведомить дежурную команду и открыть тикет.
- **Запускать задачи по расписанию** — каждые пять минут, каждую ночь, каждое утро понедельника.
- **Получать данные извне** — позволить другим системам отправлять данные в OneUptime через уникальный URL.
- **Переиспользовать типовую автоматизацию** — собрать её один раз и вызывать из любого другого рабочего процесса.

## Как устроен рабочий процесс

Каждый рабочий процесс состоит из трёх частей:

1. **Триггер** — то, что запускает рабочий процесс. Это может быть ручная кнопка, расписание, входящий webhook или событие в OneUptime (например, новый инцидент).
2. **Один или несколько компонентов** — то, что делает рабочий процесс. Отправить сообщение, выполнить HTTP-вызов, провести быструю проверку, разветвиться по условию.
3. **Связи между ними** — вы рисуете линии от одного блока к следующему, чтобы определить порядок.

Всё это вы собираете визуально на холсте. Для большинства рабочих процессов код не нужен, хотя при необходимости можно добавить фрагмент JavaScript.

## Ключевые понятия

| Термин                         | Смысл                                                                                                          |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| **Рабочий процесс (Workflow)** | Вся автоматизация целиком — имя, холст и переключатель для включения и выключения.                             |
| **Триггер**                    | Первый блок. Он определяет, когда запускается рабочий процесс. У каждого рабочего процесса ровно один триггер. |
| **Компонент**                  | Блок действия — отправляет сообщение, выполняет запрос, проверяет условие.                                     |
| **Запуск (Run)**               | Одно выполнение рабочего процесса. Сохраняется с метками времени и выводом каждого блока.                      |
| **Глобальная переменная**      | Значение (например, API-ключ), которое вы сохраняете один раз и переиспользуете в любом рабочем процессе.      |

## Где найти рабочие процессы в OneUptime

Откройте **Workflows** в левой панели навигации. Оттуда доступно:

- **Workflows** — ваш список рабочих процессов. Создайте новый или откройте существующий.
- **Вкладка Builder** — холст, на котором вы проектируете рабочий процесс.
- **Вкладка Logs** — каждое выполнение этого рабочего процесса с подробностями.
- **Вкладка Settings** — имя, описание, владельцы, метки, включение/отключение.
- **Global Variables** — значения, общие для всех ваших рабочих процессов.
- **Runs & Logs** — история выполнений по всем рабочим процессам в вашем проекте.

## Создание первого рабочего процесса

1. **Создайте** — задайте рабочему процессу имя и короткое описание.
2. **Выберите триггер** — ручной, по расписанию, webhook или событие из OneUptime.
3. **Добавьте компоненты** — перетащите действия на холст и соедините их.
4. **Протестируйте** — нажмите **Run Manually** и понаблюдайте за происходящим в журналах.
5. **Включите** — переключите **Enabled** в Settings, когда будете готовы.

## Быстрый пример

Допустим, вы хотите публиковать сообщение в Slack каждый раз, когда создаётся критический инцидент:

1. Создайте рабочий процесс с именем «Critical incidents to Slack».
2. Выберите триггер **Incident → On Create**.
3. Добавьте блок **Conditions**. Настройте его так, чтобы он проверял, содержит ли заголовок инцидента «Sev 1».
4. Из ветви **Yes** добавьте блок **Slack**. Выберите канал и напишите сообщение.
5. Включите рабочий процесс.

В следующий раз, когда кто-то откроет инцидент с «Sev 1» в заголовке, Slack засветится сообщением.

## Как рабочие процессы вписываются в остальной OneUptime

- **Мониторы** замечают проблему. **Инциденты** её фиксируют. **Рабочие процессы** на неё реагируют.
- **Runbook-и** — это пошаговые инструкции для людей. Рабочие процессы — это автоматизация без участия человека. Используйте runbook, когда решения должен принимать человек; используйте рабочий процесс, когда шаги автоматические.
- **Подключения рабочих пространств** (Slack, Teams) — это то, куда рабочие процессы отправляют свои сообщения.
