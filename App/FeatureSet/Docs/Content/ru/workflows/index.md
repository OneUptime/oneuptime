# Обзор рабочих процессов

Рабочие процессы (workflows) — это визуальный конструктор автоматизации в OneUptime. Перетащите триггер на холст, свяжите его с цепочкой действий — HTTP-вызовами, сообщениями в Slack, JavaScript-сниппетами, условными ветвлениями, запросами к базе данных — и вы получите автоматизацию, которая запускается каждый раз, когда происходит событие в OneUptime (или во внешнем мире).

Если runbook-и — это чек-листы для людей во время инцидента, то рабочие процессы — это фоновые задачи для вашего проекта: они выполняются без присмотра, реагируют на события и связывают OneUptime с остальной частью вашего стека.

## Кратко

- **Функция верхнего уровня** в панели OneUptime: **Workflows**.
- **Три стиля триггеров**: ручной (Manual), по расписанию (Schedule, cron), webhook — плюс **триггер событий модели**, срабатывающий, когда любая сущность OneUptime (инцидент, оповещение, монитор, страница статуса и т. д.) создаётся, обновляется или удаляется.
- **Визуальный холст**: перетаскивайте узлы из палитры компонентов, соединяйте выходные порты с входными.
- **Смешанная автоматизация**: HTTP-запросы, сообщения в Slack / Discord / Microsoft Teams / Telegram, собственный JavaScript, разбор JSON, условия, email, вызовы вложенных рабочих процессов и CRUD-операции над моделями OneUptime.
- **Глобальные переменные**: общие для проекта секреты и конфигурация, на которые можно ссылаться из любого рабочего процесса без копирования-вставки.
- **Запуски и журналы**: каждое выполнение записывается со статусом, временем и пошаговым выводом.

## Зачем нужны рабочие процессы?

Большинство команд берутся за рабочие процессы, когда хотят:

- **Подключить OneUptime к другой системе** — отправить инцидент в PagerDuty, отзеркалить оповещение в Jira, дёрнуть webhook в вашем стеке.
- **Реагировать на события OneUptime** — когда открывается инцидент `Sev 1`, вызвать дежурного менеджера, *и* создать тикет в Linear, *и* заблокировать фича-флаг.
- **Запускать повторяющиеся задачи по расписанию** — каждые пять минут опрашивать внутренний API и записывать результат во внешнюю систему.
- **Получать данные извне OneUptime** — webhook от CI-системы запускает цепочку обновлений в OneUptime.
- **Переиспользовать небольшие куски «склеивающей» логики** — один рабочий процесс вызывает другой, и типовые паттерны живут в одном месте.

## Ключевые понятия

| Термин | Смысл |
| --- | --- |
| **Workflow** | Холст. Названный многоразовый граф триггеров и компонентов с флагом `isEnabled`. |
| **Триггер** | Узел, который запускает выполнение рабочего процесса. Ручной, по расписанию, webhook или событие модели. У каждого рабочего процесса ровно один триггер. |
| **Компонент** | Узел, выполняющий работу — HTTP-вызов, сообщение в Slack, JavaScript-сниппет, условие и т. д. |
| **Порт** | Входной или выходной разъём узла. У компонентов есть выходные порты вроде `success` и `error`; вы соединяете порт со входным портом следующего узла. |
| **Запуск / журнал** | Одно выполнение рабочего процесса. Содержит метку времени, статус (Running, Success, Failed, Timeout) и захваченный вывод каждого выполнившегося узла. |
| **Глобальная переменная** | Именованное значение (часто секрет или API-ключ), определяемое один раз на уровне проекта и доступное из любого рабочего процесса как `{{variable.NAME}}`. |
| **Локальная переменная** | Значение, ограниченное одним запуском рабочего процесса — обычно возвращаемое значение более раннего узла, доступное как `{{ComponentId.portName}}`. |

## Где рабочие процессы живут в дашборде

| Страница | Что вы там делаете |
| --- | --- |
| **Workflows** | Просматривать, создавать и искать шаблоны рабочих процессов. |
| **Вкладка «Builder» рабочего процесса** | Холст с drag-and-drop. Добавлять узлы, соединять порты, конфигурировать аргументы. |
| **Вкладка «Logs» рабочего процесса** | Все запуски этого рабочего процесса с фильтрами по статусу и временному диапазону. Кликните по запуску, чтобы увидеть пошаговый вывод. |
| **Вкладка «Settings» рабочего процесса** | Переименовать, включить/выключить, изменить описание, управлять метками, удалить. |
| **Workflows → Global Variables** | Определять значения, общие для проекта, на которые ссылается любой рабочий процесс. Отметьте значение как секрет, чтобы скрыть его в UI после сохранения. |
| **Workflows → Runs & Logs** | История выполнения по всему проекту для всех рабочих процессов. |

## Жизненный цикл рабочего процесса

1. **Создать** — создайте рабочий процесс, бросьте триггер на холст, добавьте нужные компоненты, соедините их и сконфигурируйте каждый.
2. **Включить** — рабочие процессы поставляются выключенными. Переключите тумблер в Settings, когда уверены, что схема собрана правильно.
3. **Запустить** — ручной: нажмите **Run Manually** с опциональным JSON-payload. По расписанию: срабатывает cron. Webhook: внешняя система делает `POST` на URL рабочего процесса. Событие модели: кто-то (или другой рабочий процесс) создаёт/обновляет/удаляет монитор, инцидент, оповещение и т. д.
4. **Выполнить** — Workflow Worker обходит граф по порядку. Каждый компонент читает свои аргументы (литеральные значения или интерполированные переменные), выполняет свою работу, записывает возвращаемое значение и выбирает выходной порт. Следующий узел срабатывает.
5. **Аудит** — запуск появляется в **Logs**. Статус, общая длительность, пошаговый вывод и любые ошибки сохраняются на всё время жизни проекта.

## Развёрнутый пример

Цель: когда создаётся инцидент с `Sev 1` в заголовке, опубликовать сообщение в канале Slack и открыть тикет в вашем внутреннем admin-инструменте.

**1. Создайте рабочий процесс** под названием «Sev 1 fan-out».

**2. Бросьте триггер.** Возьмите триггер **Incident → On Create** из палитры. Триггер делает новый инцидент доступным как возвращаемое значение.

**3. Бросьте компонент Conditional.** Соедините выходной порт триггера с его входом. Установите условие: `{{Incident.title}}` *contains* `Sev 1`.

**4. От порта `yes` Conditional бросьте компонент Slack.** Канал: `#incident-room`. Текст сообщения: `Sev 1 declared: {{Incident.title}} — {{Incident.dashboardUrl}}`.

**5. От того же порта `yes` (параллельно) бросьте компонент API.** `POST` на `https://admin.internal/incidents`. Тело: небольшой JSON-объект, собранный из инцидента.

**6. Включите рабочий процесс.** Откройте в другой вкладке инцидент с заголовком «Sev 1 — checkout 500s». Через несколько секунд приходит сообщение в Slack, а в **Logs** появляется новый запуск с захваченным выводом каждого узла.

## Как рабочие процессы стыкуются с остальной OneUptime

- **Мониторы** обнаруживают проблемы; **инциденты/оповещения** их фиксируют; **рабочие процессы** на них реагируют — публикуют сообщения, открывают тикеты, запускают автоматизацию.
- **Runbook-и** — это процедуры реагирования для людей (с опциональными скриптовыми шагами). Рабочие процессы — это фоновая автоматизация без присмотра. Они дополняют друг друга — шаг runbook-а может сделать `POST` на webhook-триггер рабочего процесса.
- **Workspace-подключения** (Slack, Microsoft Teams) — типичные адресаты уведомлений рабочего процесса.
- **Дашборды** — это представления только для чтения; рабочие процессы — это сторона записи: они обновляют состояние OneUptime, вызывают внешние API и перемещают данные.
