# Конфигурация и безопасность рабочего процесса

На этой странице собраны настройки и пределы безопасности, о которых стоит знать, прежде чем направить рабочий процесс на production-трафик.

## Включение / выключение

У каждого рабочего процесса есть флаг **isEnabled** в **Settings**. Выключенные рабочие процессы никогда не срабатывают — события моделей, webhook-и и запуски по расписанию игнорируются. Новые рабочие процессы поставляются выключенными.

Считайте это своим переключателем «готов к prod»:

1. Соберите рабочий процесс.
2. Нажмите **Run Manually** с репрезентативным payload.
3. Проверьте **Logs** — убедитесь, что каждый узел пошёл в ожидаемый порт.
4. Включите **isEnabled**.

Выключение рабочего процесса не влияет на уже выполняющиеся запуски; оно только останавливает создание новых.

## Владение и метки

- **Owners** — пользователи и команды, перечисленные как владельцы, получают доступ по правам и (опционально) уведомления при сбое рабочего процесса. Настройте в **Settings → Owners**.
- **Labels** — many-to-many теги для организации рабочих процессов. Фильтруйте список рабочих процессов по метке. Полезно, когда в проекте десятки рабочих процессов, организованных по команде, по интеграции или по окружению.
- **Label rules** — в **Workflows → Settings → Label Rules** автоматически применяйте метки к новым рабочим процессам на основе regex-совпадения по имени или описанию.
- **Owner rules** — в **Workflows → Settings → Owner Rules** автоматически назначайте владельцев новым рабочим процессам.

## Секреты

Глобальные переменные можно отметить как **secret**. Значение шифруется в покое, становится write-only в UI после сохранения и редактируется в журналах запусков (заменяется на `[REDACTED]`).

Используйте секретные переменные для:

- API-ключей исходящих интеграций.
- Bearer-токенов.
- Ключей подписи webhook-ов.
- Любого значения, которое атакующий с read-доступом к рабочему процессу не должен видеть.

Не вставляйте секрет прямо в аргумент компонента — ссылки вроде `Authorization: Bearer eyJh...` появляются в JSON рабочего процесса и в журналах запусков открытым текстом. Ссылайтесь на `{{variable.MY_SECRET}}` вместо этого.

## Тайм-аут запуска

У каждого запуска есть максимальная длительность. Если запуск не завершился в течение тайм-аута, он помечается `Timeout`, а любой выполняющийся компонент отменяется. Значение по умолчанию щедрое (минуты, не секунды) — точное значение в вашей инсталляции смотрите в конфигурации окружения воркера.

У большинства компонентов есть собственные тайм-ауты на вызов внутри тайм-аута запуска — например, компонент API сдастся на зависший исходящий запрос задолго до того, как сдастся весь запуск.

## Лимит рекурсии

Компонент **Execute Workflow** позволяет одному рабочему процессу вызывать другой. Чтобы предотвратить разнос-цикл, где A вызывает B, который вызывает A бесконечно, воркер отслеживает цепочку вызовов и останавливает цепочку, превышающую фиксированную глубину (обычно небольшое число вроде 5). Завершающий запуск помечается `Error` с ясным сообщением о лимите рекурсии.

Если у вас есть законная потребность в длинной цепочке (например, рекурсивный обход папок, обрабатывающий по одному уровню за запуск), переделайте это в один рабочий процесс, который итерирует внутри через **Custom Code** — этот паттерн не подпадает под лимит цепочки.

## Безопасность webhook-ов

Триггеры webhook открывают уникальный HTTPS-URL. Любой, кто узнает URL, может его дёрнуть. Чтобы защититься от случайных или враждебных вызывающих:

- Относитесь к URL как к общему секрету. Не вставляйте его в публичный чат и не коммитьте в публичный репо.
- Для высокоценных рабочих процессов попросите вызывающую систему включать общий секрет в заголовок (например, `X-Webhook-Token`) и проверяйте его в узле **Conditions** перед любыми деструктивными действиями. Определите ожидаемый токен как секретную глобальную переменную.
- Для очень высокоценных рабочих процессов предпочитайте триггер события модели и ручной шаг импорта вместо публичного webhook.

## Исходящий сетевой egress

Компоненты API и другие HTTP-стайл-компоненты отправляют запросы из сети OneUptime Workflow Worker. Если вы self-host OneUptime, исходящая сеть воркера — это ваша забота: убедитесь, что она достигает сторонних API, которые вы вызываете. Если вы используете OneUptime Cloud, наш диапазон egress-IP опубликован в [IP Addresses](/docs/configuration/ip-addresses), так что вы можете внести его в allowlist на стороне получателя.

## Права

Рабочие процессы — это first-class ресурсы, подчиняющиеся ролевому контролю доступа на уровне проекта:

- `CreateWorkflow`, `ReadWorkflow`, `EditWorkflow`, `DeleteWorkflow` — четыре CRUD-права на шаблоны рабочих процессов.
- `RunWorkflow` — нужно, чтобы кликнуть **Run Manually** или отправить рабочий процесс через API.
- `ReadWorkflowLog` — нужно, чтобы посмотреть страницу **Runs & Logs**.
- `ReadWorkflowVariable`, `CreateWorkflowVariable`, `EditWorkflowVariable`, `DeleteWorkflowVariable` — контроль над списком глобальных переменных.

У большинства инженеров должны быть create/edit/read на рабочих процессах, но не на переменных. Зарезервируйте edit-доступ к переменным для людей, которые управляют секретами вашего проекта.

## Квоты

OneUptime Cloud ограничивает число запусков в месяц на проект на меньших планах. Лимит виден в **Project Settings → Billing**. Когда вы упираетесь в него, новые триггеры отклоняются (и записываются с причиной «quota exceeded» на затронутом рабочем процессе) до следующего биллингового цикла. Self-hosted-инсталляции не подпадают под квоту.

## К чему рабочие процессы *не* подходят

Несколько паттернов, для которых стоит взять другой инструмент:

- **Долгие вычисления** — рабочие процессы ориентированы на «склейку» между системами, а не на перемалывание больших наборов данных. Тяжёлую работу запускайте в собственной инфраструктуре, а рабочий процесс используйте, чтобы её инициировать.
- **Stateful-процессы, растянутые на минуты/часы** — одиночный запуск должен заканчиваться быстро. Если нужно «сделать A, потом подождать два часа, потом сделать B», моделируйте ожидание внешним планировщиком, который дёрнет webhook-триггер обратно.
- **Пошаговое реагирование на инциденты с человеческими чек-поинтами** — для этого есть [Runbook-и](/docs/runbooks/index). Используйте рабочий процесс, если человека в цикле нет; используйте runbook, если он есть.
