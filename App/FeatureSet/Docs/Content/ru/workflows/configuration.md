# Конфигурация и безопасность

На этой странице собраны настройки и предохранители, о которых стоит знать, прежде чем направлять рабочий процесс на реальный трафик.

## Включение и выключение рабочего процесса

У каждого рабочего процесса есть переключатель **Enabled** в **Settings**. Когда он выключен, рабочий процесс не запускается — вызовы webhook-ов, сработавшие расписания и события OneUptime игнорируются. Новые рабочие процессы создаются выключенными.

Используйте этот переключатель как «ворота готовности»:

1. Соберите рабочий процесс.
2. Нажмите **Run Manually** с реалистичной нагрузкой.
3. Проверьте **Logs** — убедитесь, что каждый блок пошёл туда, куда вы ожидали.
4. Включите **Enabled**.

Выключение рабочего процесса не останавливает уже идущие запуски; оно лишь не даёт начаться новым.

## Владельцы и метки

- **Владельцы (Owners)** — пользователи и команды, перечисленные в качестве владельцев, получают доступ к рабочему процессу и могут подписаться на уведомления о сбоях. Задаются в **Settings → Owners**.
- **Метки (Labels)** — теги для группировки рабочих процессов. Список рабочих процессов позволяет фильтровать по метке, что значительно упрощает навигацию в загруженном проекте. Полезно, когда рабочие процессы организованы по командам, интеграциям или окружениям.
- **Правила меток (Label rules)** — в разделе **Workflows → Settings → Label Rules** автоматически назначайте метки новым рабочим процессам по шаблонам имени или описания.
- **Правила владельцев (Owner rules)** — в разделе **Workflows → Settings → Owner Rules** автоматически назначайте владельцев новым рабочим процессам.

## Секреты

Отметьте глобальную переменную как **секрет**, если в ней лежит что-то чувствительное. Значение шифруется, скрывается в интерфейсе после сохранения и скрывается в журналах запусков (отображается как `[REDACTED]`).

Используйте секретные переменные для:

- API-ключей к внешним сервисам.
- Токенов авторизации.
- Ключей подписи webhook-ов.
- Всего, что вы не хотели бы показывать тому, у кого есть только доступ на чтение.

Не вставляйте секрет прямо в блок — значения вроде `Authorization: Bearer eyJh...` оказываются видны в рабочем процессе и журналах. Используйте `{{variable.MY_SECRET}}`.

## Сколько может длиться запуск

У каждого запуска есть максимальная длительность. Если запуск не завершился вовремя, он помечается как **Timeout**, а текущий блок отменяется. Значение по умолчанию щедрое — его хватает на обычные HTTP-вызовы и цепочки блоков.

У отдельных блоков есть свои внутренние тайм-ауты — например, блок API сдаётся на зависшем исходящем запросе задолго до того, как сдастся весь запуск.

## Лимит на вызовы других рабочих процессов

Компонент **Execute Workflow** позволяет одному рабочему процессу вызывать другой. Чтобы предотвратить случайные циклы, когда A вызывает B, а B снова вызывает A, есть ограничение на глубину цепочки. Запуск, выходящий за этот предел, завершается с понятной ошибкой.

Если у вас есть законная потребность в длинной цепочке (например, задача, обрабатывающая по одному элементу за запуск), обычно проще организовать цикл внутри одного рабочего процесса через **Custom Code**.

## Безопасность webhook-ов

Триггер Webhook даёт уникальный URL. Любой, кто знает этот URL, может его дёрнуть. Чтобы защититься от случайных или нежелательных вызовов:

- Относитесь к URL как к паролю. Не публикуйте его и не коммитьте в публичные репозитории.
- Для чувствительных рабочих процессов попросите вызывающую систему отправлять общий токен в заголовке (например, `X-Webhook-Token`) и проверяйте его блоком **Conditions** перед тем, как делать что-то важное. Ожидаемый токен храните как секретную переменную.
- Для очень чувствительных рабочих процессов лучше использовать триггер события OneUptime и ручной шаг импорта вместо публичного webhook-а.

## Исходящий сетевой доступ

API и другие HTTP-блоки делают свои запросы из OneUptime. Если вы хостите OneUptime самостоятельно, убедитесь, что ваша инсталляция может достучаться до нужных сервисов. Если вы используете OneUptime Cloud, диапазоны наших исходящих IP перечислены в разделе [IP-адреса](/docs/configuration/ip-addresses), чтобы вы могли разрешить их на другой стороне.

## Разрешения

Рабочие процессы уважают ролевой контроль доступа вашего проекта. Связанные разрешения:

- **Create / Read / Edit / Delete Workflow** — базовые разрешения на сам рабочий процесс.
- **Run Workflow** — нужно, чтобы нажимать **Run Manually** или запускать рабочий процесс через API.
- **Read Workflow Log** — нужно, чтобы просматривать запуски.
- **Read / Create / Edit / Delete Workflow Variable** — управление списком глобальных переменных.

У большинства инженеров должно быть create/edit/read на рабочие процессы, но не на переменные. Право на редактирование переменных оставьте людям, которые управляют секретами вашего проекта.

## Лимиты тарифа

OneUptime Cloud ограничивает число запусков в месяц на младших тарифах. Ваш текущий лимит показан в **Project Settings → Billing**. Когда вы его достигаете, новые срабатывания триггеров отклоняются до следующего расчётного периода. На самостоятельных инсталляциях этого лимита нет.

## Когда рабочие процессы — не тот инструмент

Несколько случаев, когда стоит выбрать что-то другое:

- **Тяжёлые вычисления или большие наборы данных** — рабочие процессы спроектированы для лёгкой «склеивающей» работы, а не для перемалывания чисел. Запускайте тяжёлую работу в собственной инфраструктуре, а рабочий процесс пусть её инициирует.
- **Долгоиграющие процессы длиной в часы** — один запуск должен завершаться быстро. Если нужно «сделать A, подождать два часа, сделать B», используйте внешний планировщик, который отправит webhook обратно в OneUptime, когда настанет время.
- **Пошаговый разбор инцидентов с участием людей** — для этого есть [Runbook-и](/docs/runbooks/index). Рабочие процессы — для автоматизации без присмотра.
