# Триггеры

Триггер — это первый блок в рабочем процессе, он определяет, когда рабочий процесс запускается. У каждого рабочего процесса ровно один триггер. Вы выбираете из четырёх видов.

## Manual

Запускает рабочий процесс по требованию — кнопкой **Run Manually** на странице рабочего процесса. Вы можете вставить JSON-нагрузку, которую сможет прочитать остальная часть рабочего процесса.

Подходит для: автоматизаций в один клик, для которых нужна кнопка, — например, «провернуть этот ключ» или «отправить тестовое оповещение».

**Выход**: тот JSON, который вы вставили, либо пустой объект, если ничего не передали.

## Schedule

Запускает рабочий процесс по повторяющемуся расписанию с помощью cron-выражения.

Подходит для: ночной очистки, ежечасной синхронизации, еженедельных отчётов.

**Параметр**: cron-выражение. Несколько распространённых примеров:

- `0 * * * *` — каждый час, в начале часа.
- `*/5 * * * *` — каждые 5 минут.
- `0 9 * * 1` — каждый понедельник в 9:00 утра.

Если система ненадолго недоступна, запуск подхватится сразу после восстановления — за пропущенные срабатывания при коротких сбоях беспокоиться не нужно.

## Webhook

OneUptime создаёт уникальный URL. Любой запрос к этому URL запускает рабочий процесс. Заголовки, параметры запроса и тело передаются внутрь.

Подходит для: приёма данных в OneUptime из другого инструмента — обратных вызовов CI/CD, оповещений из других систем мониторинга, регистраций в вашей CRM.

**Выход**:

- **Request Headers** — все заголовки входящего запроса.
- **Request Query Params** — разобранная строка запроса.
- **Request Body** — разобранное тело (или необработанный текст, если это не JSON).

URL принимает как `GET`, так и `POST`. Вызывающая сторона получает быстрое подтверждение — сам рабочий процесс выполняется в фоне.

Относитесь к URL как к паролю. Любой, у кого он есть, может запустить ваш рабочий процесс.

## Триггеры событий OneUptime

Почти любая сущность в OneUptime — мониторы, инциденты, оповещения, плановые работы, страницы статуса, политики дежурств, команды — может запустить рабочий процесс. Каждая из них предлагает три события:

- **On Create** — срабатывает при добавлении новой записи.
- **On Update** — срабатывает при изменении записи.
- **On Delete** — срабатывает при удалении записи.

Так вы строите логику «когда в OneUptime происходит X — делать Y», не опрашивая что-либо в цикле.

В следующий блок передаётся полная запись. Например, триггер **Incident → On Create** передаёт новый инцидент, поэтому следующий блок может прочитать его заголовок, описание, уровень критичности и любое другое поле.

### События, которые команды используют чаще всего

- **Incident** — реагировать на открытие, изменение (подтверждение, разрешение) или удаление инцидента.
- **Alert** — те же три события для оповещений.
- **Monitor** — реагировать на добавление, редактирование или удаление монитора.
- **Scheduled Maintenance** — автоматически анонсировать окно плановых работ при его создании.
- **Status Page Subscriber** — приветствовать тех, кто подписался на страницу статуса.
- **On-Call Duty Policy** — синхронизировать изменения расписания с другой системой управления дежурствами.

Чтобы найти нужный триггер, ищите по имени в палитре триггеров.

## Какой триггер выбрать?

| Если вы хотите…                        | Выбирайте             |
| -------------------------------------- | --------------------- |
| Запускать рабочий процесс по кнопке    | **Manual**            |
| Запускать по повторяющемуся расписанию | **Schedule**          |
| Принимать данные из другой системы     | **Webhook**           |
| Реагировать на что-то внутри OneUptime | **Событие OneUptime** |

У рабочего процесса может быть только один триггер. Если вам нужно два способа запуска одной и той же автоматизации, соберите общую логику в одном рабочем процессе и вызывайте её из двух тонких «обёрток» через компонент **Execute Workflow**.
