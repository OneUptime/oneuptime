# Триггеры рабочего процесса

Триггер — это стартовый узел рабочего процесса. У него нет входного порта — выполнение начинается здесь. OneUptime поддерживает четыре семейства триггеров; в каждом рабочем процессе используется ровно один.

## Manual

Запуск рабочего процесса по запросу через кнопку **Run Manually** на странице рабочего процесса. Можно вставить опциональный JSON-payload, который рабочий процесс прочитает как `{{Manual.JSON}}`.

Используйте, когда нужна кнопка, запускающая кусок автоматизации, — рабочий процесс «в один клик ротировать on-call ключ» или «перестроить поисковый индекс», которому не нужны расписание или событие для срабатывания.

**Аргументы**: нет.

**Возвращаемые значения**:

| Имя | Тип | Описание |
| --- | --- | --- |
| `JSON` | JSON | JSON-payload, переданный во время запуска, или пустой объект. |

## Schedule

Запуск рабочего процесса по расписанию cron. Настройте частоту с помощью стандартного cron-выражения.

Используйте для повторяющихся задач: ночная очистка, ежечасная синхронизация, еженедельный экспорт.

**Аргументы**:

| Имя | Тип | Описание |
| --- | --- | --- |
| `Schedule at` | CronTab | Стандартное cron-выражение из 5 полей. Например, `0 * * * *` запускается в начале каждого часа, `*/5 * * * *` — каждые пять минут. |

**Возвращаемые значения**:

| Имя | Тип | Описание |
| --- | --- | --- |
| `executedAt` | Date | Запланированное время запуска. |

Запланированные рабочие процессы выполняются на Workflow Worker в регионе проекта. Если воркер кратковременно недоступен, запуск отправляется, когда он восстанавливается — для коротких сбоев не нужно специально защищаться от пропущенных тиков.

## Webhook

Открывает уникальный HTTPS-URL, на который внешняя система делает `POST`. Заголовки запроса, query-параметры и тело становятся доступными как возвращаемые значения для чтения компонентами ниже по потоку.

Используйте для приёма данных *в* OneUptime от стороннего сервиса: колбэки CI/CD, оповещения от другого инструмента мониторинга, регистрации клиентов в вашей CRM.

**Аргументы**: нет. URL выделяется автоматически при сохранении рабочего процесса и показан на узле триггера. Относитесь к нему как к секрету — любой, у кого есть URL, может запустить рабочий процесс.

**Возвращаемые значения**:

| Имя | Тип | Описание |
| --- | --- | --- |
| `Request Headers` | JSON | Все заголовки из входящего HTTP-запроса. |
| `Request Query Params` | JSON | Разобранная query-строка. |
| `Request Body` | JSON | Разобранное тело запроса. Если тело не является валидным JSON, оно приходит как строка под ключом `raw`. |

Webhook принимает `GET` и `POST`. Ответ вызывающей стороне — `200 OK` с JSON-подтверждением, как только запуск помещён в очередь, — сам рабочий процесс выполняется асинхронно, поэтому не ожидайте читать результат компонентов ниже по потоку в HTTP-ответе.

## Триггеры событий модели

Почти каждая сущность OneUptime — мониторы, инциденты, оповещения, плановые работы, страницы статуса, политики дежурств, команды, телеметрические сервисы и многие другие — открывает три триггера:

- **On Create** — срабатывает при создании новой записи этого типа.
- **On Update** — срабатывает при изменении существующей записи. Триггер делает доступными как старые, так и новые значения.
- **On Delete** — срабатывает при удалении записи.

Так строится автоматизация «когда происходит X в OneUptime, делай Y» без опроса.

Сама модель доступна как возвращаемое значение с теми же именами полей, которые вы видите на ресурсе. Например, триггер **Incident → On Create** возвращает полный объект `Incident`, чтобы узлы ниже по потоку могли читать `{{Incident.title}}`, `{{Incident.description}}`, `{{Incident.incidentSeverityId}}` и т. д.

**Аргументы**: обычно для create/delete их нет. Триггеры Update могут позволить сузить список полей, на которые вы хотите реагировать, чтобы не срабатывать на косметические изменения.

**Возвращаемые значения** (варьируется по модели):

| Имя | Тип | Описание |
| --- | --- | --- |
| Поля модели | (варьируется) | Каждая колонка сущности — название, статус, метки времени, внешние ключи. |
| `previous` (только Update) | JSON | Запись в том виде, в каком она была до изменения. |

### Распространённые триггеры моделей

Неполный список событий моделей, к которым команды обращаются чаще всего:

- **Incident** — `On Create`, `On Update` (используется для реакции на изменения состояния, такие как Acknowledged или Resolved), `On Delete`.
- **Alert** — те же три события на модели оповещений.
- **Monitor** — реагируйте, когда монитор добавлен, отредактирован или удалён; комбинируйте с условиями, чтобы действовать только на production-мониторах.
- **Scheduled Maintenance** — автоматизация последующих анонсов при создании окна обслуживания или изменении его состояния.
- **Status Page Subscriber** — запуск приветственного потока при подписке.
- **On-Call Duty Policy** — синхронизация изменений расписания с внешним реестром.

Если модель доступна в API OneUptime, она почти наверняка может запустить рабочий процесс — найдите её в палитре триггеров по имени сущности.

## Выбор правильного триггера

| Если вы хотите… | Используйте |
| --- | --- |
| Сделать на рабочем процессе кнопку, по которой кто-то кликает | **Manual** |
| Запускать задачу каждые N минут/часов/дней | **Schedule** |
| Чтобы внешняя система отправляла данные в OneUptime | **Webhook** |
| Реагировать на что-то, что происходит *внутри* OneUptime | **Model event** |

В рабочем процессе может быть только один триггер. Если нужны два разных стартовых сигнала, делящих большую часть логики, вынесите общие шаги в один рабочий процесс и вызывайте его из двух тонких «обёрток» через компонент **Execute Workflow** (см. [Компоненты](/docs/workflows/components)).
