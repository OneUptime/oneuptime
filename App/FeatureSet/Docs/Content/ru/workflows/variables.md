# Переменные рабочего процесса

Рабочий процесс полезен только тогда, когда через него течёт информация. Переменные — это то, как эта информация движется: от триггера в первый компонент, от выхода одного компонента во вход следующего и от секретов уровня проекта в любое место, где на них ссылаются.

В OneUptime есть два вида переменных и один синтаксис интерполяции, работающий для обоих.

## Глобальные переменные

Значения в масштабе проекта, определяемые один раз в **Workflows → Global Variables**. Подумайте об API-ключах, базовых URL, именах каналов — обо всём, что не хочется зашивать в код десяти рабочих процессов.

У глобальной переменной есть:

- **Name** — идентификатор, по которому вы на неё ссылаетесь. Используйте `UPPER_SNAKE_CASE`, чтобы её было хорошо видно в шаблонах.
- **Value** — строковое значение. Многострочные значения поддерживаются.
- **Is Secret** — когда включено, значение становится write-only в UI после сохранения и редактируется в журналах запусков.

Ссылайтесь на глобальную переменную откуда угодно в любом рабочем процессе так:

```
{{variable.NAME}}
```

Например, если вы определили `PAGERDUTY_KEY` как секретную переменную, каждый компонент API, который вызывает PagerDuty, может прочитать его как `{{variable.PAGERDUTY_KEY}}`, причём никто не увидит сам ключ в JSON рабочего процесса.

## Локальные переменные

Локальные переменные — это возвращаемые значения узлов, уже выполнившихся в этом запуске. Каждый триггер и каждый компонент публикуют такое значение — см. [Триггеры](/docs/workflows/triggers) и [Компоненты](/docs/workflows/components) для списков по конкретным узлам.

Ссылайтесь на локальную переменную так:

```
{{NodeId.fieldName}}
```

`NodeId` — это имя триггера или компонента на холсте (можно переименовать его для удобочитаемости — держите имя коротким и в `PascalCase`, чтобы ссылки оставались чистыми). `fieldName` — это то, что публикует данный узел.

Примеры:

- После того как компонент **API** с именем `LookupUser` отработал успешно, узлы ниже по потоку могут прочитать его код статуса как `{{LookupUser.response-status}}`, а разобранное тело — как `{{LookupUser.response-body}}`.
- После триггера **Incident → On Create** с именем `Incident` можно прочитать `{{Incident.title}}`, `{{Incident.description}}`, `{{Incident.incidentSeverityId}}` и любую другую колонку инцидента.
- После компонента **Custom Code** с именем `Transform` возвращаемое значение доступно как `{{Transform.value}}`.

Локальные переменные ограничены одним запуском. Следующий запуск начинается с чистого листа.

## Где работает интерполяция

Почти каждый аргумент в текстовом стиле поддерживает интерполяцию:

- Поля URL на компоненте API
- Текст сообщений в Slack / Teams / Discord / Telegram / Email
- Тема и тело в Email
- Заголовки и поля тела (используйте внутри строковых значений JSON)
- Левый и правый операнды в Conditions

Чисто-JSON-аргументы принимают интерполяцию внутри строковых значений; нельзя интерполировать ключ. Если нужно собрать динамическую структуру, используйте **Custom Code** для сборки payload и подавайте возвращаемое значение в следующий узел.

Компонент **Custom Code** читает переменные иначе — глобальные переменные доступны через `args.variables`, а возвращаемые значения узлов выше по потоку передаются как именованные аргументы, настроенные на компоненте.

## Примеры

### Сборка payload из триггера

Webhook получает результат CI-сборки. Тело — это JSON вроде `{ "service": "checkout", "status": "failed" }`. Чтобы превратить его в инцидент OneUptime:

1. Триггер **Webhook** с именем `CIWebhook`.
2. Компонент **Conditions**: left `{{CIWebhook.Request Body.status}}`, оператор `==`, right `failed`.
3. От порта `yes` — компонент **Create Incident** с:
   - Title: `CI build failed: {{CIWebhook.Request Body.service}}`
   - Description: `See {{CIWebhook.Request Body.url}} for the build logs.`

### Использование секрета в исходящем API-вызове

Рабочий процесс, вызывающий PagerDuty:

1. Определите `PAGERDUTY_KEY` как секретную глобальную переменную.
2. На компоненте **API** установите заголовок `Authorization` в `Token token={{variable.PAGERDUTY_KEY}}`.

Ключ никогда не появится в JSON рабочего процесса или в журналах запусков.

### Цепочка из двух API-вызовов

Первый вызов возвращает ID, который нужен второму:

1. Компонент **API** `LookupOrder`: `GET /orders?email={{Manual.JSON.email}}`.
2. Компонент **API** `CancelOrder`: `POST /orders/{{LookupOrder.response-body.id}}/cancel`.

Если `LookupOrder` вернёт ответ не-2xx, его порт `error` сработает вместо `success` — проведите эту ветку в Email или Slack-компонент, чтобы сбои не были тихими.

## Несколько подводных камней

- **Опечатки в именах узлов ломают ссылки тихо.** Если переименовать узел после того, как ниже по потоку настроен `{{OldName.field}}`, обновите каждую ссылку. Посмотрите в журнале запуска — если в захваченных аргументах виден буквальный `{{OldName.field}}`, значит подстановка не сработала.
- **Секреты чувствительны к регистру.** `{{variable.MyKey}}` и `{{variable.mykey}}` — разные переменные.
- **Отсутствующие поля пустые.** Ссылка на `{{Foo.nonexistent}}` даёт пустую строку, а не ошибку. Полезно, но это может маскировать баги — используйте узел **Conditions**, чтобы убедиться в наличии поля, если оно обязательно для следующего шага.
