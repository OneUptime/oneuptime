# Отправка данных непрерывного профилирования в OneUptime

## Обзор

Непрерывное профилирование — это четвёртый столп наблюдаемости наряду с журналами, метриками и трассировками. Профили фиксируют, как ваше приложение расходует время CPU, выделяет память и использует системные ресурсы на уровне функций. OneUptime принимает данные профилирования через OpenTelemetry Protocol (OTLP) и хранит их вместе с другими телеметрическими данными для единого анализа.

С данными профилирования в OneUptime вы можете выявлять «горячие» функции, потребляющие CPU, обнаруживать утечки памяти, находить узкие места конкуренции и коррелировать проблемы производительности с конкретными трассировками и спанами.

## Поддерживаемые типы профилей

OneUptime поддерживает следующие типы профилей:

| Тип профиля | Описание | Единица |
| --- | --- | --- |
| cpu | Время CPU, затраченное на выполнение кода | наносекунды |
| wall | Реальное время (включая ожидание/простой) | наносекунды |
| alloc_objects | Количество выделений в куче | количество |
| alloc_space | Байты выделенной памяти в куче | байты |
| goroutine | Количество активных горутин (Go) | количество |
| contention | Время ожидания на блокировках/мьютексах | наносекунды |

## Начало работы

### Шаг 1 — Создание токена приёма телеметрии

После регистрации в OneUptime и создания проекта нажмите «Ещё» в панели навигации и выберите «Настройки проекта».

На странице ключей приёма телеметрии нажмите «Создать ключ приёма» для создания токена.

![Создание сервиса](/docs/static/images/TelemetryIngestionKeys.png)

После создания токена нажмите «Просмотреть» для его отображения.

![Просмотр сервиса](/docs/static/images/TelemetryIngestionKeyView.png)

### Шаг 2 — Настройка профилировщика

OneUptime принимает данные профилирования как по gRPC, так и по HTTP, используя протокол профилей OTLP.

| Протокол | Конечная точка |
| --- | --- |
| gRPC | `your-oneuptime-host:4317` (стандартный порт gRPC OTLP) |
| HTTP | `https://your-oneuptime-host/otlp/v1/profiles` |

**Переменные среды**

Установите следующие переменные среды, чтобы направить профилировщик на OneUptime:

```bash
export OTEL_EXPORTER_OTLP_HEADERS=x-oneuptime-token=YOUR_ONEUPTIME_SERVICE_TOKEN
export OTEL_EXPORTER_OTLP_ENDPOINT=https://oneuptime.com/otlp
export OTEL_SERVICE_NAME=my-service
```

**Самостоятельный хостинг OneUptime**

При самостоятельном хостинге замените конечную точку на свой хост (например, `http(s)://YOUR-ONEUPTIME-HOST/otlp`). Для gRPC подключитесь напрямую к порту 4317 вашего хоста OneUptime.

## Руководство по инструментированию

### Использование Grafana Alloy (профилирование на основе eBPF)

Grafana Alloy (ранее Grafana Agent) может собирать профили CPU со всех процессов Linux-хоста с помощью eBPF без изменения кода. Настройте его для экспорта через OTLP в OneUptime.

Пример конфигурации Alloy:

```hcl
pyroscope.ebpf "default" {
  forward_to = [pyroscope.write.oneuptime.receiver]
  targets    = discovery.process.all.targets
}

pyroscope.write "oneuptime" {
  endpoint {
    url = "https://oneuptime.com/pyroscope"
    headers = {
      "x-oneuptime-token" = "YOUR_ONEUPTIME_SERVICE_TOKEN",
    }
  }
}
```

### Использование async-profiler (Java)

Для Java-приложений используйте [async-profiler](https://github.com/async-profiler/async-profiler) с агентом OpenTelemetry Java для отправки данных профилирования через OTLP.

```bash
# Запустите Java-приложение с агентом OpenTelemetry Java
java -javaagent:opentelemetry-javaagent.jar \
  -Dotel.exporter.otlp.endpoint=https://oneuptime.com/otlp \
  -Dotel.exporter.otlp.headers=x-oneuptime-token=YOUR_ONEUPTIME_SERVICE_TOKEN \
  -Dotel.service.name=my-java-service \
  -jar my-app.jar
```

### Использование Go pprof с экспортом OTLP

Для Go-приложений можно использовать стандартный пакет `net/http/pprof` вместе с экспортёром OTLP. Настройте непрерывное профилирование, периодически собирая данные pprof и пересылая их в OneUptime.

```go
import (
    "runtime/pprof"
    "bytes"
    "time"
)

// Собирайте 30-секундный профиль CPU и экспортируйте периодически
func collectProfile() {
    var buf bytes.Buffer
    pprof.StartCPUProfile(&buf)
    time.Sleep(30 * time.Second)
    pprof.StopCPUProfile()
    // Преобразуйте вывод pprof в формат OTLP и отправьте в OneUptime
}
```

Как вариант, используйте коллектор OpenTelemetry с приёмником профилирования, который опрашивает конечную точку `/debug/pprof` вашего Go-приложения и экспортирует через OTLP.

### Использование py-spy (Python)

Для Python-приложений [py-spy](https://github.com/benfred/py-spy) может снимать профили CPU без изменения кода. Используйте коллектор OpenTelemetry для приёма и пересылки данных профилей.

```bash
# Снимайте профили и отправляйте в локальный OTLP-коллектор
py-spy record --format speedscope --pid $PID -o profile.json
```

Для непрерывного профилирования запускайте py-spy рядом с вашим приложением и настройте коллектор OpenTelemetry для приёма и пересылки профилей в OneUptime.

## Использование коллектора OpenTelemetry

Вы можете использовать коллектор OpenTelemetry как прокси для приёма профилей от приложений и их пересылки в OneUptime.

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

exporters:
  otlphttp:
    endpoint: "https://oneuptime.com/otlp"
    encoding: json
    headers:
      "Content-Type": "application/json"
      "x-oneuptime-token": "YOUR_ONEUPTIME_SERVICE_TOKEN"

service:
  pipelines:
    profiles:
      receivers: [otlp]
      exporters: [otlphttp]
```

## Функциональность

### Визуализация в виде диаграммы пламени

OneUptime отображает данные профилей в виде интерактивных диаграмм пламени. Каждая полоса представляет функцию в стеке вызовов, а её ширина пропорциональна потреблённому времени или ресурсам. Вы можете нажать на любую функцию для детального просмотра вызывающих и вызываемых функций.

### Список функций

Просматривайте сортируемую таблицу всех функций, захваченных в профиле, упорядоченных по собственному времени, суммарному времени или количеству выделений. Это позволяет быстро определить наиболее затратные функции в вашем приложении.

### Корреляция с трассировками

Профили в OneUptime можно коррелировать с распределёнными трассировками. Когда профиль содержит идентификаторы трассировки и спана (через таблицу ссылок OTLP), вы можете переходить непосредственно от медленного спана трассировки к соответствующему профилю CPU или памяти, чтобы понять, какой именно код выполнялся.

### Фильтрация по типу профиля

Фильтруйте профили по типу (cpu, wall, alloc_objects, alloc_space, goroutine, contention) для сосредоточения на конкретном интересующем измерении ресурсов.

## Хранение данных

Срок хранения данных профилирования настраивается для каждого телеметрического сервиса в настройках проекта OneUptime. По умолчанию данные хранятся 15 дней. По истечении срока хранения данные автоматически удаляются.

Для изменения срока хранения перейдите в **Телеметрия > Сервисы > [Ваш сервис] > Настройки** и обновите значение хранения данных.
