# Устройства IoT OneUptime

## Обзор

OneUptime отслеживает флоты IoT-устройств — датчики, шлюзы, контроллеры и пограничные устройства — путём приёма стандартных метрик OpenTelemetry (OTLP). Каждое устройство (или шлюз от его имени) отправляет небольшой набор метрик `iot_*` по OTLP HTTP, помеченных тем, к какому **флоту** оно относится, и собственным **идентификатором устройства**. OneUptime группирует эти метрики во флот, формирует актуальную инвентаризацию устройств и отслеживает по каждому устройству заряд батареи, подключение, температуру, CPU, память и доступность.

На стороне устройства не нужно устанавливать никакой агент — подойдёт всё, что умеет работать с OTLP (OpenTelemetry SDK на устройстве или OpenTelemetry Collector, запущенный на шлюзе и распределяющий данные между множеством устройств). Эта страница — **руководство по приёму данных**. Чтобы настроить мониторы и оповещения IoT поверх отправляемых данных, см. [Монитор устройства IoT](/docs/monitor/iot-device-monitor).

## Предварительные требования

- Устройство, шлюз или коллектор, способный отправлять OTLP/HTTP в OneUptime
- Сетевая доступность от устройства/шлюза до вашего экземпляра OneUptime
- **Токен приёма телеметрии OneUptime** — создайте его в _Project Settings → Telemetry Ingestion Keys_ и скопируйте значение `x-oneuptime-token`

## Как OneUptime моделирует IoT

OneUptime сопоставляет ваши устройства с двумя понятиями, используя атрибуты ресурса OpenTelemetry:

- **Флот** — логическая группа устройств (например, `building-a-sensors` или `field-gateways`). Флот выводится из атрибута ресурса `iot.fleet.name` и отображается в OneUptime как сервис телеметрии `iot/<fleet>`. Задайте `service.name=iot/<fleet>`, чтобы логи и метрики совпадали в рамках одного сервиса.
- **Устройство** — отдельное устройство внутри флота, идентифицируемое атрибутом `device.id`. OneUptime формирует и поддерживает инвентаризацию устройств по каждому флоту с ключом `device.id`.

Необязательные атрибуты уточняют, как каждое устройство классифицируется и охватывается мониторами:

| Атрибут              | Обязательный | Описание                                                                         |
| -------------------- | -------- | -------------------------------------------------------------------------------- |
| `iot.fleet.name`     | Да      | Флот, к которому относится это устройство. Становится сервисом OneUptime `iot/<fleet>`    |
| `device.id`          | Да      | Стабильный уникальный идентификатор устройства внутри флота                                |
| `iot.device.kind`    | Нет       | Класс устройства — например, `Device`, `Sensor` или `Gateway`. По умолчанию `Device` |
| `iot.device.type`    | Нет       | Более точный тип/модель устройства, используемый для фильтрации мониторов (например, `temp-sensor`) |
| `iot.device.firmware`| Нет       | Версия прошивки, сообщаемая устройством                                          |

## Отправка метрик через OpenTelemetry SDK

Если на вашем устройстве напрямую работает OpenTelemetry SDK, направьте его на OneUptime и проставьте атрибуты ресурса IoT через стандартные переменные окружения `OTEL_*`. Замените токен, эндпоинт, имя флота и идентификатор устройства значениями для вашей среды.

```bash
export OTEL_EXPORTER_OTLP_ENDPOINT=https://oneuptime.com/otlp
export OTEL_EXPORTER_OTLP_HEADERS=x-oneuptime-token=YOUR_TELEMETRY_INGESTION_TOKEN
export OTEL_RESOURCE_ATTRIBUTES=iot.fleet.name=building-a-sensors,device.id=sensor-001,service.name=iot/building-a-sensors
```

| Переменная окружения          | Обязательная | Описание                                                                                          |
| ----------------------------- | -------- | ---------------------------------------------------------------------------------------------------- |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | Да      | Эндпоинт OTLP OneUptime (`https://oneuptime.com/otlp` или `http(s)://YOUR-ONEUPTIME-HOST/otlp` для self-hosted) |
| `OTEL_EXPORTER_OTLP_HEADERS`  | Да      | `x-oneuptime-token=YOUR_TELEMETRY_INGESTION_TOKEN`                                                    |
| `OTEL_RESOURCE_ATTRIBUTES`    | Да      | Атрибуты ресурса через запятую. Должны включать `iot.fleet.name`, `device.id` и `service.name=iot/<fleet>` |

Отправляйте свои показания в виде метрик, используя имена `iot_*`, приведённые ниже (см. [Соглашения о метриках](#соглашения-о-метриках)). Примерно в течение минуты устройство появится в разделе **IoT** панели управления OneUptime.

## Отправка метрик через OpenTelemetry Collector

Когда множество устройств отправляют данные через шлюз, запустите на шлюзе OpenTelemetry Collector и экспортируйте данные в OneUptime. Процессор `resource` проставляет атрибуты флота; принимайте показания от своих устройств (OTLP, мост MQTT, файловые логи и т. д.) и пересылайте их дальше:

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    send_batch_size: 512
    timeout: 5s
  resource:
    attributes:
      - key: iot.fleet.name
        value: field-gateways
        action: upsert
      - key: service.name
        value: iot/field-gateways
        action: upsert

exporters:
  otlphttp:
    endpoint: "https://oneuptime.com/otlp"
    # OneUptime требует кодировщик JSON вместо Proto(buf) по умолчанию
    encoding: json
    headers:
      "Content-Type": "application/json"
      "x-oneuptime-token": "YOUR_TELEMETRY_INGESTION_TOKEN"

service:
  pipelines:
    metrics:
      receivers: [otlp]
      processors: [resource, batch]
      exporters: [otlphttp]
```

- **`resource`** проставляет атрибуты флота для каждой записи. Задавайте `iot.fleet.name` (и соответствующий `service.name=iot/<fleet>`) для каждого шлюза, чтобы устройства каждого шлюза попадали в правильный флот.
- Сохраняйте `device.id` (и при необходимости `iot.device.kind` / `iot.device.type` / `iot.device.firmware`) на каждой точке данных, чтобы OneUptime мог определить отдельное устройство внутри флота.
- **`otlphttp`** отправляет данные в OneUptime по HTTPS с приложенным токеном приёма. Обратите внимание, что `encoding: json` и заголовок `Content-Type: application/json` обязательны.

## Отправка метрик через MQTT

OneUptime поставляется со встроенным эндпоинтом MQTT, поэтому устройства, которые уже умеют работать с MQTT, могут отправлять показания напрямую — без OpenTelemetry SDK, коллектора или моста. Всё, что публикуется по MQTT, попадает в тот же конвейер, что и OTLP: флоты создаются автоматически, инвентаризация устройств обновляется, а все мониторы и шаблоны оповещений IoT работают без изменений.

**Эндпоинты**

| Транспорт             | Адрес                                  | Примечания                                                                                |
| --------------------- | -------------------------------------- | ----------------------------------------------------------------------------------------- |
| MQTT поверх WebSocket | `wss://<your-host>/mqtt`               | Работает в любой инсталляции — использует обычный порт HTTPS через ingress OneUptime      |
| MQTT поверх TCP       | `<app-host>:1883` (`MQTT_INGEST_PORT`) | Self-hosted: по умолчанию доступен только внутри сети кластера/compose; откройте его, если он вам нужен |

**Аутентификация** — два варианта:

- **На уровне проекта**: отправляйте ваш **токен приёма телеметрии** в качестве пароля MQTT (имя пользователя игнорируется; если ваш клиент предоставляет только поле имени пользователя, укажите токен там). Подходит для шлюзов, которые публикуют данные от имени множества устройств.
- **На уровне устройства** (рекомендуется для устройств, подключающихся напрямую): зарегистрируйте устройство на вкладке **Device Registry** флота в панели управления. При регистрации выдаются учётные данные для конкретного устройства — идентификатор учётных данных является **именем пользователя** MQTT, а секрет — **паролем**. Клиенты, аутентифицированные как устройство, могут публиковать данные только в своих собственных топиках `oneuptime/<fleet>/<device>/…`, одно скомпрометированное устройство можно отозвать из панели управления, не затрагивая остальной флот (отзыв вступает в силу примерно в течение минуты, даже для подключённых сессий), а зарегистрированные устройства получают **обнаружение «тихой смерти»**: они остаются в инвентаризации со статусом Offline, а не исчезают, когда перестают сообщать данные, и шаблон оповещения Device Offline срабатывает для них, даже если они умирают без Last Will.

Неверные учётные данные отклоняются на этапе CONNECT с кодом возврата 4 (неверное имя пользователя или пароль), поэтому неправильно настроенное устройство сообщает об ошибке явно.

**Топики** — публикуйте данные под фиксированным префиксом `oneuptime/`. Сегменты флота и устройства не должны содержать `/`, `+` или `#` и ограничены 100 символами:

| Топик                                            | Полезная нагрузка                                                                                    |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------- |
| `oneuptime/<fleet>/<device>/telemetry`           | JSON-объект с показаниями — `{ "metrics": { "iot_temperature_celsius": 21.5 } }` или плоский объект, числовые поля которого являются метриками |
| `oneuptime/<fleet>/<device>/metrics/<metricName>`| Одно значение — просто число (`23.4`) или `{ "value": 23.4 }`                                         |
| `oneuptime/<fleet>/<device>/status`              | `"online"` или `"offline"` (также `1`/`0`, `true`/`false`, `up`/`down`) — соответствует `iot_device_up` |

Полезная нагрузка телеметрии также может содержать `"attributes"` (карта строк, проставляемая на каждой точке данных — используйте её для `iot.device.kind`, `iot.device.type`, `iot.device.firmware` или собственных меток) и `"timestamp"` (ISO-8601 либо unix-секунды/миллисекунды). Оба поля необязательны; при отсутствии `timestamp` используется время приёма.

**Обнаружение офлайна с помощью Last Will** — зарегистрируйте MQTT Last Will на топике `oneuptime/<fleet>/<device>/status` с полезной нагрузкой `offline`. Если устройство умирает или пропадает из сети, брокер публикует `iot_device_up = 0` от его имени в момент завершения сессии — это запускает стандартный шаблон оповещения **Device Offline** и переводит устройство в состояние Down в инвентаризации, без опроса и без ожидания пропущенного сбора данных. Публикуйте `online` в тот же топик после подключения, чтобы устройство снова отображалось как Up.

Пример с `mosquitto_pub` (прямой TCP, self-hosted):

```bash
mosquitto_pub -h YOUR-ONEUPTIME-APP-HOST -p 1883 \
  -u oneuptime -P "YOUR_TELEMETRY_INGESTION_TOKEN" \
  -t "oneuptime/building-a-sensors/sensor-001/telemetry" \
  -m '{"metrics":{"iot_device_up":1,"iot_battery_percent":87,"iot_temperature_celsius":21.5},"attributes":{"iot.device.type":"temp-sensor","iot.device.firmware":"1.4.2"}}'
```

Пример с Node.js `mqtt` поверх WebSocket (работает с oneuptime.com и любым self-hosted экземпляром):

```javascript
const mqtt = require("mqtt");

const client = mqtt.connect("wss://oneuptime.com/mqtt", {
  username: "oneuptime", // игнорируется — аутентификацию выполняет токен ниже
  password: "YOUR_TELEMETRY_INGESTION_TOKEN",
  will: {
    topic: "oneuptime/building-a-sensors/sensor-001/status",
    payload: "offline",
  },
});

client.on("connect", () => {
  client.publish("oneuptime/building-a-sensors/sensor-001/status", "online");
  setInterval(() => {
    client.publish(
      "oneuptime/building-a-sensors/sensor-001/telemetry",
      JSON.stringify({
        metrics: {
          iot_device_up: 1,
          iot_battery_percent: readBattery(),
          iot_temperature_celsius: readTemperature(),
        },
      }),
    );
  }, 60 * 1000);
});
```

Пример с Python `paho-mqtt` поверх WebSocket:

```python
import json
import paho.mqtt.client as mqtt

client = mqtt.Client(transport="websockets")
client.username_pw_set("oneuptime", "YOUR_TELEMETRY_INGESTION_TOKEN")
client.tls_set()
client.will_set("oneuptime/building-a-sensors/sensor-001/status", "offline")
client.ws_set_options(path="/mqtt")
client.connect("oneuptime.com", 443)

client.publish("oneuptime/building-a-sensors/sensor-001/status", "online")
client.publish(
    "oneuptime/building-a-sensors/sensor-001/telemetry",
    json.dumps({"metrics": {"iot_device_up": 1, "iot_temperature_celsius": 21.5}}),
)
```

Примечания:

- Эндпоинт работает **только на приём**: подписки отклоняются (ошибка SUBACK). Используйте QoS 1, если хотите, чтобы брокер подтверждал получение. Приём осуществляется **не менее одного раза** — повторная передача QoS 1/2 после потерянного подтверждения может создать дублирующиеся точки данных.
- Публикации вне контракта топиков или с некорректной полезной нагрузкой принимаются и **отбрасываются** (в MQTT 3.1.1 нет ответа об ошибке для отдельного сообщения) — сервер записывает в лог предупреждение с указанием причины, поэтому проверяйте логи приложения OneUptime, если данные не поступают.
- На эндпоинте WebSocket держите keepalive MQTT **менее 5 минут** — ingress OneUptime закрывает простаивающие WebSocket-соединения через 300 секунд, что запустит ваш Last Will и ложное оповещение Device Offline. Значения по умолчанию в клиентских библиотеках (60 с для `mqtt` и `paho-mqtt`) подходят. У эндпоинта на чистом TCP такого ограничения нет.
- Полезная нагрузка ограничена 128 КБ и 100 метриками на одну публикацию; пакеты превышающего размера обрывают соединение.

## Соглашения о метриках

OneUptime распознаёт следующие имена метрик `iot_*`. Каждая точка данных должна нести метку `device.id`, чтобы показание было отнесено к правильному устройству. Достаточно отправлять только те метрики, которые имеют смысл для вашего устройства — отсутствующие просто не отображаются на графиках.

| Имя метрики                 | Значение                                                                        |
| --------------------------- | ------------------------------------------------------------------------------ |
| `iot_device_up`             | Доступность устройства. `1` = работает/доступно, `0` = недоступно. Управляет монитором устройства IoT |
| `iot_device_info`           | Сигнал, содержащий только идентификацию. Несёт `device.id` / kind / type / firmware, чтобы устройство появлялось в инвентаризации ещё до того, как оно сообщит показания |
| `iot_battery_percent`       | Уровень заряда батареи, `0`–`100` (%)                                            |
| `iot_signal_strength_dbm`   | Уровень беспроводного сигнала в dBm (например, RSSI Wi-Fi / LoRa / сотовой связи)      |
| `iot_temperature_celsius`   | Температура устройства или датчика в °C                                             |
| `iot_cpu_usage_ratio`       | Загрузка CPU в виде доли `0`–`1` (OneUptime хранит её в процентах)        |
| `iot_memory_usage_bytes`    | Объём используемой в данный момент памяти, в байтах                                                |
| `iot_memory_size_bytes`     | Общий объём доступной памяти устройства, в байтах                                 |
| `iot_uptime_seconds`        | Секунды с момента последней загрузки устройства                                           |

## Проверка установки

1. Убедитесь, что ваше устройство или шлюз экспортирует данные без ошибок (проверьте логи SDK/коллектора на наличие сбоев экспорта и ответов HTTP `401`/`403`).
2. В панели управления OneUptime откройте раздел **IoT** — ваш флот должен появиться как `iot/<fleet>` примерно в течение минуты.
3. Откройте вкладку **Devices** флота — каждый отправленный вами `device.id` должен быть в списке с последними значениями заряда батареи, сигнала, температуры, CPU, памяти и статуса работы/недоступности.
4. Откройте **Metrics** во флоте, чтобы построить график по любой из приведённых выше серий `iot_*`.

## Устранение неполадок

### Флот не появляется

1. Убедитесь, что `iot.fleet.name` задан как атрибут **ресурса** (а не метка точки данных) и что `service.name` равен `iot/<fleet>`.
2. Убедитесь, что эндпоинт экспортёра — `https://oneuptime.com/otlp` (или ваш self-hosted `…/otlp`), а заголовок `x-oneuptime-token` несёт действительный токен.
3. Если используете коллектор, убедитесь, что для экспортёра `otlphttp` заданы `encoding: json` и `Content-Type: application/json`.

### Устройства отсутствуют в инвентаризации

1. Убедитесь, что каждая точка данных несёт метку `device.id` — устройства идентифицируются по ней.
2. Отправляйте `iot_device_info` (только идентификация) для устройств, которые ещё не сообщили показания, чтобы они всё равно отображались в инвентаризации.
3. Проверьте, что значения `device.id` стабильны между отчётами; изменяющийся идентификатор создаёт дублирующиеся строки устройств.

### HTTP 401 / 403 от экспортёра

Токен приёма недействителен, отозван или отсутствует. Создайте новый в _Project Settings → Telemetry Ingestion Keys_ и обновите заголовок `x-oneuptime-token`.

### Метрики не отображаются на графиках

1. Убедитесь, что вы используете точные имена метрик `iot_*` из таблицы [Соглашения о метриках](#соглашения-о-метриках) — нераспознанные имена сохраняются как обычные метрики и не заполняют графики IoT.
2. Помните, что `iot_cpu_usage_ratio` — это доля `0`–`1`; отправляйте исходную долю, а OneUptime отобразит её в виде процента.
3. Дайте до минуты на появление первых точек данных после того, как устройство начинает сообщать показания.

## Self-hosted OneUptime

Если вы используете self-hosted OneUptime, направьте эндпоинт на ваш собственный экземпляр:

```bash
export OTEL_EXPORTER_OTLP_ENDPOINT=https://your-oneuptime-host.example.com/otlp
```

Или, в коллекторе:

```yaml
exporters:
  otlphttp:
    endpoint: https://your-oneuptime-host.example.com/otlp
    encoding: json
    headers:
      "Content-Type": "application/json"
      "x-oneuptime-token": "YOUR_TELEMETRY_INGESTION_TOKEN"
```

Если ваш экземпляр работает только по HTTP, измените схему на `http://` и используйте соответствующий порт.

## Дальнейшие шаги

- Настройте **Монитор устройства IoT**, чтобы получать оповещения о том, что устройство офлайн, низкий заряд батареи, слабый сигнал, высокая температура и высокая загрузка CPU — см. [Монитор устройства IoT](/docs/monitor/iot-device-monitor).
- Для неконтейнеризированных хостов (виртуальные машины и физические серверы Linux / macOS / Windows) используйте [Host OpenTelemetry Collector](/docs/telemetry/host-otel-collector).
- Чтобы глубже изучить базовую интеграцию OTLP, см. [Интеграция OpenTelemetry с OneUptime](/docs/telemetry/open-telemetry).
