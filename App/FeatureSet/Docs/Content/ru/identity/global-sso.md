# Global SSO (Единый вход в масштабе всего экземпляра)

Global SSO позволяет **администратору экземпляра** OneUptime (мастер-администратору) настроить единый поставщик удостоверений SAML 2.0 или OpenID Connect (OIDC) **один раз на уровне экземпляра** и подключить его к любому проекту на сервере. Это аналог SSO на уровне отдельного проекта, но в масштабе всего экземпляра: вместо того чтобы каждый владелец проекта настраивал собственного поставщика удостоверений, мастер-администратор настраивает одного, который может обслуживать весь экземпляр.

Global SSO — функция **OneUptime Enterprise Edition**, доступная только на экземплярах, работающих на сборке Enterprise Edition.

## Global SSO и Project SSO

|                   | Project SSO                                           | Global SSO                                                   |
| ----------------- | ----------------------------------------------------- | ------------------------------------------------------------ |
| Кем настраивается | Владельцем/администратором проекта (Project Settings) | Мастер-администратором экземпляра (Admin Dashboard)          |
| Область действия  | Один проект                                           | Весь экземпляр — подключается к любому проекту               |
| Результат входа   | Доступ к этому одному проекту                         | Доступ ко всем проектам, к которым пользователь имеет доступ |

## Настройка Global SSO

1. **Откройте Admin Dashboard**

   - Войдите как мастер-администратор и откройте **Admin** > **Settings** > **Global SSO** (для SAML) или **Global OIDC** (для OpenID Connect).

2. **Создайте поставщика**

   - Нажмите **Create Global SSO**.
   - Для SAML: введите **Name**, **Sign On URL** и **Issuer** от вашего поставщика удостоверений и вставьте **Public Certificate**. Выберите методы **Signature** и **Digest** (если не уверены, оставьте значения по умолчанию — `RSA-SHA256` / `SHA256`).
   - Для OIDC: введите **Discovery URL**, **Issuer**, **Client ID**, **Client Secret**, **Scopes** (должны включать `openid`) и имена утверждений **email** / **name**.

3. **Скопируйте URL OneUptime в вашего поставщика удостоверений**

   - Откройте поставщика (нажмите его строку в списке), чтобы открыть карточку **Identity Provider URLs**.
   - Для SAML скопируйте **ACS URL (Reply URL)** и **Issuer (Entity ID)** в ваш IdP (Okta, Azure AD, OneLogin, JumpCloud и другие).
   - Для OIDC скопируйте **Redirect URI** в список разрешённых перенаправлений вашего IdP.

4. **Проверьте поставщика**
   - Используйте ссылку **Test this SSO provider** на странице поставщика, чтобы выполнить сквозной вход через вашего поставщика удостоверений. Чтобы ссылка работала, поставщик должен быть **включён**. Включение глобального поставщика лишь добавляет опцию «Sign in with SSO» на странице входа — оно никогда не принуждает к использованию SSO и не блокирует доступ, поэтому его можно безопасно включить, протестировать и при необходимости снова отключить.

## Как пользователи входят

Поведение глобального поставщика зависит от того, привязываете ли вы к нему какие-либо проекты:

- **Проекты не привязаны (default-all / приглашение в первую очередь):** Пользователи могут войти через поставщика и получить доступ к **любому проекту, участниками которого они уже являются**. Новые пользователи **не** создаются автоматически — пользователь должен быть сначала приглашён в проект. Используйте этот вариант для общекорпоративного SSO, где членство управляется в другом месте.

- **Проекты привязаны (автоматическая подготовка учётных записей):** Откройте поставщика и используйте таблицу **Attached Projects**, чтобы привязать один или несколько проектов, каждый с набором команд по умолчанию. Пользователи, которые входят, **автоматически добавляются** в эти проекты и при первом входе включаются в команды по умолчанию. Добавляйте по одному проекту с командами за раз, чтобы сформировать список; чтобы изменить привязку, удалите её и добавьте заново.

Если вы хотите предотвратить любое автоматическое создание учётных записей даже при привязанных проектах, включите **Disable Sign Up with SSO** у поставщика — тогда пользователи должны быть приглашены, прежде чем смогут войти.

## Принудительное применение SSO

Настройка глобального поставщика никого не принуждает его использовать; вход по паролю по-прежнему работает. Чтобы сделать SSO обязательным, используйте элементы управления **Require SSO for Login**:

- **Для проекта:** проект может требовать SSO и при необходимости требовать _конкретного_ поставщика (проектного или глобального).
- **Для всего экземпляра:** в **Admin** > **Settings** > **Authentication** есть переключатель **Require SSO for Login**, который делает SSO обязательным для каждого пользователя во всём экземпляре. Мастер-администраторы остаются исключением, чтобы их нельзя было заблокировать.
