# Обновление OneUptime

В этом руководстве описывается безопасное обновление самостоятельно размещённой установки OneUptime.

## Общие рекомендации

- Обновляйтесь пошагово между основными версиями (например, 6 → 7 → 8). Не пропускайте основные версии.
- Вы можете перескакивать через минорные/патч-версии (например, 8.1 → 8.4), пока следуете примечаниям к выпуску.
- Всегда делайте резервные копии перед обновлением и проверяйте возможность их восстановления.

## Обновление с OneUptime 10 → 11

OneUptime 11 перестраивает хранилище телеметрии в ClickHouse. На этой
странице описано, что меняется, кому нужно предпринимать действия и —
для установок, которые хотят перенести историческую телеметрию, — все
запросы, необходимые для этого.

### Что меняется в v11

Телеметрия (логи, трассировки, метрики, исключения, профили, логи
мониторов, журналы аудита) переносится в новые таблицы ClickHouse с
партиционированием по времени, кодеками сжатия для отдельных столбцов и
новыми столбцами модели сущностей:

| Старая таблица        | Новая таблица         |
| --------------------- | --------------------- |
| `LogItemV2`           | `LogItemV3`           |
| `MetricItemV2`        | `MetricItemV3`        |
| `SpanItemV2`          | `SpanItemV3`          |
| `ExceptionItemV2`     | `ExceptionItemV3`     |
| `ProfileItemV2`       | `ProfileItemV3`       |
| `ProfileSampleItemV2` | `ProfileSampleItemV3` |
| `MonitorLogV2`        | `MonitorLogV3`        |
| `AuditLogV1`          | `AuditLogV2`          |

В каждой таблице телеметрии переименованы два столбца: `serviceId` →
`primaryEntityId` и `serviceType` → `primaryEntityType`. Это жёсткое
переименование — **если вы напрямую обращаетесь к аналитическому API
OneUptime с фильтрами `serviceId`/`serviceType`, обновите их на новые
имена.** Дашборды, мониторы и оповещения внутри OneUptime мигрируются
автоматически.

Переход выполняется **только вперёд**: новые таблицы создаются пустыми,
вся телеметрия, поступающая после обновления, сразу записывается в них,
а история со временем накапливается естественным образом. Старые таблицы
сохраняются и постепенно удаляются сами через TTL хранения.

### Кому нужно что-то делать

- **Новые установки:** ничего делать не нужно.
- **Обновления, которым не нужна телеметрия за период до обновления в
  интерфейсе:** ничего делать не нужно. Страницы телеметрии просто
  показывают данные с момента обновления; более старые данные незаметно
  устаревают в старых таблицах.
- **Обновления, для которых телеметрия за период до обновления должна
  быть видна:** выполните ручное копирование, описанное ниже, в любой
  момент после обновления.

Как всегда: обновляйте основные версии пошагово (10 → 11, не
пропускайте) и делайте резервные копии Postgres и ClickHouse перед
обновлением.

### Опционально: перенос истории телеметрии

Выполняйте эти запросы **после полного запуска обновлённой системы**
(новые таблицы и их материализованные представления должны существовать).
Подключайтесь напрямую на хосте ClickHouse — у нативного протокола нет
HTTP-таймаутов, поэтому многочасовые выражения выполняются без проблем:

```bash
clickhouse-client --database oneuptime
```

Что полезно знать перед началом:

- Копирование можно безопасно выполнять во время работы OneUptime. Новая
  телеметрия записывается в новые таблицы независимо; скопированная
  история заполняется параллельно.
- При больших объёмах (сотни ГБ) процесс может занять часы.
- Каждое выражение ниже содержит `insert_deduplication_token`, а новые
  таблицы поставляются с окном дедупликации — поэтому **повторный запуск
  выражения, прервавшегося на середине, безопасен** (уже вставленные
  блоки пропускаются, в том числе в агрегатах метрик), при условии, что
  вы перезапускаете его достаточно быстро. При интенсивном живом потоке
  данных окно (последние 10 000 блоков вставки на таблицу) со временем
  вытесняет старые токены.
- Копирование метрик также автоматически перестраивает предагрегированные
  сводки для дашбордов (каждая скопированная строка повторно подаётся в
  материализованные представления агрегатов) — поэтому копирование метрик
  медленнее остальных; выполняйте его последним.

#### Шаг 1 — список исходных партиций

Каждая старая таблица содержит не более 16 партиций. Для каждой исходной
таблицы:

```sql
SELECT DISTINCT _partition_id FROM LogItemV2 ORDER BY _partition_id;
```

#### Шаг 2 — генерация выражения копирования

Наборы столбцов могут немного отличаться между установками (в более
старых развёртываниях могут отсутствовать недавно добавленные столбцы),
поэтому генерируйте выражение из вашей текущей схемы, а не копируйте
фиксированное. Укажите `src` и `dst` в выражении `WITH`, подставив одну
из пар таблиц из таблицы выше, и выполните:

```sql
WITH 'LogItemV2' AS src, 'LogItemV3' AS dst
SELECT concat(
  'INSERT INTO ', dst, ' (`', arrayStringConcat(groupArray(name), '`, `'), '`)',
  ' SELECT ', arrayStringConcat(groupArray(selectExpr), ', '),
  ' FROM ', src,
  ' WHERE _partition_id = ''{PARTITION}''',
  ' ORDER BY ', (SELECT sorting_key FROM system.tables WHERE database = currentDatabase() AND name = dst), ', _id',
  ' SETTINGS max_execution_time = 0, max_partitions_per_insert_block = 0, insert_deduplication_token = ''v3copy:', dst, ':{PARTITION}'', deduplicate_blocks_in_dependent_materialized_views = 1'
) AS copy_sql
FROM (
  SELECT name,
    multiIf(name = 'primaryEntityId', 'serviceId', name = 'primaryEntityType', 'serviceType', name) AS srcName,
    if(srcName = name, concat('`', name, '`'), concat('`', srcName, '` AS `', name, '`')) AS selectExpr,
    position
  FROM system.columns
  WHERE database = currentDatabase() AND table = dst
    AND srcName IN (SELECT name FROM system.columns WHERE database = currentDatabase() AND table = src)
  ORDER BY position
);
```

Сгенерированное выражение копирует только столбцы, общие для обеих таблиц
(новые столбцы получают значения по умолчанию), переименовывает
`serviceId`/`serviceType` на лету, упорядочивает строки детерминированно,
чтобы повторный запуск создавал идентичные, дедуплицируемые блоки, и
снимает ограничения на время выполнения и количество партиций,
необходимые для выражения такого размера.

#### Шаг 3 — выполнение, по одной партиции за раз

Возьмите сгенерированное выражение и подставьте вместо `{PARTITION}` (он
встречается дважды — в `WHERE` и в токене) каждый идентификатор партиции
из шага 1. Выполняйте выражения по одному, затем повторите шаги 1–3 для
каждой пары таблиц.

Если выражение прервалось на середине, незамедлительно перезапустите **то
же самое** выражение — уже зафиксированные блоки дедуплицируются. Если
перезапуск выполняется значительно позже, сначала сравните количество
строк (шаг 5).

#### Шаг 4 (опционально) — история по-хостовых агрегатов метрик

Скопированные сырые строки метрик автоматически перестраивают агрегаты
уровня сервисов, но не **по-хостовый** агрегат (у старых строк нет ключа
сущности хоста). Обновление намеренно оставляет старую таблицу
по-хостовых агрегатов на месте, чтобы вы могли перенести её, вычислив
новый ключ из имени хоста:

```sql
INSERT INTO MetricItemAggMV1mByHostV2 (projectId, name, hostEntityKey, bucketTime, valueSumState, valueCountState, valueMinState, valueMaxState, retentionDate)
SELECT
  projectId,
  name,
  substring(lower(hex(SHA256(concat(projectId, '|host|host.name=', lower(trimBoth(hostIdentifier)))))), 1, 16) AS hostEntityKey,
  bucketTime,
  valueSumState,
  valueCountState,
  valueMinState,
  valueMaxState,
  retentionDate
FROM MetricItemAggMV1mByHost
SETTINGS max_execution_time = 0, insert_deduplication_token = 'v3copy:MetricItemAggMV1mByHostV2:all';
```

#### Шаг 5 — проверка

Сравните итоговые значения по каждой паре таблиц (новая таблица также
содержит строки, появившиеся после обновления, поэтому её значение должно
быть больше или равно значению старой):

```sql
SELECT
  (SELECT count() FROM LogItemV2) AS old_rows,
  (SELECT count() FROM LogItemV3) AS new_rows;
```

#### Шаг 6 (опционально) — досрочное освобождение диска

Старые таблицы опустошаются сами по себе через TTL, но как только вы
убедитесь в корректности копии, их можно удалить сразу:

```sql
DROP TABLE IF EXISTS LogItemV2;
DROP TABLE IF EXISTS MetricItemV2;
DROP TABLE IF EXISTS SpanItemV2;
DROP TABLE IF EXISTS ExceptionItemV2;
DROP TABLE IF EXISTS ProfileItemV2;
DROP TABLE IF EXISTS ProfileSampleItemV2;
DROP TABLE IF EXISTS MonitorLogV2;
DROP TABLE IF EXISTS AuditLogV1;
DROP TABLE IF EXISTS MetricItemAggMV1mByHost;
```

> Совет: как и при любом обновлении основной версии, сначала протестируйте
> процесс в тестовой среде и убедитесь, что телеметрия поступает в новые
> таблицы, прежде чем полагаться на копию в production.



## Обновление с OneUptime 9 → 10

Изменений, требующих ручных действий, нет. Просто следуйте стандартному процессу обновления.

## Обновление с OneUptime 8 → 9

Helm-чарт больше не создаёт ресурс Kubernetes Ingress. OneUptime поставляется с контейнером шлюза ingress, который уже завершает TLS, управляет доменами страниц статуса и маршрутизирует трафик платформы, поэтому контроллер ingress кластера больше не нужен.

- Удалите все переопределения `oneuptimeIngress` из ваших пользовательских файлов `values.yaml` перед обновлением. Эти ключи теперь игнорируются и вызовут ошибки валидации, если оставить их.
- Убедитесь, что `nginx.service.type` отражает способ, которым вы хотите открыть встроенный шлюз ingress (например, `LoadBalancer`, `NodePort` или `ClusterIP` с внешним балансировщиком нагрузки).
- Проверьте, что записи DNS для страниц статуса или основных хостов по-прежнему указывают на Service или балансировщик нагрузки перед шлюзом ingress OneUptime.
- После обновления убедитесь, что TLS-сертификаты продолжают обновляться через встроенный шлюз и что домены страниц статуса разрешаются корректно.


## Обновление с OneUptime 7 → 8

Если вы работаете на Kubernetes, есть важные несовместимые изменения:

- Мы больше не используем чарты Bitnami для Postgres, Redis и ClickHouse из-за [изменений лицензии Bitnami](https://github.com/bitnami/charts/issues/35164)
- Эти изменения обратно несовместимы. Вы должны следовать новой структуре в `values.yaml` Helm-чарта.
- Создайте резервную копию ваших данных (Postgres, ClickHouse и все постоянные тома) перед обновлением.


> Совет: Сначала протестируйте обновление в тестовой среде. Убедитесь, что ваши рабочие нагрузки работоспособны и данные целы, прежде чем обновлять production.
