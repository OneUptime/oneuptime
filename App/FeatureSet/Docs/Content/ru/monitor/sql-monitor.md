# Монитор SQL-запросов

Монитор SQL-запросов по расписанию выполняет запрос SQL только для чтения с зонда и формирует оповещения на основе результата — количества возвращённых строк, скалярного значения, времени выполнения запроса или ошибки запроса. Он создан для сценария «выполнить запрос и создать инцидент»: например, для оповещения о резком росте числа отменённых заказов за последние пять минут, о чрезмерном увеличении таблицы очереди или об исчезновении критически важной строки.

Поскольку запрос выполняется с зонда внутри вашей сети, OneUptime не требуется прямое подключение к вашей базе данных, а полный набор результатов никогда не покидает зонд — обратно передаётся лишь небольшая ограниченная проекция результата.

## Поддерживаемые базы данных

Монитор SQL-запросов поддерживает следующие СУБД:

- **PostgreSQL** (порт по умолчанию `5432`)
- **MySQL** (порт по умолчанию `3306`)
- **Microsoft SQL Server** (порт по умолчанию `1433`)

MySQL-совместимые и PostgreSQL-совместимые СУБД, использующие тот же сетевой протокол и диалект SQL, как правило, тоже работают, но официально протестированы только три перечисленные выше СУБД.

## Как это работает

При каждой проверке зонд подключается к вашей базе данных, выполняет запрос в контексте только для чтения, считывает не более ограниченного количества строк и передаёт в OneUptime компактную проекцию. Затем на основе этой проекции вычисляются критерии вашего монитора.

Зонд передаёт только:

- **Количество строк** — количество строк, возвращённых запросом (ограничено параметром «Максимум строк»).
- **Скалярное значение** — первый столбец первой строки. Это естественное значение для запросов вида `SELECT COUNT(*)`.
- **Первая строка** — первая строка в виде набора пар «столбец/значение», отображается в сводке проверки для контекста.
- **Время выполнения** — длительность выполнения запроса в миллисекундах.
- **Ошибка запроса** — очищенное сообщение об ошибке, если запрос завершился неудачно.

Полный набор результатов никогда не отправляется в OneUptime, поэтому данные клиентов не копируются в хранилище OneUptime.

## Модель безопасности

Выполнение предоставленного пользователем запроса к рабочей базе данных — операция чувствительная, поэтому Монитор SQL-запросов изначально работает только для чтения и использует несколько уровней защиты:

- **Пользователь базы данных с минимальными привилегиями (основная мера).** Всегда подключайтесь через выделенного пользователя базы данных с правами только для чтения, у которого есть доступ только к тем таблицам, которые нужны запросу. Это самая важная мера — см. раздел Создание пользователя только для чтения ниже.
- **Выполнение только для чтения.** В PostgreSQL и MySQL зонд открывает транзакцию `READ ONLY`, которая отклоняет любые операции записи (включая записывающие CTE) независимо от текста запроса. В Microsoft SQL Server, где нет транзакций только для чтения, зонд работает внутри транзакции, которая всегда откатывается.
- **Одиночные запросы из разрешённого списка.** Запрос должен быть одиночной инструкцией, начинающейся с `SELECT`, `WITH`, `VALUES` или `TABLE`. Составные инструкции (`SELECT 1; DROP TABLE …`), а также операции записи и DDL отклоняются ещё до подключения зонда. Проверка учитывает комментарии и строковые литералы, поэтому ключевое слово, спрятанное в комментарии или строке, не пройдёт.
- **Тайм-аут выполнения.** Для каждого запроса действует жёсткое ограничение по времени. Слишком долгий запрос прерывается.
- **Ограничение по строкам.** Считывается не более чем «Максимум строк» (плюс одна строка — для обнаружения усечения), что ограничивает потребление памяти зондом и размер передаваемых данных.
- **Скрытие учётных данных.** Ошибки базы данных очищаются перед сохранением — пароль и любые строки подключения удаляются, поэтому учётные данные никогда не попадают в сообщения об ошибках.

## Предварительные требования

- **Зонд** с сетевым доступом к хосту и порту вашей базы данных. Это может быть зонд, размещённый в OneUptime (если ваша база данных доступна из интернета), или собственный зонд, работающий внутри вашей сети. О том, как установить собственный зонд, см. документацию по зондам.
- **Пользователь базы данных с правами только для чтения** и параметры подключения (хост, порт, имя базы данных, имя пользователя, пароль).

## Настройка

Создайте новый монитор, выберите тип монитора **SQL-запрос** и заполните параметры подключения:

- **Тип базы данных** — PostgreSQL, MySQL или Microsoft SQL Server. Выбор типа задаёт порт по умолчанию.
- **Хост** — хост базы данных, доступный с зонда (например, `db.internal`).
- **Порт** — порт базы данных.
- **Имя базы данных** — база данных, к которой выполняется запрос.
- **Имя пользователя** — пользователь базы данных с правами только для чтения и минимальными привилегиями.
- **Пароль** — пароль базы данных. Настоятельно рекомендуем ссылаться на [Секрет монитора](/docs/monitor/monitor-secrets) с помощью `{{monitorSecrets.name}}` вместо ввода пароля в открытом виде (см. ниже).
- **SQL-запрос** — запрос только для чтения, который нужно выполнить (см. Составление запроса).
- **Использовать SSL/TLS** — включите для подключения по TLS. При включении можно отключить **Проверять сертификат сервера**, если база данных использует самоподписанный сертификат.

### Дополнительные параметры

- **Тайм-аут соединения (мс)** — сколько ждать установления соединения. По умолчанию `10000`, максимум `30000`.
- **Тайм-аут выполнения запроса (мс)** — жёсткое ограничение на длительность выполнения запроса. По умолчанию `15000`, максимум `60000`.
- **Максимум строк** — верхняя граница числа строк, считываемых из базы данных. По умолчанию `100`, максимум `1000`.

## Составление запроса

Запрос должен быть **одиночной инструкцией только для чтения**. Он должен начинаться с одного из ключевых слов `SELECT`, `WITH`, `VALUES` или `TABLE`. Допускается одна завершающая точка с запятой; несколько инструкций не допускаются.

Делайте запросы лёгкими и узконаправленными — они выполняются при каждой проверке, поэтому отдавайте предпочтение индексированным столбцам и узким временным окнам.

```sql
-- Подсчёт недавних отмен (PostgreSQL)
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL '5 minutes';
```

```sql
-- Та же идея для MySQL
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > NOW() - INTERVAL 5 MINUTE;
```

```sql
-- Та же идея для Microsoft SQL Server
SELECT COUNT(*) AS cancelled
FROM orders
WHERE status = 'CANCELLED'
  AND created_at > DATEADD(minute, -5, GETDATE());
```

Для запроса вида `COUNT(*)` результат доступен и как **Количество строк** (равное `1`, поскольку возвращается одна строка), и как **Скалярное значение** (сам счётчик из первого столбца). Для оповещений по принципу «сколько именно» сравнивайте со **Скалярным значением**.

## Использование секрета монитора для пароля

Чтобы пароль базы данных никогда не хранился в мониторе в открытом виде, создайте [Секрет монитора](/docs/monitor/monitor-secrets) и сошлитесь на него в поле «Пароль»:

1. Перейдите в OneUptime Dashboard → Настройки проекта → Секреты монитора → Создать секрет монитора.
2. Создайте секрет (например, `dbPassword`) и предоставьте этому монитору доступ к нему.
3. В поле «Пароль» монитора введите `{{monitorSecrets.dbPassword}}`.

OneUptime разрешает секрет на стороне сервера до того, как конфигурация будет передана зонду. OneUptime никогда не создаёт эти секреты за вас — ссылаться на секрет вы решаете сами.

## Настройка критериев

Добавьте критерии, чтобы определить, когда монитор считается доступным, деградированным или недоступным. Для Монитора SQL-запросов доступны следующие проверки:

- **SQL доступен** — была ли база данных доступна и успешно ли выполнился запрос.
- **Количество строк SQL-запроса** — количество возвращённых строк. Сравнивается операторами «больше чем», «меньше чем» или «равно».
- **Скалярное значение SQL-запроса** — первый столбец первой строки. Сравнивается численно, если обе стороны выглядят как числа, иначе — как строки. Эту проверку следует использовать для запросов вида `COUNT(*)`.
- **Время выполнения SQL-запроса (в мс)** — длительность выполнения запроса. Полезно для выявления медленной базы данных.
- **Ошибка SQL-запроса** — сообщение об ошибке запроса. Оповещайте, когда оно пустое (или не пустое) либо совпадает с определённой строкой.
- **Выражение JavaScript** — вычисление пользовательского выражения JavaScript для полного контроля. См. [Выражения JavaScript](/docs/monitor/javascript-expression).

### Пример: оповещение при резком росте отмен

Используя приведённый выше запрос:

- **Критерий: Деградирован** — `Скалярное значение SQL-запроса` больше `10`.
- **Критерий: Недоступен** — `Скалярное значение SQL-запроса` больше `50` или `SQL доступен` равно `false`.

Прикрепите к критериям политику дежурств, чтобы уведомления получали нужные люди.

## Создание пользователя только для чтения

Всегда подключайтесь через выделенного пользователя с правами только для чтения. Примеры:

```sql
-- PostgreSQL
CREATE USER oneuptime_ro WITH PASSWORD 'a-strong-password';
GRANT CONNECT ON DATABASE orders TO oneuptime_ro;
GRANT USAGE ON SCHEMA public TO oneuptime_ro;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO oneuptime_ro;
-- Включая таблицы, создаваемые в будущем:
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO oneuptime_ro;
```

```sql
-- MySQL
CREATE USER 'oneuptime_ro'@'%' IDENTIFIED BY 'a-strong-password';
GRANT SELECT ON orders.* TO 'oneuptime_ro'@'%';
FLUSH PRIVILEGES;
```

```sql
-- Microsoft SQL Server
CREATE LOGIN oneuptime_ro WITH PASSWORD = 'a-strong-password';
USE orders;
CREATE USER oneuptime_ro FOR LOGIN oneuptime_ro;
ALTER ROLE db_datareader ADD MEMBER oneuptime_ro;
```

## Что следует учитывать

- Запрос выполняется при каждой проверке, поэтому делайте его лёгким. Используйте индексы и узкие временные окна, а «Тайм-аут выполнения запроса» используйте как страховку.
- Передаются только количество строк, первая ячейка (скалярное значение) и первая строка — составляйте запрос так, чтобы значение, по которому нужно оповещать, было в первом столбце.
- Если результат усечён из-за превышения «Максимума строк», в сводке проверки это отмечается как ограничение. Увеличивайте «Максимум строк» только при необходимости; большие наборы результатов требуют больше памяти на зонде.
- Операции записи и DDL всегда отклоняются. Если нужно протестировать путь записи, этот монитор для этого не предназначен.
- Предпочитайте секрет монитора паролю в открытом виде, чтобы учётные данные оставались зашифрованными при хранении.
