# Монитор входящих запросов

Мониторинг входящих запросов (также известный как мониторинг пульса) позволяет отслеживать сервисы, которые сами периодически отправляют HTTP-запросы в OneUptime. Вместо того чтобы OneUptime обращался к вашему сервису, сам сервис отправляет сигнал в OneUptime, подтверждая свою работоспособность.

## Обзор

Мониторы входящих запросов предоставляют уникальный URL-адрес вебхука, на который ваши сервисы обращаются по расписанию. Это позволяет:

- Мониторинг cron-заданий и запланированных задач
- Проверка работы фоновых воркеров
- Мониторинг сервисов за файерволом, недоступных снаружи
- Интеграция со сторонними инструментами мониторинга
- Отслеживание сигналов пульса от любых систем, поддерживающих HTTP

## Создание монитора входящих запросов

1. Перейдите в раздел **Мониторы** на панели управления OneUptime
2. Нажмите **Создать монитор**
3. Выберите тип монитора **Входящий запрос**
4. Для данного монитора будет сгенерирован **Секретный ключ** и URL пульса
5. Настройте свой сервис для отправки запросов на URL пульса
6. При необходимости настройте критерии мониторинга

## URL пульса

После создания у монитора появится уникальный URL пульса в формате:

```
https://oneuptime.com/heartbeat/YOUR_SECRET_KEY
```

Ваш сервис должен отправлять HTTP-запросы **GET** или **POST** на этот URL через регулярные промежутки времени.

### Отправка сигнала пульса

#### Через curl

```bash
# Простой GET-запрос
curl https://oneuptime.com/heartbeat/YOUR_SECRET_KEY

# POST-запрос с телом запроса
curl -X POST https://oneuptime.com/heartbeat/YOUR_SECRET_KEY \
  -H "Content-Type: application/json" \
  -d '{"status": "healthy", "version": "1.2.3"}'
```

#### Из cron-задания

```bash
# Добавьте в crontab для отправки пульса каждые 5 минут
*/5 * * * * curl -s https://oneuptime.com/heartbeat/YOUR_SECRET_KEY > /dev/null
```

#### Из кода приложения

```javascript
// Пример на Node.js
const https = require("https");
https.get("https://oneuptime.com/heartbeat/YOUR_SECRET_KEY");
```

```python
# Пример на Python
import requests
requests.get('https://oneuptime.com/heartbeat/YOUR_SECRET_KEY')
```

При использовании самостоятельного хостинга замените `https://oneuptime.com` URL-адресом вашего экземпляра OneUptime.

## Критерии мониторинга

Можно настроить критерии, определяющие, считается ли сервис доступным, деградированным или недоступным, на основе:

### Доступные типы проверок

| Тип проверки               | Описание                                         |
| -------------------------- | ------------------------------------------------ |
| Входящий запрос            | Был ли получен пульс в пределах временного окна  |
| Тело запроса               | Содержимое тела запроса, отправленного с пульсом |
| Заголовок запроса          | Имя конкретного заголовка запроса                |
| Значение заголовка запроса | Значение конкретного заголовка запроса           |

### Типы фильтров

Для **Входящего запроса**:

- **Получено в течение минут** — пульс получен в течение указанного количества минут
- **Не получено в течение минут** — пульс не получен в течение указанного количества минут

Для **Тела запроса**, **Заголовка запроса** и **Значения заголовка запроса**:

- **Содержит** — значение содержит указанный текст
- **Не содержит** — значение не содержит указанный текст

### Примеры критериев

#### Статус «Недоступен» при отсутствии пульса в течение 10 минут

- **Проверять**: Входящий запрос
- **Тип фильтра**: Не получено в течение минут
- **Значение**: 10

#### Статус «Деградирован» на основе содержимого тела запроса

- **Проверять**: Тело запроса
- **Тип фильтра**: Содержит
- **Значение**: `"status": "degraded"`

## Рекомендации по использованию

1. **Устанавливайте подходящее временное окно** — если cron-задание выполняется каждые 5 минут, установите порог «Не получено в течение минут» равным 10–15 минутам, чтобы учесть возможные задержки
2. **Включайте значимые данные** — отправляйте информацию о статусе в теле запроса для настройки детальных критериев
3. **Используйте POST для детальных данных** — используйте POST-запросы с телами JSON, когда нужно отправить подробную информацию о статусе
4. **Мониторинг монитора** — убедитесь, что сервис, отправляющий пульсы, имеет корректную обработку ошибок, чтобы неудачные запросы пульса не оставались незамеченными
