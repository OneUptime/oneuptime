# Монитор внешних страниц статуса

Мониторинг внешних страниц статуса позволяет отслеживать статус-страницы сторонних поставщиков и получать оповещения при сбоях или деградации производительности зависимых сервисов. OneUptime периодически проверяет внешние страницы статуса (такие как AWS, GCP, Azure, GitHub, OpenAI, Anthropic и другие) и оценивает их состояние.

## Обзор

Мониторы внешних страниц статуса проверяют работоспособность сервисов, от которых вы зависите, запрашивая их публичные страницы статуса. Это позволяет:

- Мониторить доступность сторонних сервисов, от которых зависит ваше приложение
- Получать оповещения при сбоях у поставщиков вышестоящего уровня
- Отслеживать статусы отдельных компонентов (например, «AWS EC2 us-east-1»)
- Ограничивать мониторинг одной группой компонентов (например, только группой «APIs» у OpenAI), чтобы не связанные инциденты в других частях страницы не срабатывали в вашем мониторе
- Обнаруживать деградацию производительности до того, как она затронет ваших пользователей
- Сопоставлять собственные инциденты с проблемами у поставщиков вышестоящего уровня

## Поддерживаемые поставщики

OneUptime поддерживает мониторинг страниц статуса через следующие методы:

| Тип поставщика           | Описание                                                           |
| ------------------------ | ------------------------------------------------------------------ |
| **Авто** (по умолчанию)  | Автоматически определяет формат страницы статуса                   |
| **Atlassian Statuspage** | Страницы статуса на основе Atlassian Statuspage (JSON API)         |
| **incident.io**          | Страницы статуса на основе incident.io (например, `https://status.openai.com`) |
| **RSS**                  | Страницы статуса, предоставляющие RSS-канал                        |
| **Atom**                 | Страницы статуса, предоставляющие Atom-канал                       |

### Автоматическое определение

При выборе варианта **Авто** OneUptime пытается автоматически определить формат страницы статуса в следующем порядке:

1. Сначала пробует JSON API Atlassian Statuspage (`/api/v2/status.json`, `/api/v2/components.json` и `/api/v2/incidents/unresolved.json`)
2. Затем пробует API страницы статуса incident.io (`/proxy/<host>`)
3. При неудаче пытается разобрать страницу как RSS- или Atom-канал
4. В качестве последнего варианта выполняет базовую проверку доступности HTTP

## Создание монитора внешней страницы статуса

1. Перейдите в раздел **Мониторы** на панели управления OneUptime
2. Нажмите **Создать монитор**
3. Выберите тип монитора **Внешняя страница статуса**
4. Введите URL страницы статуса для мониторинга
5. При желании выберите конкретный тип поставщика (или оставьте **Авто**)
6. При желании введите **группу компонентов** для ограничения мониторинга группой, такой как «APIs»
7. При желании введите **имя компонента** для фильтрации до одного компонента (внутри группы, если группа задана)
8. При необходимости настройте критерии мониторинга

## Параметры конфигурации

### URL страницы статуса

Введите URL внешней страницы статуса для мониторинга. Для сайтов на платформе Atlassian Statuspage и incident.io это обычно корневой URL (например, `https://status.example.com`). Для RSS/Atom-каналов введите URL канала напрямую.

### Тип поставщика

Выберите тип поставщика страницы статуса. Используйте **Авто** (по умолчанию), чтобы позволить OneUptime определить формат автоматически, или укажите **Atlassian Statuspage**, **incident.io**, **RSS** или **Atom**, если он вам известен.

### Фильтр по группе компонентов

Если страница статуса организует свои компоненты в группы, можно ограничить монитор одной группой. Например, на `https://status.openai.com` ввод `APIs` ограничит монитор API-сервисами OpenAI.

Когда задана группа компонентов, **количество активных инцидентов** и **общий статус** вычисляются только по компонентам этой группы — инцидент, затрагивающий не связанную группу (например, ChatGPT), не сработает в мониторе, ограниченном группой «APIs».

Фильтрация по группе компонентов поддерживается для поставщиков **Atlassian Statuspage** и **incident.io**. (RSS/Atom-каналы не предоставляют группы компонентов.)

### Фильтр по имени компонента

Если страница статуса отображает несколько компонентов, можно при желании указать имя компонента для мониторинга только этого конкретного компонента. Например, для мониторинга только AWS EC2 в us-east-1 введите `EC2 us-east-1` (точное имя компонента, как указано на странице статуса).

Когда также задана группа компонентов, фильтр по имени компонента применяется **внутри** этой группы, позволяя нацелиться на один компонент внутри более крупной группы. Если ни один из фильтров не указан, мониторятся все компоненты в области видимости.

### Дополнительные параметры

#### Тайм-аут

Максимальное время (в миллисекундах) ожидания ответа от страницы статуса. По умолчанию 10000 мс (10 секунд).

#### Повторные попытки

Количество повторных попыток при неудачном запросе. По умолчанию 3 попытки.

## Критерии мониторинга

Можно настроить критерии, определяющие, считается ли внешний сервис доступным или недоступным, на основе:

- **Доступен** — доступна ли страница статуса и возвращает ли данные о состоянии
- **Общий статус** — общий индикатор статуса страницы (например, `operational`, `degraded_performance`, `partial_outage`, `major_outage`)
- **Статус компонента** — статус компонентов в области видимости (с учётом фильтров по группе компонентов / имени компонента)
- **Активные инциденты** — количество активных инцидентов, отображаемых на странице статуса (ограниченное группой компонентов / компонентом, когда задан фильтр)
- **Время отклика** — время, необходимое для получения данных страницы статуса

### Критерии по умолчанию

По умолчанию OneUptime задаёт критерии на основе того, что действительно важно для страницы статуса — её активных инцидентов и состояния компонентов, а не простой доступности:

- Монитор помечается как **Operational**, когда в области видимости нет активных инцидентов.
- Монитор помечается как **Down** (и создаётся инцидент), когда в области видимости есть хотя бы один активный инцидент или когда компонент в области видимости сообщает о `degraded_performance`, `partial_outage`, `major_outage` или `full_outage`.

Поскольку количество активных инцидентов и статусы компонентов учитывают фильтры по группе компонентов / имени компонента, эти критерии по умолчанию автоматически нацеливаются только на интересующие вас компоненты.

## Шаблонизация инцидентов и оповещений

При создании инцидентов или оповещений на основе мониторов внешних страниц статуса можно использовать следующие переменные шаблона:

| Переменная                | Описание                                                     |
| ------------------------- | ------------------------------------------------------------ |
| `{{isOnline}}`            | Доступна ли страница статуса (true/false)                    |
| `{{responseTimeInMs}}`    | Время отклика в миллисекундах                                |
| `{{failureCause}}`        | Причина сбоя, если применимо                                 |
| `{{overallStatus}}`       | Значение общего индикатора статуса                           |
| `{{activeIncidentCount}}` | Количество активных инцидентов (ограниченное фильтром, если задан) |
| `{{componentStatuses}}`   | Массив JSON со статусами компонентов (`name`, `status`, `description`, `groupName`) |
| `{{provider}}`            | Определённый поставщик (Atlassian Statuspage, incident.io, RSS, Atom) |
| `{{componentGroup}}`      | Группа компонентов, которой ограничен монитор, если задана   |
| `{{componentName}}`       | Компонент, которым ограничен монитор, если задан             |

## Рекомендации по использованию

- **Используйте тип поставщика «Авто»**, если точный формат неизвестен — автоматическое определение хорошо работает для большинства страниц статуса
- **Ограничивайте мониторинг группой компонентов**, если вы зависите только от части поставщика (например, только от группы «APIs» у OpenAI), чтобы не связанные инциденты не создавали шум
- **Отслеживайте конкретные компоненты**, если вы зависите только от отдельных сервисов (например, конкретного региона AWS)
- **Настройте корреляцию инцидентов** — когда ваши мониторы обнаруживают проблемы, а страница статуса поставщика вышестоящего уровня также показывает неполадки, это помогает быстрее определить первопричину
- **Комбинируйте с другими мониторами** — используйте мониторы внешних страниц статуса совместно с собственными мониторами API и сайтов для комплексного контроля
