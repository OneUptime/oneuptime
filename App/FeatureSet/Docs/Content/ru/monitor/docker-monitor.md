# Монитор Docker

Мониторинг Docker позволяет отслеживать работоспособность и производительность хостов Docker и выполняемых на них контейнеров. OneUptime собирает метрики и журналы контейнеров через предварительно настроенный OpenTelemetry Collector (**OneUptime Docker Agent**) и оценивает их на соответствие заданным критериям.

## Обзор

Мониторы Docker используют метрики и журналы с ваших хостов для обеспечения видимости контейнерных рабочих нагрузок. Это позволяет:

- Мониторинг работоспособности хоста Docker и отдельных контейнеров
- Отслеживание CPU, памяти, сети, блочного ввода-вывода и количества процессов по контейнерам
- Обнаружение перезапусков, сбоев и троттлинга CPU контейнеров
- Потоковая передача структурированных журналов контейнеров в нативном формате OpenTelemetry
- Оповещения при высокой нагрузке на CPU и память, цикличных перезапусках и других событиях

## Создание монитора Docker

1. Перейдите в раздел **Мониторы** на панели управления OneUptime
2. Нажмите **Создать монитор**
3. Выберите тип монитора **Docker**
4. Выберите хост Docker и область ресурсов для мониторинга
5. Настройте метрические запросы и агрегацию
6. При необходимости настройте критерии мониторинга

## Параметры конфигурации

### Хост Docker

Выберите хост Docker для мониторинга. Хосты автоматически регистрируются при первой отправке телеметрии агентом OneUptime Docker Agent — создавать их вручную не требуется.

### Область ресурсов

Выберите уровень, на котором следует отслеживать ресурсы:

| Область   | Описание                                                       |
| --------- | -------------------------------------------------------------- |
| Хост      | Мониторинг всего хоста Docker с агрегацией по всем контейнерам |
| Контейнер | Мониторинг конкретного контейнера по имени или образу          |

### Метрические запросы

Настройте один или несколько метрических запросов. Каждый запрос задаёт:

- **Имя метрики** — метрика контейнера для запроса
- **Агрегация** — способ агрегации значений метрик (Avg, Sum, Max, Min)
- **Фильтры** — дополнительная фильтрация по атрибутам (например, по имени контейнера, образу или хосту)
- **Группировка** — необязательная группировка по `resource.container.name` для независимой оценки каждого контейнера

Также можно создавать **формулы**, объединяющие несколько метрических запросов с помощью математических выражений.

### Скользящее временное окно

Выберите временное окно для оценки метрик:

- Последняя 1 минута
- Последние 5 минут
- Последние 10 минут
- Последние 15 минут
- Последние 30 минут
- Последние 60 минут

## Собираемые метрики

Агент Docker использует приёмник OpenTelemetry `docker_stats`, который опрашивает Docker Engine API с настраиваемым интервалом (по умолчанию каждые 30 секунд).

### CPU

| Метрика                                           | Описание                                                  |
| ------------------------------------------------- | --------------------------------------------------------- |
| `container.cpu.utilization`                       | Использование CPU в процентах от CPU хоста                |
| `container.cpu.usage.total`                       | Суммарное время CPU, потреблённое контейнером             |
| `container.cpu.throttling_data.throttled_time`    | Время, в течение которого контейнер был ограничен cgroups |
| `container.cpu.throttling_data.throttled_periods` | Количество периодов троттлинга                            |

### Память

| Метрика                        | Описание                                   |
| ------------------------------ | ------------------------------------------ |
| `container.memory.usage.total` | Текущее использование памяти в байтах      |
| `container.memory.usage.limit` | Лимит памяти в байтах                      |
| `container.memory.percent`     | Использование памяти в процентах от лимита |

### Сеть

| Метрика                               | Описание                    |
| ------------------------------------- | --------------------------- |
| `container.network.io.usage.rx_bytes` | Общий объём полученных байт |
| `container.network.io.usage.tx_bytes` | Общий объём переданных байт |

### Блочный ввод-вывод

| Метрика                                              | Описание                                |
| ---------------------------------------------------- | --------------------------------------- |
| `container.blockio.io_service_bytes_recursive.read`  | Байты, прочитанные с блочных устройств  |
| `container.blockio.io_service_bytes_recursive.write` | Байты, записанные на блочные устройства |

### Информация о контейнере

| Метрика                | Описание                               |
| ---------------------- | -------------------------------------- |
| `container.uptime`     | Время работы контейнера в секундах     |
| `container.restarts`   | Количество перезапусков контейнера     |
| `container.pids.count` | Количество процессов внутри контейнера |

## Критерии мониторинга

### Доступные типы проверок

| Тип проверки     | Описание                                               |
| ---------------- | ------------------------------------------------------ |
| Значение метрики | Значение настроенного метрического запроса или формулы |

### Типы агрегации

| Агрегация             | Описание                                              |
| --------------------- | ----------------------------------------------------- |
| Среднее               | Среднее значение за временное окно                    |
| Сумма                 | Сумма всех значений                                   |
| Максимальное значение | Наибольшее значение за временное окно                 |
| Минимальное значение  | Наименьшее значение за временное окно                 |
| Все значения          | Все значения должны соответствовать критерию          |
| Любое значение        | Хотя бы одно значение должно соответствовать критерию |

### Типы фильтров

- **Больше чем**, **Меньше чем**, **Больше или равно**, **Меньше или равно**, **Равно**, **Не равно**

## Готовые шаблоны оповещений

OneUptime предоставляет шаблоны для типовых сценариев мониторинга Docker:

| Шаблон                                  | Описание                              | Порог | Агрегация                |
| --------------------------------------- | ------------------------------------- | ----- | ------------------------ |
| Высокая нагрузка CPU контейнера         | Использование CPU контейнером         | > 90% | Максимум (по контейнеру) |
| Высокое использование памяти контейнера | Использование памяти в % от лимита    | > 85% | Максимум (по контейнеру) |
| Высокий троттлинг CPU                   | Периоды троттлинга CPU                | > 0   | Максимум (по контейнеру) |
| Цикличные перезапуски контейнера        | Количество перезапусков               | > 3   | Сумма                    |
| Контейнер недоступен                    | Время работы контейнера сброшено до 0 | = 0   | Минимум                  |

> Примечание: шаблоны для CPU, памяти и троттлинга используют агрегацию **Максимум** с группировкой по `resource.container.name`. Это предотвращает размывание сигнала от одного перегруженного контейнера большим количеством простаивающих контейнеров на том же хосте.

## Сбор журналов

Помимо метрик, агент Docker отслеживает файл `*-json.log` каждого контейнера через приёмник OpenTelemetry filelog и отправляет записи журналов в нативном формате OTLP. Каждая запись обогащается:

- `resource.host.name` — идентификатор хоста Docker
- `resource.container.id` — полный идентификатор контейнера
- `resource.container.runtime` — всегда `docker`
- `attributes["log.iostream"]` — `stdout` или `stderr`
- `severityText` / `severityNumber` — определяется по потоку: `stderr` → `ERROR`, `stdout` → `INFO`
- `body` — необработанная строка журнала, выведенная процессом контейнера
- `time` — временная метка Docker daemon для данной строки

Журналы отображаются на вкладке **Журналы** хоста Docker и на странице сведений о каждом контейнере.

### Требование к драйверу журналирования

**Агент Docker принимает журналы только от контейнеров, использующих драйвер журналирования `json-file` Docker.** Это драйвер Docker по умолчанию, но его можно переопределить для отдельных контейнеров или глобально:

- Драйвер **`local`** — записывает бинарные блоки protobuf в `/var/lib/docker/containers/<id>/local-logs/container.log`. Приёмник filelog не может разобрать этот формат.
- **`journald`**, **`syslog`**, **`fluentd`**, **`gelf`**, **`awslogs`**, **`splunk`** и др. — отправляют журналы на удалённый адрес; файла для отслеживания нет.
- **`none`** — полностью отбрасывает журналы.

При использовании любого из вышеперечисленных драйверов метрики будут отображаться на странице хоста Docker, но вкладка **Журналы** будет пустой (или содержать только журналы самого агента Docker).

**Проверка драйвера журналирования конкретного контейнера:**

```bash
docker inspect <container> --format '{{.HostConfig.LogConfig.Type}}'
```

**Проверка драйвера по умолчанию:**

```bash
docker info --format '{{.LoggingDriver}}'
```

**Переключение сервиса Docker Compose на `json-file` с разумной ротацией:**

```yaml
services:
  my-app:
    image: my-app:latest
    logging:
      driver: "json-file"
      options:
        max-size: "100m"
        max-file: "5"
```

**Изменение драйвера по умолчанию** (применяется ко всем последующим контейнерам) путём редактирования `/etc/docker/daemon.json`:

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "5"
  }
}
```

Затем перезапустите Docker daemon и **пересоздайте** затронутые контейнеры. Docker назначает драйвер журналирования при создании контейнера, поэтому существующий контейнер сохраняет старый драйвер до его удаления и пересоздания:

```bash
# Docker Compose
docker compose up -d --force-recreate <service>

# Plain docker
docker rm -f <container>
docker run ... <image>
```

## Требования для настройки

Для использования мониторинга Docker необходимо:

1. Установить агент OneUptime Docker Agent на каждый хост Docker, который требуется отслеживать
2. Передать `ONEUPTIME_URL`, `ONEUPTIME_SERVICE_TOKEN` и `DOCKER_HOST_NAME` в качестве переменных среды
3. Убедиться, что отслеживаемые контейнеры используют драйвер журналирования `json-file` (см. выше)

Агент публикуется как `oneuptime/docker-agent:release` на Docker Hub. Полные примеры команд `docker run` и `docker compose` см. в [руководстве по установке агента Docker](https://github.com/OneUptime/oneuptime/tree/master/DockerAgent).

## Устранение неполадок

### Метрики отображаются, но вкладка «Журналы» пуста

Скорее всего, ваши контейнеры не используют драйвер журналирования `json-file`. Выполните диагностические команды из раздела [Требование к драйверу журналирования](#требование-к-драйверу-журналирования) и переключите контейнеры, журналы которых необходимо собирать.

### Приёмник filelog выводит `no files match the configured criteria`

Это означает, что шаблон `/var/lib/docker/containers/*/*-json.log` не совпал ни с одним файлом при запуске агента. Возможные причины:

1. Ни один контейнер на этом хосте не использует `json-file`, либо
2. Привязка `-v /var/lib/docker/containers:/var/lib/docker/containers:ro` отсутствует или указывает на пустой каталог, либо
3. Агент запущен на Docker Desktop для macOS без доступа к каталогу контейнеров виртуальной машины Linux.

### Журналы поступают, но сгруппированы под неверным именем хоста

OneUptime автоматически регистрирует хосты Docker по `resource.host.name`, значение которого берётся из переменной среды `DOCKER_HOST_NAME`. Изменение `DOCKER_HOST_NAME` после первой отправки телеметрии создаст новую строку хоста вместо переименования существующей.

### Инциденты «Высокая нагрузка CPU» не срабатывают

Убедитесь, что агрегация в метрическом запросе установлена на **Максимум** (не Среднее) и что задана группировка по `resource.container.name`. Среднее по всем контейнерам на загруженном хосте размывается простаивающими контейнерами и редко превышает порог.
