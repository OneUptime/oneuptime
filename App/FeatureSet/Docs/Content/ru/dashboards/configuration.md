# Конфигурация и разрешения

На этой странице собраны настройки и средства контроля доступа, о которых стоит знать, когда у вас уже есть дашборд, который вы планируете поддерживать.

## Владельцы

**Владельцы (Owners)** дашборда — это пользователи и команды, которым вы дали явный доступ (поверх их общей роли в проекте).

В разделе **Dashboard → Owners**:

- Добавьте **владельца-пользователя**, чтобы дать одному человеку дополнительный доступ к этому дашборду.
- Добавьте **владельца-команду**, чтобы дать то же самое каждому участнику команды.

Используйте владельцев, когда общая по проекту роль «чтения» слишком широка — например, для дашборда с деталями по клиентам, который должен видеть только отдел по работе с клиентами.

## Метки

Метки — это теги для упорядочивания дашбордов. Применяйте их в разделе **Dashboard → Overview**.

Распространённые схемы:

- **По командам**: `team:platform`, `team:checkout`, `team:growth`.
- **По окружениям**: `env:prod`, `env:staging`.
- **По назначению**: `purpose:oncall`, `purpose:exec`, `purpose:investigation`.

Список **Dashboards** позволяет фильтровать по метке — это самый быстрый способ найти нужный дашборд в проекте, где их накопилось много.

## Разрешения

Дашборды работают с ролевым контролем доступа вашего проекта. Связанные разрешения:

| Разрешение | Что позволяет |
| --- | --- |
| **Create Dashboard** | Создавать новые дашборды. |
| **Read Dashboard** | Просматривать дашборды (в приватном режиме). |
| **Edit Dashboard** | Изменять виджеты, переменные и настройки. |
| **Delete Dashboard** | Удалять дашборд. |

Есть отдельные разрешения для владельцев дашборда и кастомных доменов, поэтому можно выдать «управлять владельцами», не выдавая «редактировать дашборд».

Назначайте их на ролях проекта в **Project Settings → Teams & Roles**.

## Доступ к публичным дашбордам

Когда вы делаете дашборд публичным (см. [Совместное использование и публичные дашборды](/docs/dashboards/sharing)), тем, кто его видит, управляют три настройки:

1. Переключатель **Public Dashboard** — если выключен, публичный URL возвращает 404.
2. **Master Password** — если задан, посетители вводят пароль, прежде чем увидят дашборд.
3. **IP Whitelist** (тариф Scale) — если задан, запросы с других IP отклоняются.

Эти настройки можно сочетать. Самый защищённый сценарий — «Public включён, пароль задан, белый список IP активен» — полезен для партнёрских порталов, где нужны все три уровня.

## Хранение данных

Сами дашборды не имеют срока годности. Данные, которые они показывают, подчиняются настройкам хранения вашего проекта — метрики, логи и трейсы доступны для запроса столько, сколько хранит ваш тариф. Виджет, нацеленный на «прошедшие 90 дней» на тарифе с 30-дневным хранением, покажет то, что ещё сохранилось.

## Дублирование дашборда

Чтобы скопировать существующий дашборд, откройте список дашбордов и выберите **Duplicate**. Копия включает все виджеты, переменные и настройки, кроме публичного доступа — он всегда начинается выключенным, чтобы вы могли решить, нужно ли его включать.

Это правильный шаг, когда хочется форкнуть шаблон (например, «наш дашборд для дежурного») в копию для конкретного сервиса.

## Удаление дашборда

В разделе **Dashboard → Delete**. Это нельзя отменить — раскладка дашборда и все привязанные к нему кастомные домены будут удалены. Ваши данные телеметрии не затронуты.

Если дашборд опубликован на кастомном домене, URL перестаёт открываться сразу после удаления. Если хотите сохранить URL рабочим, сначала перенесите домен на другой дашборд.

## Резервное копирование

Если вы хостите OneUptime самостоятельно, обычной резервной копии базы данных достаточно — конфигурация дашборда хранится вместе с остальной частью вашего проекта.

В OneUptime Cloud резервные копии делаются за вас. Если хочется иметь собственную копию, дашборд можно прочитать через [OneUptime API](/docs/api-reference/api-reference).
