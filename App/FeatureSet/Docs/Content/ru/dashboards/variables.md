# Переменные и фильтры дашборда

Переменная превращает один дашборд в шаблон. Определите переменную `service` — и один и тот же график перерисуется для `checkout`, `payments` и `search`: выбирайте из выпадающего списка наверху вместо того, чтобы строить три почти одинаковых дашборда.

Эта страница покрывает четыре типа переменных, как их значения подставляются в запросы виджетов, а также глобальные контролы временного диапазона и обновления, стоящие рядом.

## Типы переменных

Переменные добавляются в **Dashboard → Settings → Variables**. У каждой есть имя (на которое ссылаются как `{{name}}` в запросах виджетов), опциональный label и тип.

### Custom List

Статический выпадающий список. Вы поставляете список значений через запятую; зритель выбирает одно.

Используйте, когда: набор вариантов небольшой, фиксированный и осмысленный только для вашей команды. `environment` со значениями `prod, staging, dev`. `region` со значениями `us-east-1, eu-west-1, ap-south-1`.

### Query

Опции для выпадающего списка вычисляются ClickHouse-запросом во время рендеринга.

Используйте, когда: варианты динамические и живут в вашей телеметрии. «Каждый customer ID, который логировался за последние 24 часа» через `SELECT DISTINCT customer_id FROM ...`. Запрос выполняется по данным вашего проекта; относитесь к результату как к недоверенному входу, даже если это ваши собственные данные.

### Text Input

Поле свободного текста. Что зритель введёт, то и подставляется.

Используйте, когда: хотите, чтобы дашборд работал как поисковый инструмент. Дашборд «фильтр по IP» или «фильтр по request ID».

### Telemetry Attribute

Опции — это различные значения ключа атрибута OpenTelemetry по всей телеметрии проекта за временной диапазон дашборда.

Настройте **ключ атрибута** (например, `k8s.cluster.name`, `service.name`, `host.name`). Виджет получает различные значения из логов / метрик / трейсов и предлагает их в выпадающем списке.

Используйте, когда: варианты — это именно сущности, которыми вы уже тегировали свою телеметрию. Имя кластера, имя сервиса, регион, customer ID, окружение — всё, что вы уже отправляете как ресурсный или span-атрибут OpenTelemetry.

Это самый распространённый тип переменных для сервис-ориентированных дашбордов, потому что он автообновляется: когда вы выкатываете новый сервис с тегом `service.name = inventory`, это значение появляется в выпадающем списке, без редактирования дашборда.

## Multi-select

Каждая переменная может быть настроена как **multi-select**. Когда это включено, зритель выбирает одно или несколько значений; дашборд фильтрует по `value IN (...)` вместо `value = ...`.

Используйте multi-select, когда: вы хотите смотреть «checkout + payments вместе», не уходя с дашборда. Избегайте его, когда математика графика не складывается между выбранными значениями — например, при усреднении средних.

## Значения по умолчанию

Каждая переменная принимает опциональное значение по умолчанию. Дашборд рисуется со значением по умолчанию, пока зритель не изменит выпадающий список. Для публичных дашбордов значение по умолчанию — это то, с чем посетители попадают на страницу.

## Как работает интерполяция

Везде, где запрос виджета принимает строковый фильтр — клауза `WHERE` запроса метрики, фильтр виджета-списка, совпадение по атрибуту в потоке логов — можно ссылаться на `{{variable_name}}`.

Например, запрос метрики Chart может выглядеть так:

```
SELECT avg(latency_ms) FROM spans WHERE service.name = '{{service}}'
```

Когда `service` установлен в `checkout`, запрос выполняется с `service.name = 'checkout'`. Когда зритель переключается на `payments`, запрос перезапускается с `service.name = 'payments'`.

Для переменных **Telemetry Attribute** OneUptime знает ключ атрибута и подставляет фильтр в каждый виджет, упоминающий тот же атрибут, — не нужно вручную править запрос каждого виджета при смене переменной. Это и есть та магия, благодаря которой сервис-шаблонные дашборды работают «из коробки».

## Временной диапазон

В заголовке дашборда есть глобальный селектор **time range**. Каждый виджет-метрика делает запрос по этому окну. Варианты:

- **Пресеты** — последний час, 24 часа, 7 дней, 30 дней, 90 дней (в зависимости от вашего retention).
- **Произвольный диапазон** — выберите метки времени начала и конца.

Временной диапазон — часть URL дашборда; поделившись URL, вы делитесь и окном. Это удобно во время инцидента: закрепите временной диапазон на «10:00–10:30 UTC сегодня» и поделитесь ссылкой в канале инцидента.

## Интервал обновления

Рядом с временным диапазоном выберите, как часто виджеты переопрашивают:

- **Off** — виджеты опрашивают один раз при загрузке.
- **5с / 10с / 30с / 1м / 5м / 15м** — автообновление.

Автообновление удобно для настенного экрана и для текущего вида инцидента. Для ad-hoc-расследования оставьте выключенным, чтобы вид оставался стабильным во время прокрутки.

## Собираем вместе

Сервис-шаблонный дашборд обычно содержит:

1. Переменную `service` типа **Telemetry Attribute**, привязанную к `service.name`. По умолчанию: ваш самый наблюдаемый сервис. Multi-select: off (чтобы графики всегда показывали один сервис).
2. Переменную `environment` типа **Custom List**. По умолчанию: `prod`.
3. Переменную `cluster` типа **Telemetry Attribute**, привязанную к `k8s.cluster.name`. Multi-select: on (чтобы можно было сворачивать по кластерам).
4. Виджеты дашборда ссылаются на эти переменные в своих фильтрах.

Результат: один дашборд, покрытие всего парка, несколько выпадающих списков сверху.
