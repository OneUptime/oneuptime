# Обзор дашбордов

Дашборды — это способ превратить телеметрию, которую OneUptime уже собирает (метрики, логи, трейсы, инциденты, мониторы, ресурсы Kubernetes и Docker), в одну страницу, на которую можно взглянуть и понять состояние системы.

Разместите график задержки запросов рядом со списком открытых инцидентов, рядом с индикатором утилизации CPU, рядом с предложением статуса простым языком. Сохраните. Поделитесь ссылкой.

## Кратко

- **Функция верхнего уровня** в панели OneUptime: **Dashboards**.
- **Холст на сетке** — 12 единиц в ширину на 60 единиц в высоту по умолчанию. Перетаскивайте виджеты, изменяйте их размер, привязывайтесь к сетке.
- **Более 20 типов виджетов** — графики, единичные значения, индикаторы, таблицы, текстовые блоки, потоки логов, списки трейсов и live-списки ресурсов для инцидентов, оповещений, мониторов, Kubernetes (поды, ноды, deployment-ы …), Docker и хостов.
- **Переменные и фильтры** — превращайте один дашборд в шаблонизированное представление, переиспользуемое для каждого кластера, сервиса, клиента или окружения.
- **Публичный доступ** — переключите тумблер, и дашборд станет доступен по публичному URL с опциональной защитой паролем и IP-allowlist-ом.
- **Собственные домены** — размещайте публичный дашборд на `status.your-domain.com` вместо OneUptime.

## Зачем нужны дашборды?

Дашборды оправдывают себя, когда верно одно из:

- **Вам нужна страница «всё ли в порядке?»** для on-call-ротации, ежедневного стендапа или CEO, проходящего мимо настенного телевизора.
- **Вам нужно сопоставлять сигналы** — скачок CPU в ту же минуту, что и рост задержки трейсов, и открытый инцидент гораздо заметнее на одном дашборде, чем в трёх вкладках.
- **Вы расследуете** — свободно собранный дашборд во время отладки быстрее, чем десять ручных запросов.
- **Вы публикуете наружу** — клиентский дашборд производительности, партнёрский свод, публичная страница «здоровья» open-source-сервиса.

## Ключевые понятия

| Термин | Смысл |
| --- | --- |
| **Dashboard** | Холст. Названное многоразовое представление, содержащее список виджетов, контрол временного диапазона и набор переменных. |
| **Widget** | Один компонент на холсте — график, значение, таблица, текстовый блок, список. У каждого есть тип и конфигурация в стиле JSON. |
| **Dashboard unit** | Клетка сетки. Виджеты измеряются в единицах дашборда (например, «4 в ширину × 6 в высоту»). Единицы конвертируются в пиксели в зависимости от viewport-а. |
| **Variable** | Именованное значение, которое зритель выбирает из выпадающего списка (или вводит), и которое дашборд подставляет в запрос каждого виджета. Кластер, сервис, клиент, окружение — всё, по чему вы фильтруете. |
| **Time range** | Окно времени, по которому каждый виджет делает запрос. Выберите пресет («последние 24 часа») или произвольный диапазон. |
| **Refresh interval** | Как часто виджеты переопрашивают в режиме **View**. Off, 5с, 10с, 30с, 1м, 5м, 15м. |
| **Mode** | `Edit` (перетаскивать, изменять размер, конфигурировать) или `View` (только чтение). У них один холст. |

## Каталог виджетов

Неполная карта того, что можно разместить на дашборде:

| Категория | Виджеты |
| --- | --- |
| **Временные ряды** | Chart |
| **Единичное число** | Value, Gauge |
| **Табличные** | Table |
| **Аннотация** | Text |
| **Логи и трейсы** | LogStream, TraceList |
| **Операционные списки** | IncidentList, AlertList, MonitorList |
| **Kubernetes** | KubernetesPodList, KubernetesNodeList, KubernetesNamespaceList, KubernetesDeploymentList, KubernetesStatefulSetList, KubernetesDaemonSetList, KubernetesJobList, KubernetesCronJobList |
| **Docker** | DockerHostList, DockerContainerList, DockerImageList, DockerNetworkList, DockerVolumeList |
| **Инфраструктура** | HostList |

Аргументы каждого виджета и когда к нему обращаться — см. [Виджеты](/docs/dashboards/widgets).

## Где дашборды живут в панели

| Страница | Что вы там делаете |
| --- | --- |
| **Dashboards** | Просматривать, создавать, искать дашборды, ставить метки. |
| **Дашборд → View** | Холст — режим Edit для авторов, режим View для всех остальных. Переключение в заголовке. |
| **Дашборд → Overview** | Описание, владение, метки. |
| **Дашборд → Settings** | Публичный доступ, мастер-пароль, IP-allowlist, собственные домены, брендинг (заголовок страницы, описание, лого, фавикон). |
| **Дашборд → Owners** | Пользователи и команды с явным владением. |
| **Дашборд → Delete** | Удалить дашборд (необратимо). |

## Жизненный цикл дашборда

1. **Создать** — в **Dashboards → Create Dashboard** дайте имя. Холст откроется пустым.
2. **Бросить виджеты** — из палитры виджетов выберите тип, настройте источник (запрос метрики, фильтр списка, тело свободного текста). Расположите и измените размер.
3. **(Опционально) добавить переменные** — определите выпадающий список вроде `cluster` или `service`, чтобы один и тот же дашборд рисовался для каждого значения.
4. **Задайте временной диапазон и интервал обновления** — значения по умолчанию подходят; настройку оставьте на потом.
5. **(Опционально) поделиться публично** — в **Settings** включите **Public Dashboard**. Добавьте мастер-пароль, если нужен барьер, или ограничьте по IP.
6. **(Опционально) собственный домен** — добавьте запись `dashboard.your-domain.com` и подтвердите DNS, затем подавайте дашборд на своём URL.

## Развёрнутый пример

Цель: страница on-call для сервиса checkout с задержкой, частотой ошибок, открытыми инцидентами и недавним хвостом логов.

1. Создайте дашборд «Checkout oncall».
2. Добавьте переменную `service` типа **Telemetry Attribute**, привязанную к ключу атрибута `service.name`. Значение по умолчанию — `checkout`.
3. Добавьте виджет **Chart**: P95 задержка из вашей APM-метрики, отфильтрованная по `service.name = {{service}}`. Временной диапазон следует за дашбордом.
4. Рядом добавьте виджет **Value**: процент ошибок с порогом warning на 1% и порогом critical на 5%.
5. Ниже добавьте виджет **IncidentList**, отфильтрованный по меткам, включающим `checkout`.
6. Под ним — виджет **LogStream**, отфильтрованный по `service.name = {{service}}`.
7. Сохраните. Переключите выпадающий список переменной на `payments` — весь дашборд перерисуется для сервиса payments. Тот же шаблон, другой фильтр.

## Как дашборды стыкуются с остальной OneUptime

- **Мониторы и телеметрия** подают на дашборды сырые данные — каждая настроенная вами метрика, каждая принятая строка лога, каждый span трейса запрашивается на виджете.
- **Инциденты и оповещения** появляются в виджетах **IncidentList** и **AlertList** — дашборды — это представления только для чтения по ним; создавать/редактировать эти сущности нужно в другом месте.
- **Страницы статуса** — это инструмент общения с клиентами («работает ли система прямо сейчас?»). Дашборды — это аналитический инструмент («как система ведёт себя в деталях?»). Они дополняют друг друга, а не заменяют.
- **Рабочие процессы** — сторона записи в OneUptime; дашборды — сторона чтения.
