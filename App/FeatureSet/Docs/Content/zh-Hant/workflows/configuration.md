# 配置與安全

本頁涵蓋在你讓工作流對接真實流量之前值得了解的設置與安全限制。

## 啓用或停用工作流

每個工作流在 **設置** 中都有一個 **已啓用** 開關。關閉時,工作流不會運行——對 webhook 調用、定時時間、OneUptime 事件都視而不見。新工作流默認是停用的。

把這個開關當作你的"準備就緒"門:

1. 構建工作流。
2. 用真實的負載點擊 **手動運行**。
3. 查看 **日誌**——確保每個模塊都走到了預期的位置。
4. 把 **已啓用** 切到開啓。

關閉工作流不會停止已經在進行中的運行;它只會阻止新的運行開始。

## 所有者與標籤

- **所有者**——被列爲所有者的用戶和團隊可以訪問該工作流,並且可以選擇在它失敗時收到通知。在 **設置 → 所有者** 下設置。
- **標籤**——用於給工作流分組的標記。工作流列表允許按標籤篩選,讓繁忙的項目更容易導航。當你按團隊、集成或環境組織工作流時很有用。
- **標籤規則**——在 **工作流 → 設置 → 標籤規則** 下,根據名稱或描述的模式,自動爲新工作流應用標籤。
- **所有者規則**——在 **工作流 → 設置 → 所有者規則** 下,自動爲新工作流分配所有者。

## 密鑰

如果某個全局變量包含敏感內容,把它標記爲 **密鑰**。值會被加密、保存後在 UI 中隱藏,並在運行日誌中隱藏(顯示爲 `[REDACTED]`)。

把密鑰變量用於:

- 外部服務的 API 密鑰。
- 認證令牌。
- Webhook 簽名密鑰。
- 任何你不想讓只讀訪問者看到的內容。

不要把密鑰直接粘貼到模塊裏——像 `Authorization: Bearer eyJh...` 這樣的值會在工作流和日誌中可見。改用 `{{variable.MY_SECRET}}`。

## 一次運行可以持續多久

每次運行都有最長時長。如果運行沒有及時完成,會被標記爲 **超時**,進行中的模塊會被取消。默認值已經很寬鬆——足以容納正常的 HTTP 調用和成串的模塊。

各模塊在這個時長內還有自己的時間限制——例如,API 模塊對掛起的出站請求會在整個運行超時之前就放棄。

## 調用其他工作流的限制

**執行工作流** 組件讓一個工作流可以調用另一個。爲了防止 A 調用 B、B 又調用 A 這種意外的循環,鏈路深度有一個上限。超過限制的運行會以明確的錯誤結束。

如果你確實有合理的長鏈需求(比如一個工作每次運行處理一項),通常更簡單的做法是在單個工作流裏用 **自定義代碼** 循環。

## Webhook 安全

Webhook 觸發器會給你一個專屬的 URL。任何知道這個 URL 的人都可以觸發它。爲了防止意外或不受歡迎的調用方:

- 把 URL 當作密碼對待。不要公開分享或提交到公共倉庫。
- 對敏感工作流,要求調用方在頭部裏發送一個共享令牌(比如 `X-Webhook-Token`),在做任何重要事情之前用 **條件** 模塊檢查它。把預期的令牌保存爲密鑰變量。
- 對極敏感的工作流,優先使用 OneUptime 事件觸發器加手動導入步驟,而不是公開 webhook。

## 出站網絡訪問

API 等 HTTP 模塊的請求都是從 OneUptime 發出的。如果你自託管,確保你的安裝能訪問你正在調用的服務。如果你使用 OneUptime Cloud,我們的出站 IP 段列在 [IP 地址](/docs/configuration/ip-addresses) 中,你可以在另一端把它們加入白名單。

## 權限

工作流遵循你項目的基於角色的訪問控制。相關權限:

- **創建 / 讀取 / 編輯 / 刪除 Workflow**——工作流本身的基本權限。
- **Run Workflow**——點擊 **手動運行** 或通過 API 觸發工作流所需。
- **Read Workflow Log**——查看運行所需。
- **讀取 / 創建 / 編輯 / 刪除 Workflow Variable**——對全局變量列表的控制權限。

大多數工程師應該對工作流有創建/編輯/讀取權限,但對變量則沒有。把變量的編輯權限留給管理項目密鑰的人。

## 計劃限制

OneUptime Cloud 對較小套餐每月的運行次數有上限。當前的限額會顯示在 **項目設置 → 計費** 下。達到上限後,新的觸發會被拒絕直到下一個計費週期。自託管的安裝沒有這個限制。

## 工作流不適合的場景

幾種情況下你應該選擇別的工具:

- **重計算或大數據集**——工作流是爲輕量粘合而設計的,不是爲大量計算。把繁重的工作放在你自己的基礎設施裏跑,讓工作流去啓動它。
- **跨數小時的長時進程**——單次運行的目標是快速完成。如果你需要"做 A、等兩小時、做 B",使用外部調度器,在時間到了時向 OneUptime 發回 webhook。
- **需要人蔘與的逐步事件響應**——這是 [Runbook](/docs/runbooks/index) 的職責。工作流是用於無人值守的自動化。
