# 組件

組件是你在觸發器之後添加的構建模塊。每一個只做一件事——發送消息、調用 API、檢查條件——並連接到接下來的模塊。

本頁是組件目錄。關於如何在畫布上拖動、放置和連接它們,見 [創作工作流](/docs/workflows/authoring)。

## API

對任意 URL 發起 HTTP 請求。

**設置**:

- **Method**——`GET`、`POST`、`PUT`、`PATCH` 或 `DELETE`。
- **URL**——要調用的地址。
- **Headers**——要發送的任意頭部。
- **Body**——用於 `POST` / `PUT` / `PATCH` 的請求體。

**輸出**:

- **成功**——調用成功(2xx 響應)時觸發。一併傳出狀態、頭部和響應體。
- **錯誤**——發生網絡錯誤或非 2xx 響應時觸發。一併傳出錯誤信息。

適合用於:任意外部 API、你自己的管理端點,或任何沒有專用組件的集成。

## Webhook(出站)

API 組件的簡化版,適合"發完即走"的場景。向某個 URL 發送一段 JSON 正文。

如果你需要讀取響應,使用 **API**。如果你只想發個通知就接着幹別的,使用 **Webhook**。

## Slack

向 Slack 頻道發送一條消息。

**設置**:

- **頻道**——頻道名稱。機器人必須已經在那個頻道里。
- **消息**——要發送的文本。支持 Slack 格式。

請先在 **項目設置 → 工作區連接 → Slack** 下把 Slack 連接到你的項目。見 [Slack 工作區連接](/docs/workspace-connections/slack)。

## Microsoft Teams

向 Microsoft Teams 頻道發送一條消息。

**設置**:

- **團隊和頻道**——發佈的位置。
- **消息**——要發送的文本。

設置方式見 [Microsoft Teams 工作區連接](/docs/workspace-connections/microsoft-teams)。

## Discord

通過入站 webhook URL 向 Discord 頻道發送一條消息。

## Telegram

使用機器人令牌和聊天 ID 向 Telegram 聊天發送消息。

## 郵件

通過 OneUptime 發送郵件。

**設置**:

- **收件人**——收件人的郵箱地址。
- **主題**——郵件主題。
- **正文**——Markdown 或 HTML 格式的內容。

郵件會從你項目配置的發件人發出——見 [SMTP](/docs/emails/smtp)。

## 自定義代碼

當其他模塊辦不到時,運行一小段 JavaScript。

**設置**:

- **Code**——你的 JavaScript 代碼。最後的值(或異步函數中返回的值)會成爲模塊的輸出。
- **Arguments**——你可以傳入的命名值。

**輸出**:成功(你的返回值)和錯誤(任何異常)。

適合用於:在兩個系統之間整形數據、做小計算,或任何不值得爲它單獨做一個模塊的事情。需要更重的腳本時,改用 [Runbook](/docs/runbooks/index)。

## JSON

在文本和 JSON 之間轉換。

- **JSON → Text**——把 JSON 對象轉成字符串。當下一個模塊需要文本時有用。
- **Text → JSON**——把字符串解析爲 JSON 對象。當某個內容以文本形式傳來、而你需要讀取它的字段時有用。

## 條件

基於比較進行分支。

**設置**:

- **左側值**——通常來自前面模塊的某個值。
- **運算符**——`==`、`!=`、`>`、`>=`、`<`、`<=`、`contains`、`starts with`、`ends with`。
- **右側值**——要比較的對象。

**輸出**:**是** 和 **否**。把後續模塊連接到你想要的分支。

## 延遲

讓工作流暫停一段時間後再繼續。當你需要給另一個系統一點時間追上時很有用。

## 日誌

向運行日誌寫一行。沒有外部副作用——它只會出現在工作流的日誌裏,供你查看。便於調試。

## 執行工作流

從當前工作流中調用另一個工作流。被調用的工作流獨立運行——你的工作流不會等它完成。

用它來共享通用邏輯。構建一次"發佈到事件頻道"的工作流,然後在所有需要通知該頻道的其他工作流中調用它。

爲了防止工作流循環互調,有一個安全限制。見 [配置與安全](/docs/workflows/configuration)。

## OneUptime 數據組件

對 OneUptime 中的每種記錄類型(監控、事件、警報、狀態頁、值班策略,以及更多),面板裏都有這些組件——按類型名稱搜索:

- **Find One**——按 ID 或篩選條件獲取一條記錄。
- **Find**——獲取記錄列表。
- **Create**——添加一條新記錄。
- **Update**——更新一條記錄。
- **Delete**——刪除一條記錄。
- **Count**——統計符合篩選條件的記錄數。

這就是工作流讀取和修改 OneUptime 數據的方式。例如:來自 CI 工具的 webhook 可以用 **Create Incident** 創建一個帶有失敗詳情的事件。

## 我該用哪個組件?

幾條簡單規則:

- 如果有專門的模塊可用(Slack、郵件、某個 OneUptime 記錄),就用它——你會得到更好的錯誤處理和更清晰的日誌。
- 對任何其他外部 API,使用 **API**。
- 要在模塊間整形數據,使用 **自定義代碼** 或 **JSON**。
- 要根據某個值採取不同動作,使用 **條件**。
