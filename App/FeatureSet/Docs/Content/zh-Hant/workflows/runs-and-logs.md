# 運行與日誌

每當工作流運行時,OneUptime 都會保存一條記錄——什麼時候運行的、是否成功、每個模塊做了什麼。這條記錄就叫一次 **運行**。運行就是你確認工作流是否工作、排查沒有工作的工作流,以及回看過往活動的依據。

## 在哪裏查看

| 頁面 | 你會看到什麼 |
| --- | --- |
| **工作流 → 運行與日誌** | 項目中所有工作流的每一次運行。可以按工作流、狀態和時間篩選。 |
| **工作流 → 日誌標籤** | 僅當前這個工作流的運行。 |
| **單次運行** | 一次執行,包含每個模塊的輸出。 |

## 運行狀態

| 狀態 | 含義 |
| --- | --- |
| **已排隊** | 觸發器已觸發,運行即將開始。通常只需要不到一秒。 |
| **運行中** | 工作流正在進行。長時間運行的模塊會讓運行保持這個狀態。 |
| **成功** | 所有運行過的模塊都沒有錯誤地完成了。(故意走 **錯誤** 分支仍然算成功——工作流本身沒有失敗。) |
| **錯誤** | 某個模塊失敗了,而且沒有連接 **錯誤** 路徑來處理它。運行就此停止。 |
| **超時** | 運行時間超過了允許的上限。見 [配置與安全](/docs/workflows/configuration)。 |

## 閱讀運行詳情

點擊任意運行可以打開詳情。你會看到:

- **頭部**——觸發器、開始和結束時間、總耗時,以及狀態。
- **模塊列表**——每個運行過的模塊,按順序排列。每個都顯示它接收的值、它的輸出,以及它走了哪條路徑。
- **錯誤**——如果某個模塊失敗,會顯示錯誤信息和(可用時)更多細節。

顯示的值就是模塊實際看到的值——所有變量都已經被填充。這是最有用的調試視圖:如果一條 Slack 消息顯示的是字面文本 `{{Incident.title}}` 而不是實際標題,你就知道變量沒解析。

## 常見調試

### "我的工作流沒運行。"

1. 確認設置中工作流是 **已啓用** 的。新工作流默認是停用的。
2. 對於 OneUptime 事件觸發器:確認事件確實發生了。打開記錄查看它的歷史。
3. 對於 webhook 觸發器:確認對方系統正在向正確的 URL 發送。大多數工具會記錄它們發送 webhook 的情況——去那裏檢查。
4. 對於定時觸發器:確認 cron 表達式與你預期的時間匹配。

如果觸發器觸發了但沒有運行出現,檢查 **項目設置 → 計費** 下的運行配額。

### "後續的模塊沒有運行。"

某個模塊沒有運行通常是連線問題。打開畫布並檢查:

- 前一個模塊的輸出連接到這個模塊的輸入了嗎?
- 前一個模塊走的輸出是否和你預期的不同(例如走了 **錯誤** 而不是 **成功**,或者走了 **否** 而不是 **是**)?運行詳情會顯示走了哪條路徑。

### "變量傳過來是空的。"

打開運行,查看失敗模塊的值。

- 如果你看到字面文本 `{{BlockName.field}}`,說明引用沒解析——可能是模塊名或字段名拼寫錯誤。
- 如果你看到空字符串,說明前一個模塊運行了,但沒有產生那個字段。

### "手動運行可以工作,但被觸發器觸發時不行。"

使用 **手動運行**,帶上和真實觸發器發送的內容相似的 JSON 負載。然後把手動運行和真實運行的值並排比較。差異通常只是一個字段名或類型。

## 重新運行工作流

沒有"重試這次運行"按鈕。我們不會自動重跑過去的執行,因爲副作用(Slack 消息、API 調用、工單)可能不安全。要重做這些工作,修復工作流然後讓下一次真實的觸發器觸發它。

對於手動工作流,只需用相同的負載再次點擊 **手動運行**。

## 運行保留多久?

運行會在項目中無限期保留。如果某個工作流運行非常頻繁、把歷史搞亂了(比如每分鐘觸發一次的調試工作流),停用或刪除它來停止增加噪音。
