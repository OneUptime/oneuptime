# Runbook 規則

Runbook 規則會在 **事件**、**警報** 或 **計劃維護事件** 創建時自動附加 Runbook。規則從每種實體的設置菜單管理：

- 事件 → 設置 → **Runbook 規則**
- 警報 → 設置 → **Runbook 規則**
- 計劃維護 → 設置 → **Runbook 規則**

三個頁面編輯的是同一底層規則模型——只是按實體類型過濾顯示而已。

## 規則的結構

| 字段 | 用途 |
| --- | --- |
| **名稱** | 簡短的人類可讀標籤。會顯示在審計日誌中。 |
| **描述** | 給同事的可選上下文。 |
| **啓用** | 不刪除而暫停規則的開關。 |
| **標題模式** | 對實體標題做大小寫不敏感的正則匹配。空 = 匹配任意標題。 |
| **描述模式** | 對實體描述做大小寫不敏感的正則匹配。空 = 匹配任意描述。 |
| **要啓動的 Runbook** | 規則觸發時啓動的一個或多個 Runbook。 |

## 匹配語義

當 **所有指定的條件都通過** 時規則匹配。空條件會被跳過，所以：

- 沒有設置任何模式的規則在該類型的每一個事件上都會運行（一個全局的"總是運行"規則）。
- 只設置了標題模式的規則在標題匹配該正則的事件上觸發。
- 同一個事件可能匹配多個規則——每個匹配都觸發，所有規則下 Runbook 的並集會被運行（每個 Runbook 都有自己的執行）。

## 示例：數據庫事件的 DB 故障切換

```
名稱：       Start DB failover for DB incidents
觸發：       事件
標題模式：    (?:^|\b)(db|database|postgres|mysql|mongo)
Runbooks：   [DB failover playbook, Notify DBA team]
```

只要創建的事件標題裏包含 "db"、"database"、"postgres" 等，就會創建兩個 Runbook 執行。

## 示例：總是運行的衛生規則

```
名稱：           Always-run pre-flight check
觸發：           事件
標題模式：       （空）
描述模式：       （空）
Runbooks：      [Capture pre-incident state]
```

在每個事件上都觸發——適合捕獲系統狀態快照、頁面指標等。

## 規則觸發時會發生什麼

1. 加載 Runbook。
2. 它的步驟被 **快照** 到一個新的 Runbook 執行上。
3. 執行被入隊到 Runbook 隊列 worker。
4. 執行被鏈接到來源實體——它會出現在事件、警報或計劃維護事件的頁面上，以及 Runbook 的執行列表裏。

你可以在 **Runbooks → 執行** 中查看所有由規則觸發的運行，並按狀態、Runbook 或日期過濾。

## 被禁用的 Runbook

如果規則引用的 Runbook `isEnabled = false`，規則仍然會匹配但該 Runbook 的執行會被跳過。重新啓用 Runbook 即可恢復。

## 測試規則

在生產上依賴規則之前，請用一個標題能匹配模式的測試事件（或警報）來確認預期的 Runbook 會被觸發。規則是在創建時刻評估的——之後改動事件標題不會重新觸發規則。
