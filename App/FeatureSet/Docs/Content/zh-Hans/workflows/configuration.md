# 配置与安全

本页涵盖在你让工作流对接真实流量之前值得了解的设置与安全限制。

## 启用或停用工作流

每个工作流在 **设置** 中都有一个 **已启用** 开关。关闭时,工作流不会运行——对 webhook 调用、定时时间、OneUptime 事件都视而不见。新工作流默认是停用的。

把这个开关当作你的"准备就绪"门:

1. 构建工作流。
2. 用真实的负载点击 **手动运行**。
3. 查看 **日志**——确保每个模块都走到了预期的位置。
4. 把 **已启用** 切到开启。

关闭工作流不会停止已经在进行中的运行;它只会阻止新的运行开始。

## 所有者与标签

- **所有者**——被列为所有者的用户和团队可以访问该工作流,并且可以选择在它失败时收到通知。在 **设置 → 所有者** 下设置。
- **标签**——用于给工作流分组的标记。工作流列表允许按标签筛选,让繁忙的项目更容易导航。当你按团队、集成或环境组织工作流时很有用。
- **标签规则**——在 **工作流 → 设置 → 标签规则** 下,根据名称或描述的模式,自动为新工作流应用标签。
- **所有者规则**——在 **工作流 → 设置 → 所有者规则** 下,自动为新工作流分配所有者。

## 密钥

如果某个全局变量包含敏感内容,把它标记为 **密钥**。值会被加密、保存后在 UI 中隐藏,并在运行日志中隐藏(显示为 `[REDACTED]`)。

把密钥变量用于:

- 外部服务的 API 密钥。
- 认证令牌。
- Webhook 签名密钥。
- 任何你不想让只读访问者看到的内容。

不要把密钥直接粘贴到模块里——像 `Authorization: Bearer eyJh...` 这样的值会在工作流和日志中可见。改用 `{{variable.MY_SECRET}}`。

## 一次运行可以持续多久

每次运行都有最长时长。如果运行没有及时完成,会被标记为 **超时**,进行中的模块会被取消。默认值已经很宽松——足以容纳正常的 HTTP 调用和成串的模块。

各模块在这个时长内还有自己的时间限制——例如,API 模块对挂起的出站请求会在整个运行超时之前就放弃。

## 调用其他工作流的限制

**执行工作流** 组件让一个工作流可以调用另一个。为了防止 A 调用 B、B 又调用 A 这种意外的循环,链路深度有一个上限。超过限制的运行会以明确的错误结束。

如果你确实有合理的长链需求(比如一个工作每次运行处理一项),通常更简单的做法是在单个工作流里用 **自定义代码** 循环。

## Webhook 安全

Webhook 触发器会给你一个专属的 URL。任何知道这个 URL 的人都可以触发它。为了防止意外或不受欢迎的调用方:

- 把 URL 当作密码对待。不要公开分享或提交到公共仓库。
- 对敏感工作流,要求调用方在头部里发送一个共享令牌(比如 `X-Webhook-Token`),在做任何重要事情之前用 **条件** 模块检查它。把预期的令牌保存为密钥变量。
- 对极敏感的工作流,优先使用 OneUptime 事件触发器加手动导入步骤,而不是公开 webhook。

## 出站网络访问

API 等 HTTP 模块的请求都是从 OneUptime 发出的。如果你自托管,确保你的安装能访问你正在调用的服务。如果你使用 OneUptime Cloud,我们的出站 IP 段列在 [IP 地址](/docs/configuration/ip-addresses) 中,你可以在另一端把它们加入白名单。

## 权限

工作流遵循你项目的基于角色的访问控制。相关权限:

- **创建 / 读取 / 编辑 / 删除 Workflow**——工作流本身的基本权限。
- **Run Workflow**——点击 **手动运行** 或通过 API 触发工作流所需。
- **Read Workflow Log**——查看运行所需。
- **读取 / 创建 / 编辑 / 删除 Workflow Variable**——对全局变量列表的控制权限。

大多数工程师应该对工作流有创建/编辑/读取权限,但对变量则没有。把变量的编辑权限留给管理项目密钥的人。

## 计划限制

OneUptime Cloud 对较小套餐每月的运行次数有上限。当前的限额会显示在 **项目设置 → 计费** 下。达到上限后,新的触发会被拒绝直到下一个计费周期。自托管的安装没有这个限制。

## 工作流不适合的场景

几种情况下你应该选择别的工具:

- **重计算或大数据集**——工作流是为轻量粘合而设计的,不是为大量计算。把繁重的工作放在你自己的基础设施里跑,让工作流去启动它。
- **跨数小时的长时进程**——单次运行的目标是快速完成。如果你需要"做 A、等两小时、做 B",使用外部调度器,在时间到了时向 OneUptime 发回 webhook。
- **需要人参与的逐步事件响应**——这是 [Runbook](/docs/runbooks/index) 的职责。工作流是用于无人值守的自动化。
