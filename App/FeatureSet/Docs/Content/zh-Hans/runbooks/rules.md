# Runbook 规则

Runbook 规则会在 **事件**、**告警** 或 **计划维护事件** 创建时自动附加 Runbook。规则从每种实体的设置菜单管理：

- 事件 → 设置 → **Runbook 规则**
- 告警 → 设置 → **Runbook 规则**
- 计划维护 → 设置 → **Runbook 规则**

三个页面编辑的是同一底层规则模型——只是按实体类型过滤显示而已。

## 规则的结构

| 字段 | 用途 |
| --- | --- |
| **名称** | 简短的人类可读标签。会显示在审计日志中。 |
| **描述** | 给同事的可选上下文。 |
| **启用** | 不删除而暂停规则的开关。 |
| **标题模式** | 对实体标题做大小写不敏感的正则匹配。空 = 匹配任意标题。 |
| **描述模式** | 对实体描述做大小写不敏感的正则匹配。空 = 匹配任意描述。 |
| **要启动的 Runbook** | 规则触发时启动的一个或多个 Runbook。 |

## 匹配语义

当 **所有指定的条件都通过** 时规则匹配。空条件会被跳过，所以：

- 没有设置任何模式的规则在该类型的每一个事件上都会运行（一个全局的"总是运行"规则）。
- 只设置了标题模式的规则在标题匹配该正则的事件上触发。
- 同一个事件可能匹配多个规则——每个匹配都触发，所有规则下 Runbook 的并集会被运行（每个 Runbook 都有自己的执行）。

## 示例：数据库事件的 DB 故障切换

```
名称：       Start DB failover for DB incidents
触发：       事件
标题模式：    (?:^|\b)(db|database|postgres|mysql|mongo)
Runbooks：   [DB failover playbook, Notify DBA team]
```

只要创建的事件标题里包含 "db"、"database"、"postgres" 等，就会创建两个 Runbook 执行。

## 示例：总是运行的卫生规则

```
名称：           Always-run pre-flight check
触发：           事件
标题模式：       （空）
描述模式：       （空）
Runbooks：      [Capture pre-incident state]
```

在每个事件上都触发——适合捕获系统状态快照、页面指标等。

## 规则触发时会发生什么

1. 加载 Runbook。
2. 它的步骤被 **快照** 到一个新的 Runbook 执行上。
3. 执行被入队到 Runbook 队列 worker。
4. 执行被链接到来源实体——它会出现在事件、告警或计划维护事件的页面上，以及 Runbook 的执行列表里。

你可以在 **Runbooks → 执行** 中查看所有由规则触发的运行，并按状态、Runbook 或日期过滤。

## 被禁用的 Runbook

如果规则引用的 Runbook `isEnabled = false`，规则仍然会匹配但该 Runbook 的执行会被跳过。重新启用 Runbook 即可恢复。

## 测试规则

在生产上依赖规则之前，请用一个标题能匹配模式的测试事件（或告警）来确认预期的 Runbook 会被触发。规则是在创建时刻评估的——之后改动事件标题不会重新触发规则。
