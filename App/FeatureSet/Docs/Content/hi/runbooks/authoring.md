# Runbook लिखना

**Runbooks → Create Runbook** में Runbook बनाएँ, फिर इसे खोलकर **Steps** टैब पर जाएँ।

## चरण की संरचना

प्रत्येक चरण में होते हैं:

| फ़ील्ड | उद्देश्य |
| --- | --- |
| **शीर्षक** | चेकलिस्ट UI में दिखने वाला छोटा लेबल। आवश्यक। |
| **विवरण** | रिस्पॉन्डर के लिए वैकल्पिक संदर्भ। Markdown-सुरक्षित टेक्स्ट। |
| **Continue on failure** | चालू होने पर असफल चरण run को नहीं रोकता — अगला चरण फिर भी चलता है। |
| **Require approval** | चालू होने पर Runbook इस चरण के बाद रुक जाता है और अगले चरण को चलाने से पहले किसी उपयोगकर्ता की अनुमति का इंतज़ार करता है। |
| **प्रकार-विशिष्ट कॉन्फ़िग** | स्क्रिप्ट, URL, एजेंट आदि — नीचे देखें। |

चरण **क्रम में** चलते हैं। Steps एडिटर में ऊपर/नीचे तीरों से क्रम बदलें।

## चरण प्रकार

### Manual

एक चेकबॉक्स जिसे रिस्पॉन्डर टिक करता है। Manual चरण पर पहुँचते ही Runbook execution रुक जाता है और तब तक `WaitingForManualStep` में रहता है जब तक कोई इसे पूर्ण के रूप में चिह्नित नहीं करता (या छोड़ देता)।

ऐसी चीज़ों के लिए जिनकी सत्यापना केवल इंसान कर सकता है: "लोड बैलेंसर डैशबोर्ड में पुष्ट किया कि ट्रैफ़िक सेकंडरी रीजन पर चला गया।"

### JavaScript

`isolated-vm` सैंडबॉक्स में चलने वाला JavaScript स्निपेट। sandbox आपकी अपनी इन्फ्रास्ट्रक्चर के [Runbook एजेंट](/docs/runbooks/agents) पर रहता है — OneUptime Worker पर नहीं।

JavaScript चरण पर दो चीज़ें कॉन्फ़िगर करें:

- **Runbook Agent** — ड्रॉपडाउन से वह एजेंट चुनें जिसे यह चरण चलाना है। केवल चुना हुआ एजेंट ही job को claim कर सकता है।
- **Script** — चलाने के लिए JavaScript।

```js
const start = Date.now();
// ... आपकी लॉजिक ...
return { durationMs: Date.now() - start };
```

लौटाया गया मान चरण execution पर कैप्चर होता है। `console.log` आउटपुट लॉग लाइनों के रूप में सहेजा जाता है। डिफ़ॉल्ट execution timeout: 30 सेकंड। डिफ़ॉल्ट claim timeout (Worker कितनी देर एजेंट के job लेने का इंतज़ार करता है): 2 मिनट।

### HTTP रिक्वेस्ट

एक आउटबाउंड HTTP कॉल। विधि (GET/POST/PUT/PATCH/DELETE/HEAD), URL, वैकल्पिक JSON हेडर और वैकल्पिक body कॉन्फ़िगर करें। रिस्पॉन्स स्थिति, हेडर और body कैप्चर होते हैं (कुल 50KB तक)।

उपयोगी: PagerDuty incident खोलना, Slack पर पोस्ट करना, अपनी एडमिन API बुलाना आदि। HTTP चरण सीधे OneUptime Worker पर चलते हैं; किसी एजेंट की आवश्यकता नहीं।

### Bash

एक bash स्क्रिप्ट (`bash -c <script>`) जो आपकी अपनी इन्फ्रास्ट्रक्चर के [Runbook एजेंट](/docs/runbooks/agents) पर चलती है। Bash कभी भी OneUptime Worker पर नहीं चलता।

Bash चरण पर दो चीज़ें कॉन्फ़िगर करें:

- **Runbook Agent** — ड्रॉपडाउन से वह एजेंट चुनें जिसे यह चरण चलाना है। केवल चुना हुआ एजेंट ही job को claim कर सकता है।
- **Script** — चलाने के लिए bash। आउटपुट (stdout + stderr) 50 KB तक कैप्चर होता है; टाइमआउट पर प्रक्रिया मार दी जाती है।

यदि Runbook इस चरण पर पहुँचने पर चुना हुआ एजेंट offline है, चरण **claim timeout** (डिफ़ॉल्ट 2 मिनट) तक प्रतीक्षा करता है और फिर `TimedOut` के साथ fail हो जाता है। Bash चरण पर निर्भर होने से पहले **Runbooks → Settings → Agents** के तहत एक एजेंट जोड़ें।

## सहेजना और संपादन

संरक्षित करने के लिए **Save Steps** दबाएँ। पुराने Runbook संस्करणों के चलते हुए execution प्रभावित नहीं होते — वे अपने स्नैपशॉट से चलते रहते हैं।

## कई चरण और त्रुटि प्रबंधन

डिफ़ॉल्ट रूप से असफल चरण run को रोक देता है और execution को `Failed` के रूप में चिह्नित करता है। यदि आप किसी चरण पर **Continue on failure** सेट करते हैं, तो विफलता दर्ज होती है लेकिन अगला चरण चलता है। "इन तीन चीज़ों को आज़माएँ, फिर सूचित करें" जैसे पैटर्न के लिए उपयोगी।

## विस्तृत उदाहरण

"primary DB तक नहीं पहुँचा जा सकता" के लिए सरल Runbook:

1. **JavaScript** — कॉन्फ़िग सेवा से वर्तमान primary होस्ट लाएँ और लॉग करें।
2. **Manual** — "सेकंडरी पर replication lag 5 सेकंड से कम — पुष्ट करें।"
3. **HTTP रिक्वेस्ट** — failover ऑर्केस्ट्रेटर की API पर POST।
4. **Manual** — "पुष्ट करें कि लेखन अब नए primary पर जा रहे हैं।"
5. **HTTP रिक्वेस्ट** — Slack पर "सब ठीक" संदेश POST।

रिस्पॉन्डर ऑटोमेटेड चरण को चलते देखता है, एक Manual चरण टिक करता है, अगला ऑटोमेटेड चरण देखता है, और इसी तरह आगे। प्रत्येक चरण का आउटपुट पोस्टमोर्टम के लिए सुरक्षित रहता है।
