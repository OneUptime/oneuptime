# Runbook लिखना

**Runbooks → Create Runbook** में Runbook बनाएँ, फिर इसे खोलकर **Steps** टैब पर जाएँ।

## चरण की संरचना

प्रत्येक चरण में होते हैं:

| फ़ील्ड | उद्देश्य |
| --- | --- |
| **शीर्षक** | चेकलिस्ट UI में दिखने वाला छोटा लेबल। आवश्यक। |
| **विवरण** | रिस्पॉन्डर के लिए वैकल्पिक संदर्भ। Markdown टेक्स्ट। |
| **त्रुटि पर जारी रखें** | चालू होने पर असफल चरण run को नहीं रोकता — अगला चरण फिर भी चलता है। |
| **प्रकार-विशिष्ट कॉन्फ़िग** | स्क्रिप्ट, URL आदि — नीचे देखें। |

चरण **क्रम में** चलते हैं। Steps एडिटर में ऊपर/नीचे तीरों से क्रम बदलें।

## चरण प्रकार

### Manual

एक चेकबॉक्स जिसे रिस्पॉन्डर टिक करता है। Manual चरण पर पहुँचते ही execution रुक जाता है और तब तक `WaitingForManualStep` में रहता है जब तक कोई इसे पूर्ण के रूप में चिह्नित नहीं करता (या छोड़ देता)।

ऐसे चीज़ों के लिए जिनकी सत्यापना केवल इंसान कर सकता है: "लोड बैलेंसर डैशबोर्ड में पुष्ट किया कि ट्रैफ़िक सेकंडरी रीजन पर चला गया — पुष्ट।"

### JavaScript

`isolated-vm` सैंडबॉक्स में चलने वाला JavaScript स्निपेट (कोई फ़ाइलसिस्टम नहीं, कोई नेटवर्क नहीं जब तक आप API न लाएँ) — पर sandbox आपकी अपनी इन्फ्रास्ट्रक्चर के [Runbook एजेंट](/docs/runbooks/agents) पर रहता है, OneUptime Worker पर नहीं। step पर एक **Agent Tag** सेट करें जो उन एजेंट(ओं) की ओर इशारा करे जिन्हें इसे चलाना है। अगर उस tag वाला कोई एजेंट online नहीं है, तो step **claim timeout** (डिफ़ॉल्ट 2 मिनट) तक प्रतीक्षा करता है और फिर fail हो जाता है।

```js
const start = Date.now();
// ... आपकी लॉजिक ...
return { durationMs: Date.now() - start };
```

लौटाया गया मान चरण execution पर कैप्चर होता है। `console.log` आउटपुट लॉग लाइनों के रूप में सहेजा जाता है। डिफ़ॉल्ट टाइमआउट: 30 सेकंड।

### HTTP रिक्वेस्ट

एक आउटबाउंड HTTP कॉल। विधि (GET/POST/PUT/PATCH/DELETE/HEAD), URL, वैकल्पिक JSON हेडर और वैकल्पिक body कॉन्फ़िगर करें। रिस्पॉन्स स्थिति, हेडर और body कैप्चर होते हैं (कुल 50KB तक)।

उपयोगी: PagerDuty incident खोलना, Slack पर पोस्ट करना, अपनी एडमिन API बुलाना आदि।

### Bash

एक bash स्क्रिप्ट जो [Runbook एजेंट](/docs/runbooks/agents) पर चलती है — एक छोटी प्रक्रिया जिसे आप अपनी अपनी इन्फ्रास्ट्रक्चर के एक होस्ट पर इंस्टॉल करते हैं। Bash चरण OneUptime Worker पर कभी नहीं चलते।

Bash चरण पर दो चीज़ें कॉन्फ़िगर करें:

- **Agent Tag** — यह tag पहचानता है कि कौन सा/से एजेंट इस चरण को चलाएँगे। प्रोजेक्ट में उस tag को धारण करने वाला कोई भी स्वस्थ एजेंट job को claim करके चलाएगा।
- **Script** — चलाने के लिए bash। आउटपुट (stdout + stderr) 50 KB तक कैप्चर होता है; टाइमआउट पर प्रक्रिया मार दी जाती है।

यदि Runbook इस चरण पर पहुँचने पर चुने हुए tag का कोई एजेंट online नहीं है, चरण **claim timeout** (डिफ़ॉल्ट 2 मिनट) तक प्रतीक्षा करता है और फिर fail हो जाता है। Bash चरण पर निर्भर होने से पहले **Runbooks → Agents** के तहत एक एजेंट जोड़ें।

## सहेजना और संपादन

संरक्षित करने के लिए **Save Steps** दबाएँ। पुराने Runbook संस्करणों के चलते हुए execution प्रभावित नहीं होते — वे अपने स्नैपशॉट से चलते रहते हैं।

## कई चरण और त्रुटि प्रबंधन

डिफ़ॉल्ट रूप से असफल चरण execution को रोक देता है और execution को `Failed` के रूप में चिह्नित करता है। यदि आप किसी चरण पर **त्रुटि पर जारी रखें** सेट करते हैं, तो विफलता दर्ज होती है लेकिन अगला चरण चलता है। "इन तीन चीज़ों को आज़माएँ, फिर सूचित करें" जैसे पैटर्न के लिए उपयोगी।

## विस्तृत उदाहरण

"primary DB तक नहीं पहुँचा जा सकता" के लिए सरल Runbook:

1. **JavaScript** — कॉन्फ़िग सेवा से वर्तमान primary होस्ट लाएँ और लॉग करें।
2. **Manual** — "सेकंडरी पर replication lag 5 सेकंड से कम — पुष्ट।"
3. **HTTP रिक्वेस्ट** — failover ऑर्केस्ट्रेटर की API पर POST।
4. **Manual** — "लेखन नए primary पर जा रहे हैं — पुष्ट।"
5. **HTTP रिक्वेस्ट** — Slack पर "सब ठीक" संदेश POST।

रिस्पॉन्डर ऑटोमेटेड चरण को चलते देखता है, एक Manual चरण टिक करता है, अगला ऑटोमेटेड चरण देखता है, और इसी तरह आगे। प्रत्येक चरण का आउटपुट पोस्टमोर्टम के लिए सुरक्षित रहता है।
