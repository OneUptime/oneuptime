# Runbook का अवलोकन

Runbook दोबारा इस्तेमाल होने वाली प्रतिक्रिया प्रक्रियाएँ हैं — मैनुअल या ऑटोमेटेड चरणों की क्रमबद्ध सूची — जिन्हें आप घटनाओं, अलर्ट या निर्धारित रखरखाव से जोड़ते हैं। ये "अब हम क्या करें?" जैसी तदर्थ Slack थ्रेड्स को ऐसी चीज़ में बदल देती हैं जिसे एक टीममेट रात के 3 बजे बिना पूर्व जानकारी के उठाकर चला सके।

## एक नज़र में

- OneUptime डैशबोर्ड में **Analytics & Automation → Runbooks** के अंतर्गत **टॉप-लेवल फीचर**।
- **चार चरण प्रकार**: मैनुअल चेकलिस्ट, JavaScript (सैंडबॉक्स), HTTP रिक्वेस्ट, Bash (आपकी अपनी इन्फ्रास्ट्रक्चर में [Runbook एजेंट](/docs/runbooks/agents) पर चलता है)।
- **तीन ट्रिगर रास्ते**: घटनाओं/अलर्ट/निर्धारित रखरखाव से मिलने वाले नियम, या किसी भी इवेंट पर मैनुअल "Run Runbook" बटन।
- **स्नैपशॉट सिमेंटिक्स**: जब Runbook शुरू होता है तो इसके चरण execution में कॉपी हो जाते हैं। बाद में टेम्पलेट संपादित करना कभी भी चल रही execution को नहीं बदलता।
- **पूर्ण ऑडिट ट्रेल**: हर चरण की स्थिति, आउटपुट, त्रुटि संदेश और अवधि हमेशा execution पर संग्रहीत रहती है।

## Runbook क्यों इस्तेमाल करें?

घटना प्रतिक्रिया अक्सर एक मिनट की रुकावट और कई घंटे के आउटेज के बीच का अंतर होती है। Runbook आपकी मदद करते हैं:

- **अंतर्निहित ज्ञान को संहिताबद्ध करें** — "कतार जाम होने पर क्या करें" का जवाब उस जगह रहता है जहाँ टीम उसे ढूँढ सके।
- **औसत पुनर्प्राप्ति समय (MTTR) घटाएँ** — ऑटोमेटेड चरण सेकंडों में चलते हैं; मैनुअल चरण निर्णय पक्षाघात हटाते हैं।
- **प्रतिक्रिया कार्रवाइयों का ऑडिट** — हर चलाया हुआ चरण, हर आउटपुट, हर क्लिक execution में दर्ज होता है।
- **जूनियर इंजीनियरों को सक्षम बनाएँ** — वे आत्मविश्वास से Runbook चला सकते हैं, बजाय इसके कि रात के 3 बजे सीनियर को कॉल करें।
- **डेटा से, याद से नहीं, पोस्टमोर्टम लिखें** — कैप्चर की गई execution घटित घटना का जमा हुआ रिकॉर्ड है।

## मुख्य अवधारणाएँ

बाक़ी डॉक्युमेंटेशन में दोहराए जाने वाले कुछ शब्द। पहले इन्हें स्पष्ट कर लें:

| शब्द | अर्थ |
| --- | --- |
| **Runbook** | टेम्पलेट। एक नामांकित, पुन: प्रयोज्य प्रक्रिया जिसमें चरणों की क्रमबद्ध सूची और `isEnabled` फ़्लैग होता है। |
| **चरण** | Runbook में एक आइटम। इसका एक प्रकार (Manual / JavaScript / HTTP / Bash), शीर्षक, विवरण और प्रकार-विशिष्ट कॉन्फ़िग होता है। |
| **Runbook नियम** | एक पैटर्न जो घटना, अलर्ट या निर्धारित रखरखाव के शीर्षक/विवरण के regex से मेल खाने पर एक या अधिक Runbook को स्वतः जोड़ता है। |
| **Execution** | Runbook का एक रन। जब नियम चालू होता है, कोई इवेंट पर "Run Runbook" क्लिक करता है, या Runbook पर "Run Now" क्लिक करता है — तब बनता है। चरणों के स्नैपशॉट और प्रति-चरण स्थिति/आउटपुट रखता है। |
| **स्नैपशॉट** | प्रत्येक execution पर रहने वाली Runbook के चरणों की जमी हुई कॉपी। इससे आप टेम्पलेट बाद में बदल सकते हैं बिना इतिहास को बदले। |

## Runbook का जीवनचक्र

1. **लिखें** — एक Runbook बनाएँ, Manual, JavaScript, HTTP और Bash चरणों का मिश्रण डालें, सहेजें।
2. **(वैकल्पिक) नियम जोड़ें** — Incidents, Alerts या Scheduled Maintenance की सेटिंग्स में, OneUptime को बताएँ कि जब किसी इवेंट का शीर्षक/विवरण regex से मेल खाए तो यह Runbook शुरू करे।
3. **ट्रिगर करें** — या तो मिलते-जुलते इवेंट के बनते ही नियम स्वतः चलता है, या रिस्पॉन्डर इवेंट पर **Run Runbook** क्लिक करता है।
4. **चलाएँ** — चरणों के स्नैपशॉट के साथ नया execution बनता है। ऑटोमेटेड चरण Runbook वर्कर पर चलते हैं; execution प्रत्येक Manual चरण पर तब तक रुकता है जब तक कोई उसे टिक नहीं करता।
5. **ऑडिट करें** — execution इवेंट के **Runbooks** टैब और Runbook की execution सूची में हमेशा रहता है। प्रत्येक चरण का आउटपुट, त्रुटियाँ और समय पोस्टमोर्टम के लिए सुरक्षित रहते हैं।

## कौन-सा चरण प्रकार कब

एक तेज़ निर्णय गाइड। विस्तार से [Runbook लिखना](/docs/runbooks/authoring) में है।

| चरण प्रकार | इसे कब चुनें… | उदाहरण |
| --- | --- | --- |
| **Manual** | जब किसी इंसान को सत्यापन, निर्णय या कोई ऐसी कार्रवाई करनी हो जिसे OneUptime देख नहीं सकता। | "लोड बैलेंसर डैशबोर्ड पर सेकंडरी रीजन का ट्रैफ़िक पुष्ट किया।" |
| **JavaScript** | छोटी, स्वतंत्र गणना चाहिए — कॉन्फ़िग सेवा से क्वेरी, payload परिवर्तन, अगले चरण से पहले लॉजिक। | वर्तमान रेप्लिका लैग की गणना और आगे बढ़ने का निर्णय। |
| **HTTP रिक्वेस्ट** | आप मौजूदा API को बुला रहे हैं — खुद का एडमिन एंडपॉइंट, क्लाउड प्रोवाइडर, PagerDuty, Slack। | अपने failover ऑर्केस्ट्रेटर को `POST`। |
| **Bash** | आपको अपनी इन्फ्रास्ट्रक्चर पर shell कमांड चलाने हैं — एक सेवा रिस्टार्ट करना, `kubectl` चलाना, एक डिप्लॉय स्क्रिप्ट कॉल करना। आपके वातावरण में इंस्टॉल किए गए [Runbook एजेंट](/docs/runbooks/agents) की आवश्यकता है। | सेवा रिस्टार्ट, `kubectl rollout restart`, recovery स्क्रिप्ट चलाना। |

एक ही Runbook में चारों को मिला सकते हैं — Runbook की ताक़त मानवीय सत्यापन और ऑटोमेशन को आपस में बुनना है।

## डैशबोर्ड में Runbook कहाँ रहते हैं

| पेज | वहाँ आप क्या करते हैं |
| --- | --- |
| **Analytics & Automation → Runbooks** | Runbook टेम्पलेट देखना, बनाना और संपादित करना। |
| **Runbook का Steps टैब** | चरण सूची लिखना और पुनः क्रम लगाना। |
| **Runbook का Executions टैब** | स्थिति फ़िल्टर्स के साथ इस Runbook की हर रन देखना। |
| **Runbook का Run Now बटन** | किसी इवेंट से न जुड़ा एक तदर्थ execution शुरू करना। |
| **Incidents / Alerts / Scheduled Maintenance → Settings → Runbook Rules** | प्रत्येक एंटिटी प्रकार के लिए ऑटो-ट्रिगर नियम बनाना। |
| **कोई incident / alert / maintenance इवेंट → Runbooks टैब** | इस इवेंट से जुड़े executions देखना और मैनुअल रन के लिए **Run Runbook** क्लिक करना। |

## आम उपयोग

टीमें इन पैटर्न के लिए Runbook का सहारा लेती हैं:

- **डेटाबेस failover** — JavaScript से वर्तमान स्थिति कैप्चर, ऑन-कॉल DBA से रेप्लिका स्वास्थ्य की पुष्टि (Manual), ऑर्केस्ट्रेटर API कॉल (HTTP), "DNS अपडेट" टिक करना (Manual), Slack पर "सब ठीक" पोस्ट (HTTP)।
- **कैश फ्लश** — एक HTTP चरण और एक Manual "डैशबोर्ड पर कैश हिट रेट की रिकवरी पुष्ट करें"।
- **ग्राहक-प्रभावित घटना** — Manual: "स्टेटस पेज पर अपडेट पोस्ट करें।" HTTP: "#customer-incidents में CS टीम को सूचित करें।" JavaScript: "इंटरनल API से प्रभावित अकाउंट की सूची लाएँ।"
- **निर्धारित रखरखाव से पहले की जाँच** — JavaScript: वर्तमान मेट्रिक्स का स्नैपशॉट। Manual: "हितधारकों के साथ बदलाव विंडो की पुष्टि।" HTTP: लोड बैलेंसर पर रखरखाव मोड चालू।
- **हमेशा चलने वाला स्वच्छता नियम** — खाली शीर्षक पैटर्न वाला नियम जो प्रत्येक घटना पर सिस्टम स्थिति कैप्चर करता है — पोस्टमोर्टम के लिए बहुत कीमती।

## विस्तृत उदाहरण

मान लीजिए आप चाहते हैं कि शीर्षक में "db-primary" वाली हर incident अपने आप पाँच-चरण DB failover Runbook शुरू करे।

**1. Runbook बनाएँ।** **Runbooks → Create Runbook** में इसे "DB primary failover" नाम दें और ये चरण जोड़ें:

| # | प्रकार | शीर्षक |
| --- | --- | --- |
| 1 | JavaScript | failover से पहले रेप्लिका लैग कैप्चर करें |
| 2 | Manual | DBA डैशबोर्ड पर रेप्लिका स्वास्थ्य पुष्ट करें |
| 3 | HTTP | failover ऑर्केस्ट्रेटर को `POST` |
| 4 | Manual | पुष्ट करें कि लेखन नए primary पर जा रहे हैं |
| 5 | HTTP | Slack `#db-incidents` में "सब ठीक" पोस्ट |

**2. नियम जोड़ें।** **Incidents → Settings → Runbook Rules** में बनाएँ:

```
शीर्षक पैटर्न:  ^db-primary
Runbook:        [DB primary failover]
```

**3. ट्रिगर।** एक मॉनिटर अलर्ट से incident `INC-4821 · db-primary connection timeout` खुलती है। नियम मेल खाता है, execution बनता है, और:

- चरण 1 (JavaScript) तुरंत वर्कर पर चलता है — `return { lagMs: 412 }` मान कैप्चर होता है।
- चरण 2 (Manual) execution रोक देता है। ऑन-कॉल इंजीनियर incident पेज पर "आपका इंतज़ार" बैज देखता है, डैशबोर्ड चेक करता है और चरण को टिक करता है।
- चरण 3 (HTTP) चरण 2 के टिक होते ही चलता है — `POST` का रिस्पॉन्स बॉडी कैप्चर होता है।
- चरण 4 (Manual) फिर रोकता है।
- चरण 5 (HTTP) चलता है और execution समाप्त।

**4. ऑडिट।** execution incident के **Runbooks** टैब पर रहता है। हर चरण का आउटपुट एक क्लिक दूर है। अगले हफ़्ते पोस्टमोर्टम लिखते समय आपको "उस स्क्रिप्ट ने क्या लौटाया?" पूछना नहीं पड़ता — वह वहीं है।

## Runbook OneUptime के बाकी हिस्सों से कैसे जुड़ते हैं

- **मॉनिटर** incident और alert खोलते हैं; **Runbook नियम** उन इवेंट्स को Runbook execution में बदलते हैं। मिलकर वे एक बंद लूप बनाते हैं: पहचान → ट्रिगर → प्रतिक्रिया → रिकॉर्ड।
- **Workspace कनेक्शन** (Slack, Microsoft Teams) HTTP चरणों का स्वाभाविक लक्ष्य हैं — स्थिति अपडेट पोस्ट करना, चैनलों को सूचित करना।
- **स्टेटस पेज** अक्सर ग्राहक-प्रभावित Runbook में Manual चरण के रूप में अपडेट होते हैं।
- **ऑन-कॉल शेड्यूल** तय करते हैं कि किसे पेज किया जाएगा; Runbook तय करते हैं कि जागने के बाद वह व्यक्ति क्या करता है।

## आगे क्या पढ़ें

- [Runbook लिखना](/docs/runbooks/authoring) — Runbook बनाना, चार चरण प्रकार और प्रत्येक का काम।
- [Runbook नियम](/docs/runbooks/rules) — incident, alert और निर्धारित रखरखाव में Runbook अपने आप जोड़ना।
- [Runbook चलाना](/docs/runbooks/running) — मैनुअल ट्रिगर, execution दृश्य, और मैनुअल/ऑटोमेटेड चरण कैसे एक-दूसरे से मिलते हैं।
- [Runbook एजेंट](/docs/runbooks/agents) — अपनी इन्फ्रास्ट्रक्चर में Bash चरण चलाने वाले एजेंट स्थापित करना।
- [कॉन्फ़िगरेशन और सुरक्षा](/docs/runbooks/configuration) — आउटपुट सीमाएँ, अनुमतियाँ, हार्डनिंग नोट्स।
