# Runs और Logs

हर बार जब कोई वर्कफ़्लो चलता है, तो OneUptime जो हुआ उसका रिकॉर्ड सहेजता है — कब चला, क्या काम किया, और हर ब्लॉक ने क्या किया। उस रिकॉर्ड को **run** कहा जाता है। Runs वह तरीका हैं जिससे आप पुष्टि करते हैं कि एक वर्कफ़्लो ने काम किया, उसे डीबग करें जो नहीं चला, और पिछली गतिविधि पर पीछे मुड़कर देखें।

## इन्हें कहाँ खोजें

| पेज                         | आप क्या देखते हैं                                                                       |
| --------------------------- | --------------------------------------------------------------------------------------- |
| **Workflows → Runs & Logs** | प्रोजेक्ट के हर वर्कफ़्लो से हर run। वर्कफ़्लो, स्थिति, और समय के आधार पर फ़िल्टर करें। |
| **Workflow → Logs tab**     | केवल इस एक वर्कफ़्लो के runs।                                                           |
| **एक एकल run**              | एक execution, हर ब्लॉक के आउटपुट के साथ।                                                |

## Run स्थितियाँ

| स्थिति        | इसका अर्थ                                                                                                                                           |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Scheduled** | ट्रिगर चालू हुआ और run शुरू होने वाला है। आमतौर पर एक सेकंड के अंश ही लगते हैं।                                                                     |
| **Running**   | वर्कफ़्लो प्रगति पर है। लंबे समय तक चलने वाले ब्लॉक एक run को इस स्थिति में रखते हैं।                                                               |
| **Success**   | जो भी ब्लॉक चले वे बिना त्रुटि के समाप्त हुए। (जान-बूझकर **error** शाखा लेना अभी भी सफलता के रूप में गिना जाता है — वर्कफ़्लो स्वयं विफल नहीं हुआ।) |
| **Error**     | एक ब्लॉक विफल हुआ और इसे संभालने के लिए कोई **error** पथ कनेक्ट नहीं था। Run वहीं रुक गया।                                                          |
| **Timeout**   | Run अनुमत से अधिक समय तक चला। [कॉन्फ़िगरेशन और सुरक्षा](/docs/workflows/configuration) देखें।                                                       |

## एक run पढ़ना

विवरण खोलने के लिए किसी run पर क्लिक करें। आप देखेंगे:

- **Header** — ट्रिगर, शुरू और समाप्ति समय, कुल अवधि, और स्थिति।
- **Block list** — हर ब्लॉक जो क्रम में चला। हर एक उसे दिए गए मान, इसका आउटपुट, और इसने कौन सा पथ लिया दिखाता है।
- **Errors** — यदि कोई ब्लॉक विफल हुआ, तो त्रुटि संदेश और (जब उपलब्ध हो) अधिक विवरण।

दिखाए गए मान वही हैं जो ब्लॉक ने देखे — सभी वेरिएबल भरे जाने के बाद। यह सबसे उपयोगी डीबगिंग दृश्य है: यदि एक Slack संदेश वास्तविक शीर्षक के बजाय शाब्दिक टेक्स्ट `{{Incident.title}}` दिखाता है, तो आपको पता चलता है कि वेरिएबल हल नहीं हुआ।

## सामान्य डीबगिंग

### "मेरा वर्कफ़्लो नहीं चला।"

1. सुनिश्चित करें कि वर्कफ़्लो Settings में **enabled** है। नए वर्कफ़्लो अक्षम अवस्था से शुरू होते हैं।
2. OneUptime इवेंट ट्रिगर के लिए: पुष्टि करें कि इवेंट वास्तव में हुआ था। रिकॉर्ड खोलें और इसका इतिहास जाँचें।
3. Webhook ट्रिगर के लिए: पुष्टि करें कि अन्य सिस्टम सही URL पर भेज रहा है। ज़्यादातर टूल लॉग करते हैं जब वे एक webhook भेजते हैं — वहाँ जाँचें।
4. Schedule ट्रिगर के लिए: पुष्टि करें कि cron एक्सप्रेशन आपकी अपेक्षित समय से मेल खाती है।

यदि ट्रिगर चालू हुआ लेकिन कोई run नहीं दिखता, तो **Project Settings → Billing** के अंतर्गत अपना run कोटा जाँचें।

### "बाद का एक ब्लॉक कभी नहीं चला।"

एक ब्लॉक जो नहीं चलता आमतौर पर एक वायरिंग समस्या होती है। कैनवास खोलें और जाँचें:

- क्या पहले के ब्लॉक का आउटपुट इस ब्लॉक के इनपुट से कनेक्ट है?
- क्या पहले के ब्लॉक ने आपकी अपेक्षा से अलग आउटपुट लिया (उदाहरण के लिए, **success** के बजाय **error**, या **Yes** के बजाय **No**)? Run विवरण दिखाता है कि कौन सा पथ लिया गया था।

### "एक वेरिएबल खाली आया।"

Run खोलें और विफल हो रहे ब्लॉक के मानों को देखें।

- यदि आप शाब्दिक `{{BlockName.field}}` टेक्स्ट देखते हैं, तो संदर्भ हल नहीं हुआ — शायद ब्लॉक नाम या फ़ील्ड नाम में टाइपो।
- यदि आप एक खाली स्ट्रिंग देखते हैं, तो पहले का ब्लॉक चला लेकिन वह फ़ील्ड उत्पन्न नहीं किया।

### "जब मैं इसे मैन्युअली चलाता हूँ तो काम करता है लेकिन ट्रिगर से नहीं।"

एक JSON पेलोड के साथ **Run Manually** का उपयोग करें जो असली ट्रिगर भेजने जैसा दिखता है। फिर मैनुअल run में मानों की तुलना असली run से साथ-साथ करें। अंतर आमतौर पर एक एकल फ़ील्ड नाम या प्रकार होता है।

## एक वर्कफ़्लो को फिर से चलाना

"इस run को पुनः प्रयास करें" बटन नहीं है। हम पुराने execution स्वचालित रूप से दोबारा नहीं चलाते क्योंकि साइड इफ़ेक्ट्स (Slack संदेश, API कॉल, टिकट) दोहराने के लिए सुरक्षित नहीं हो सकते। काम को फिर से करने के लिए, वर्कफ़्लो ठीक करें और अगले असली ट्रिगर को इसे चालू करने दें।

मैनुअल वर्कफ़्लो के लिए, बस उसी पेलोड के साथ **Run Manually** पर क्लिक करें।

## Runs कितने समय तक रखे जाते हैं?

Runs प्रोजेक्ट के लिए अनिश्चित काल तक रखे जाते हैं। यदि एक वर्कफ़्लो बहुत बार चलता है और आपके इतिहास को अव्यवस्थित करता है (जैसे एक डीबग वर्कफ़्लो जो हर मिनट चालू होता है), तो शोर में जोड़ना बंद करने के लिए इसे अक्षम या डिलीट करें।
