# वर्कफ़्लो अवलोकन

वर्कफ़्लो OneUptime का विज़ुअल ऑटोमेशन बिल्डर है। एक ट्रिगर को कैनवास पर खींचें, उसे क्रियाओं की एक श्रृंखला से जोड़ें — HTTP कॉल, Slack संदेश, JavaScript स्निपेट, कंडीशनल ब्रांच, डेटाबेस लुकअप — और आपके पास ऐसा ऑटोमेशन है जो OneUptime (या बाहरी दुनिया) में कोई इवेंट होते ही चलता है।

अगर Runbook घटना के दौरान इंसानों के लिए चेकलिस्ट हैं, तो वर्कफ़्लो आपके प्रोजेक्ट के लिए बैकग्राउंड जॉब हैं — वे बिना देखरेख चलते हैं, चीज़ों पर प्रतिक्रिया देते हैं, और OneUptime को आपके बाकी स्टैक से जोड़ते हैं।

## एक नज़र में

- OneUptime डैशबोर्ड में **Workflows** के अंतर्गत **टॉप-लेवल फीचर**।
- **तीन ट्रिगर शैलियाँ**: Manual, Schedule (cron), Webhook — साथ ही एक **मॉडल-इवेंट ट्रिगर** जो तब चालू होता है जब कोई OneUptime एंटिटी (incident, alert, monitor, status page, आदि) बनती, अपडेट होती या डिलीट होती है।
- **विज़ुअल कैनवास**: कंपोनेंट पैलेट से नोड्स खींचें, आउटपुट पोर्ट्स को इनपुट पोर्ट्स से जोड़ें।
- **मिश्रित ऑटोमेशन**: HTTP रिक्वेस्ट, Slack / Discord / Microsoft Teams / Telegram संदेश, कस्टम JavaScript, JSON पार्सिंग, कंडीशनल, ईमेल, सब-वर्कफ़्लो कॉल, और OneUptime मॉडल्स पर CRUD ऑपरेशन।
- **Global Variables**: प्रोजेक्ट-स्तरीय सीक्रेट और कॉन्फ़िग जिन्हें आप किसी भी वर्कफ़्लो से बिना कॉपी-पेस्ट किए संदर्भित कर सकते हैं।
- **Runs & Logs**: हर execution स्थिति, समय और प्रति-चरण आउटपुट के साथ रिकॉर्ड होता है।

## वर्कफ़्लो क्यों इस्तेमाल करें?

ज़्यादातर टीमें वर्कफ़्लो का सहारा तब लेती हैं जब वे चाहती हैं:

- **OneUptime को किसी दूसरे सिस्टम से जोड़ना** — किसी incident को PagerDuty पर पोस्ट करना, alert को Jira में मिरर करना, अपने स्टैक में किसी webhook को पिंग करना।
- **OneUptime इवेंट्स पर प्रतिक्रिया देना** — जब `Sev 1` incident खुले, ऑन-कॉल मैनेजर को पेज करें *और* एक Linear टिकट बनाएँ *और* एक फीचर फ़्लैग लॉक करें।
- **आवर्ती जॉब शेड्यूल करना** — हर पाँच मिनट में एक आंतरिक API से क्वेरी करें और परिणाम को बाहरी सिस्टम में लिखें।
- **OneUptime के बाहर से डेटा प्राप्त करना** — किसी CI सिस्टम से आया webhook OneUptime अपडेट्स की श्रृंखला शुरू करता है।
- **छोटे ग्लू लॉजिक के टुकड़े दोबारा इस्तेमाल करना** — एक वर्कफ़्लो दूसरे को कॉल करता है, इसलिए सामान्य पैटर्न एक ही जगह रहते हैं।

## मुख्य अवधारणाएँ

| शब्द | अर्थ |
| --- | --- |
| **Workflow** | कैनवास। ट्रिगर और घटकों का एक नामित, पुन: प्रयोज्य ग्राफ़ जिसमें `isEnabled` फ़्लैग होता है। |
| **Trigger** | वह नोड जो वर्कफ़्लो रन शुरू करता है। Manual, Schedule, Webhook, या एक मॉडल इवेंट। हर वर्कफ़्लो में बिल्कुल एक ट्रिगर होता है। |
| **Component** | वह नोड जो काम करता है — एक HTTP कॉल, एक Slack संदेश, एक JavaScript स्निपेट, एक कंडीशनल, आदि। |
| **Port** | किसी नोड पर इनपुट या आउटपुट सॉकेट। घटकों के पास `success` और `error` जैसे आउटपुट पोर्ट्स होते हैं; आप एक पोर्ट को अगले नोड के इनपुट पोर्ट से जोड़ते हैं। |
| **Run / Log** | वर्कफ़्लो का एक execution। टाइमस्टैम्प, स्थिति (Running, Success, Failed, Timeout) और चले हर नोड का कैप्चर किया गया आउटपुट रखता है। |
| **Global Variable** | प्रोजेक्ट स्तर पर एक बार परिभाषित नामांकित मान (अक्सर एक सीक्रेट या API कुंजी) जिसे किसी भी वर्कफ़्लो से `{{variable.NAME}}` के रूप में संदर्भित किया जाता है। |
| **Local Variable** | एकल वर्कफ़्लो रन तक सीमित मान — आमतौर पर पहले के नोड का रिटर्न मान, `{{ComponentId.portName}}` के रूप में संदर्भित। |

## डैशबोर्ड में वर्कफ़्लो कहाँ रहते हैं

| पेज | वहाँ आप क्या करते हैं |
| --- | --- |
| **Workflows** | वर्कफ़्लो टेम्पलेट्स देखना, बनाना और खोजना। |
| **वर्कफ़्लो का Builder टैब** | ड्रैग-एंड-ड्रॉप कैनवास। नोड्स जोड़ें, पोर्ट्स कनेक्ट करें, आर्ग्युमेंट्स कॉन्फ़िगर करें। |
| **वर्कफ़्लो का Logs टैब** | स्थिति और समय सीमा के फ़िल्टर्स के साथ इस वर्कफ़्लो का हर रन। प्रति-नोड आउटपुट देखने के लिए किसी रन पर क्लिक करें। |
| **वर्कफ़्लो का Settings टैब** | नाम बदलें, सक्षम/अक्षम करें, विवरण बदलें, लेबल प्रबंधित करें, हटाएँ। |
| **Workflows → Global Variables** | प्रोजेक्ट-स्तरीय मान परिभाषित करें जिन्हें किसी भी वर्कफ़्लो से संदर्भित किया जाता है। सहेजने के बाद UI से छिपाने के लिए मान को सीक्रेट के रूप में चिह्नित करें। |
| **Workflows → Runs & Logs** | सभी वर्कफ़्लो के लिए प्रोजेक्ट-स्तरीय execution इतिहास। |

## वर्कफ़्लो का जीवनचक्र

1. **लिखें** — एक वर्कफ़्लो बनाएँ, कैनवास पर ट्रिगर डालें, आवश्यक घटकों को खींचें, उन्हें जोड़ें, और प्रत्येक को कॉन्फ़िगर करें।
2. **सक्षम करें** — वर्कफ़्लो अक्षम होकर आते हैं। एक बार जब आप आश्वस्त हों कि वायरिंग सही है, तो Settings में स्विच पलटें।
3. **ट्रिगर करें** — Manual: वैकल्पिक JSON payload के साथ **Run Manually** पर क्लिक करें। Schedule: cron चालू होता है। Webhook: कोई बाहरी सिस्टम वर्कफ़्लो URL पर `POST` करता है। Model event: कोई (या दूसरा वर्कफ़्लो) मॉनिटर, incident, alert आदि बनाता/अपडेट करता/हटाता है।
4. **चलाएँ** — Workflow Worker ग्राफ़ को क्रम में चलाता है। प्रत्येक घटक अपने आर्ग्युमेंट्स (शाब्दिक मान या इंटरपोलेट किए गए वेरिएबल) पढ़ता है, अपना काम करता है, अपना रिटर्न मान लिखता है, और एक आउटपुट पोर्ट चुनता है। अगला नोड चलता है।
5. **ऑडिट करें** — रन **Logs** में दिखाई देता है। स्थिति, कुल अवधि, प्रति-घटक आउटपुट और कोई भी त्रुटियाँ प्रोजेक्ट के जीवनकाल तक रखी जाती हैं।

## विस्तृत उदाहरण

लक्ष्य: जब शीर्षक में `Sev 1` के साथ कोई incident बनती है, तो एक Slack चैनल पर पोस्ट करें और अपने आंतरिक एडमिन टूल में एक टिकट खोलें।

**1. एक वर्कफ़्लो बनाएँ** जिसका नाम "Sev 1 fan-out" हो।

**2. एक ट्रिगर डालें।** पैलेट से **Incident → On Create** ट्रिगर चुनें। ट्रिगर नई incident को रिटर्न मान के रूप में उजागर करता है।

**3. एक Conditional घटक डालें।** ट्रिगर के आउटपुट पोर्ट को इसके इनपुट से जोड़ें। शर्त सेट करें: `{{Incident.title}}` *contains* `Sev 1`।

**4. Conditional के `yes` पोर्ट से, एक Slack घटक डालें।** चैनल: `#incident-room`। संदेश बॉडी: `Sev 1 declared: {{Incident.title}} — {{Incident.dashboardUrl}}`।

**5. उसी `yes` पोर्ट से (समानांतर में), एक API घटक डालें।** `https://admin.internal/incidents` पर `POST`। बॉडी: incident से बना एक छोटा JSON ऑब्जेक्ट।

**6. वर्कफ़्लो सक्षम करें।** दूसरे टैब में "Sev 1 — checkout 500s" शीर्षक वाली एक incident खोलें। कुछ सेकंड के भीतर Slack संदेश आ जाता है, और प्रत्येक नोड के कैप्चर किए गए आउटपुट के साथ **Logs** के अंतर्गत एक नया रन दिखाई देता है।

## वर्कफ़्लो OneUptime के बाकी हिस्सों से कैसे जुड़ते हैं

- **मॉनिटर** समस्याओं का पता लगाते हैं; **incidents/alerts** उन्हें रिकॉर्ड करते हैं; **वर्कफ़्लो** उन पर प्रतिक्रिया देते हैं — संदेश पोस्ट करते हैं, टिकट खोलते हैं, ऑटोमेशन शुरू करते हैं।
- **Runbook** इंसानों के लिए प्रतिक्रिया प्रक्रियाएँ हैं (वैकल्पिक स्क्रिप्ट चरणों के साथ)। वर्कफ़्लो बिना देखरेख वाला बैकग्राउंड ऑटोमेशन हैं। वे पूरक हैं — एक Runbook चरण किसी वर्कफ़्लो के webhook ट्रिगर पर `POST` कर सकता है।
- **Workspace कनेक्शन** (Slack, Microsoft Teams) वर्कफ़्लो सूचनाओं के लिए विशिष्ट गंतव्य हैं।
- **डैशबोर्ड** केवल-पढ़ने वाले दृश्य हैं; वर्कफ़्लो लिखने वाली ओर हैं — वे OneUptime की स्थिति को अपडेट करते हैं, बाहरी APIs को कॉल करते हैं, और डेटा को इधर-उधर ले जाते हैं।
