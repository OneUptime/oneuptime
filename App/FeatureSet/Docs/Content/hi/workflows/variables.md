# वेरिएबल

वर्कफ़्लो डेटा को स्थानांतरित करने के बारे में हैं — ट्रिगर से पहले ब्लॉक में, एक ब्लॉक से दूसरे में, और साझा मानों से जहाँ भी आपको चाहिए। वेरिएबल वह तरीका हैं जिससे डेटा चलता है।

दो प्रकार हैं, और वे एक ही सिंटैक्स साझा करते हैं।

## Global वेरिएबल

प्रोजेक्ट-व्यापी मान जिन्हें आप एक बार सहेजते हैं और कहीं भी फिर से इस्तेमाल करते हैं। API कुंजी, URL, चैनल नाम के बारे में सोचें — कुछ भी जिसे आप दस अलग-अलग वर्कफ़्लो में कॉपी नहीं करना चाहते।

उन्हें **Workflows → Global Variables** के अंतर्गत खोजें। प्रत्येक में:

- **Name** — आप इसे कैसे संदर्भित करेंगे। `UPPER_SNAKE_CASE` का उपयोग करें ताकि यह आपके ब्लॉक्स में अलग दिखे।
- **Value** — वास्तविक मान। बहु-पंक्ति मान भी काम करते हैं।
- **Is Secret** — चालू होने पर, सहेजने के बाद UI में मान छुपा होता है और run लॉग से छुपा होता है।

किसी भी वर्कफ़्लो में एक global वेरिएबल का उपयोग करें:

```
{{variable.NAME}}
```

उदाहरण के लिए, यदि आपने अपनी PagerDuty कुंजी `PAGERDUTY_KEY` के रूप में सहेजी है, तो कोई भी ब्लॉक इसे `{{variable.PAGERDUTY_KEY}}` के रूप में उपयोग कर सकता है — असली कुंजी कभी भी वर्कफ़्लो या इसके लॉग में नहीं दिखती।

## Local वेरिएबल (पहले के ब्लॉक्स से डेटा)

Local वेरिएबल इस execution में पहले से चले ब्लॉक्स के आउटपुट हैं। हर ट्रिगर और हर कंपोनेंट कुछ आउटपुट उत्पन्न करता है जिसे आप पढ़ सकते हैं।

पहले के ब्लॉक के आउटपुट को इस तरह संदर्भित करें:

```
{{BlockName.fieldName}}
```

`BlockName` कैनवास पर ट्रिगर या कंपोनेंट का नाम है (आप इसे कुछ छोटा और स्पष्ट में बदल सकते हैं)। `fieldName` जो भी वह ब्लॉक उत्पन्न करता है।

उदाहरण:

- `LookupUser` नामक एक **API** ब्लॉक चलने के बाद, आप स्थिति कोड को `{{LookupUser.response-status}}` के रूप में और बॉडी को `{{LookupUser.response-body}}` के रूप में पढ़ सकते हैं।
- `Incident` नामक एक **Incident → On Create** ट्रिगर के बाद, आप `{{Incident.title}}`, `{{Incident.description}}`, और incident पर कोई भी अन्य फ़ील्ड पढ़ सकते हैं।
- `Transform` नामक एक **Custom Code** ब्लॉक के बाद, रिटर्न किया गया मान `{{Transform.value}}` पर है।

Local वेरिएबल केवल वर्तमान run के दौरान मौजूद होते हैं। हर नया run नए सिरे से शुरू होता है।

## वेरिएबल कहाँ काम करते हैं

लगभग हर टेक्स्ट फ़ील्ड वेरिएबल स्वीकार करती है:

- API ब्लॉक पर URL।
- Slack, Teams, Discord, Telegram, Email पर संदेश टेक्स्ट।
- एक ईमेल का विषय और बॉडी।
- हेडर और बॉडी फ़ील्ड्स (स्ट्रिंग मानों के अंदर)।
- एक Conditions ब्लॉक के दोनों किनारे।

शुद्ध JSON फ़ील्ड्स स्ट्रिंग मानों के अंदर वेरिएबल स्वीकार करती हैं, लेकिन आप एक कुंजी के रूप में एक वेरिएबल का उपयोग नहीं कर सकते। यदि आपको गतिशील रूप से एक संरचना बनाने की आवश्यकता है, तो इसे बनाने के लिए **Custom Code** ब्लॉक का उपयोग करें, फिर इसके आउटपुट को अगले ब्लॉक में पास करें।

**Custom Code** ब्लॉक वेरिएबल को अलग तरीके से पढ़ता है — global वेरिएबल `args.variables` पर आते हैं, और आप तय करते हैं कि कौन से पहले के आउटपुट आर्ग्यूमेंट्स के रूप में पास करने हैं।

## उदाहरण

### एक webhook से पेलोड बनाना

एक webhook `{ "service": "checkout", "status": "failed" }` जैसी बॉडी के साथ आता है। उसे एक OneUptime incident में बदलने के लिए:

1. `CIWebhook` नामक **Webhook** ट्रिगर।
2. **Conditions** ब्लॉक: बायाँ `{{CIWebhook.Request Body.status}}`, ऑपरेटर `==`, दायाँ `failed`।
3. **Yes** शाखा से, एक **Create Incident** ब्लॉक:
   - शीर्षक: `CI build failed: {{CIWebhook.Request Body.service}}`
   - विवरण: `See {{CIWebhook.Request Body.url}} for the logs.`

### API कॉल में सीक्रेट का उपयोग करना

एक वर्कफ़्लो जो PagerDuty को कॉल करता है:

1. एक सीक्रेट global वेरिएबल के रूप में `PAGERDUTY_KEY` सहेजें।
2. **API** ब्लॉक पर, `Authorization` हेडर को `Token token={{variable.PAGERDUTY_KEY}}` पर सेट करें।

कुंजी वर्कफ़्लो और लॉग से बाहर रहती है।

### दो API कॉल्स को चेन करना

पहली कॉल आपको एक ID देती है जिसकी दूसरी को आवश्यकता है:

1. **API** ब्लॉक `LookupOrder`: `GET /orders?email={{Manual.JSON.email}}`।
2. **API** ब्लॉक `CancelOrder`: `POST /orders/{{LookupOrder.response-body.id}}/cancel`।

यदि `LookupOrder` विफल होता है, तो इसका **error** आउटपुट **success** के बजाय चालू होता है। उसे एक Email या Slack ब्लॉक से जोड़ें ताकि विफलताएँ अनदेखी न रहें।

## ध्यान देने योग्य बातें

- **एक ब्लॉक का नाम बदलने से संदर्भ टूट जाते हैं।** यदि आप एक ब्लॉक का नाम बदलते हैं, तो हर जगह जहाँ उसका उपयोग किया गया है उसे अपडेट करें। run लॉग में, एक अनसुलझा संदर्भ शाब्दिक `{{BlockName.field}}` टेक्स्ट के रूप में दिखाई देता है।
- **वेरिएबल नाम केस-संवेदनशील हैं।** `{{variable.MyKey}}` और `{{variable.mykey}}` अलग हैं।
- **गायब फ़ील्ड्स खाली हो जाती हैं।** एक फ़ील्ड का संदर्भ देना जो मौजूद नहीं है आपको एक खाली स्ट्रिंग देता है, त्रुटि नहीं। सुविधाजनक — लेकिन यह बग छुपा सकता है। जारी रखने से पहले महत्वपूर्ण फ़ील्ड्स की जाँच करने के लिए **Conditions** ब्लॉक का उपयोग करें।
