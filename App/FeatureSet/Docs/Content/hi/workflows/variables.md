# वर्कफ़्लो वेरिएबल

एक वर्कफ़्लो केवल तभी उपयोगी होता है जब डेटा उसमें से होकर बहता है। वेरिएबल वह तरीका है जिससे डेटा चलता है — ट्रिगर से पहले घटक में, एक घटक के आउटपुट से अगले घटक के इनपुट में, और प्रोजेक्ट-स्तरीय सीक्रेट से जहाँ भी उन्हें संदर्भित किया जाता है।

OneUptime में दो प्रकार के वेरिएबल और एक इंटरपोलेशन सिंटैक्स है जो दोनों के लिए काम करता है।

## ग्लोबल वेरिएबल

प्रोजेक्ट-स्तरीय मान **Workflows → Global Variables** के अंतर्गत एक बार परिभाषित किए जाते हैं। API कुंजी, बेस URL, चैनल नाम, कुछ भी जिसे आप दस वर्कफ़्लो में हार्ड-कोड नहीं करना चाहते।

एक ग्लोबल वेरिएबल में:

- **Name** — वह पहचानकर्ता जिसके द्वारा आप इसे संदर्भित करते हैं। टेम्पलेट्स में स्पष्ट करने के लिए `UPPER_SNAKE_CASE` का उपयोग करें।
- **Value** — स्ट्रिंग मान। मल्टी-लाइन मान समर्थित हैं।
- **Is Secret** — चालू होने पर, सहेजने के बाद UI में मान केवल-लिखने के लिए होता है और रन लॉग से छुपा दिया जाता है।

किसी भी वर्कफ़्लो में कहीं से भी एक ग्लोबल वेरिएबल को संदर्भित करें:

```
{{variable.NAME}}
```

उदाहरण के लिए, यदि आपने `PAGERDUTY_KEY` को सीक्रेट वेरिएबल के रूप में परिभाषित किया है, तो PagerDuty को कॉल करने वाला हर API घटक इसे `{{variable.PAGERDUTY_KEY}}` के रूप में पढ़ सकता है, बिना किसी को वर्कफ़्लो JSON में वास्तविक कुंजी देखे।

## लोकल वेरिएबल

लोकल वेरिएबल वे नोड्स के रिटर्न मान हैं जो इस execution में पहले ही चल चुके हैं। हर ट्रिगर और हर घटक एक प्रकाशित करता है — प्रति-नोड सूचियों के लिए [Triggers](/docs/workflows/triggers) और [Components](/docs/workflows/components) देखें।

एक लोकल वेरिएबल को इस रूप में संदर्भित करें:

```
{{NodeId.fieldName}}
```

`NodeId` कैनवास पर ट्रिगर या घटक का नाम है (आप इसे पठनीयता के लिए नाम बदल सकते हैं — इसे छोटा और `PascalCase` रखें ताकि संदर्भ साफ़ रहें)। `fieldName` वह कुछ भी है जो वह नोड प्रकाशित करता है।

उदाहरण:

- `LookupUser` नामक **API** घटक के सफलतापूर्वक लौटने के बाद, आगे के नोड्स इसके स्थिति कोड को `{{LookupUser.response-status}}` के रूप में और पार्स की गई बॉडी को `{{LookupUser.response-body}}` के रूप में पढ़ सकते हैं।
- `Incident` नामक **Incident → On Create** ट्रिगर के बाद, आप `{{Incident.title}}`, `{{Incident.description}}`, `{{Incident.incidentSeverityId}}`, और incident पर किसी भी अन्य कॉलम को पढ़ सकते हैं।
- `Transform` नामक **Custom Code** घटक के बाद, लौटाए गए मान को `{{Transform.value}}` के रूप में उजागर किया जाता है।

लोकल वेरिएबल एकल रन तक सीमित हैं। अगला रन एक नए स्लेट से शुरू होता है।

## इंटरपोलेशन कहाँ काम करता है

लगभग हर टेक्स्ट-स्टाइल आर्ग्युमेंट इंटरपोलेशन का समर्थन करता है:

- API घटक पर URL फ़ील्ड्स
- Slack / Teams / Discord / Telegram / Email पर संदेश पाठ
- Email पर विषय और बॉडी
- हेडर और बॉडी फ़ील्ड्स (इसे JSON मानों के अंदर उपयोग करें)
- Conditions पर बायाँ और दायाँ ऑपरेंड

शुद्ध-JSON आर्ग्युमेंट्स स्ट्रिंग मानों के अंदर इंटरपोलेशन स्वीकार करते हैं; आप एक कुंजी को इंटरपोलेट नहीं कर सकते। यदि आपको एक गतिशील संरचना बनाने की आवश्यकता है, तो payload को असेंबल करने के लिए **Custom Code** का उपयोग करें और फिर उसके रिटर्न मान को अगले नोड में पाइप करें।

**Custom Code** घटक वेरिएबल्स को अलग तरह से पढ़ता है — ग्लोबल वेरिएबल `args.variables` पर उजागर होते हैं, और अपस्ट्रीम रिटर्न मान आपके द्वारा घटक पर कॉन्फ़िगर किए गए नामांकित आर्ग्युमेंट्स के रूप में पास होते हैं।

## उदाहरण

### एक ट्रिगर से payload बनाएँ

एक webhook एक CI बिल्ड परिणाम प्राप्त करता है। बॉडी `{ "service": "checkout", "status": "failed" }` जैसा JSON है। इसे OneUptime incident में बदलने के लिए:

1. `CIWebhook` नामक **Webhook** ट्रिगर।
2. **Conditions** घटक: बायाँ `{{CIWebhook.Request Body.status}}`, ऑपरेटर `==`, दायाँ `failed`।
3. `yes` पोर्ट से, एक **Create Incident** घटक:
   - शीर्षक: `CI build failed: {{CIWebhook.Request Body.service}}`
   - विवरण: `See {{CIWebhook.Request Body.url}} for the build logs.`

### एक आउटबाउंड API कॉल में सीक्रेट का उपयोग करें

एक वर्कफ़्लो जो PagerDuty को कॉल करता है:

1. `PAGERDUTY_KEY` को सीक्रेट ग्लोबल वेरिएबल के रूप में परिभाषित करें।
2. **API** घटक पर, `Authorization` हेडर को `Token token={{variable.PAGERDUTY_KEY}}` पर सेट करें।

कुंजी कभी भी वर्कफ़्लो JSON या रन लॉग में नहीं दिखाई देती।

### दो API कॉल चेन करें

पहला कॉल एक ID लौटाता है जिसकी दूसरे कॉल को आवश्यकता होती है:

1. **API** घटक `LookupOrder`: `GET /orders?email={{Manual.JSON.email}}`।
2. **API** घटक `CancelOrder`: `POST /orders/{{LookupOrder.response-body.id}}/cancel`।

यदि `LookupOrder` एक गैर-2xx रिस्पॉन्स लौटाता है, तो उसका `error` पोर्ट `success` के बजाय चालू होता है — उस शाखा को एक Email या Slack घटक से जोड़ें ताकि विफलताएँ चुप न हों।

## कुछ सावधानियाँ

- **नोड नामों में टाइपो संदर्भों को चुपचाप तोड़ देते हैं।** यदि आप `{{OldName.field}}` को नीचे जोड़ने के बाद किसी नोड का नाम बदलते हैं, तो हर संदर्भ अपडेट करें। रन लॉग देखें — यदि आप कैप्चर किए गए आर्ग्युमेंट में शाब्दिक `{{OldName.field}}` देखते हैं, तो लुकअप ने हल नहीं किया।
- **सीक्रेट केस-संवेदनशील हैं।** `{{variable.MyKey}}` और `{{variable.mykey}}` अलग वेरिएबल हैं।
- **गायब फ़ील्ड्स खाली हैं।** `{{Foo.nonexistent}}` को संदर्भित करने से एक खाली स्ट्रिंग बनती है, त्रुटि नहीं। उपयोगी, लेकिन यह बग्स को छिपा सकता है — यदि अगले चरण के लिए फ़ील्ड आवश्यक है, तो उपस्थिति की पुष्टि करने के लिए **Conditions** नोड का उपयोग करें।
