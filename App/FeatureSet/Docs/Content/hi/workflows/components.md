# वर्कफ़्लो घटक

घटक वे एक्शन नोड्स हैं जिन्हें आप ट्रिगर के बाद रखते हैं। प्रत्येक एक काम करता है — एक HTTP रिक्वेस्ट बनाता है, एक Slack संदेश भेजता है, किसी शर्त पर शाखा करता है, एक JavaScript स्निपेट चलाता है — और अगले नोड के कनेक्ट होने के लिए एक या अधिक आउटपुट पोर्ट उजागर करता है।

यह पेज एक कैटलॉग है। वायरिंग नियमों और कैनवास के लिए, [वर्कफ़्लो बनाना](/docs/workflows/authoring) देखें।

## API

किसी भी URL पर एक आउटबाउंड HTTP रिक्वेस्ट बनाएँ।

**Arguments**:

- **Method** — `GET`, `POST`, `PUT`, `PATCH`, `DELETE`।
- **URL** — रिक्वेस्ट URL। इंटरपोलेट किया गया।
- **Request Headers** — हेडर्स का JSON ऑब्जेक्ट।
- **Request Body** — `POST` / `PUT` / `PATCH` के लिए JSON या टेक्स्ट बॉडी।

**Output ports**:

- `success` — जब रिस्पॉन्स स्थिति 2xx होती है तो चालू होता है। रिटर्न मान: `response-status`, `response-headers`, `response-body`।
- `error` — नेटवर्क विफलता या गैर-2xx रिस्पॉन्स पर चालू होता है। रिटर्न मान: `error` संदेश।

इसका उपयोग करें: किसी भी तृतीय-पक्ष REST API के लिए, अपने स्वयं के एडमिन एंडपॉइंट्स, हल्के एकीकरण जिनमें समर्पित घटक नहीं है।

## Webhook (आउटबाउंड)

सामान्य "fire and forget" मामले के लिए API घटक के चारों ओर एक पतला रैपर। एक URL पर एक JSON बॉडी पोस्ट करता है और एकल `success` / `error` जोड़ी उजागर करता है।

यदि आपको आगे रिस्पॉन्स बॉडी पढ़ने की आवश्यकता है तो **API** को प्राथमिकता दें; यदि आप केवल किसी अन्य सिस्टम को सूचित करना चाहते हैं तो **Webhook** को प्राथमिकता दें।

## Slack

अपने प्रोजेक्ट के Slack workspace कनेक्शन का उपयोग करके एक Slack चैनल पर संदेश पोस्ट करें।

**Arguments**:

- **Channel name** — पोस्ट करने के लिए चैनल। बॉट को पहले से ही उस चैनल का सदस्य होना चाहिए।
- **Message text** — बॉडी। इंटरपोलेट किया गया; Slack mrkdwn का समर्थन करता है।

पहले **Project Settings → Workspace Connections → Slack** में workspace कनेक्शन सेट करें। [Slack Workspace Connection](/docs/workspace-connections/slack) देखें।

## Microsoft Teams

अपने प्रोजेक्ट के Teams कनेक्शन का उपयोग करके एक Microsoft Teams चैनल पर संदेश पोस्ट करें।

**Arguments**:

- **Team & channel** — गंतव्य।
- **Message text** — बॉडी।

कनेक्शन सेटअप के लिए [Microsoft Teams Workspace Connection](/docs/workspace-connections/microsoft-teams) देखें।

## Discord

घटक पर कॉन्फ़िगर किए गए एक आने वाले webhook URL के माध्यम से एक Discord चैनल पर संदेश पोस्ट करें।

## Telegram

घटक पर कॉन्फ़िगर किए गए एक बॉट टोकन और चैट ID के माध्यम से Telegram चैट को संदेश भेजें।

## Email

OneUptime के SMTP कॉन्फ़िगरेशन के माध्यम से ईमेल भेजें।

**Arguments**:

- **To** — प्राप्तकर्ता का ईमेल पता।
- **Subject** — इंटरपोलेट किया गया।
- **Body** — Markdown या HTML।

ईमेल प्रोजेक्ट के कॉन्फ़िगर किए गए प्रेषक पते से भेजा जाता है ([SMTP](/docs/emails/smtp) देखें)।

## Custom Code

वर्कफ़्लो के वेरिएबल्स और अपस्ट्रीम नोड के रिटर्न मानों तक पहुँच के साथ JavaScript का एक स्निपेट चलाएँ।

**Arguments**:

- **Code** — JavaScript बॉडी। अंतिम अभिव्यक्ति का मान (या `(async () => { ... })()` से लौटाया गया कुछ भी) घटक का रिटर्न मान बन जाता है।
- **Arguments** — `args` के रूप में पास किए गए वैकल्पिक नामांकित मान।

**Output ports**: `success` (रिटर्न मान), `error` (पकड़ा गया अपवाद)।

इसका उपयोग करें: दो सिस्टमों के बीच एक payload बदलने के लिए, एक छोटी गणना करने के लिए जो अपने घटक के योग्य नहीं है, JS-केवल लॉजिक को कॉल करने के लिए। भारी स्क्रिप्टिंग जो आपकी अपनी इन्फ्रास्ट्रक्चर के अंदर चलनी चाहिए, वह [Runbook](/docs/runbooks/index) Bash या JavaScript चरण में होती है।

## JSON

टेक्स्ट और JSON के बीच कनवर्ट करें।

- **JSON → Text** — एक JSON ऑब्जेक्ट को स्ट्रिंग में सीरियलाइज़ करें (एक आउटबाउंड घटक के `body` आर्ग्युमेंट में पाइप करने के लिए सुविधाजनक जो टेक्स्ट की अपेक्षा करता है)।
- **Text → JSON** — एक स्ट्रिंग को JSON ऑब्जेक्ट में पार्स करें। जब अपस्ट्रीम API ने अपनी बॉडी टेक्स्ट के रूप में लौटाई हो लेकिन आपको एक फ़ील्ड पढ़ने की आवश्यकता हो तो उपयोगी।

## Conditions

तुलना पर शाखा करें। कॉन्फ़िगर करें:

- **Left value** — आमतौर पर `{{Incident.title}}` जैसा इंटरपोलेट किया गया संदर्भ।
- **Operator** — `==`, `!=`, `>`, `>=`, `<`, `<=`, `contains`, `starts with`, `ends with`।
- **Right value** — तुलना करने के लिए मान।

**Output ports**: `yes` और `no`। अपने इरादे से मेल खाने वाली शाखा के बाद बाकी वर्कफ़्लो को जोड़ें।

## Schedule (विलंब)

जारी रखने से पहले एक कॉन्फ़िगर की गई अवधि के लिए वर्कफ़्लो को रोकें। उपयोगी जब आपको किसी बाहरी सिस्टम को उसकी स्थिति जाँचने से पहले शांत होने का समय देने की आवश्यकता हो।

## Log

वर्कफ़्लो रन लॉग में एक पंक्ति लिखें। शुद्ध डीबगिंग सहायता; पंक्ति रन पर कैप्चर होती है और **Logs** के अंतर्गत दिखाई देती है। कोई बाहरी साइड इफ़ेक्ट नहीं।

## Execute Workflow

किसी अन्य वर्कफ़्लो को एक उप-चरण के रूप में कॉल करें। कॉल किया गया वर्कफ़्लो स्वतंत्र रूप से चलता है (फायर-एंड-फॉरगेट) — जैसे ही कॉल भेजी जाती है, कॉलर को नियंत्रण लौट जाता है।

इसका उपयोग कई वर्कफ़्लो से साझा लॉजिक को निकालने के लिए करें: एक "post-to-incident-channel" वर्कफ़्लो एक बार बनाएँ और इसे हर दूसरे वर्कफ़्लो से कॉल करें जिसे चैनल को सूचित करना है।

एक पुनरावृत्ति सीमा वर्कफ़्लो को एक अनंत लूप में एक दूसरे को कॉल करने से रोकती है। [Configuration & Safety](/docs/workflows/configuration) देखें।

## मॉडल घटक (OneUptime एंटिटीज़ पर CRUD)

हर OneUptime एंटिटी के लिए जो वर्कफ़्लो का समर्थन करती है (मॉनिटर, incidents, alerts, status pages, on-call policies, आदि), पैलेट स्वचालित रूप से निम्नलिखित घटकों को उजागर करता है — एंटिटी नाम से खोजा जा सकता है:

- **Find One {Entity}** — क्वेरी द्वारा एक एकल रिकॉर्ड प्राप्त करें।
- **Find {Entity}** — क्वेरी द्वारा रिकॉर्ड्स की सूची प्राप्त करें (पेजिनेटेड)।
- **Create {Entity}** — एक नया रिकॉर्ड डालें।
- **Update {Entity}** — ID द्वारा एक रिकॉर्ड अपडेट करें।
- **Delete {Entity}** — ID द्वारा एक रिकॉर्ड हटाएँ।
- **Count {Entity}** — क्वेरी से मेल खाने वाले रिकॉर्ड्स की गिनती करें।

इस तरह से एक वर्कफ़्लो प्लेटफ़ॉर्म छोड़े बिना OneUptime की स्थिति पढ़ और लिख सकता है। उदाहरण के लिए: आपके CI टूल से एक webhook बिल्ड के विफलता संदेश के साथ **Create Incident** कॉल करता है; या एक शेड्यूल किया गया वर्कफ़्लो हर पाँच मिनट में **Find Incident** चलाता है और एक सारांश ईमेल करता है।

## सही घटक चुनना

कुछ त्वरित नियम:

- यदि आप जो करना चाहते हैं उसके लिए एक समर्पित घटक मौजूद है (Slack, Email, OneUptime एंटिटी पर एक CRUD), तो इसका उपयोग करें — यह आपको अपना खुद का रोल करने की तुलना में बेहतर त्रुटि हैंडलिंग और स्पष्ट लॉग देता है।
- यदि आपको एक बाहरी HTTP API को कॉल करना है जिसमें समर्पित घटक नहीं है, तो **API** का उपयोग करें।
- यदि आपको दो घटकों के बीच डेटा को *आकार* देना है, तो **Custom Code** या **JSON** का उपयोग करें।
- यदि आपको एक मान के आधार पर अलग-अलग कार्रवाई करनी है, तो **Conditions** का उपयोग करें।
