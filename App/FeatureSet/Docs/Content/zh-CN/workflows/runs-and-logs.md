# 运行与日志

每当工作流运行时,OneUptime 都会保存一条记录——什么时候运行的、是否成功、每个模块做了什么。这条记录就叫一次 **运行**。运行就是你确认工作流是否工作、排查没有工作的工作流,以及回看过往活动的依据。

## 在哪里查看

| 页面 | 你会看到什么 |
| --- | --- |
| **工作流 → 运行与日志** | 项目中所有工作流的每一次运行。可以按工作流、状态和时间筛选。 |
| **工作流 → 日志标签** | 仅当前这个工作流的运行。 |
| **单次运行** | 一次执行,包含每个模块的输出。 |

## 运行状态

| 状态 | 含义 |
| --- | --- |
| **已排队** | 触发器已触发,运行即将开始。通常只需要不到一秒。 |
| **运行中** | 工作流正在进行。长时间运行的模块会让运行保持这个状态。 |
| **成功** | 所有运行过的模块都没有错误地完成了。(故意走 **错误** 分支仍然算成功——工作流本身没有失败。) |
| **错误** | 某个模块失败了,而且没有连接 **错误** 路径来处理它。运行就此停止。 |
| **超时** | 运行时间超过了允许的上限。见 [配置与安全](/docs/workflows/configuration)。 |

## 阅读运行详情

点击任意运行可以打开详情。你会看到:

- **头部**——触发器、开始和结束时间、总耗时,以及状态。
- **模块列表**——每个运行过的模块,按顺序排列。每个都显示它接收的值、它的输出,以及它走了哪条路径。
- **错误**——如果某个模块失败,会显示错误信息和(可用时)更多细节。

显示的值就是模块实际看到的值——所有变量都已经被填充。这是最有用的调试视图:如果一条 Slack 消息显示的是字面文本 `{{Incident.title}}` 而不是实际标题,你就知道变量没解析。

## 常见调试

### "我的工作流没运行。"

1. 确认设置中工作流是 **已启用** 的。新工作流默认是停用的。
2. 对于 OneUptime 事件触发器:确认事件确实发生了。打开记录查看它的历史。
3. 对于 webhook 触发器:确认对方系统正在向正确的 URL 发送。大多数工具会记录它们发送 webhook 的情况——去那里检查。
4. 对于定时触发器:确认 cron 表达式与你预期的时间匹配。

如果触发器触发了但没有运行出现,检查 **项目设置 → 计费** 下的运行配额。

### "后续的模块没有运行。"

某个模块没有运行通常是连线问题。打开画布并检查:

- 前一个模块的输出连接到这个模块的输入了吗?
- 前一个模块走的输出是否和你预期的不同(例如走了 **错误** 而不是 **成功**,或者走了 **否** 而不是 **是**)?运行详情会显示走了哪条路径。

### "变量传过来是空的。"

打开运行,查看失败模块的值。

- 如果你看到字面文本 `{{BlockName.field}}`,说明引用没解析——可能是模块名或字段名拼写错误。
- 如果你看到空字符串,说明前一个模块运行了,但没有产生那个字段。

### "手动运行可以工作,但被触发器触发时不行。"

使用 **手动运行**,带上和真实触发器发送的内容相似的 JSON 负载。然后把手动运行和真实运行的值并排比较。差异通常只是一个字段名或类型。

## 重新运行工作流

没有"重试这次运行"按钮。我们不会自动重跑过去的执行,因为副作用(Slack 消息、API 调用、工单)可能不安全。要重做这些工作,修复工作流然后让下一次真实的触发器触发它。

对于手动工作流,只需用相同的负载再次点击 **手动运行**。

## 运行保留多久?

运行会在项目中无限期保留。如果某个工作流运行非常频繁、把历史搞乱了(比如每分钟触发一次的调试工作流),停用或删除它来停止增加噪音。
