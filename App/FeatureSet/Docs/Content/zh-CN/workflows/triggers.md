# 触发器

触发器是工作流的第一个模块——它决定工作流何时运行。每个工作流恰好有一个触发器。你可以从四种类型中选择。

## 手动

通过在工作流页面上点击 **手动运行** 按需运行工作流。你可以粘贴一段 JSON 负载,工作流的其余部分可以读取它。

适合用于:你想要一键自动化的场景,比如"轮换这个密钥"或"发一个测试告警"。

**输出**:你粘贴的 JSON,如果没粘贴则是一个空对象。

## 定时

使用 cron 表达式按重复的计划运行工作流。

适合用于:夜间清理、每小时同步、每周报表。

**设置**:一个 cron 表达式。几个常见的:

- `0 * * * *`——每小时整点。
- `*/5 * * * *`——每 5 分钟。
- `0 9 * * 1`——每周一上午 9:00。

如果系统短时间不可用,运行会在恢复后立即被领取——短时中断不必担心错过的触发点。

## Webhook

OneUptime 会创建一个专属 URL。任何对该 URL 的访问都会启动工作流。请求的头、查询参数和请求体都会被传入。

适合用于:从其他工具接收数据到 OneUptime——CI/CD 回调、来自其他监控的告警、你 CRM 的注册事件。

**输出**:

- **Request Headers**——入站请求的所有头部。
- **Request Query Params**——解析后的查询字符串。
- **Request Body**——解析后的请求体(如果不是 JSON,则为原始文本)。

该 URL 同时接受 `GET` 和 `POST`。调用方会立即收到确认——工作流本身在后台运行。

把 URL 当作密码对待。任何拿到它的人都可以启动你的工作流。

## OneUptime 事件触发器

OneUptime 中几乎所有对象——监控、事件、告警、计划维护、状态页、值班策略、团队——都可以触发工作流。每种对象都提供三种事件:

- **创建时**——当新建一个时触发。
- **更新时**——当某个被更改时触发。
- **删除时**——当某个被删除时触发。

这就是你构建"当 OneUptime 中发生 X 时,做 Y"的方式,不需要循环轮询。

完整记录会传递给下一个模块。例如,**事件 → 创建时** 触发器会传入新建的事件,下一个模块就可以读取它的标题、描述、严重程度以及任何其他字段。

### 团队最常用的事件

- **事件**——当事件被打开、更新(已确认、已解决)或删除时响应。
- **告警**——告警的同样三种事件。
- **监控**——当监控被添加、编辑或移除时响应。
- **计划维护**——在计划好后自动公告维护窗口。
- **状态页订阅者**——欢迎订阅状态页的人。
- **值班策略**——把排班变更同步到另一个排班系统。

按名称搜索触发器面板找到你想要的那个。

## 我该用哪个触发器?

| 如果你想…                 | 选择               |
| ------------------------- | ------------------ |
| 点一个按钮运行工作流      | **手动**           |
| 按重复计划运行            | **定时**           |
| 让另一个系统把数据推进来  | **Webhook**        |
| 响应 OneUptime 内部的事件 | **OneUptime 事件** |

一个工作流只能有一个触发器。如果你需要两种方式启动同一个自动化,把共享逻辑放在一个工作流中,然后用 **执行工作流** 组件从两个轻量的"包装"工作流中调用它。
